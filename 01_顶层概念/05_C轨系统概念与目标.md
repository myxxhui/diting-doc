## 一、为什么要单独定义“个人能力训练系统”

如果只从产品角度设计，你会自然选择“最快做出来”的技术路径；  
如果只从简历角度设计，你又会走向脱离业务的炫技工程。

所以必须单独定义第二套顶层目标：

**让副业每一阶段都在为年底跳槽积累真实、可验证、可复述的 AI 工程能力。**

## 二、训练系统的总目标

### 主目标

在不脱离真实业务价值的前提下，优先把以下两类能力练到可面试、可讲案例、可写简历：

1. **岗位 1：AI 基础设施研发工程师**
2. **岗位 4：智能体基础设施工程师**

### 辅目标

在主线建设过程中，顺手补齐：

3. **岗位 3：MLOps**
4. **岗位 2：AI 技术负责人**

## 三、四类岗位对应的训练地图

### A. 岗位 1：AI 基础设施训练地图

要练到的核心能力：

- 自托管推理服务
- `vLLM` 部署与调优
- `Kubernetes` 编排
- GPU / CPU / 网络 / 存储资源治理
- 可观测性、弹性与 FinOps

项目中的对应落点：

- 本地或云端部署自托管模型
- 推理网关与模型路由
- 观测指标、性能基线、成本看板

### B. 岗位 4：智能体基础设施训练地图

要练到的核心能力：

- Agent 动作或代码的安全执行
- 沙箱、隔离、轻量虚拟化
- 长任务与异步任务生命周期
- 多租户与配额管理
- Runtime 级观测和故障恢复

项目中的对应落点：

- 让 LLM 生成结构化动作甚至代码
- 在沙箱中执行高风险动作
- 记录任务状态、输出、失败原因、资源占用

### C. 岗位 3：MLOps 训练地图

要练到的核心能力：

- Eval 数据集与回归验证
- 模型 / Prompt / Workflow 版本管理
- 灰度发布与回滚
- CT（Continuous Training）闭环

项目中的对应落点：

- 人工反馈沉淀为训练/评测数据
- 每次变更可比对
- 效果下降可快速回退

### D. 岗位 2：AI 技术负责人训练地图

要练到的核心能力：

- 业务抽象
- AI 工作流设计
- 产品与技术之间的取舍
- 需求到上线的闭环表达

项目中的对应落点：

- 你需要亲自定义什么值得做、什么不值得做
- 你要解释为什么某项复杂度值得引入
- 你要把系统从“能跑”推进到“真的有价值”

## 四、8 个月训练节奏

### 阶段 1：先做对自己有用的产品闭环

目标：

- 快速产出真实可用的投研工作流
- 积累真实使用反馈
- 确定后续最值得基础设施化的能力

主要覆盖：

- 岗位 2 的业务抽象
- 岗位 3 的初步评测意识

### 阶段 2：替换推理底座，主攻岗位 1

目标：

- 从商业 API 迁移到自托管模型或混合推理
- 建立基础观测、部署与成本治理能力

主要覆盖：

- 岗位 1 主线
- 岗位 3 的版本与部署意识

### 阶段 3：引入安全执行与长任务运行时，主攻岗位 4

目标：

- 让 Agent 不只“分析”，还能“安全执行”
- 建立 Runtime 级别的状态、隔离与恢复能力

主要覆盖：

- 岗位 4 主线

### 阶段 4：补齐评测、训练、发布闭环

目标：

- 让每次优化都有评测依据
- 让每次上线都有发布与回滚机制

主要覆盖：

- 岗位 3 主线
- 顺手增强岗位 2 的治理能力

## 五、训练系统的输出物

训练不是“学会了”，而是必须沉淀成输出物：

- 可运行代码
- 架构文档
- 指标与基准测试
- 设计取舍记录
- Demo 与面试故事

如果一个阶段没有形成这些输出物，就不算完成训练。

## 六、训练系统的约束

### 约束 1：不能脱离业务瞎练

任何训练项必须回答：

- 它服务的业务问题是什么？
- 为什么现在做而不是以后做？

### 约束 2：不能为了简历去造规模

你可以模拟多租户、长任务、发布流程，但不能虚构“百万级”“海量用户”等并不存在的规模。

### 约束 3：主线必须收敛

岗位 1 和岗位 4 是主线；  
岗位 2 和岗位 3 只做自然覆盖，不允许反客为主把精力带偏。
