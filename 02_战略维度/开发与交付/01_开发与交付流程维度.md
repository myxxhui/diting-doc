## 一、维度定义

新的开发与交付流程维度，核心不是“把系统一步到位部署到多复杂的环境”，而是：

**在 8 个月副业窗口中，用最合理的顺序，把业务闭环、Infra 训练、Runtime 训练和评测闭环逐步搭起来。**

## 二、总原则

### 原则 1：先有业务闭环，再有基础设施深度

没有真实工作流，就不该过早投入重型 Infra。  
所以流程必须先证明“系统真的有用”，再证明“系统值得被工程化”。

### 原则 2：每个阶段只引入一种主要复杂度

不要同时做：

- 新业务逻辑
- 新模型部署
- 新沙箱运行时
- 新发布流程

否则无法定位问题来源，也无法形成清晰训练故事。

### 原则 3：每个阶段都要留下可展示输出

至少包括：

- 代码
- 设计说明
- 指标或验证结果
- 可对外讲清楚的取舍说明

## 三、8 个月开发节奏

### 阶段 A：业务流跑通（Month 1-2）

目标：

- 快速建立最小可用的投研工作流
- 不追求底层最优，优先验证产品价值

建议做法：

- 可以先用商业 API
- 先把数据采集、摘要、跟踪、提醒跑通
- 建立你自己的日常使用闭环

这一阶段主要训练：

- 岗位 2 的业务抽象
- 岗位 3 的基础评测意识

### 阶段 B：替换推理底座（Month 3-5）

目标：

- 从“调用别人模型”过渡到“掌握自己的推理底座”
- 建立岗位 1 的核心案例

建议做法：

- 引入自托管模型
- 部署 `vLLM`
- 逐步引入 `Kubernetes`
- 建立基础观测、性能对比和成本记录

这一阶段主要训练：

- 岗位 1

### 阶段 C：引入 Agent Runtime（Month 6-7）

目标：

- 让系统具备安全执行动作或代码的能力
- 建立岗位 4 的核心案例

建议做法：

- 让 LLM 产出结构化动作或受限代码
- 在隔离环境里执行
- 为长任务建立状态、日志、失败恢复和超时回收机制

这一阶段主要训练：

- 岗位 4

### 阶段 D：补齐评测与发布闭环（Month 8）

目标：

- 让整个系统可演示、可迭代、可讲述
- 为年底跳槽整理可复用材料

建议做法：

- 建立评测集
- 形成版本、灰度、回滚与回归流程
- 统一整理设计文档、指标、简历要点

这一阶段主要训练：

- 岗位 3
- 同时补强岗位 2 的治理视角

## 四、环境推进顺序

推荐的环境推进顺序：

1. **本地开发 + 最小脚本/服务**
2. **Docker 统一依赖**
3. **单机或轻量集群推理环境**
4. **`Kubernetes` 编排**
5. **沙箱 / 运行时隔离**
6. **评测 / 发布 / 回滚闭环**

关键点：  
`Kubernetes`、沙箱、多租户都不是越早越好，而是要在前一层已经证明有价值之后再进入。

## 五、阶段准入规则

### 进入阶段 B 前，必须满足

- 你自己已经开始真实使用系统
- 已能明确识别最值得迁移到自托管模型的能力
- 已有可比较的基线结果

### 进入阶段 C 前，必须满足

- 已经确定确实需要 Agent 执行动作或代码
- 纯文本结论已经不够满足业务目标
- 已有可观测的推理与任务调用链

### 进入阶段 D 前，必须满足

- 系统已经有多个版本迭代点
- 已经开始遇到“改了之后到底变好还是变差”的问题
- 已有真实反馈样本可沉淀为评测集

## 七、不做什么

本维度明确反对：

- 一开始就按最终形态设计全部平台能力
- 没有业务反馈时就深挖复杂基础设施
- 没有观测和验证就盲目引入新一层技术
- 为了“看起来像大厂”而过早微服务化
