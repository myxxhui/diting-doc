## 一、本阶段用户场景核心升级

> 第二阶段是**副驾驶模式**——把"看建议 → 思考 → 操作"从 30 min 压缩到 5 min。
> 验证的核心是：**"批量决策会能否让我 5 分钟搞定一周决策？双盲 Kappa 能否证明系统真的懂我？"**

## 二、典型一周完整时间线（升级版 vs 阶段 1）

### 周一·一周开局（5 min，**对比阶段 1 的 30 min**）

```
08:00  📧 收到周报邮件 (与阶段 1 同)

       │ ☐ BBB  超短 45 天 +18% conf 0.78    │
       │ ☐ CCC  主战场 120 天 +32% conf 0.82 │
       └─────────────────────────────────────┘
       ┌─ 持仓调仓建议 (2 条) ───────────────┐
       │ ☐ XX 减仓 50% (broken_any)          │
       │ ☐ YY 加仓 30% (strong_high)         │
       └─────────────────────────────────────┘
```

- ✅ 系统建议接受率 ≥ 50%

## 五、可能的失败场景与应对

### 5.1 批量决策会平均 > 10 min

```
原因: 三组件信息过载 / thesis 卡太复杂
应对:
  1. 简化推荐池 thesis 卡的展示（仅显示 5 必填核心字段，详情下钻）
  2. 调仓建议按优先级排序，超过 5 条折叠
  3. verified 单周上限 ≤ 10 条（已强约束）
```

### 5.2 双盲 Kappa < 0.85

```
原因: 系统训练数据偏差 / 用户判断风格漂移
应对:
  1. Kappa ∈ [0.70, 0.85): 暂缓 LoRA 发布 + 架构师 review 标注规范
  2. Kappa < 0.70: 暂停所有训练 + 紧急 review
  3. 月报 P.4 飞轮训练贡献页明示 Kappa 异常
```

### 5.3 副驾驶建议接受率 < 30%

```
原因: 系统建议与用户偏好失配 / thesis 卡说服力不够
应对:
  1. 检查接受率与 confidence 的关系: 高 conf 接受率应 > 60%
  2. 用户访谈: 哪些建议总是被拒
  3. 调整推荐池排序 + 增强 verified 偏好对训练
```

### 5.4 券商 API 同步连续失败

```
原因: 券商 API 变更 / 凭证失效 / 网络问题
应对:
  1. 30 min 内 3 次重试失败 → 邮件告警
  2. 1h 失败 → 回退到手动维护 + Web 通知用户
  3. 凭证失效 → 安全引导用户重新授权
```

## 七、阶段 2 验收的"5 必看"

> 阶段验收时，请用户必看以下 5 个场景：

1. **看一次批量决策会**（5 min）：3 组件清晰 + ≤ 5 min 完成
2. **看一次双盲 Kappa 标定**（30 min）：用户独立标注 10 条 + Kappa 计算
3. **看一次战场分配审计**（月度饼图）：是否在健康范围
4. **看一次 4×4 调仓矩阵建议命中**：T+30 后看结果
5. **看一次月报"飞轮训练贡献"页**：含 verified DPO 对数 + Kappa 趋势

> 5 个场景中任一让用户"感觉不对" → 暂停阶段 3，回头优化产品体验。
