## 一、本阶段用户场景核心升级

> 第三阶段是**自动驾驶限额版**——把 30% 仓位交给 AI，70% 仍人类决策；引入议会模式 + 电话告警。
> 验证的核心是：**"自动驾驶能跑赢手动吗？议会模式真的稳健吗？如果不行，能否立刻收手？"**

## 四、本阶段不可错过的"价值证明"事件

| 事件 | 必须证明 |
|---|---|
| **首次议会模式 4 票一致**（阶段 3 W1）| 投票轨迹清晰 + 24h 撤销窗口 + 自动执行成功 |
| **首次议会模式 abstain**（任意 W）| 系统不强行执行 + 用户能看懂 abstain 原因 |
| **首次自动驾驶动作被用户撤销**（任意 W）| 撤销原因入审计 + 下次议会校准 |
| **首次电话告警**（自动驾驶异常）| < 3 min 接通 + 用户语音回执解析 |
| **首次自动 vs 手动月报对比**（M1）| 数据准确 + 分别展示 + 用户能判断"自动是否值得" |
| **首次自动驾驶 verified**（M1 周日）| 用户能区分"亏损但流程对" vs "议会决策错" |
| **首次跑赢手动 1 季度**（Q1 末）| 阶段 3 关键里程碑 |

## 五、可能的失败场景与应对

### 5.1 自动驾驶 3 月连续跑输手动 → 自动暂停

```
触发: 3 月内自动 vs 手动 SCS / EV 双输
应对:
  1. 自动驾驶进入"暂停模式" (清空自动仓位 + 移到手动)
  2. 紧急邮件 + 电话告知架构师
  3. 议会模式 + LoRA 全量复盘
  4. 阶段 3 不通关 → 回头优化阶段 2
```

### 5.2 议会模式 abstain 占比 > 50%

```
原因: 4 维度独立 LoRA 训练发散 / Judge LLM 路由失败
应对:
  1. 暂停议会模式 (单维度建议 → 用户决策)
  2. 重新 review 各维度 LoRA Holdout
  3. Judge LLM 重新训练
  4. 直到 abstain < 30% 才恢复议会
```

### 5.3 自动驾驶 24h 撤销率 > 30%

```
原因: 系统建议与用户偏好失配 / 议会模式过于激进
应对:
  1. 提高议会通过阈值 (如 4 票一致 → 4 票一致 + confidence ≥ 0.80)
  2. 收紧自动驾驶战场限制
  3. 月度 review 撤销原因
```

### 5.4 电话告警过频（> 2 次/月）

```
原因: P&L 阈值过低 / 自动驾驶动作过频
应对:
  1. 调整 P&L 异常阈值（5% → 7%）
  2. 增加自动驾驶冷却期（同一标的 24h 内不重复决策）
  3. 用户访谈是否觉得被打扰
```

## 六、本阶段对用户的"承诺清单（终极版）"

| 承诺 | 怎么实现 |
|---|---|
| **AI 可以接管 30% 本金，但不会超过** | 30% 硬上限 + 单笔 10% 硬上限 + 24h 撤销窗口 |
| **每次 AI 动作我都知道** | 红色告警 + 议会投票轨迹可下钻 |
| **AI 跑不赢就立刻收手** | 3 月连续跑输 → 自动暂停 |
| **我可以一键关闭自动驾驶** | Web 一键关闭 + 自动平仓 |
| **议会模式不是黑盒** | 4 维度投票轨迹 + reasoning 全部可见 |
| **电话告警不打扰我** | 仅自动驾驶异常 + 单日上限 2 次 + 静默时段 |
| **每月对比"自动 vs 手动"账本** | 月报独立页 + Web 跨月趋势 |

## 七、阶段 3 验收的"5 必看 + 1 终极"

### 5 必看

1. **看一次议会模式 4 票一致 → 自动执行**（端到端 24h+1 天）
2. **看一次议会模式 abstain**（用户能看懂为什么不执行）
3. **看一次自动驾驶动作被用户撤销**（撤销流程 + 审计）
4. **看一次电话告警**（< 3 min + 语音回执）
5. **看一次月报"自动 vs 手动对比"页**（数据准确 + 帮助决策）

### 1 终极验收（**项目第一年完整闭环**）

```
项目运行满 12 个月时:
  - 累计 SCS 趋势: 连续 6 月 ≥ 60 → 系统能力稳定
  - 累计 EV: 避险 + 收益 ≥ ¥10 万 (本金 20%)
  - 自动驾驶 vs 手动: ≥ 1 季度跑赢
  - 用户主观: "我会继续使用 + 推荐给同好"
  
  → 项目完整年度通关
  → 进入"扩展期"(多用户 / 议会模式增强 / 战场扩展)
```
