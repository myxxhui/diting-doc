## 一、子模块定位

| 项 | 内容 |
|---|---|
| **一句话定位** | 把用户的"赞/踩/修正"标注转化为飞轮 DPO 偏好对，让系统每月更懂你 |
| **优先级** | **P1**（与 P0 平行推进，第一阶段中后期上线） |
| **使用频率** | 每周日 21:00 一次（5-10 min）+ 紧急告警后即时 verified |
| **L1 承接** | 基石⑨演进（按象限路由训练）+ 基石④八象限（Kappa 标定）|
| **核心价值** | 闭合"系统建议 → 用户操作 → 后续结果 → verified 反馈"全链路，飞轮自转 |

## 三、双盲 Kappa 标定机制

> 严格继承 [维度五·演进实践策略规划 §4.4](../../05_维度五_演进飞轮/04_演进实践策略规划.md#四架构师-verified-kappa-标定)。

### 3.1 双盲流程

```mermaid
flowchart TD
  N[每 50 条系统判断] --> SAMPLE[随机抽 10 条]
  SAMPLE --> HIDE[隐藏系统结论<br/>仅显示原始证据]
  HIDE --> USER[用户独立标注<br/>10 条]
  USER --> REVEAL[揭示系统结论]
  REVEAL --> KAPPA[计算 Cohen's Kappa]
  KAPPA --> ROUTE{Kappa 水平}
  ROUTE -->|≥ 0.85| HEALTHY[训练数据健康]
  ROUTE -->|[0.70, 0.85)| WARN[暂缓 LoRA 发布]
  ROUTE -->|< 0.70| HALT[暂停所有训练]
```

## 四、verified 路由到飞轮训练库

> 严格按八象限路由（基石⑨）。

| verified 类型 | 路由 | 用途 |
|---|---|---|
| ✅ 认同系统结论 | 强化原归因 | A/F → gold_library（SFT 强化）|
| ❌ 不认同 | DPO 偏好对（用户结论 vs 系统结论作为正负对）| failure_library_for_dpo |
| ⚠️ 部分认同 + 修正 | 进入"待 architect 仲裁"队列 | 仲裁后定向训练库 |
| 双盲标定结果 | 计算 Kappa；进 calibration_library | Kappa 监控 |

## 五、数据接入契约

### 5.1 输出到维度五

```yaml
# 每周日 23:00 批量推送
DPOPreferencePairs:
  batch_id: str
  generated_at: datetime
  source: maintenance_zero_feedback
  pairs:
    - chosen: text          # 用户认可的结论
      rejected: text        # 系统结论 (用户不认可时)
      decision_id: str
      user_id: str
      reason: str
```

### 5.2 输入消费

| 来源 | 用途 |
|---|---|
| `decision_log` (维度零) | 抽取待 verified 的 5-10 条 |
| `events:cryo_guard:reject` | 紧急告警执行后 24h 内邀请 verified |
| `events:thrust:thesis_proposed` | 推荐被采纳/拒绝后 1 周邀请 verified |

### 5.3 推送维度五的 SLO

| SLO | 目标 |
|---|---|
| 周度 DPO 偏好对推送（周日 23:00）| 100% |
| 双盲标定数据每月汇总 | 100% |
| Kappa < 0.70 触发自动暂停训练 | < 1 小时 |

## 六、3 阶段演进

| 阶段 | 实现范围 |
|---|---|
| **阶段 1·启动期** | 周末 verified 面板（赞/踩/修正）+ DPO 偏好对推送到维度五 |
| **阶段 2·扩展期** | + 双盲 Kappa 标定面板 + 即时 verified（告警后 24h）+ 跨周 verified 历史可视化 |
| **阶段 3·完善期** | + 自动驾驶决策的 verified（用户事后赞/踩）+ 多用户 Kappa 一致性 |

## 七、与 L1 9 块基石的双向映射

| 基石 | 在本模块的体现 |
|---|---|
| ② 工程化 | verified 的原始证据可下钻；不做"凭感觉"反馈 |
| ④ 八象限 | verified 进 DPO 偏好对，与系统结论形成对比对 |
| ⑨ 演进 | 双盲 Kappa 标定 + 飞轮按象限路由 |

## 八、用户激励与防疲劳设计

| 设计 | 目的 |
|---|---|
| 单周 verified 上限 ≤ 10 条 | 防疲劳（一次 5 分钟可完成）|
| 跳过本周不惩罚 | 不强制 |
| 双盲标定每月 1 次 | 不破坏日常使用流 |
| verified 后立即看到"你的反馈进入第 X 期飞轮训练" | 即时正反馈 |
| 月报附"飞轮训练贡献"页 | 长期价值证明 |

## 九、关键技术选型

| 项 | 选型 | 理由 |
|---|---|---|
| verified 队列 | Redis Sorted Set（按权重排序）| 优先级排序：紧急告警 > 高 SCS 决策 > 其他 |
| Kappa 计算 | scikit-learn `cohen_kappa_score` | 标准实现 |
| 双盲存储 | PG 表 `double_blind_labels` | 永久审计保留 |
| 推送到维度五 | 定时 cron + Redis Stream | 周日 23:00 批量 |
