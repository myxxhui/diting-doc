## 一、本文档的位置

L2 维度三 + 维度四**同时映射到 L3 状态机监控**（这是 L2 5 维度产品视角 vs L3 4 大模块工程视角的不对齐表达——已在 [L2 双目标与战略维度关系](../00_双目标与战略维度关系.md) 中明确说明）。

本文档明确**维度三的引擎/能力运行在 L3 state_watch 哪些服务上**；维度四的映射见 [维度四·00_引擎到L3模块的映射](../04_维度四_卖出决策/00_引擎到L3模块的映射.md)。

## 二、维度三 ↔ L3 模块的总归属

| L2 维度 | L3 工程归属 |
|---|---|
| **维度三·持仓监控**（产品视角）| **L3·状态机监控 state_watch**（工程视角，与维度四共享）|

## 三、引擎/能力 → L3 state_watch 后端服务子模块映射

| L2 能力（维度三）| 优先级 | L3 主责服务 | L3 协责服务 |
|---|---|---|---|
| **叙事一致性评分引擎**（P0） | P0 | `transition_engine` | `state_machine_registry` |
| **核心 SLI 探针调度器**（P0） | P0 | `probe_scheduler` | `state_machine_registry` |
| **节点 4 态状态机** | P0 | `state_machine_registry` + `transition_engine` | — |
| **4×4 调仓矩阵建议** | P1 | `rebalance_advisor` | `transition_engine` |
| **战场分配饼图 + 月度审计** | P1 | `battlefield_auditor`（新增）| `state_machine_registry` |
| **收益仓库控制边界** | P1 | `gain_vault_controller`（新增）| `rebalance_advisor` |
| **4 战场健康度阈值 + 加速规则** | P1 | `state_machine_registry` + `transition_engine` | — |
| **失效复盘** | P2 | `invalidation_auditor` | super_evo `feedback_collector` |
| **健康度算法** | P0 | `transition_engine.compute_health_level` | — |

## 四、关键能力的协作场景

| 场景 | 涉及的 L3 服务 | 说明 |
|---|---|---|
| **节点 4 态状态迁移** | `transition_engine` 处理 SLI 探针触发 | active → validated → weakened → broken |
| **强约束 broken 缓冲期管理** | `transition_engine` + 4 战场加速规则 | 1-20 个交易日 |
| **4×4 矩阵调仓建议生成** | `rebalance_advisor` 查健康度 + 价格象限 | 16 单元 action |
| **战场分配月度审计** | `battlefield_auditor` 每月 1 日触发 | 输出 BattlefieldAuditEvent |
| **收益仓库 30% 入仓** | `gain_vault_controller` 监听 take_profit | 自动转入 |
| **与维度二 ThesisProposedEvent 集成** | thesis 通过 → `state_machine_registry` 自动注册节点 | 跨模块协作 |
| **与维度一 RejectEvent 集成** | reject → `transition_engine` 强制 broken_any | 跨模块协作 |

## 五、部署形态建议

| 服务 | 部署形态 | 资源 |
|---|---|---|
| `state_machine_registry` | 独立微服务（PG 后端 + Redis 缓存）| 1 vCPU / 2GB RAM |
| `probe_scheduler` | 独立微服务（cron + 事件触发）| 1 vCPU / 2GB RAM |
| `transition_engine` | 独立微服务（含健康度算法）| 2 vCPU / 4GB RAM |
| `rebalance_advisor` | 独立微服务（4×4 矩阵）| 1 vCPU / 2GB RAM |
| `battlefield_auditor` | 独立微服务（月度调度）| 0.5 vCPU / 1GB RAM |
| `gain_vault_controller` | 独立微服务（PG 后端）| 0.5 vCPU / 1GB RAM |
| `invalidation_auditor` | 独立微服务（与 super_evo 联动）| 1 vCPU / 2GB RAM |
| `notification_dispatcher` | 独立微服务 | 0.5 vCPU / 1GB RAM |

## 六、共享服务（与维度四共享 + 与其他维度复用）

| 共享服务 | 说明 |
|---|---|
| `state_machine_registry` | 维度三（持仓监控）+ 维度四（卖出决策）共享同一节点状态机 |
| `transition_engine` | 维度三 + 维度四共享同一状态迁移引擎 |
| `invalidation_auditor` | 维度三 + 维度四共享失效审计 |
| `notification_dispatcher` | 与维度零紧急告警通道共享 |

## 七、上下游事件流

### 7.1 维度三输出（被其他维度消费）

```yaml
HealthChangeEvent:           消费方: 维度零 4 色卡片 + 紧急告警
RebalanceAdviceEvent:        消费方: 维度零调仓建议历史
BattlefieldAuditEvent:       消费方: 维度零月度战场饼图
StateTransitionEvent:        消费方: 维度五失效审计 + DPO 训练数据
```

### 7.2 维度三输入（消费其他维度）

```yaml
events:thrust:thesis_proposed:  维度二 thesis 通过 → 注册节点状态机
events:cryo_guard:reject:        维度一 reject → 强制 broken_any
events:price:daily_close:        T+0 收盘价 → SLI 探针触发
events:flywheel:lora_updated:    维度五 LoRA 更新
```
