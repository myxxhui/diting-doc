## 四、跨服务的协作场景

| 场景 | 涉及的 L3 服务 | 说明 |
|---|---|---|
| **认知边界检查（5 维度）** | `decision_gate` → `cognitive_boundary_check` 接口 | 所有 thesis 入推荐池前必经 |
| **reject 配额管理** | `decision_gate` 内 `reject_quota_manager` | 防御不过敏（日/周/月 ≤ 50%）|
| **多源弱信号汇聚** | `risk_event_bus` 聚合多引擎信号 | ≥ 2 引擎共同触发才升级为 reject |
| **黑名单生命周期** | `audit_log_service` 持久化 | 4 态 + 解封流程 |
| **F vs B 归因** | `audit_log_service` → super_evo `quadrant_router` | T+90/180 归因 + 8 象限路由 |

## 五、部署形态建议

| 服务 | 部署形态 | 资源 |
|---|---|---|
| `decision_gate` | 独立微服务（含 reject_quota_manager + cognitive_boundary_check）| 1 vCPU / 2GB RAM |
| `risk_event_bus` | 共用 Redis Streams 实例 | 与其他模块共享 |
| `audit_log_service` | 独立微服务（PG 后端）| 1 vCPU / 2GB RAM |
| `circuit_breaker` | 独立微服务 | 1 vCPU / 1GB RAM |
| `input_sanitizer` | 独立微服务（含 LLM 上下文检测）| 1 vCPU / 4GB RAM |

> 阶段 1 可合并部署为单一微服务（cryo_guard_service），阶段 2 按需拆分。

## 六、共享服务（与其他维度复用）

| 共享服务 | 与哪些维度共享 |
|---|---|
| `audit_log_service` | 维度二 PassEvent 审计 + 维度三 monitor 审计 + 维度四 exit 审计 |
| `risk_event_bus` | 维度二/三/四（订阅 reject 事件触发降级）|
| `circuit_breaker` | 维度五（训练失败熔断）|

## 七、上下游事件流

### 7.1 维度一输出（被其他维度消费）

```yaml
RejectEvent:       消费方: 维度零紧急告警 + 维度三强制 broken_any
DegradeEvent:      消费方: 维度零橙色卡片 + 维度二降级
PassEvent:         消费方: 维度二 thesis finalize 闸门
RejectAttributionEvent (T+90/180):  消费方: 维度五 8 象限路由
```

### 7.2 维度一输入（消费其他维度）

```yaml
events:flywheel:lora_updated:   维度五 LoRA 守门人通过后更新引擎模型
events:user:verified:            维度零反馈闭环 → 影响下次 LoRA 训练
```
