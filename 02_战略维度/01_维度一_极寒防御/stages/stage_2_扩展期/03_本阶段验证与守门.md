## 四、DPO 偏好对齐效果验证（**本阶段新增**）

### 4.1 DPO 评测对比

```python
def eval_dpo_effect(engine_id, before_lora, after_lora, holdout):
    metrics_before = run_holdout(before_lora, holdout)
    metrics_after = run_holdout(after_lora, holdout)
    
    # 必须满足：Recall 不退化（避免灾难性遗忘）
    assert metrics_after["recall"] >= metrics_before["recall"] - 0.01
    
    # 必须满足：Precision 显著提升
    assert metrics_after["precision"] >= metrics_before["precision"] + 0.05
    
    # 必须满足：架构师一致性提升
    assert metrics_after["kappa_with_architect"] >= 0.85
    
    return {"before": metrics_before, "after": metrics_after}
```

### 4.2 灾难性遗忘的"快速诊断"

| 诊断信号 | 含义 |
|---|---|
| Recall 退化 > 5% | 灾难性遗忘 → 立即回滚 LoRA |
| 关键案例 FN 增加 | 偏好数据质量问题 → 重新清洗偏好对 |
| Kappa 反而下降 | DPO 偏好对方向错误 → 架构师 review |

### 4.3 DPO 数据治理

- 偏好对必须 architect-verified（不接受 AI 自标）
- 30 天后随机抽 10% 复核（一致性 ≥ 0.90）
- 每月输出 DPO 数据质量报告
