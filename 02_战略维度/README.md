## 维度说明

L2 不再围绕旧的 A/B/C 轨展开，而是围绕新的**双目标系统**展开：

1. **产品目标系统**：让谛听成为真正可用的 A 股分析追踪智能平台  
2. **个人能力训练系统**：让谛听成为年底跳槽可复述、可演示、可量化的 AI 工程训练场

L2 的作用，就是把这两个目标拆成能落到 L3/L4 的具体维度。

## L2 的三种视角

### 视角一：产品设计视角

回答：

- 产品先做什么
- 哪些能力真的服务业务价值
- 什么是 MVP，什么是平台化延展

### 视角二：能力训练视角

回答：

- 当前阶段练的是岗位 1、4 还是 2、3
- 某项技术复杂度是否真的值得引入
- 一项工程实践最终能沉淀成什么面试故事

### 视角三：开发与交付视角

回答：

- 先做业务闭环还是先搭底座
- 什么阶段引入自托管推理、沙箱、评测闭环
- 如何在 8 个月内控制节奏，不把副业做成无底洞

## L2 的核心原则

### 原则 1：每个维度都要有“双重解释”

每个维度至少要回答两件事：

- 它解决了什么产品问题？
- 它训练了什么岗位能力？

### 原则 2：主线聚焦岗位 1 和岗位 4

L2 中凡涉及高成本复杂度的设计，应优先支持：

- 自托管推理 / 推理基础设施
- Agent Runtime / 沙箱 / 隔离 / 多租户

### 原则 3：岗位 2 和岗位 3 作为自然外溢结果

不单独为“管理感”或“MLOps 感”去造复杂度；  
只有当它们成为真实产品演进与主线工程演进的自然结果时，才纳入当前阶段。

### 原则 4：历史文档允许渐进迁移

当前 `diting-doc` 下仍有大量旧叙事文档。L2 的新职责不是一次性推翻全部，而是先完成：

1. L1 统一新叙事  
2. L2 统一新维度  
3. L3/L4 按阶段逐步迁移
