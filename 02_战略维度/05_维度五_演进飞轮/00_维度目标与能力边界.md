## 一、战略目标

把"系统使用越久越值钱"做成可验证的工程闭环。三大核心目标：

1. **AI 工程能力沉淀**：让架构师在构建 diting 的过程中真正掌握 LLMOps 全栈（Teacher 蒸馏、SFT、DPO、多 LoRA、议会模式）
2. **个人成长资产化**：每一项被掌握的工程能力都能复用到下一份工作 / 下一个项目（Resume-Driven Development）
3. **数字分身雏形**：长期而言，把架构师的判断风格固化为"数字分身"模型，能在架构师不在线时辅助决策

## 二、核心使命

充当系统的"基因迭代器"：
- 从其他 4 维度采集"决策事件 + 真实结果"
- 蒸馏为训练数据
- 训练新版本 LoRA / 规则
- 经过 Holdout 回放守门后部署
- 形成"使用 → 反馈 → 进化 → 部署"的闭环

## 三、关键能力清单（13 LLMOps 组件）

### 3.1 P0 启动套件（4 组件，第 1 个月就要跑通）

1. **Teacher LLM 蒸馏服务**：用 Claude 3.5 / GPT-4o 调用 API，按 prompt 模板批量生成训练数据
2. **数据湖 + DVC 版本化**：MinIO 存数据，DVC 锁定每次训练数据快照
3. **Label Studio 人工 verified**：架构师每周抽时间 verified 一批关键样本
4. **LLaMA-Factory 手动微调**：本地用 LoRA 微调小模型（Qwen2.5-7B），跑通"数据 → 模型"链路

### 3.2 P1 流水线套件（5 组件，第 2–4 个月）

5. **K8s GPU Job 自动训练触发**：CT 流水线，数据增量到一定阈值自动触发训练
6. **vLLM 推理网关**：所有维度调用 LoRA 模型都过 vLLM
7. **MLflow Model Registry**：管理所有版本 LoRA + 元数据 + 评测结果
8. **评测回归集与回放器**：Holdout 回放守门，自动 Block 退化版本上线
9. **DPO 偏好对齐流水线**：Stage C 开始引入

### 3.3 P2 高阶套件（4 组件，第 6 个月+）

10. **vLLM 多 LoRA 多路复用**：一台 GPU 同时服务多个专科 LoRA
11. **数字分身 RAG 系统**：基于架构师历史决策日志构建个人风格 RAG
12. **gVisor / Firecracker 沙箱**：Agent 生成的代码在沙箱执行，杜绝安全风险
13. **A/B 测试与灰度发布**：新 LoRA 先 5% 流量灰度，观察一周再全量

## 四、能力边界（不做什么）

| 边界 | 不做的事 | 归属 |
|---|---|---|
| 投资决策 | 不直接做风险/机会/卖出决策 | 维度一/二/三/四 |
| 自动下单 | 不直接执行交易 | 与 L1 永久禁止 |
| 替代人工标注 | Teacher LLM 不替代人工 verified（仅做"加速器"） | 永久要求人工 verified 高质量样本 |
| 不可解释的模型 | 不引入完全黑盒的端到端模型 | 必须可解释 |
