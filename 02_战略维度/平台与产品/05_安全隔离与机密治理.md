# L2 · 安全隔离与机密治理

> [!NOTE] **[TRACEBACK] 战略维度锚点**
> - **相关维度**: [平台技术栈与系统架构](./02_平台技术栈与系统架构.md) | [外部系统接入与边界抽象](./08_外部系统接入与边界抽象.md)

## 目标

本维度的核心不是保护旧交易账户，而是保护以下四类风险面：

1. API 密钥与云资源机密
2. 原始数据与用户侧敏感材料
3. Agent 执行动作或代码的运行边界
4. 多租户或多会话之间的隔离

## 安全重点

### 机密治理
- 严禁硬编码
- 使用加密 Secret、KMS 或等价机制
- 区分开发、测试、生产机密

### 最小权限
- 按服务拆权限
- 数据采集、推理、Runtime、管理接口不要混用同一凭证

### Runtime 隔离
- 对模型生成的动作或代码使用隔离执行
- 按阶段引入 gVisor、Firecracker 或等价沙箱
- 所有高风险执行都必须可审计、可超时回收

### 多租户预留
- 即使当前只有一个用户，也要在结构上保留会话隔离与资源配额能力

## 原则

1. 安全设计先服务当前产品真实风险，再服务未来平台化风险。
2. Runtime 安全必须与产品需求绑定推进，不做脱离场景的隔离设计。
3. 安全不只是“防泄漏”，还包括“防越权执行”和“防资源失控”。
