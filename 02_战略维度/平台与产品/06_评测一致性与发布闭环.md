# L2 · 评测一致性与发布闭环

> [!NOTE] **[TRACEBACK] 战略维度锚点**
> - **相关维度**: [数据体系与分层存储](./03_数据体系与分层存储.md)

## 目标

本维度替代旧的“研产同构”表述，强调的是：

- 分析工作流在线下验证和线上运行之间尽可能一致
- 每次迭代都可评测、可灰度、可回滚
- Prompt、RAG、Agent 工作流都能形成版本闭环

## 核心能力

### 1. 评测一致性
- 统一输入样例
- Golden set / failure set
- 结构化输出验证
- 关键任务回归测试

### 2. 版本治理
- Prompt 版本
- Workflow 版本
- 检索配置版本
- 模型与 LoRA 版本
- `MLflow` 中的实验、模型与版本记录

### 3. 发布闭环
- 小流量灰度
- 指标对比
- 回滚策略
- 效果回归
- `KServe` / `BentoML` 一类服务化部署与切流能力

### 4. CT 闭环
- 人工反馈沉淀
- 样本转评测集
- 需要时触发微调或规则更新
- drift detection 触发重新评估、重训或规则更新

## 标准工具链

当前项目的 MLOps 不应只停留在“有灰度和回滚”，还必须明确以下工具链位置：

| 能力 | 当前建议 |
|---|---|
| 实验跟踪 / 模型注册 | `MLflow` |
| 流水线编排 | `Kubeflow` 或 `Airflow` |
| 在线服务化 | `KServe` 或 `BentoML` |
| 特征资产 | feature store |
| 质量漂移 | drift detection |

这些能力不要求一次性全部落地到生产，但必须在文档和阶段节奏中成为明确实施对象。

## 发布决策闭环

发布治理不只体现在“提出需求”，还体现在以下闭环：

1. 定义上线前的核心指标与最低门槛。
2. 决定灰度范围、扩量条件和回滚条件。
3. 判断一个 AI 功能是否值得继续投入数据、算力和维护成本。
4. 用评测结果而不是主观印象推动路线调整。

## 原则

1. 没有评测就不要轻易宣布“优化成功”。
2. 没有回滚就不要把新工作流直接升成默认。
3. 评测治理的价值在于“把变化管理好”，不是单纯部署更多服务。
4. 发布治理的价值在于“把上线决策做对”，不是只把功能堆上去。
