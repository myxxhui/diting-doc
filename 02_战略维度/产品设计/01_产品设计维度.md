## 一、维度定义

新的“产品设计维度”，不是单纯回答“做什么功能”，而是同时回答：

1. **这个功能为什么对产品有价值？**
2. **这个功能为什么值得现在做？**
3. **这个功能是否顺带训练了目标岗位能力？**

换句话说，产品设计维度负责管理**业务价值**与**工程训练价值**之间的平衡。

## 二、关键目标

### 目标 1：保证产品先对自己有用

任何设计都应优先服务下列真实问题：

- 看不过来
- 看不全
- 看不深
- 看了以后跟不住

### 目标 2：保证复杂度有收益

复杂度只能在以下情况下被引入：

- 明显提升产品能力
- 明显提升岗位 1/4/3/2 的训练价值
- 最好两者同时成立

### 目标 3：明确阶段优先级

产品设计必须区分：

- **现在必须做**
- **下一阶段适合做**
- **未来可以做但现在不做**

否则副业会迅速失控。

## 三、核心范围

### P0：必须优先落地的能力

- 数据采集与清洗
- 财报 / 公告 / 新闻 / 行业信息组织
- 基础研究摘要
- 风险过滤
- 标的跟踪与提醒

这些能力的共同特点是：

- 直接提升你的使用价值
- 能最快形成反馈循环
- 是后续 Agent、RAG、Eval 的必要输入

### P1：在产品闭环成立后继续下沉的能力

- 多 Agent 编排
- 自托管推理
- 证据引用与结构化输出增强
- 评测与回归验证
- 发布、回滚与 CT

这些能力一方面增强产品质量，另一方面开始显著覆盖岗位 1/3/2。

### P2：平台化与硬核基础设施能力

- 沙箱执行
- 长任务状态机
- 多租户
- 资源隔离与配额
- 更强的平台接入与运行边界

这些能力会显著覆盖岗位 4，但只有在业务闭环成立后才值得继续投入。

## 四、优先级决策原则

### 原则 1：先真实需求，后“想象中的未来用户”

当前第一用户是你自己。  
所以优先级判断标准应是：

- 你是否真的会频繁使用
- 你是否真的会因为它少犯错、少花时间
- 你是否真的愿意基于它做复盘

### 原则 2：优先做“产品价值高且训练价值也高”的功能

优先级最高的是交集区：

- `RAG` 检索与证据引用
- Agent 工作流
- 自托管推理
- 可观测性
- Eval / 回归
- 沙箱执行

### 原则 3：避免两类无效建设

#### 无业务价值的炫技建设

比如：

- 一开始就做重型多租户平台
- 在没有长任务场景时先做复杂任务编排
- 还没确认有用就先做极重的微服务拆分

#### 无训练价值的低水平重复建设

比如：

- 永远停留在最简单 API 调用
- 用最省事方案把所有复杂问题都绕开
- 没有评测、没有观测、没有发布闭环

## 五、产品设计的双重验收

### 产品验收

要回答：

- 这项能力是否让研究更快、更深、更稳？
- 是否真实降低了漏信息、漏风险、漏跟踪的概率？

### 训练验收

要回答：

- 这项能力最终能支持你讲清哪个岗位故事？
- 它是否产出代码、指标、架构、设计取舍证据？

只有两个问题都成立，才算高质量设计。

## 六、系统使用者视角

当前阶段的主要使用者不是“公众用户”，而是你自己这个高频投研用户。  
因此产品设计优先支持以下动作：

- 快速建立候选池
- 快速查看风险卡片
- 快速查看 thesis 与证据
- 快速收到逻辑变化提醒
- 快速形成复盘记录

未来若平台化，再扩展为：

- 多租户用户
- 审阅者
- 管理员
- 运行值守者

## 七、与开发与交付维度的关系

- **本维度决定做什么、为什么做、先后顺序如何排**
- [开发与交付流程维度](../开发与交付/01_开发与交付流程维度.md) 决定这些能力按什么节奏落地

产品设计不直接规定命令、环境、部署步骤，但要规定阶段目标与准入边界。
