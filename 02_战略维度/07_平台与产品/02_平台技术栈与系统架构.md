## 设计目标

技术栈必须满足两件事：

- 能支撑 A 股分析追踪平台的真实工作流
- 能在成本与复杂度可控前提下稳定演进

## 当前推荐技术栈

| 层次 | 当前推荐 | 作用 |
|---|---|---|
| 应用语言 | Python 3.11+ | 数据处理、AI 工作流、生态成熟 |
| Web/API | FastAPI | 服务化与结构化接口 |
| 工作流 | LangGraph | 多 Agent 状态流转 |
| 数据库 | PostgreSQL + pgvector | 结构化数据与向量检索 |
| 缓存/队列 | Redis | 缓存、任务状态、异步协调 |
| 对象存储 | OSS / S3 兼容存储 | 原始资料、归档、评测产物 |
| 推理 | vLLM | 自托管推理网关 |
| 编排 | Docker + K8s / K3s | 环境统一、服务编排 |
| 观测 | OpenTelemetry + Prometheus + Grafana + Langfuse | 全链路观测 |
| Runtime | gVisor / Firecracker（按阶段引入） | Agent 安全执行与隔离 |

## 关键能力映射

### 推理与资源治理能力

| 能力 | 当前必须覆盖 |
|---|---|
| GPU 调度 | K8s GPU 调度、配额、利用率治理 |
| 推理性能优化 | `vLLM` 连续批处理、缓存、并发与延迟分析 |
| 推理网关 | `vLLM` 为起点，预留 `Triton` / `TGI` 对比与替换实验 |
| 分布式训练 | 至少完成 1 条多节点训练或模拟训练链路认知 |
| checkpoint | 模型或工作流 checkpoint、恢复、失败重试 |
| RTC / 低延迟链路 | 至少保留一段实时流式交互或低延迟链路能力验证 |

### Runtime 与隔离能力

| 能力 | 当前必须覆盖 |
|---|---|
| WebSocket / 长连接 | 会话级流式响应、断线重连、背压与资源控制 |
| 会话恢复 | 长任务状态持久化、恢复、幂等执行 |
| browser / computer-use runtime | 浏览器操作或等价执行环境接入 |
| 沙箱 | `gVisor` / `Firecracker` 为主线，保留 `nsjail` / `seccomp` 细化空间 |
| Linux 隔离 | `namespace` / `cgroup` / 能力边界 |
| 细粒度安全 | `seccomp`、审计、超时、资源回收、必要时引入 `eBPF` 观测 |

### 评测与发布能力

| 能力 | 当前必须覆盖 |
|---|---|
| 实验与模型版本 | `MLflow` |
| 流水线编排 | `Kubeflow` 或 `Airflow` |
| 服务化部署 | `KServe` 或 `BentoML` |
| 特征资产 | feature store |
| 质量漂移 | drift detection |
| 发布治理 | 灰度、回滚、CT、门禁 |

### 产品治理能力

- 产品指标体系：定义效果指标、效率指标、成本指标、风险指标
- 需求优先级框架：明确 P0 / P1 / P2 的进入标准
- 上线决策：什么时候灰度、什么时候回滚、什么时候不上线
- 组织协作：产品、工程、数据、标注 / 评测之间的职责分界
- 商业化判断：某个 AI 功能是否真的值得投入算力、数据和维护成本

## 架构原则

1. 先模块化单体，再逐步服务化。
2. 推理层、Runtime 层与业务层边界明确。
3. 所有外部能力通过适配层接入，避免深度绑定单一供应商。
4. 自托管能力的引入必须服务产品价值，而不是只为“看起来高级”。
5. 底层能力不能停留在名词层，必须给出至少一轮真实工程验证。
6. 治理能力不能停留在“写 PRD”，必须落到指标、优先级、上线门禁和 ROI 判断。
