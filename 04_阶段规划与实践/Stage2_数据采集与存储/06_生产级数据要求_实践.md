## 数据采集在 K3s 的部署方式与镜像

| 问题 | 说明 |
|------|------|
| **如何部署在 K3s？** | 远程 K3s 以 **Job** 实现：**diting-infra** `charts/diting-stack` 内 `templates/ingest/` 定义 Secret `diting-ingest-conn` + Job（名称带 revision，如 `diting-ingest-5`，非 Hook 以免 Job 失败导致部署报错）；连接信息由 values 生成，集群内用 Service DNS。**云原生依赖就绪**：Job 的 Pod 带 **init 容器**（busybox），在 TimescaleDB / PostgreSQL L2 / Redis 端口可达前循环等待；主容器在 init 通过后才执行 `make ingest-deploy`。因此 stack 与 DB 可同一次 `make deploy` 顺序部署，无需两阶段 Helm。详见 diting-infra `docs/ingest-cloud-native.md`。 |
| **镜像与环境？** | diting-core **Dockerfile.ingest** 构建采集镜像（`diting-ingest:test`，`make build-images`）；Python 3.11、akshare 1.17.1、psycopg2、redis、pandas 等，与 `make deps-ingest` 一致。Job 执行 **`make ingest-deploy`**（见下节），Secret 注入 DSN 后连 L1/L2/Redis；Pod 需能访问外网行情源。 |

## 生产数据环境：前半部分（环境与数据继承）

**目标**：`make deploy diting prod` 得到竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis，连接信息写入 `prod.conn`；`make down diting prod` 仅释放 ECS/EIP、**磁盘数据保留**；再次 Up 挂载同盘实现数据继承。服务与端口由 **config/diting-prod.yaml** 控制，见 diting-infra config/README.md。设计见 [06_设计#生产数据环境与两段实现](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-prod-data-env)。prod 用独立 NAS 与 dev 隔离；完整销毁用 `FULL_DESTROY=1 make down diting prod`。

**前半可执行步骤（按序）**：

| 步骤 | 工作目录 | 命令/操作 |
|------|----------|-----------|
| 1 | diting-infra | 见下方代码块 |
| 2 | diting-core | 见下方代码块 |
| 3 | diting-core | 见下方代码块 |
| 4 | diting-infra | 见下方代码块 |
| 5 | diting-infra | 见下方代码块 |
| 6 | diting-core | 见下方代码块 |

```bash
# 前半步骤 1：部署
cd /path/to/diting-infra && make deploy diting prod

# 前半步骤 2：复制连接信息
cp /path/to/diting-infra/prod.conn /path/to/diting-core/.env

# 前半步骤 3：连通验证
cd /path/to/diting-core && make verify diting prod

# 前半步骤 4：释放（保留盘）
cd /path/to/diting-infra && make down diting prod

# 前半步骤 5：再次部署
cd /path/to/diting-infra && make deploy diting prod

# 前半步骤 6：再次验证
cp /path/to/diting-infra/prod.conn /path/to/diting-core/.env
cd /path/to/diting-core && make verify diting prod
```

**前半核心验证（三项）**：① 环境与连接 — Up 后 `cp prod.conn .env` 且 `make verify diting prod` 退出码 0。② Down 保留盘 — `make down` 后 `prod.disk_id` 存在或控制台可见数据盘。③ 再次 Up 数据继承 — 再次 deploy 后 verify 或 psql 确认数据与 Down 前一致。

<a id="l4-stage2-06-phase2"></a>
## 生产数据环境：后半部分（采集生产级 + 落库 + 数据继承）

**顺序**：先 **`make deploy diting prod`**（部署会产出 `prod.conn`）→ 再设置 .env（`cp prod.conn .env`）→ 再执行验证。任一步失败先分析修复重试，同一问题重试两次以上仍失败再 [回收](#l4-stage2-06-phase2-teardown)。

**部署顺序（避免死循环）**：`make deploy diting prod` 先 **提交 Diting Stack**（`helm install/upgrade --no-wait`），创建 PVC、Job、Deployment 后**立即返回**，终端不阻塞；再**部署数据库**（TimescaleDB / PostgreSQL L2 / Redis）。DB 就绪后，Stack 的 init 容器会自动连上并退出，业务 Pod 正常启动。若改为对 Stack 使用 `--wait`，init 会一直等 DB，而 DB 在下一步才部署，会导致「终端与 init 一起卡住、数据库永远起不来」的死循环，因此部署流程采用 **Stack 先提交不等待 → 再部署 DB**。

**采集触发**：初次部署时，若 `config/diting-prod.yaml` 中 `stack.ingest.enabled=true`，`make deploy diting prod` 会一次完成 stack 提交 + 数据库部署；**diting-ingest** Job 作为普通资源创建（不作为 Hook，避免 Job 失败导致整次部署报错），名称带 release revision（如 `diting-ingest-5`）。Pod 的 **init 容器**会等待 TimescaleDB/PostgreSQL L2/Redis 端口就绪再退出，主容器随后执行 `make ingest-deploy`（全量/增量由 [每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental) 判断）。**若 Job 状态为 Failed / BackoffLimitExceeded**：`kubectl get jobs -n default` 找到最新 `diting-ingest-<revision>`，执行 `kubectl logs job/diting-ingest-<revision> -n default -c ingest` 查看主容器日志（常见为镜像拉取失败、Python/采集脚本报错）。**若暂无日志**（Pod 已被 TTL 回收）：(1) 下次部署后立即在另一终端执行 `kubectl get pods -n default -w`，一旦出现 `diting-ingest-*` Pod 即执行 `kubectl logs -f pod/<该 Pod 名> -n default -c ingest` 抓取输出；(2) 或运行一次性 Debug Pod（同镜像、同 Secret、同 command）直接看报错，见下「无日志时排查」。**后续**也可在本机随时触发：在 diting-core 内执行 `cp $INFRA/prod.conn .env && make ingest-deploy`。

**无日志时排查**（Job Failed 且 `kubectl logs job/diting-ingest-<revision>` 无输出时）：在已配置 KUBECONFIG 的本机执行下面命令，会启动一个临时 Pod、注入与 Job 相同的 Secret 与命令，终端直接输出报错（镜像、Secret 名与 config 一致时替换 `<REVISION>` 为当前任意 revision 或省略后缀即可）：

```bash
kubectl run diting-ingest-debug --rm -it --restart=Never -n default \
  --image=crpi-7vifw4ok9jkcxr60.cn-hongkong.personal.cr.aliyuncs.com/titan-core/diting-ingest:latest \
  --overrides='{"spec":{"imagePullSecrets":[{"name":"acr-titan"}],"containers":[{"name":"ingest","image":"crpi-7vifw4ok9jkcxr60.cn-hongkong.personal.cr.aliyuncs.com/titan-core/diting-ingest:latest","command":["make","ingest-deploy"],"envFrom":[{"secretRef":{"name":"diting-ingest-conn"}}],"env":[{"name":"INGEST_DEPLOY_FULL_DAYS_THRESHOLD","value":"7"}]}]}}'
```

退出后可根据终端报错修复（如缺依赖、连库失败、脚本异常等）。Chart 已将该 Job 的 `ttlSecondsAfterFinished` 调为 86400，失败后 Job/Pod 会保留更久以便补查日志。

### 运行步骤与验证

| 步序 | 工作目录 | 命令/操作 |
|------|----------|-----------|
| 1 | diting-core | `make push-images` |
| 2 | diting-infra | `make deploy diting prod`（产出 `prod.conn`；若 `stack.ingest.enabled=true` 会启动采集 Job） |
| 3 | diting-core | `cp $INFRA/prod.conn .env` |
| 4 | diting-core | `make verify diting prod` |
| 5 | 本机 | 等待 Job 完成：`kubectl get jobs -n default`（名称形如 `diting-ingest-<revision>`）、`kubectl logs job/diting-ingest-<revision> -n default` |
| 6 | diting-core | `make verify-data-production` |
| 7（可选） | diting-infra / diting-core | 数据继承验证：`make down diting prod` → `make deploy diting prod` → `cp prod.conn .env` → `make verify diting prod` → `make verify-data-production` |
| 8 | diting-infra | 回收：`make down diting prod` |
| **全量生产级采集**（推荐本机） | diting-core | `make ingest-production`（**最佳分批模式**：全 A 股 + 单标≥5 年日线；分批拉取、批间暂停、断点续跑，仍从 AkShare 采集但绕过限流/断连；须先 `cp $INFRA/prod.conn .env`） |
| **生产级日终增量**（每交易日结束后） | diting-core | `make ingest-production-incremental`（全 A 股仅补最近 N 天，默认 7 天；约 1h；须先 `cp $INFRA/prod.conn .env`） |

**数据与模块验证须有证据**（见 §7.2 第 8 条、规则生效要点「数据/模块验证结论（须有证据）」）：断言「数据符合预期」「模块验证通过」时，须同时提供**数据库查询结果**（表、条件、行数或样本）与**模块执行输出**（命令与日志摘要）；不得在模块明显为降级结果（如 Module A 全「未知」、L2 行业名为空未补全）时断言符合预期。可执行证据脚本（工作目录 diting-core，使用 .env 的 DSN 与 DITING_SYMBOLS）：

```bash
cd diting-core && python3 scripts/verify_production_data_evidence.py
```

脚本输出：目标标的数、L1 ohlcv 有数据标的数与样本、L2 industry_revenue_summary 有数据标的数与 industry_name 样本、L2 data_versions 新闻条数、Module A 执行退出码与分类结果摘要。据此判断数据是否落库、Module A 是否使用 L2 并输出非降级结果。

**DSN 与表一致性**（采集、验证、Module A 是否用同一库）：采集脚本、验证/证据脚本、Module A 均通过 **diting-core/diting/ingestion/config.py** 的 `get_timescale_dsn()` / `get_pg_l2_dsn()` 读取 DSN（优先 .env，与 `make ingest-production` 同源）。写入与查询的表一致：L1 为 **ohlcv**（period='day'），L2 为 **industry_revenue_summary**（Module A 读）、**data_versions**（新闻）。若 DSN 错误则连不上库，不会出现「写入成功但查不到」；若查不到目标标的，多为 **symbol 格式**（须带后缀如 002371.SZ）或 **日期区间** 与配置不一致。**诊断脚本**（与 ingest 同源 DSN，列出 ohlcv 全表概况与 symbol 样本）：`cd diting-core && python3 scripts/check_ohlcv_table_consistency.py`，可核对 L1 实际存在的 symbol 格式与行数、L2 industry_revenue_summary 行数与 industry_name 样本。

**跳过逻辑与日期区间**：Phase 1（K 线）是否跳过由 **配置日期区间内** 是否有数据决定（`INGEST_JQDATA_DATE_START` / `INGEST_JQDATA_DATE_END`）。若库中仅有其他区间（如历史 2024）而无该区间（如 2026-01-01～2026-03-07），则 `_count_ohlcv` 为 0，不会跳过，会写入新区间数据；写入后再次执行同区间会跳过。

```bash
# 5. 可选：数据继承验证（down → deploy → verify）
# cd "$INFRA" && make down diting prod && make deploy diting prod
# cp -f "$INFRA/prod.conn" "$CORE/.env" && cd "$CORE" && make verify diting prod && make verify-data-production

# 6. 回收

```

**全量生产级采集**（本机执行，满足 AB 模块生产级数据）：`cp $INFRA/prod.conn "$CORE/.env"` 后 `cd "$CORE" && make ingest-production`。采集采用**最佳分批模式**（见下方「[一键生产级采集（最佳分批模式）](#l4-stage2-06-ingest-production-batch-mode)」）：数据源仍为 AkShare，通过分批、批间暂停与断点续跑绕过限流/断连。**生产级日终增量**（每交易日结束后）：`cd "$CORE" && make ingest-production-incremental`。**其他触发**（非部署时）：`cp $INFRA/prod.conn .env && make ingest-deploy`，会检查 L1 并执行全量或增量（注意当前增量仅为 15 标，非生产级）。

**本地一键生产全量（diting-infra 一条命令）**：在 **diting-infra** 目录设置 `REPO_I_ROOT` 指向 diting-core 根目录后，执行 **`make local-ingest-production-prod`**，会复制 prod.conn 到 diting-core 的 .env 并执行 **make ingest-production**（全 A 股 + 单标≥5 年 + 行业/财务 + 新闻，符合 AB 模块数据要求）。**后台运行 + 日志**：执行 **`make local-ingest-production-prod-background`**，采集在后台运行，输出写入 **ingest-production.log**（默认 diting-infra 根目录，可覆盖 `INGEST_PROD_LOG=路径`）；关终端不杀进程，查看进度：**`tail -f ingest-production.log`**。说明：合上笔记本盖后本机可能休眠会暂停进程；若需合盖也继续请用 **make trigger-ingest**（集群内 Job）或 SSH 到服务器执行上述后台 target。

### 部署流程卡住（init 与终端一起卡、INSTALLATION FAILED: timed out）

若之前出现 **「部署 Diting Stack」时终端卡住**、**`diting-ingest` / `diting-module-a` 一直 Init:0/1**、**Helm 报错 `INSTALLATION FAILED: timed out waiting for the condition`**，原因是：Stack 的 Pod 带 init 容器在等 DB 就绪，而 DB 在下一步才部署，形成死循环。**当前 Makefile 已改为**：先对 Stack 执行 **`helm install/upgrade --no-wait`**（提交后立即返回），再部署数据库；DB 就绪后 init 自动通过。若你仍在使用旧流程或自行改过 Makefile，请确保 **Stack 使用 `--no-wait`、数据库在 Stack 之后部署**，见上文 [部署顺序（避免死循环）](#l4-stage2-06-phase2)。

### 采集 Job 网络类报错（RemoteDisconnected / Connection aborted）

若日志出现 **`RemoteDisconnected`**、**`Connection aborted`**、**`Remote end closed connection without response`**，多为 **Pod/节点或本机访问东方财富接口异常**：节点未出网、安全组未放行出站、东方财富限流或对当前 IP 断连。**处理**：① 确认 K3s 节点（ECS）能访问外网（如从节点或 Pod 内 `curl -I https://quote.eastmoney.com`）；② 检查安全组/路由是否有出站限制；③ **本机全量请用 `make ingest-production`**（已启用**最佳分批模式**：分批+批间暂停+断点续跑），可减轻限流/断连；中断后重跑会从未完成批继续；④ 代码已加重试（5 次、退避 5s～25s），可先重跑 Job 看是否偶发；⑤ 若集群长期无法稳定访问行情源，可改为在本机执行 `cp prod.conn .env && make ingest-deploy`；⑥ **东方财富持续断连时改用 Baostock**：在 `.env` 中设置 **`INGEST_OHLCV_SOURCE=baostock`**，并在 diting-core 执行 **`make deps-ingest`**（会安装 baostock），再执行 `make ingest-production` 或 `make ingest-production-incremental`。Baostock 为免费 A 股日线接口，通常更稳定，适合本地或受限网络环境。

### 标间延迟与全 A 股耗时

当前默认**标与标之间**间隔 2 秒（`INGEST_OHLCV_DELAY_BETWEEN_SYMBOLS=2`），用于减轻东方财富限流。**影响**：15 标约 +28 秒；若全 A 股约 5000+ 标，仅间隔约 **2.8 小时**（5000×2s），再加单标请求与写入时间，整轮可能 3～4 小时以上。若要缩短全市场采集时间，可在环境变量中设 **`INGEST_OHLCV_DELAY_BETWEEN_SYMBOLS=0`** 或 **`0.5`**（Job 的 env 或 values 中注入），需自担限流/断连风险；当前生产数据环境仍使用 **15 标**（`REAL_TEST_SYMBOLS_15`），不跑全 A 股。

### 更好方案（全 A 股 / 大批量）

东方财富/akshare 无「一次请求多标的日线」的批量接口，只能逐只请求。**`make ingest-production` 已采用最佳分批模式**（见 [一键生产级采集（最佳分批模式）](#l4-stage2-06-ingest-production-batch-mode)），在保证限流可控前提下可采用：

| 方案 | 说明 | 预期耗时（5000 标量级） |
|------|------|-------------------------|
| **最佳分批模式（默认）** | **`make ingest-production`** 默认：每批 150 标、批间暂停 45s、断点续跑；单批内仍用 CONCURRENT=3、RATE=1.5。数据源仍为 AkShare，通过分批+批间暂停绕过限流/断连。 | 约 **1.5～2 小时**（含批间暂停）；中断后可重跑只补未完成批 |
| **可控并发 + 全局限速** | 单批内设置 **`INGEST_OHLCV_CONCURRENT=3`**（或 5）、**`INGEST_OHLCV_RATE_PER_SEC=1.5`**：多线程并发拉取，全局限速。 | 单批 150 标约 100s；整轮见上 |
| 标间延迟调低 | 仅串行时可将 `INGEST_OHLCV_DELAY_BETWEEN_SYMBOLS=0.5` 或 `0`，自担限流风险。 | 2s 间隔约 2.8h；0.5s 约 42min 仅间隔 |
| 断点续跑 | 已实现：进度写入 `.ingest_production_progress`，再次执行从未完成标的继续。 | 单次耗时不变，重跑时只补漏 |
| 数据源选型 | **JQData（聚宽）**：设 **`INGEST_SOURCE=jqdata`** 并配置 `JQDATA_USER`、`JQDATA_PASSWORD`，标的/行情/行业财务均从 JQData 拉取，需 `pip install jqdatasdk`。或 Baostock（`INGEST_OHLCV_SOURCE=baostock`）、Tushare、万得等。 | 视接口与配额而定 |
| **行业（Module A）与网络** | 行业名来自东方财富 **stock_individual_info_em**；当 **INGEST_SOURCE=jqdata** 且 JQData 无行业数据时会自动回退到该接口。东方财富在某些网络（如境外/香港节点）可能不可达（RemoteDisconnected），此时 L2 行业为空、Module A 输出「未知」；在**可访问东方财富的环境**（如境内本机）执行 **make ingest-production** 即可写入有效行业并使 Module A 得到 AGRI/TECH/GEO。 | 见 diting-core `industry_revenue.py` |

**推荐**：全 A 股直接执行 **`make ingest-production`**（已含分批+批间暂停+断点续跑）；需微调时在 `.env` 中设置 `INGEST_OHLCV_BATCH_SIZE`、`INGEST_OHLCV_BATCH_PAUSE_SEC` 及单批内 `INGEST_OHLCV_CONCURRENT`、`INGEST_OHLCV_RATE_PER_SEC`。

### 采集数据与生产级要求对照（每标的：量化 + 行业/财务 + 新闻）

**生产标准**：每个标的均采集 **量化（OHLCV）、行业/财务、新闻**，与 06_ 设计、11_ 规约一致。

| 要求项 | 规约约定 | 当前 ingest-production 实际采集 | 是否满足生产要求 |
|--------|----------|----------------------------------|------------------|
| **量化（OHLCV）** | 单标日线 ≥5 年、全 A 股标的 | ✅ 每批内 step2：按 universe **逐批**拉取该批约 5 年日线写入 L1；分批（默认 15 只/批）、批间暂停、断点续跑 | ✅ **满足**：全 A 股 + 单标 ≥5 年 |
| **标的池（universe）** | 全 A 股、updated_at 有效 | ✅ step1 仅执行一次，刷新全 A 股列表写入 a_share_universe 表（约 5500 标） | ✅ **满足** |
| **复权** | 与 02_/回测一致 | ✅ 与 akshare/baostock 约定一致 | ✅ **满足** |
| **行业/财务（Module A 输入）** | 申万行业、营收占比等 | ✅ 每批内 step3：对该批**逐只**拉取并写入 L2。优先用东方财富 **stock_individual_info_em**（含「行业」）解析写入 industry_revenue_summary，使 Module A 能分类 AGRI/TECH/GEO；失败时回退 stock_financial_abstract 仅作版本记录。同批大小与批间暂停，标间延迟可配（默认 2s） | ✅ **满足**：每标的均有 |
| **新闻/公告** | 国内新闻、公告、研报 → L2/知识库 | ✅ step4 全市场+宏观在首批前执行一次；每批内 step4 对该批**逐只**拉取个股新闻（stock_news_em）写入 L2；标间延迟与批间暂停同上 | ✅ **满足**：每标的均有个股新闻版本 |
| **Module B 输入** | L1 OHLCV、板块与 A 同源 | ✅ 全 A 股 OHLCV 在 L1；行业/财务与新闻按标写入 L2 | ✅ **满足** |

**结论**：**`make ingest-production`** 已按「每标的：量化 + 行业/财务 + 新闻」生产标准执行：step1 仅执行一次刷新 universe；之后**每批**内依次执行 step2（K 线）→ step3（行业/财务）→ step4（该批个股新闻），step4 全市场+宏观新闻在首批前执行一次；断点续跑以已完成标的数为单位，每批完成本批四阶段后再更新进度。可供 Module A/B/C 生产级使用。

### 每个交易日结束后的增量采集（生产级）

| 项目 | 说明 |

| **目标** | 标的覆盖 = 全 A 股；仅补最近 1～5 个交易日数据，维持 L1 新鲜度；universe 可同日刷新；行业/新闻按需更新。 |

### 全量全 A 股更新（怎么跑、耗时与限流）

<a id="l4-stage2-06-ingest-production-batch-mode"></a>
### 一键生产级采集（最佳分批模式）

**设计目标**：仍从 **AkShare** 采集数据，但通过**分批策略**绕过东方财富限流、断连与长时间连续请求导致的失败，提升全 A 股全量采集成功率与可恢复性。

| 策略 | 说明 | 绕过 AkShare 限制的方式 |
|------|------|--------------------------|
| **分批** | 全 A 股拆成多批（默认每批 15 只），OHLCV 逐批拉取并**立即写入 L1**；行业/财务、个股新闻同批大小逐批拉取并写入 L2 | 单批请求量可控，避免长时间高密度请求触发限流或 RemoteDisconnected |
| **批间暂停** | 每批完成后暂停（默认 60 秒）再发下一批 | 给数据源“冷却”，降低 IP 被限流或断连的概率 |
| **标间延迟** | 行业/财务、个股新闻阶段，每两只标的之间延迟（默认 2 秒） | 进一步减轻限流，仅用于 step3/step4 |
| **断点续跑** | 每批完成本批的 step2+step3+step4 后将进度写入 `.ingest_production_progress`；再次执行时从未完成的批继续 | 中途中断后从下一批起补跑，该批内仍会执行 K 线 + 行业/财务 + 个股新闻 |

**环境变量**（可选）：`make ingest-production` 启动时会**自动加载 diting-core 根目录下的 `.env`**（仅对未在 shell 中设置的变量写入环境），因此 **`.env` 中的配置均生效**。也可在执行前 `export`。

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `INGEST_OHLCV_BATCH_SIZE` | 15 | 每批标的数量（OHLCV / 行业/财务 / 新闻共用）；小批断连率低 |
| `INGEST_OHLCV_BATCH_PAUSE_SEC` | 60 | 批间暂停秒数；网络不稳可调高 |
| `INGEST_PHASE_PAUSE_SEC` | 30 | 阶段间暂停（仅 INGEST_SEPARATE_PHASES=1）：K线→行业、行业→新闻 之间的秒数。东方财富行业接口在境外/限流下常**首次请求即** RemoteDisconnected，长暂停无法避免断连，默认 30 以节省时间；行业会走 fallback 写入 |
| `INGEST_EXTRA_DELAY_SEC` | 2 | 行业/财务、个股新闻的标间延迟（秒） |
| `INGEST_PRODUCTION_RESUME` | 1 | 是否启用 OHLCV 断点续跑（1/0 或 true/false） |
| `INGEST_OHLCV_CONCURRENT` | 1 | 单批内 OHLCV 并发数（保守以减断连） |
| `INGEST_OHLCV_RATE_PER_SEC` | 0.5 | 单批内 OHLCV 全局限速（次/秒） |

**生产采集开关与时间线**（0 或不写 = 关闭，其它数字 = 开启且表示数据长度/天数）：

| 变量 | 说明 |
|------|------|
| `INGEST_PRODUCTION_UNIVERSE` | 0 或不写=不拉标的池；非 0=拉取全 A 股列表（无数值含义） |
| `INGEST_PRODUCTION_OHLCV` | 0 或不写=不拉 K 线；正数=拉取且为回溯天数（如 30、1825≈5 年） |
| `INGEST_PRODUCTION_INDUSTRY_REVENUE` | 0 或不写=不拉行业/财务；非 0=拉取（无数值含义） |
| `INGEST_PRODUCTION_NEWS` | 0 或不写=不拉新闻；正数=拉取且只保留最近 N 天（如 30） |

**远程 K3s 采集 Job**：Pod 内无 `.env`，上述四个开关及**数据源相关变量**（`INGEST_SOURCE`、`JQDATA_USER`/`JQDATA_PASSWORD`、`INGEST_JQDATA_DATE_START`/`INGEST_JQDATA_DATE_END`）须通过 Chart 注入。在 **config/diting-prod.yaml** 的 **stack.ingest** 下配置：**productionUniverse**、**productionOhlcv**、**productionIndustryRevenue**、**productionNews**（Step 开关），以及 **ingestSource**（如 `jqdata`）、**jqdataDateStart** / **jqdataDateEnd**、**jqdataUser** / **jqdataPassword**（用 JQData 时必填）。与本地 `.env` 中同名变量含义一致，见 diting-stack values.yaml。

**JQData 日期范围**（当 `INGEST_SOURCE=jqdata` 时）：在 `.env` 中直接配置起止日期（格式 `YYYY-MM-DD` 或 `YYYYMMDD`）。试用版通常仅支持「约 3 个月前～15 个月前」区间，按实际可拉区间填写。

| 变量 | 说明 |
|------|------|
| `INGEST_JQDATA_DATE_START` | 拉取起始日期，如 `2025-05-01` |
| `INGEST_JQDATA_DATE_END` | 拉取结束日期，如 `2025-12-01` |
| `INGEST_JQDATA_DAYS_BACK_START` / `INGEST_JQDATA_DAYS_BACK_END` | 未配置上面日期时按「距今天数」计算（默认 450 / 90） |

**断点续跑**：进度文件为 diting-core 根目录下的 `.ingest_production_progress`（已加入 .gitignore）。全量正常跑完后会自动删除该文件，下次 `make ingest-production` 仍为完整全量。若希望强制从头跑，可删除该文件或设置 `INGEST_PRODUCTION_RESUME=0`。

**耗时粗估**（约 5500 标）：批数 ≈ 5500÷15 ≈ 367 批；OHLCV 单批 15 标、批间 60s；step3 行业/财务、step4 个股新闻同批数，标间延迟 2s、批间 60s。整轮为**数小时级**；OHLCV 支持断点续跑。

### 本地加速（尽量不触发限流）

在**本机**执行时，出口 IP 多为家庭/企业网，东方财富对这类 IP 限流通常比机房宽松，可适当提高并发与限速以缩短时间；若出现 `RemoteDisconnected` 再降回保守档。

| 档位 | CONCURRENT | RATE_PER_SEC | 5500 标 OHLCV 粗估 | 用法 |
|------|------------|--------------|---------------------|------|
| **保守**（默认） | 3 | 1.5 | 约 55 min～1 h | `make ingest-production` / `make ingest-production-incremental`（不改 .env） |
| **加速** | 5 | 2.0 | 约 45 min | 在 diting-core 下执行：`INGEST_OHLCV_CONCURRENT=5 INGEST_OHLCV_RATE_PER_SEC=2.0 make ingest-production`（或 `make ingest-production-incremental`）；或把两变量写入 `.env` 后直接 `make ingest-production` |
| **更快**（自担限流风险） | 6 | 2.5 | 约 35 min | 同上，设为 6 与 2.5；若频繁断连则改回 5/2.0 或 3/1.5 |

**建议**：先试**加速**档（5 / 2.0），无报错再视需要试**更快**档；代码允许 CONCURRENT 最大 16、RATE 最大 5.0，但过高易触发限流，本地一般 5～6、2.0～2.5 为宜。**一键加速**：不改 .env 可直接用 **`make ingest-production-fast`**、**`make ingest-production-incremental-fast`**（内部 5/2.0），若出现断连则改回 `make ingest-production` / `make ingest-production-incremental`。

### 写入报错：column "period" of relation "ohlcv" does not exist

若采集拉数成功但写入时报 **`column "period" of relation "ohlcv" does not exist`**，说明当前库里的 **ohlcv 表是按旧版建的**（无 `period` 列），与规约（diting-infra 的 `schemas/sql/01_l1_ohlcv.sql`）不一致。**处理**：用当前库的连接串执行迁移（在 diting-infra 目录下）：  
`psql "<连接串>" -f schemas/sql/01_l1_ohlcv_add_period.sql`  
连接串可从 `prod.conn` 或 `.env` 中的 `TIMESCALE_DSN` 取得。迁移仅在表已存在且缺少 `period` 时生效（旧表为 `trade_date` 时会重命名为 `datetime` 并改主键）；新建表请直接用 `01_l1_ohlcv.sql`。

若写入 **a_share_universe** 时报 **`column "market" of relation "a_share_universe" does not exist`**，说明表为旧结构（仅有 symbol/name/list_date/updated_at）。**处理**：对同一 L1 库执行  
`psql "<连接串>" -f schemas/sql/02_a_share_universe_add_market.sql`  
会补列 `market`、`count`、`source`。
