| 问题 | 说明 |
|------|------|
| **如何部署在 K3s？** | 远程 K3s 以 **Job** 实现：**diting-infra** `charts/diting-stack` 内 `templates/ingest/` 定义 Secret `diting-ingest-conn` + Job `diting-ingest`；连接信息由 values 生成，集群内用 Service DNS。`make deploy diting prod` 就绪后若 `stack.ingest.enabled=true`，post-install/post-upgrade Hook 会创建并运行该 Job。详见 diting-infra `docs/ingest-cloud-native.md`。 |

<a id="l4-stage2-06-deploy-check-full-incremental"></a>
### 每次部署检查全量/增量（当前策略）

**触发时机**：每次部署（Helm post-install/post-upgrade）触发一次采集 Job。

**决策逻辑**（diting-core `scripts/run_ingest_deploy.py`，`make ingest-deploy`）：连 L1 查 ohlcv 及 `MAX(datetime)`；无数据或超过 N 天（默认 7 天，可配 `INGEST_DEPLOY_FULL_DAYS_THRESHOLD` / values `ingest.fullDaysThreshold`）→ 全量（`run_ingest_production.py`）；有数据且未过期 → 增量。部署即检查并补齐，与 06_dna/06_ 设计一致。diting-infra：Chart 从 values 生成 Secret `diting-ingest-conn`；storage init（weight -5）先于 ingest Job（weight 5）。

<a id="l4-stage2-06-phase1"></a>
## 生产数据环境：前半部分（环境与数据继承）

**目标**：`make deploy diting prod` 得到竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis，连接信息写入 `prod.conn`；`make down diting prod` 仅释放 ECS/EIP、**磁盘数据保留**；再次 Up 挂载同盘实现数据继承。服务与端口由 **config/diting-prod.yaml** 控制，见 diting-infra config/README.md。设计见 [06_设计#生产数据环境与两段实现](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-prod-data-env)。prod 用独立 NAS 与 dev 隔离；完整销毁用 `FULL_DESTROY=1 make down diting prod`。

**前半可执行步骤（按序）**：

| 步骤 | 工作目录 | 命令/操作 |
|------|----------|-----------|
| 1 | diting-infra | 见下方代码块 |
| 2 | diting-core | 见下方代码块 |
| 3 | diting-core | 见下方代码块 |
| 4 | diting-infra | 见下方代码块 |
| 5 | diting-infra | 见下方代码块 |
| 6 | diting-core | 见下方代码块 |

```bash
# 前半步骤 1：部署
cd /path/to/diting-infra && make deploy diting prod

# 前半步骤 2：复制连接信息
cp /path/to/diting-infra/prod.conn /path/to/diting-core/.env

# 前半步骤 3：连通验证
cd /path/to/diting-core && make verify diting prod

# 前半步骤 4：释放（保留盘）
cd /path/to/diting-infra && make down diting prod

# 前半步骤 5：再次部署
cd /path/to/diting-infra && make deploy diting prod

# 前半步骤 6：再次验证
cp /path/to/diting-infra/prod.conn /path/to/diting-core/.env
cd /path/to/diting-core && make verify diting prod
```

**前半核心验证（三项）**：① 环境与连接 — Up 后 `cp prod.conn .env` 且 `make verify diting prod` 退出码 0。② Down 保留盘 — `make down` 后 `prod.disk_id` 存在或控制台可见数据盘。③ 再次 Up 数据继承 — 再次 deploy 后 verify 或 psql 确认数据与 Down 前一致。

<a id="l4-stage2-06-phase2"></a>
## 生产数据环境：后半部分（采集生产级 + 落库 + 数据继承）

顺序：**① 前置检查 → ② 采集逻辑与数量质量（先写）→ ③ 环境自检与按需部署 → ④ 运行与验证 → ⑤ 回收**。运行表：步骤 1 推送镜像；步骤 2～3 验证集群存在与 prod.conn/DSN（不足则 deploy）；步骤 4～5 cp .env 与连通验证；步骤 6～7 等待 Job 完成并做数据数量与质量验证；步骤 8～10 释放后重新部署并再次验证全量（通过即采集与数据继承合格）；步骤 11 回收。首个依赖环境的操作前完成 [前置检查](#l4-stage2-06-phase2-precheck)；任一步失败先分析修复重试，同一问题重试两次以上仍失败再 [回收](#l4-stage2-06-phase2-teardown)；正常结束须执行步骤 11。

<a id="l4-stage2-06-phase2-precheck"></a>
### ① 前置检查（五项）

首个依赖环境的操作前确认；不满足按「不满足时」处理。**环境是否就绪以 C5（本机 `make verify diting prod` 通过）为准**，不得仅凭 prod.conn 存在即跳过部署。

| 条件 | 检查方法 | 不满足时 |
|------|----------|----------|
| **C1 prod.conn 存在** | diting-infra 有 `prod.conn` 且含非空 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` | `make deploy diting prod` |
| **C2 表已初始化** | 由 Bitnami/diting-stack 部署与 Hook 顺序保证（storage init weight -5，ingest weight 5） | 无需处理 |
| **C3 采集镜像 K3s 可用** | diting-core 执行 `make push-images`；config 中 `stack.ingest.image` 指向 ACR，集群需 imagePullSecrets | `make push-images`；未配置凭证则在 diting-infra 配置 ACR imagePullSecrets |
| **C4 集群出网** | Pod 能访问外网（AkShare 等） | 查节点网络/安全组/代理 |
| **C5 本机可连 prod** | diting-core 内 `cp prod.conn .env` 后 `make verify diting prod` 退出码 0 | 部署时会 `make apply-security-group-rules`；仍超时见 diting-infra docs/安全组规则说明.md |

<a id="l4-stage2-06-phase2-run"></a>
### ④ 运行步骤与验证（数据继承 + 全量采集与质量验证）

真实行情、禁止 mock。采集由**单次部署触发的 Job** 完成：Job 内按 [每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental) 判断全量或增量并执行，不区分两次采集。流程：先确认集群与 prod.conn → 连通与 cp .env → 等待 Job 完成 → 数据数量与质量验证 → 释放后重新部署 → 再次验证全量（通过则采集合格、数据继承合格）→ 回收。

失败时先分析修复重试；同一问题重试两次以上仍失败再执行 [回收环境](#l4-stage2-06-phase2-teardown)。按序执行。

| 步序 | 工作目录 | 命令/操作（概要） |
|------|----------|-------------------|
| **1** | diting-core | `make push-images` |
| **2** | diting-core / diting-infra | `cp $INFRA/prod.conn $CORE/.env` → `make verify diting prod`；失败则 `make deploy diting prod` |
| **3** | diting-infra | 缺 prod.conn/DSN 时：`make deploy diting prod` |
| **4** | diting-core | `cp $INFRA/prod.conn .env` |
| **5** | diting-core | `make verify diting prod` |
| **6** | 本机 | `kubectl get jobs -n default`；`kubectl logs job/diting-ingest -n default`；未部署则先 `make deploy diting prod` |
| **7** | diting-core | `make verify-data-production` |
| **8** | diting-infra | `make down diting prod` |
| **9** | diting-infra | `make deploy diting prod` |
| **10** | diting-core | `cp $INFRA/prod.conn .env`；`make verify diting prod`；`make verify-data-production` |
| **11** | diting-infra | `make down diting prod` |

**按步序可复制命令**（将 `$INFRA`、`$CORE` 换为实际路径；每步为独立代码块，可逐条复制执行）（将 `$INFRA`、`$CORE` 换为实际路径；步骤 2/3 为条件分支，按需执行其一）：


```bash

  cd "$INFRA" && make deploy diting prod

```

```bash

```

```bash
# --- 步骤 5：连通验证 ---

```


<a id="l4-stage2-06-phase2-teardown"></a>
### 回收环境（正常结束或任一步失败时执行）

正常结束：步骤 1～10 验证通过后执行步骤 11（回收）。中途失败：先分析修复重试；同一问题重试两次以上仍无法解决时再执行回收。

| 操作 | 工作目录 | 命令 |
|------|----------|------|
| 回收生产数据环境 | **diting-infra** | `make down diting prod` |

`make down` 仅释放 ECS/EIP，其余在 tfvars 中固定；`prod.disk_id` 保留。完整销毁用 `FULL_DESTROY=1 make down diting prod`。

<a id="l4-stage2-06-phase2-checklist"></a>
### 后半部分可执行清单（复制执行）

在能访问 prod 公网 IP 的机器上执行。**层次**：① 步骤 1 推送镜像 → ② 步骤 2～3 集群与 prod.conn（不足则 deploy）→ ③ 步骤 4～5 连接信息与连通 → ④ 步骤 6～7 Job 完成与数据验证 → ⑤ 步骤 8～10 释放→再部署→再验证全量 → ⑥ 步骤 11 回收。将 `INFRA`、`CORE` 替换为实际路径后整段复制执行或按块分步执行。

```bash
export INFRA=/path/to/diting-infra
export CORE=/path/to/diting-core

# ========== ① 步骤 1：推送采集镜像 ==========
cd "$CORE" && make push-images

# ========== ② 步骤 2～3：验证集群与 prod.conn，不足则 deploy ==========
if [ -f "$INFRA/prod.conn" ]; then
  cp -f "$INFRA/prod.conn" "$CORE/.env"
  cd "$CORE" && make verify diting prod || { cd "$INFRA" && make deploy diting prod; }
else
  cd "$INFRA" && make deploy diting prod
fi
if ! [ -f "$INFRA/prod.conn" ] || ! grep -q 'TIMESCALE_DSN=' "$INFRA/prod.conn" 2>/dev/null; then
  cd "$INFRA" && make deploy diting prod
fi

# ========== ③ 步骤 4～5：连接信息与连通验证 ==========
cp -f "$INFRA/prod.conn" "$CORE/.env"
cd "$CORE" && make verify diting prod

# ========== ④ 步骤 6～7：等待 Job 完成 + 数据数量与质量验证 ==========
# 步骤 6：确认 Job 完成（可手动执行后等待）
# kubectl get jobs -n default
# kubectl logs job/diting-ingest -n default --tail=100
cd "$CORE" && make verify-data-production

# ========== ⑤ 步骤 8～10：释放 → 再部署 → 再验证全量 ==========
cd "$INFRA" && make down diting prod
cd "$INFRA" && make deploy diting prod
cp -f "$INFRA/prod.conn" "$CORE/.env"
cd "$CORE" && make verify diting prod && make verify-data-production

# ========== ⑥ 步骤 11：回收环境 ==========
cd "$INFRA" && make down diting prod
```

**说明**：`prod.conn` 在 diting-infra 根目录。步骤 6 需在 Job 完成后再执行步骤 7；若需 KUBECONFIG：`eval $(make -C "$INFRA" print-kubeconfig)`。

### ② 部署环境与依赖组件（参考）

与前半部分共用 Up/Down；自检不通过时经 [环境自检](#l4-stage2-06-phase2-env-check) 的 E3 部署。部署：`make deploy/down diting prod`（down 仅释放 ECS/EIP，保留数据盘；`prod.conn` 在 infra 根目录）。配置：`config/diting-prod.yaml`、`config/terraform-diting-prod.tfvars`。落库：`stack.ingest.enabled=true` 时 Hook 创建 Job 执行 `make ingest-deploy`，见 [每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental)。

#### 静态存储与数据继承（实施依据）

生产环境数据继承采用**混合存储策略**：需继承的库（TimescaleDB L1、PostgreSQL L2、Redis）使用**固定路径静态 PV/PVC**，Down 后数据保留在数据盘 hostPath，再次 Up 时重新创建同名 PV/PVC 并挂载同盘，实现继承。实现以 **diting-infra** 为准。

| 维度 | 约定与实现位置 |
|------|----------------|
| **架构** | 独立数据盘（如 `/mnt/titan-data`）下固定路径：`postgres/timescaledb`、`postgres/postgresql-l2`、`postgres/redis`；静态 PV `ReclaimPolicy: Retain`，PVC 名称固定以便数据库 Chart 通过 `existingClaim` 引用。 |
| **实现** | diting-infra `charts/diting-stack` 负责静态 PV/PVC 与 init Job；数据库用 Bitnami Chart + `existingClaim` 引用；配置见 `config/diting-prod.yaml` 的 `stack.storage`、`stack.databases`。 |
| **验证** | [运行步骤](#l4-stage2-06-phase2-run) 步骤 7～10：数据验证 → Down → 再 Up → 再 verify（数据继承）。 |
| **故障排查** | PV Released：`kubectl patch pv <pv-name> -p '{"spec":{"claimRef":null}}'`。权限：查 hostPath 或重跑 init-job。 |

更多背景与动态 PVC 无法继承的原因分析见 [StorageClass 配置与数据继承验证](StorageClass配置与数据继承验证.md)。diting-infra 侧说明见其 `charts/diting-stack/README.md` 与 `config/README.md`。
