## 数据采集在 K3s 的部署方式与镜像

| 问题 | 说明 |
|------|------|
| **如何部署在 K3s？** | **当前实现**：数据采集在远程 K3s 以 **Job** 实现，Job 由 **diting-infra** 的 Helm Chart **`charts/diting-stack`** 内 **`templates/ingest/`** 定义（Secret `diting-ingest-conn` + Job `diting-ingest`）。连接信息由 values 生成 Secret，**集群内使用 Service DNS**（如 `timescaledb-primary.default.svc.cluster.local`、`postgresql-l2-primary.default.svc.cluster.local`、`redis-master.default.svc.cluster.local`）。`make deploy diting prod` 在栈就绪后会 `helm upgrade` diting-stack，若 **`stack.ingest.enabled=true`** 则 post-install/post-upgrade Hook 会创建并运行该 Job。详见 **diting-infra** 仓库内 `docs/ingest-cloud-native.md`。 |
| **有相关镜像吗？** | **有**。diting-core 的 **Dockerfile.ingest** 构建出采集镜像（默认 tag：`diting-ingest:test`；`make build-images` 一键构建）。镜像内已安装 Python 3.11、akshare 1.17.1、psycopg2、redis、pandas 等，与本地 `make deps-ingest` 的依赖策略一致。 |
| **镜像环境能否满足要求？** | **能**。Dockerfile.ingest 使用 **Python 3.11**（≥3.8）、先 `pip install --no-deps akshare==1.17.1` 再 `pip install -r requirements-ingest-core.txt`，避免 akshare 1.18+ 依赖的 curl_cffi≥0.13 在部分构建环境不可用；镜像内具备 akshare、psycopg2、redis、pandas、py-mini-racer 等，满足 [前置条件](#l4-stage2-06-phase2-run) 中的环境要求。 |
| **能顺利完成数据采集任务吗？** | **能**。Job 默认执行 **`make ingest-deploy`**（见下节「每次部署检查全量/增量」）；运行时通过 Secret 注入 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 后即可连集群内 L1/L2/Redis。集群内 Pod 需能访问外网行情源（如 AkShare 所用接口）方可拉取真实数据。 |

<a id="l4-stage2-06-deploy-check-full-incremental"></a>
### 每次部署检查全量/增量（当前策略）

**触发时机**：每次部署（Helm post-install/post-upgrade）触发一次采集 Job。

**决策逻辑**（diting-core **`scripts/run_ingest_deploy.py`**，Make target **`make ingest-deploy`**）：
- 连接 L1，检查 **`ohlcv`** 表是否存在、是否有数据、**`MAX(datetime)`** 是否在阈值内。
- **无数据或超过 N 天未更新**（默认 7 天，可配 `INGEST_DEPLOY_FULL_DAYS_THRESHOLD` / values **`ingest.fullDaysThreshold`**）→ 执行**全量**（`run_ingest_production.py`）。
- **有数据且未过期** → 执行**增量**（当前用 `run_ingest_test.py` 做轻量刷新；后续可替换为真实增量逻辑）。

无需单独 CronJob 或常驻 Pod；部署即检查并补齐数据。与 06_dna、06_ 设计中的「C3 落库触发方式」一致，见 [06_ 设计#后半部分](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-prod-data-env)。

**实现说明**（diting-infra）：① **Secret**：由 Chart 模板从 values（`ingest.timescaleHost`、`ingest.postgresL2Host`、`ingest.redisHost` 及账号密码）生成 `diting-ingest-conn`，供 Job 的 `envFrom` 引用。② **存储与 init**：diting-stack 的 storage init Job（hook weight `-5`）先于 ingest Job（weight `5`）执行。③ **本机/CI 验证**：仍可使用 **REPO_I_ROOT** + 本机 `make ingest-test` 或 `make ingest-deploy`（将 `prod.conn` 复制为 diting-core 的 `.env`）做验证；生产环境以集群内 Job 为准。

<a id="l4-stage2-06-phase1"></a>
## 生产数据环境：前半部分（环境与数据继承）

**目标**：在 diting-infra 执行 `make deploy diting prod` 后，得到竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis，NodePort+公网 IP 暴露，连接信息写入 `prod.conn`；执行 `make down diting prod` **回收资源且磁盘数据保留**；再次执行 Up 时新起竞价实例挂载同一块数据盘，K3s 与数据库继续使用该盘上的数据，实现数据继承。部署哪些服务（PG/Redis 等）及 K3s 存储、PG/Redis 存储配置由 **config/diting-prod.yaml** 控制。**依赖组件暴露端口**在 `diting-prod.yaml` 顶层 **`ports`** 中固定（TimescaleDB 30432、PostgreSQL L2 30433、Redis 30379、SSH 22、K3s API 6443），与 `prod.conn`、变量文档一致，见 diting-infra **config/README.md**「端口与连接变量」小节。设计细节与 C1/C2 约定见 [06_生产级数据要求_设计#生产数据环境与两段实现](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-prod-data-env)。

**前半部分·实际验证流程**（部署 → 验证 → 回收且保留磁盘）：

| 阶段 | 工作目录 | 命令 | 说明 |
|------|----------|------|------|
| **部署** | diting-infra | `make deploy diting prod` | 创建竞价实例 + 独立数据盘 + K3s + DB；产出 `prod.conn`、`prod.disk_id` |
| **验证** | diting-core | `cp <diting-infra>/prod.conn .env` 后执行 `make verify diting prod` | 验证 L1/L2 连通及 **Redis 连通与缓存/队列效果性能测试**（见 11_ 规约） |
| **回收（磁盘保留）** | diting-infra | `make down diting prod` | 仅回收 ECS/K3s/EIP，**磁盘数据保留**；`prod.disk_id` 保留供再次 Up 挂载同盘 |

**说明**：prod 使用**独立 NAS**（config/terraform-diting-prod.tfvars 中不设或 `nas_use_existing_access_group = false`），与 dev 隔离。`FULL_DESTROY=1 make down diting prod` 会完整销毁 prod 资源（含 prod NAS）。若 state 中仍残留 dev 共享的 NAS 访问组（diting_nas_group_dev），Make 会在 destroy 前自动从 state 移除，避免 `InvalidAccessGroup.AlreadyAttached`。

**前半可执行步骤（按序执行）**：

| 步骤 | 工作目录 | 命令/操作 | 说明 |
|------|----------|-----------|------|
| 1 | **diting-infra** | `make deploy diting prod` | 创建竞价实例 + 独立数据盘 + K3s + DB；连接信息写入 `prod.conn`，数据盘 ID 写入 `prod.disk_id` |
| 2 | **diting-core** | 将 `diting-infra/prod.conn` 中 DSN 写入 `.env`：`cp /path/to/diting-infra/prod.conn .env`（须含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL`） | 供 `make verify diting prod`（L1/L2/Redis 必验+Redis 性能测试）/ ingest-test 使用 |
| 3 | **diting-core** | `make verify diting prod` | 验证实际实现：L1/L2 连通（前半验证 ①） |
| 4 | **diting-infra** | `make down diting prod` | 回收资源且**磁盘数据保留**；`prod.disk_id` 保留（前半验证 ②） |
| 5 | 云控制台或再次 Up | 可选：在云控制台确认数据盘仍存在；或直接再次 Up | 前半验证 ② 可选确认 |
| 6 | **diting-infra** | `make deploy diting prod` | 再次 Up 时自动读取 `prod.disk_id` 挂载同盘 |
| 7 | **diting-core** | 再次复制 `prod.conn` → `.env` 后执行 `make verify diting prod` 或 psql 查表 | 前半验证 ③：数据继承成立 |

**前半核心验证（三项）**：

| 序号 | 验证项 | 命令/方式 | 期望 |
|------|--------|-----------|------|
| ① | 环境可用、连接正确 | Up 后，在 diting-core 使用 `prod.conn` 中的 DSN 配置 `.env`，执行 `make verify diting prod` | 退出码 0，能连通 L1/L2 |
| ② | Down 保留盘 | 执行 `make down diting prod` 后，在云控制台或约定方式确认数据盘未删除；或再次 Up 时可挂载同盘 | 数据盘仍存在 |
| ③ | 再次 Up 数据可访问一致 | 再次执行 `make deploy diting prod` 后，通过 psql 或约定查询确认 L1/L2 表数据与 Down 前一致（如行数、样例） | 数据继承成立 |

**前半部分·目标测试步骤**（功能/验收测试，与上表三项一一对应）：

| 测试项 | 目标 | 执行步骤（可复制执行） | 通过标准 |
|--------|------|------------------------|----------|
| **T1 环境与连接** | 验证 Up 后环境可用、DSN 正确 | 1）diting-infra：`make deploy diting prod`<br>2）diting-core：`cp <diting-infra>/prod.conn .env`<br>3）diting-core：`make verify diting prod` | 退出码 0；终端输出含 L1/L2 verify OK |
| **T2 Down 保留盘** | 验证回收资源且磁盘数据保留 | 1）diting-infra：`make down diting prod`<br>2）确认 `prod.disk_id` 文件仍存在且非空；或云控制台查看该盘未删除 | `prod.disk_id` 存在；或控制台可见数据盘 |
| **T3 再次 Up 数据继承** | 验证再次 Up 挂载同盘、数据可访问一致 | 1）diting-infra：`make deploy diting prod`<br>2）diting-core：再次 `cp prod.conn .env` 后 `make verify diting prod`<br>3）可选：psql 连接 L1/L2 查表行数或样例，与 Down 前对比 | `make verify diting prod` 退出码 0；可选：表行数/样例与 Down 前一致 |

<a id="l4-stage2-06-phase2"></a>
## 生产数据环境：后半部分（采集生产级 + 落库 + 数据继承）

后半部分按以下顺序执行：**① 前置检查（五项）→ ② 采集逻辑与数量质量要求（先写）→ ③ 环境自检与按需部署 → ④ 运行与验证（数据继承 + 全量采集与质量）→ ⑤ 回收环境**。**运行步骤表**中**先步骤 1 推送镜像**，**再步骤 2 保证 prod 环境就绪**（缺则部署、有则跳过），**然后步骤 3～12**。

**约定**：在**第一个依赖生产环境的操作**之前，须完成 [前置检查（五项）](#l4-stage2-06-phase2-precheck)；不满足项按表中「不满足时」修复后再继续。**任一步骤失败时**，须先执行 [回收环境](#l4-stage2-06-phase2-teardown)，再结束本流程并排查原因。**正常完成所有运行与验证后**，必须执行回收环境。

### ① 前置检查（五项）

在运行 [环境自检](#l4-stage2-06-phase2-env-check) 或 [运行步骤](#l4-stage2-06-phase2-run) 之前，确认下列五项条件；任一项不满足时按「不满足时」处理后再继续。

| 条件 | 检查方法 | 不满足时 |
|------|----------|----------|
| **C1 生产环境已 Up** | **diting-infra** 存在 `prod.conn`，且含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 非空 | 执行 `make deploy diting prod`（或先配置 tfvars/yaml 再执行） |
| **C2 表已初始化（L1/L2）** | 数据库由 **diting-infra** Makefile 使用 Bitnami Chart 部署，建表由各 Chart 或 diting-stack 约定保证；**diting-stack** 的 ingest Job 在 hook weight 5 执行，晚于存储 init（weight -5） | 无需处理，由部署顺序保证 |
| **C3 采集镜像对 K3s 可用** | 在 **diting-core** 执行 **`make push-images`** 将镜像推送到项目 ACR（见 [镜像仓库（ACR）与本地构建推送](#l4-stage2-06-acr-and-push)）；**config/diting-prod.yaml** 中 `stack.ingest.image` 已指向 ACR 且 tag 为 `latest`，K3s 节点需能拉取该私有仓库（配置 imagePullSecrets 或节点已 docker login） | 先 `make push-images` 推送至 ACR；若集群未配置拉取凭证，需在 diting-infra 为 diting-stack 配置 ACR 的 imagePullSecrets |
| **C4 集群出网** | K3s 内 Pod 能访问外网（AkShare 等行情源） | 检查节点网络/安全组/代理；无外网时无法拉取真实行情 |
| **C5 本机可连 prod** | 在 **diting-core** 将 `prod.conn` 复制为 `.env` 后执行 `make verify diting prod` 退出码 0 | 确保本机与 prod 公网 IP 或 VPC 互通、安全组放行；或在与 ECS 同 VPC 的机器上执行 |

**建议执行顺序**：先完成 C1（必要时 E3 部署）→ C2 由 Chart Hook 自动满足 → **C3 在首次部署前完成**（在 diting-core 执行 `make push-images` 将镜像推送到 ACR，见 [镜像仓库（ACR）与本地构建推送](#l4-stage2-06-acr-and-push)）→ C4/C5 按环境确认。

<a id="l4-stage2-06-phase2-env-check"></a>
### ③ 环境自检与按需部署（首个依赖环境操作之前执行）

在首次依赖生产环境的操作（如复制 `prod.conn`、执行 `make verify diting prod`）之前，先完成 [前置检查（五项）](#l4-stage2-06-phase2-precheck)，再按下列顺序**自行判断**是否具备满足需要的基础环境；若不存在则实施部署。

| 步序 | 工作目录 | 判断/操作 | 说明 |
|------|----------|----------|------|
| **E1** | **diting-infra** | 检查 `prod.conn` 是否存在，且内容含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 非空 | 若文件不存在或上述变量缺失/为空 → 基础环境未就绪，进入 E3 |
| **E2** | **diting-core** | 将 `prod.conn` 复制为 `.env`：`cp <diting-infra 路径>/prod.conn .env`；执行 `make verify diting prod` | 若命令**失败**（退出码非 0，如连接超时、连接拒绝）→ 基础环境不满足，进入 E3；若**成功** → 环境已就绪，进入 [运行与验证步骤](#l4-stage2-06-phase2-run) 步骤 3 |
| **E3** | **diting-infra** | 实施部署：`make deploy diting prod`（即 [运行步骤](#l4-stage2-06-phase2-run) 步骤 2）。Up 后若 **`stack.ingest.enabled=true`**（在 config/diting-prod.yaml），**diting-stack** 的 post-install/post-upgrade Hook 会创建采集 Job 并执行 **`make ingest-deploy`**（检查 L1 后全量或增量）；连接信息由 Chart 生成 Secret `diting-ingest-conn`，集群内 Service DNS。采集镜像须已构建且 K3s 可拉取（见 [#数据采集在 K3s 的部署方式与镜像](#l4-stage2-06-k3s-and-image)）。 | 若部署**失败** → 若有部分资源已创建，执行 [回收环境](#l4-stage2-06-phase2-teardown)，结束流程并排查；若部署**成功** → 回到 E2 再次执行 verify，通过后进入 [运行与验证步骤](#l4-stage2-06-phase2-run) 步骤 3 |

<a id="l4-stage2-06-phase2-run"></a>
### ④ 运行步骤与验证（数据继承 + 全量采集与质量验证）

**执行顺序**：本流程一律使用**真实行情数据**；**采集均在远程 K3s 以 Job 实现**，Job 执行 **`make ingest-deploy`**（[每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental)）：无数据或过期则全量，否则增量。**禁止**设置 DITING_INGEST_MOCK。前置检查 C2～C5 见 [① 前置检查](#l4-stage2-06-phase2-precheck)。**先步骤 1 推送镜像，再步骤 2 保证 prod 环境就绪（缺则部署、有则跳过），然后步骤 3～12**。

**步骤失败时的处理**（与系统规则 §8.4f 一致，适用于 04_阶段规划与实践 下所有实践阶段的所有实践文档）：任一步执行失败时，**先分析问题、修复后重试**，不直接回收环境退出。仅当**同一问题**在**修复并重试两次以上**仍无法解决时，才执行 [回收环境](#l4-stage2-06-phase2-teardown) 并结束流程。

以下步骤须**按序执行**。

| 步序 | 工作目录 | 命令/操作 | 失败时 |
|------|----------|-----------|--------|
| **1** | **diting-core** | **`make push-images`**（满足 C3：将采集镜像构建并推送到项目 ACR；ACR 凭证由 `DITING_ACR_PASSWORD` 或 `docker login` 提供，见 [镜像仓库（ACR）与本地构建推送](#l4-stage2-06-acr-and-push)）。部署时 K3s 拉取该镜像。 | 分析原因（如 ACR 凭证、网络）并修复后重试；同一问题修复重试超过 2 次仍失败则 [回收环境](#l4-stage2-06-phase2-teardown) 并退出 |
| **2** | **diting-infra** | **保证 prod 环境就绪（缺则部署、有则跳过）**：若 `prod.conn` 不存在或未含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 非空，则执行 **`make deploy diting prod`**（创建竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis，产出 `prod.conn`、`prod.disk_id`；若 `stack.ingest.enabled=true` 会顺带创建采集 Job 并执行 `make ingest-deploy`）。若已存在且有效则**跳过**本步。 | 分析原因（如 tfvars、云凭证）并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **3** | **diting-core** | `cp <diting-infra 路径>/prod.conn .env` | 分析原因（路径、权限）并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **4** | **diting-core** | `make verify diting prod` | 分析原因（连接、DSN、网络）并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **5** | **diting-infra** | **部署时采集（K3s Job）**：步骤 2 或 8 的 `make deploy diting prod` 在 `stack.ingest.enabled=true` 时会由 diting-stack Hook 创建 Job 并执行 `make ingest-deploy`（检查后自动全量或增量）。若需单独补跑：在 diting-infra 执行 `helm upgrade diting-stack ... -f config/diting-prod.yaml` 并确保 `ingest.enabled=true`。 | 分析原因（Job 日志、镜像拉取、Hook）并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **6** | **diting-core** | `make verify-data-test`（仅验测试集约 15 标真实数据存在与口径，不要求 5 年深度） | 分析原因（数据、表结构）并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **7** | **diting-infra** | `make down diting prod`（回收环境但**保留数据盘**） | 分析原因并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **8** | **diting-infra** | `make deploy diting prod`（再次 Up，挂载同盘；Up 后若 `stack.ingest.enabled=true` 会再次跑 ingest-deploy Job） | 分析原因并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **9** | **diting-core** | 再次 `cp …/prod.conn .env`；`make verify diting prod`；**数据继承验证**：`make verify-data-test`（验测试集 15 标数据仍可访问）。若 Down 后数据未保留需补采：再次部署或单独触发 ingest Job 后再执行 `make verify-data-test` | 分析原因并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **10** | **diting-infra** / **diting-core** | **全量生产级**：若步骤 2 或 8（部署）触发的 ingest-deploy 已执行全量则无需额外操作；否则在 diting-core（`.env` 已指向 prod）执行 `make ingest-production`。**禁止** DITING_INGEST_MOCK。 | 分析原因并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **11** | **diting-core** | **全量数据的生产级数量质量检测**：`make verify-data-production`（L1 单标≥5 年、标的与 universe 一致等）；**必须在全量采集完成后执行** | 分析原因并修复后重试；同一问题修复重试超过 2 次仍失败则回收环境并退出 |
| **12** | **diting-infra** | **[回收环境]** `make down diting prod` | 正常结束时**必须**执行；见 [回收环境](#l4-stage2-06-phase2-teardown) |

**步骤 1 与 2**：先执行**步骤 1**（push-images）满足 C3，再执行**步骤 2**（prod 环境就绪：无 prod.conn 或 DSN 缺失则部署，否则跳过），然后步骤 3～12。

<a id="l4-stage2-06-phase2-teardown"></a>
### 回收环境（正常结束或任一步失败时执行）

**以下两种情况均须执行回收**：  
- **正常结束**：完成步骤 1～11 后，执行步骤 12（`make down diting prod`）。  
- **中途失败**：步骤 1～11 中任一步失败时，按「步骤失败时的处理」分析问题、修复后重试；**仅当同一问题修复重试两次以上仍无法解决**时，执行回收再结束流程。

| 操作 | 工作目录 | 命令 |
|------|----------|------|
| 回收生产数据环境 | **diting-infra** | `make down diting prod` |

说明：`make down diting prod` 仅回收 ECS/K3s/EIP，**数据盘保留**；`prod.disk_id` 保留供再次 Up 挂载同盘。完整销毁含 prod NAS 时使用 `FULL_DESTROY=1 make down diting prod`。

<a id="l4-stage2-06-phase2-checklist"></a>
### 后半部分可执行清单（复制执行）

在**能访问 prod 公网 IP** 的机器上（或与 ECS 同 VPC/已配置安全组放行），将 `INFRA`、`CORE` 替换为实际路径后**按序执行**下列命令。**先步骤 1 推送镜像，再步骤 2 保证 prod 环境就绪（缺则部署、有则跳过），然后步骤 3～12**；无额外「需人工另行完成」项。

```bash
export INFRA=/path/to/diting-infra
export CORE=/path/to/diting-core

# ---------- 步骤 1：采集镜像推送到 ACR（满足 C3）----------
cd "$CORE" && make push-images

# ---------- 步骤 2：保证 prod 环境就绪（缺则部署、有则跳过）----------
[ -f "$INFRA/prod.conn" ] && grep -q 'TIMESCALE_DSN=' "$INFRA/prod.conn" || { cd "$INFRA" && make deploy diting prod; }

# ---------- 步骤 3-4：连接与连通验证 ----------
cp -f "$INFRA/prod.conn" "$CORE/.env"
cd "$CORE" && make verify diting prod

# ---------- 步骤 5-6：部署时采集（Up 时已跑 ingest-deploy Job）+ 测试集验证 ----------
cd "$CORE" && make verify-data-test

# ---------- 步骤 7-8：Down 保留盘 → 再次 Up ----------
cd "$INFRA" && make down diting prod
cd "$INFRA" && make deploy diting prod

# ---------- 步骤 9：数据继承验证 ----------
cp -f "$INFRA/prod.conn" "$CORE/.env"
cd "$CORE" && make verify diting prod && make verify-data-test

# ---------- 步骤 10-11：全量生产级（若未跑全量可先 cd "$CORE" && make ingest-production）→ 再验证 ----------
cd "$CORE" && make verify-data-production

# ---------- 步骤 12：回收环境 ----------
cd "$INFRA" && make down diting prod
```

**说明**：**步骤 1** 完成镜像推送；**步骤 2** 保证 prod 环境就绪（无 `prod.conn` 或 DSN 缺失则执行 `make deploy diting prod`，否则跳过）。config 中 **`stack.ingest.enabled=true`** 且 **`stack.ingest.image`** 指向 ACR 的 `latest` 时，部署会拉取最新镜像并由 Hook 创建采集 Job。本流程**一律使用真实行情数据**，**采集在 K3s 内以 Job 实现**，Job 执行 **`make ingest-deploy`**（[每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental)）。**禁止** DITING_INGEST_MOCK。

### ② 部署环境与依赖组件（参考）

生产数据环境与前半部分共用同一套 Up/Down 流程，依赖以下组件与配置。环境自检不通过时，通过 [环境自检与按需部署](#l4-stage2-06-phase2-env-check) 中的 E3 实施部署。

| 依赖项 | 说明 |
|--------|------|
| **部署流程** | **diting-infra**：`make deploy diting prod`（竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis）；`make down diting prod` 仅回收 ECS/K3s/EIP，**保留数据盘**。连接信息输出至 `prod.conn`。 |
| **配置** | **diting-infra** 的 `config/diting-prod.yaml`（部署哪些服务、存储与数据库参数）、`config/terraform-diting-prod.tfvars`（实例与数据盘）。数据继承由**静态 PV/PVC** 实现，配置与 Chart 见下方「静态存储与数据继承」。 |
| **落库触发（C3）** | Up 就绪后，**在远程 K3s 以 Job 实现**：**diting-stack** 在 `stack.ingest.enabled=true` 时通过 post-install/post-upgrade Hook 创建 Job，执行 **`make ingest-deploy`**（检查 L1 后自动全量或增量）；连接信息由 Chart 生成 Secret `diting-ingest-conn`，集群内 Service DNS。镜像由 `stack.ingest.image` 指定。详见 [每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental)。 |

#### 静态存储与数据继承（实施依据）

生产环境数据继承采用**混合存储策略**：需继承的库（TimescaleDB L1、PostgreSQL L2、Redis）使用**固定路径静态 PV/PVC**，Down 后数据保留在数据盘 hostPath，再次 Up 时重新创建同名 PV/PVC 并挂载同盘，实现继承。实现以 **diting-infra** 为准。

| 维度 | 约定与实现位置 |
|------|----------------|
| **架构** | 独立数据盘（如 `/mnt/titan-data`）下固定路径：`postgres/timescaledb`、`postgres/postgresql-l2`、`postgres/redis`；静态 PV `ReclaimPolicy: Retain`，PVC 名称固定以便数据库 Chart 通过 `existingClaim` 引用。 |
| **实现** | **diting-infra** 仓库内 Helm Chart **`charts/diting-stack`** 仅负责静态 PV/PVC 与存储目录初始化 Job；数据库由 Makefile 使用 Bitnami Chart 部署，并通过 `primary.persistence.existingClaim` / `master.persistence.existingClaim` 引用上述 PVC。配置由 `config/diting-prod.yaml` 的 `stack.storage`、`stack.databases` 驱动。 |
| **验证** | 见 [运行步骤与验证](#l4-stage2-06-phase2-run) 步骤 7～9：先写数据 → Down → 再 Up → 再 verify。 |
| **故障排查** | **PV 状态 Released**：`kubectl patch pv <pv-name> -p '{"spec":{"claimRef":null}}'` 后重新绑定。**权限问题**：Pod 无法写盘时，检查 hostPath 目录权限或重新执行 diting-stack 的 init-job（Helm Hook）。 |

更多背景与动态 PVC 无法继承的原因分析见 [StorageClass 配置与数据继承验证](StorageClass配置与数据继承验证.md)。diting-infra 侧说明见其 `charts/diting-stack/README.md` 与 `config/README.md`。

| 序号 | 验证项 | 命令/方式 | 期望 |
|------|--------|-----------|------|

## 验证路线说明（目标功能）

## 本步骤失败时

- **回退目标**：回退到 Stage2-05 准出状态。
- **生产数据环境后半部分**（与系统规则 §8.4f 一致：04_阶段规划与实践 下所有实践文档均遵循「先修复重试，同一问题两次以上仍无法解决则回收退出」）：任一步失败时**先分析问题、修复后重试**，不直接回收退出。仅当**同一问题**在**修复并重试两次以上**仍无法解决时，才执行 [回收环境](#l4-stage2-06-phase2-teardown)（`make down diting prod`）并结束流程。
- **重试**：规约/文档类修正后重试，建议上限 3 次；与「同一问题修复重试超过 2 次仍失败则回收退出」一致。
- **具体回退操作**：见 [03_ 工作流详细规划 八、失败与回退策略](../../03_原子目标与规约/开发与交付/03_项目全功能开发测试实践工作流详细规划.md)。
