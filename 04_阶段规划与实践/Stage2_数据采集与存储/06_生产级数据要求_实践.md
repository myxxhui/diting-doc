## 数据采集在 K3s 的部署方式与镜像

| 问题 | 说明 |
|------|------|
| **如何部署在 K3s？** | 远程 K3s 以 **Job** 实现：**diting-infra** `charts/diting-stack` 内 `templates/ingest/` 定义 Secret `diting-ingest-conn` + Job（名称带 revision，如 `diting-ingest-5`，非 Hook 以免 Job 失败导致部署报错）；连接信息由 values 生成，集群内用 Service DNS。**云原生依赖就绪**：Job 的 Pod 带 **init 容器**（busybox），在 TimescaleDB / PostgreSQL L2 / Redis 端口可达前循环等待；主容器在 init 通过后才执行 `make ingest-deploy`。因此 stack 与 DB 可同一次 `make deploy` 顺序部署，无需两阶段 Helm。详见 diting-infra `docs/ingest-cloud-native.md`。 |
| **镜像与环境？** | diting-core **Dockerfile.ingest** 构建采集镜像（`diting-ingest:test`，`make build-images`）；Python 3.11、akshare 1.17.1、psycopg2、redis、pandas 等，与 `make deps-ingest` 一致。Job 执行 **`make ingest-deploy`**（见下节），Secret 注入 DSN 后连 L1/L2/Redis；Pod 需能访问外网行情源。 |

### 每次部署检查全量/增量（当前策略）

**触发时机**：每次部署（Helm post-install/post-upgrade）触发一次采集 Job。

**决策逻辑**（diting-core `scripts/run_ingest_deploy.py`，`make ingest-deploy`）：连 L1 查 ohlcv 及 `MAX(datetime)`；无数据或超过 N 天（默认 7 天，可配 `INGEST_DEPLOY_FULL_DAYS_THRESHOLD` / values `ingest.fullDaysThreshold`）→ 全量（`run_ingest_production.py`）；有数据且未过期 → 增量。部署即检查并补齐，与 06_dna/06_ 设计一致。diting-infra：Chart 从 values 生成 Secret `diting-ingest-conn`；storage init（weight -5）先于 ingest Job（weight 5）。

## 生产数据环境：前半部分（环境与数据继承）

**目标**：`make deploy diting prod` 得到竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis，连接信息写入 `prod.conn`；`make down diting prod` 仅释放 ECS/EIP、**磁盘数据保留**；再次 Up 挂载同盘实现数据继承。服务与端口由 **config/diting-prod.yaml** 控制，见 diting-infra config/README.md。设计见 [06_设计#生产数据环境与两段实现](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-prod-data-env)。prod 用独立 NAS 与 dev 隔离；完整销毁用 `FULL_DESTROY=1 make down diting prod`。

**前半可执行步骤（按序）**：

| 步骤 | 工作目录 | 命令/操作 |
|------|----------|-----------|
| 1 | diting-infra | 见下方代码块 |
| 2 | diting-core | 见下方代码块 |
| 3 | diting-core | 见下方代码块 |
| 4 | diting-infra | 见下方代码块 |
| 5 | diting-infra | 见下方代码块 |
| 6 | diting-core | 见下方代码块 |

```bash
# 前半步骤 1：部署
cd /path/to/diting-infra && make deploy diting prod

# 前半步骤 2：复制连接信息
cp /path/to/diting-infra/prod.conn /path/to/diting-core/.env

# 前半步骤 3：连通验证
cd /path/to/diting-core && make verify diting prod

# 前半步骤 4：释放（保留盘）
cd /path/to/diting-infra && make down diting prod

# 前半步骤 5：再次部署
cd /path/to/diting-infra && make deploy diting prod

# 前半步骤 6：再次验证
cp /path/to/diting-infra/prod.conn /path/to/diting-core/.env
cd /path/to/diting-core && make verify diting prod
```

**前半核心验证（三项）**：① 环境与连接 — Up 后 `cp prod.conn .env` 且 `make verify diting prod` 退出码 0。② Down 保留盘 — `make down` 后 `prod.disk_id` 存在或控制台可见数据盘。③ 再次 Up 数据继承 — 再次 deploy 后 verify 或 psql 确认数据与 Down 前一致。

<a id="l4-stage2-06-phase2"></a>
## 生产数据环境：后半部分（采集生产级 + 落库 + 数据继承）

**顺序**：先 **`make deploy diting prod`**（部署会产出 `prod.conn`）→ 再设置 .env（`cp prod.conn .env`）→ 再执行验证。任一步失败先分析修复重试，同一问题重试两次以上仍失败再 [回收](#l4-stage2-06-phase2-teardown)。

**采集触发**：初次部署时，若 `config/diting-prod.yaml` 中 `stack.ingest.enabled=true`，`make deploy diting prod` 会一次完成 stack + 数据库部署；**diting-ingest** Job 作为普通资源创建（不作为 Hook，避免 Job 失败导致整次部署报错），名称带 release revision（如 `diting-ingest-5`）。Pod 的 **init 容器**会等待 TimescaleDB/PostgreSQL L2/Redis 端口就绪再退出，主容器随后执行 `make ingest-deploy`（全量/增量由 [每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental) 判断）。**若 Job 状态为 Failed / BackoffLimitExceeded**：`kubectl get jobs -n default` 找到最新 `diting-ingest-<revision>`，执行 `kubectl logs job/diting-ingest-<revision> -n default -c ingest` 查看主容器日志（常见为镜像拉取失败、Python/采集脚本报错）。**若暂无日志**（Pod 已被 TTL 回收）：(1) 下次部署后立即在另一终端执行 `kubectl get pods -n default -w`，一旦出现 `diting-ingest-*` Pod 即执行 `kubectl logs -f pod/<该 Pod 名> -n default -c ingest` 抓取输出；(2) 或运行一次性 Debug Pod（同镜像、同 Secret、同 command）直接看报错，见下「无日志时排查」。**后续**也可在本机随时触发：在 diting-core 内执行 `cp $INFRA/prod.conn .env && make ingest-deploy`。

**无日志时排查**（Job Failed 且 `kubectl logs job/diting-ingest-<revision>` 无输出时）：在已配置 KUBECONFIG 的本机执行下面命令，会启动一个临时 Pod、注入与 Job 相同的 Secret 与命令，终端直接输出报错（镜像、Secret 名与 config 一致时替换 `<REVISION>` 为当前任意 revision 或省略后缀即可）：

```bash
kubectl run diting-ingest-debug --rm -it --restart=Never -n default \
  --image=crpi-7vifw4ok9jkcxr60.cn-hongkong.personal.cr.aliyuncs.com/titan-core/diting-ingest:latest \
  --overrides='{"spec":{"imagePullSecrets":[{"name":"acr-titan"}],"containers":[{"name":"ingest","image":"crpi-7vifw4ok9jkcxr60.cn-hongkong.personal.cr.aliyuncs.com/titan-core/diting-ingest:latest","command":["make","ingest-deploy"],"envFrom":[{"secretRef":{"name":"diting-ingest-conn"}}],"env":[{"name":"INGEST_DEPLOY_FULL_DAYS_THRESHOLD","value":"7"}]}]}}'
```

退出后可根据终端报错修复（如缺依赖、连库失败、脚本异常等）。Chart 已将该 Job 的 `ttlSecondsAfterFinished` 调为 86400，失败后 Job/Pod 会保留更久以便补查日志。

### 部署环境与依赖组件（参考）

与前半部分共用 Up/Down。部署：`make deploy diting prod`（产出 `prod.conn`；down 仅释放 ECS/EIP，保留数据盘）。配置：`config/diting-prod.yaml`、`config/terraform-diting-prod.tfvars`。落库：`stack.ingest.enabled=true` 时部署完成后 Hook 创建 Job 执行 `make ingest-deploy`，见 [每次部署检查全量/增量](#l4-stage2-06-deploy-check-full-incremental)。

#### 静态存储与数据继承（实施依据）

生产环境数据继承采用**混合存储策略**：需继承的库（TimescaleDB L1、PostgreSQL L2、Redis）使用**固定路径静态 PV/PVC**，Down 后数据保留在数据盘 hostPath，再次 Up 时重新创建同名 PV/PVC 并挂载同盘，实现继承。实现以 **diting-infra** 为准。

| 维度 | 约定与实现位置 |
|------|----------------|
| **架构** | 独立数据盘（如 `/mnt/titan-data`）下固定路径：`postgres/timescaledb`、`postgres/postgresql-l2`、`postgres/redis`；静态 PV `ReclaimPolicy: Retain`，PVC 名称固定以便数据库 Chart 通过 `existingClaim` 引用。 |
| **实现** | diting-infra `charts/diting-stack` 负责静态 PV/PVC 与 init Job；数据库用 Bitnami Chart + `existingClaim` 引用；配置见 `config/diting-prod.yaml` 的 `stack.storage`、`stack.databases`。 |
| **验证** | [运行步骤](#l4-stage2-06-phase2-run) 步骤 7（可选）：Down → 再 Deploy → 再 verify（数据继承）。 |
| **故障排查** | PV Released：`kubectl patch pv <pv-name> -p '{"spec":{"claimRef":null}}'`。权限：查 hostPath 或重跑 init-job。 |

更多背景与动态 PVC 无法继承的原因分析见 [StorageClass 配置与数据继承验证](StorageClass配置与数据继承验证.md)。diting-infra 侧说明见其 `charts/diting-stack/README.md` 与 `config/README.md`。

### 采集 Job 网络类报错（RemoteDisconnected / Connection aborted）

若日志出现 **`RemoteDisconnected`**、**`Connection aborted`**、**`Remote end closed connection without response`**，多为 **Pod/节点或本机访问东方财富接口异常**：节点未出网、安全组未放行出站、东方财富限流或对当前 IP 断连。**处理**：① 确认 K3s 节点（ECS）能访问外网（如从节点或 Pod 内 `curl -I https://quote.eastmoney.com`）；② 检查安全组/路由是否有出站限制；③ 代码已加重试（5 次、退避 5s～25s），可先重跑 Job 看是否偶发；④ 若集群长期无法稳定访问行情源，可改为在本机执行 `cp prod.conn .env && make ingest-deploy`；⑤ **东方财富持续断连时改用 Baostock**：在 `.env` 中设置 **`INGEST_OHLCV_SOURCE=baostock`**，并在 diting-core 执行 **`make deps-ingest`**（会安装 baostock），再执行 `make ingest-production` 或 `make ingest-production-incremental`。Baostock 为免费 A 股日线接口，通常更稳定，适合本地或受限网络环境。

### 标间延迟与全 A 股耗时

当前默认**标与标之间**间隔 2 秒（`INGEST_OHLCV_DELAY_BETWEEN_SYMBOLS=2`），用于减轻东方财富限流。**影响**：15 标约 +28 秒；若全 A 股约 5000+ 标，仅间隔约 **2.8 小时**（5000×2s），再加单标请求与写入时间，整轮可能 3～4 小时以上。若要缩短全市场采集时间，可在环境变量中设 **`INGEST_OHLCV_DELAY_BETWEEN_SYMBOLS=0`** 或 **`0.5`**（Job 的 env 或 values 中注入），需自担限流/断连风险；当前生产数据环境仍使用 **15 标**（`REAL_TEST_SYMBOLS_15`），不跑全 A 股。

### 更好方案（全 A 股 / 大批量）

东方财富/akshare 无「一次请求多标的日线」的批量接口，只能逐只请求。在保证限流可控前提下，可采用：

| 方案 | 说明 | 预期耗时（5000 标量级） |
|------|------|-------------------------|
| **可控并发 + 全局限速** | 设置 **`INGEST_OHLCV_CONCURRENT=3`**（或 5）、**`INGEST_OHLCV_RATE_PER_SEC=1.5`**：多线程并发拉取，但全局不超过 1.5 次/秒，既缩短总时长又降低被限流概率。 | 约 **55 分钟**（5000÷1.5≈3333s） |
| 标间延迟调低 | 仅串行时可将 `INGEST_OHLCV_DELAY_BETWEEN_SYMBOLS=0.5` 或 `0`，自担限流风险。 | 2s 间隔约 2.8h；0.5s 约 42min 仅间隔 |
| 断点续跑（规划） | 全量跑一半失败时，可记录已成功标的，下次只拉未完成部分，避免重复请求。当前未实现，可按需在脚本层维护进度文件或 Redis。 | 单次耗时不变，重跑时只补漏 |
| 数据源选型 | 若需更稳或批量接口，可评估 Tushare、Baostock、万得/Choice 等，部分提供批量或更高 QPS。 | 视接口与配额而定 |

**推荐**：全 A 股时在 Job 或本机环境中设置 **`INGEST_OHLCV_CONCURRENT=3`**、**`INGEST_OHLCV_RATE_PER_SEC=1.5`**（或 1.0 更保守），在 diting-stack values 或 ingest Job 的 env 中注入即可。

### 采集数据与 AB 模块生产级要求对照

设计中的生产级要求（06_ 设计、11_ 规约）与当前采集的对应关系如下。

| 要求项 | 规约约定 | 全量采集（ingest-production） | 当前增量（ingest-test / make ingest-deploy 有数据时） |
|--------|----------|-------------------------------|------------------------------------------------------|
| **OHLCV 历史深度** | 单标日线 ≥5 年 | ✅ 拉取约 5 年（days_back=5×365） | ❌ 仅 15 标、最近 30 天，不维持 5 年深度 |
| **标的覆盖** | 与 get_current_a_share_universe() 一致（全 A 股） | ✅ 先刷新 universe，再按 universe 全量拉取 | ❌ 仅 15 标（REAL_TEST_SYMBOLS_15），非全 A 股 |
| **universe 表** | 全 A 股、updated_at 有效条件 | ✅ 全量刷新约 5500 标 | ✅ 同流程会刷新全量 universe |
| **复权** | 与回测/实盘一致 | ✅ 与 02_/akshare 约定一致 | ✅ 同源 |
| **Module A 输入** | 申万/营收占比等，与 02_ 一致 | ✅ industry_revenue、news 等写入 L2 | ✅ 同流程 |
| **Module B 输入** | L1 OHLCV、板块与 A 同源 | ✅ 全 A 股 + 5 年写入 L1；行业同源 | ❌ 仅 15 标有最新数据，B 全量扫描时其余标无当日/近期数据 |

**结论**：**全量采集**（`make ingest-production`，已含并发与限速）满足设计的生产级要求，可供 Module A/B 生产级功能使用。**当前“增量”**（`make ingest-deploy` 在 L1 有数据且未过期时执行的 ingest-test）仅更新 **15 标、30 天**，**不符合**「标的覆盖 = 全 A 股」与「单标深度维持」的生产级要求，仅适合**验证流水线/数据继承**，不能作为每个交易日后的生产级日终增量。

### 每个交易日结束后的增量采集（生产级）

若要在**每个交易日结束后**做**生产级增量**（全 A 股标的、仅补最近 N 天，保证 AB 模块可用最新全市场数据）：

| 项目 | 说明 |
|------|------|
| **目标** | 标的覆盖 = 全 A 股；仅补最近 1～5 个交易日数据，维持 L1 新鲜度；universe 可同日刷新；行业/新闻按需更新。 |
| **执行速度与时间** | 与全量类似：universe 刷新约 1 次请求；OHLCV 约 5500 标 × 1 次请求/标，用 **INGEST_OHLCV_CONCURRENT=3**、**INGEST_OHLCV_RATE_PER_SEC=1.5** 时约 **55 分钟～1 小时**（仅 OHLCV 部分）。建议**日终收盘后**（如 18:00 后）在本机执行，避免交易时段接口不稳定。 |
| **数据质量** | 满足生产级：标的 = 全 A 股、深度已有 5 年（首轮全量已写入），本次仅追加最近 N 日；`make verify-data-production` 仍可通过（单标 bar 数≥5 年、标的口径与 universe 一致）。 |
| **入口** | 已提供 **`make ingest-production-incremental`**（diting-core）：先刷新 universe，再按全 A 股标的仅补最近 N 天（默认 7 天，可设 `INGEST_PRODUCTION_INCREMENTAL_DAYS=5`）；内部默认启用 CONCURRENT=3、RATE=1.5，耗时约 1 小时。每个交易日结束后在本机执行即可。 |

### 全量全 A 股更新（怎么跑、耗时与限流）

需要**全 A 股、全量更新**（先刷新 universe，再按 universe 拉取每只标的约 5 年日线）时：

| 项目 | 说明 |
|------|------|
| **入口** | **`make ingest-production`**（见 [运行步骤与验证](#l4-stage2-06-phase2-run)）。流程：① 刷新 a_share_universe（约 5500 标）② 从表内读全量标的 ③ 对每只标的拉取约 5 年日线写入 ohlcv ④ 行业/新闻等。或由 `make ingest-deploy` 在 L1 无数据/过期时自动触发。 |
| **推荐本地执行** | 东方财富数据接口对机房/云 IP 易断连，全量建议在**本机**执行：`cp prod.conn .env` 后 **`make ingest-production`**（已默认启用并发+限速）。 |
| **速度与限流** | **`make ingest-production`** 默认 `INGEST_OHLCV_CONCURRENT=3`、`INGEST_OHLCV_RATE_PER_SEC=1.5`，OHLCV 部分约 **1 小时内**完成，全局限速降低被限流/断连概率。可在 `.env` 中覆盖上述变量。 |
| **总耗时粗估** | universe 刷新 + 5500 标 OHLCV（1.5 req/s 约 1h）+ 行业/新闻等，整轮约 **1～1.5 小时**（本地网络正常时）。 |

### 本地加速（尽量不触发限流）

在**本机**执行时，出口 IP 多为家庭/企业网，东方财富对这类 IP 限流通常比机房宽松，可适当提高并发与限速以缩短时间；若出现 `RemoteDisconnected` 再降回保守档。

| 档位 | CONCURRENT | RATE_PER_SEC | 5500 标 OHLCV 粗估 | 用法 |
|------|------------|--------------|---------------------|------|
| **保守**（默认） | 3 | 1.5 | 约 55 min～1 h | `make ingest-production` / `make ingest-production-incremental`（不改 .env） |
| **加速** | 5 | 2.0 | 约 45 min | 在 diting-core 下执行：`INGEST_OHLCV_CONCURRENT=5 INGEST_OHLCV_RATE_PER_SEC=2.0 make ingest-production`（或 `make ingest-production-incremental`）；或把两变量写入 `.env` 后直接 `make ingest-production` |
| **更快**（自担限流风险） | 6 | 2.5 | 约 35 min | 同上，设为 6 与 2.5；若频繁断连则改回 5/2.0 或 3/1.5 |

**建议**：先试**加速**档（5 / 2.0），无报错再视需要试**更快**档；代码允许 CONCURRENT 最大 16、RATE 最大 5.0，但过高易触发限流，本地一般 5～6、2.0～2.5 为宜。**一键加速**：不改 .env 可直接用 **`make ingest-production-fast`**、**`make ingest-production-incremental-fast`**（内部 5/2.0），若出现断连则改回 `make ingest-production` / `make ingest-production-incremental`。

### 写入报错：column "period" of relation "ohlcv" does not exist

若采集拉数成功但写入时报 **`column "period" of relation "ohlcv" does not exist`**，说明当前库里的 **ohlcv 表是按旧版建的**（无 `period` 列），与规约（diting-infra 的 `schemas/sql/01_l1_ohlcv.sql`）不一致。**处理**：用当前库的连接串执行迁移（在 diting-infra 目录下）：  
`psql "<连接串>" -f schemas/sql/01_l1_ohlcv_add_period.sql`  
连接串可从 `prod.conn` 或 `.env` 中的 `TIMESCALE_DSN` 取得。迁移仅在表已存在且缺少 `period` 时生效（旧表为 `trade_date` 时会重命名为 `datetime` 并改主键）；新建表请直接用 `01_l1_ohlcv.sql`。

若写入 **a_share_universe** 时报 **`column "market" of relation "a_share_universe" does not exist`**，说明表为旧结构（仅有 symbol/name/list_date/updated_at）。**处理**：对同一 L1 库执行  
`psql "<连接串>" -f schemas/sql/02_a_share_universe_add_market.sql`  
会补列 `market`、`count`、`source`。
