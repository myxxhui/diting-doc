## 数据采集在 K3s 的部署方式与镜像

| 问题 | 说明 |
|------|------|
| **如何部署在 K3s？** | **默认**：数据采集**一律在远程 K3s 以 Job 实现**，**Job 由 Helm Chart** `diting-infra/charts/ingest` 定义。**Secret** 由 Makefile 的 `make prod-sync-conn-secret` 从 `prod.conn` 同步（kubectl --from-env-file）；**Schema 建表**由 Chart 的 **pre-install Hook** 自动执行；脚本 `prod-apply-ingest-job.sh` 仅调用 Helm。`make deploy diting prod` 在栈就绪后会执行 `deploy-ingest-job`；单独触发：`make deploy-ingest-job [WAIT=wait] [INGEST_TARGET=...]`。见 [11_ 数据采集与输入层规约](../../03_原子目标与规约/_共享规约/11_数据采集与输入层规约.md)。 |
| **有相关镜像吗？** | **有**。diting-core 的 **Dockerfile.ingest** 构建出采集镜像（默认 tag：`diting-ingest:test`；`make build-images` 一键构建）。镜像内已安装 Python 3.11、akshare 1.17.1、psycopg2、redis、pandas 等，与本地 `make deps-ingest` 的依赖策略一致。 |
| **镜像环境能否满足要求？** | **能**。Dockerfile.ingest 使用 **Python 3.11**（≥3.8）、先 `pip install --no-deps akshare==1.17.1` 再 `pip install -r requirements-ingest-core.txt`，避免 akshare 1.18+ 依赖的 curl_cffi≥0.13 在部分构建环境不可用；镜像内具备 akshare、psycopg2、redis、pandas、py-mini-racer 等，满足 [前置条件](#l4-stage2-06-phase2-run) 中的环境要求。 |
| **能顺利完成数据采集任务吗？** | **能**。镜像默认 `CMD ["make", "ingest-test"]`；运行时通过环境变量注入 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 后即可连 K3s 内或 NodePort 暴露的 L1/L2/Redis 执行采集。若需跑 `ingest-test-real` 或 `ingest-production`，在 Job 中覆盖 command 为 `["make", "ingest-test-real"]` 或 `["make", "ingest-production"]` 即可。集群内 Pod 需能访问外网行情源（如 AkShare 所用接口）方可拉取真实数据。 |

**实现说明**（diting-infra）：① **Secret 集群内 DSN**：`make prod-sync-conn-secret` 同步到 K8s 的 Secret 会将 `prod.conn` 中的主机地址替换为集群内 Service（`timescaledb-postgresql.default.svc:5432`、`postgresql-l2.default.svc:5432`、`redis-master.default.svc:6379`），供集群内 Job 连接；`prod.conn` 文件本身仍为公网 NodePort，供本机 `make verify diting prod` 使用。② **Schema-init 顺序**：ConfigMap `diting-schema-init-sql` 使用 pre-install hook 且 weight `-10`，先于 schema-init Job（weight `-5`）创建，避免 Pod 挂载 ConfigMap 时找不到。③ **L1 建表兼容**：`charts/ingest/sql/01_l1_ohlcv.sql` 在存在 TimescaleDB 扩展时才调用 `create_hypertable`，L1 若为普通 PostgreSQL（如 Bitnami）仍可建表使用。

<a id="l4-stage2-06-phase1"></a>
## 生产数据环境：前半部分（环境与数据继承）

**目标**：在 diting-infra 执行 `make deploy diting prod` 后，得到竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis，NodePort+公网 IP 暴露，连接信息写入 `prod.conn`；执行 `make down diting prod` **回收资源且磁盘数据保留**；再次执行 Up 时新起竞价实例挂载同一块数据盘，K3s 与数据库继续使用该盘上的数据，实现数据继承。部署哪些服务（PG/Redis 等）及 K3s 存储、PG/Redis 存储配置由 **config/diting-prod.yaml** 控制。**依赖组件暴露端口**在 `diting-prod.yaml` 顶层 **`ports`** 中固定（TimescaleDB 30432、PostgreSQL L2 30433、Redis 30379、SSH 22、K3s API 6443），与 `prod.conn`、变量文档一致，见 diting-infra **config/README.md**「端口与连接变量」小节。设计细节与 C1/C2 约定见 [06_生产级数据要求_设计#生产数据环境与两段实现](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-prod-data-env)。

**前半部分·实际验证流程**（部署 → 验证 → 回收且保留磁盘）：

| 阶段 | 工作目录 | 命令 | 说明 |
|------|----------|------|------|
| **部署** | diting-infra | `make deploy diting prod` | 创建竞价实例 + 独立数据盘 + K3s + DB；产出 `prod.conn`、`prod.disk_id` |
| **验证** | diting-core | `cp <diting-infra>/prod.conn .env` 后执行 `make verify diting prod` | 验证 L1/L2 连通及 **Redis 连通与缓存/队列效果性能测试**（见 11_ 规约） |
| **回收（磁盘保留）** | diting-infra | `make down diting prod` | 仅回收 ECS/K3s/EIP，**磁盘数据保留**；`prod.disk_id` 保留供再次 Up 挂载同盘 |

**说明**：prod 使用**独立 NAS**（config/terraform-diting-prod.tfvars 中不设或 `nas_use_existing_access_group = false`），与 dev 隔离。`FULL_DESTROY=1 make down diting prod` 会完整销毁 prod 资源（含 prod NAS）。若 state 中仍残留 dev 共享的 NAS 访问组（diting_nas_group_dev），Make 会在 destroy 前自动从 state 移除，避免 `InvalidAccessGroup.AlreadyAttached`。

**前半可执行步骤（按序执行）**：

| 步骤 | 工作目录 | 命令/操作 | 说明 |
|------|----------|-----------|------|
| 1 | **diting-infra** | `make deploy diting prod` | 创建竞价实例 + 独立数据盘 + K3s + DB；连接信息写入 `prod.conn`，数据盘 ID 写入 `prod.disk_id` |
| 2 | **diting-core** | 将 `diting-infra/prod.conn` 中 DSN 写入 `.env`：`cp /path/to/diting-infra/prod.conn .env`（须含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL`） | 供 `make verify diting prod`（L1/L2/Redis 必验+Redis 性能测试）/ ingest-test 使用 |
| 3 | **diting-core** | `make verify diting prod` | 验证实际实现：L1/L2 连通（前半验证 ①） |
| 4 | **diting-infra** | `make down diting prod` | 回收资源且**磁盘数据保留**；`prod.disk_id` 保留（前半验证 ②） |
| 5 | 云控制台或再次 Up | 可选：在云控制台确认数据盘仍存在；或直接再次 Up | 前半验证 ② 可选确认 |
| 6 | **diting-infra** | `make deploy diting prod` | 再次 Up 时自动读取 `prod.disk_id` 挂载同盘 |
| 7 | **diting-core** | 再次复制 `prod.conn` → `.env` 后执行 `make verify diting prod` 或 psql 查表 | 前半验证 ③：数据继承成立 |

**前半核心验证（三项）**：

| 序号 | 验证项 | 命令/方式 | 期望 |
|------|--------|-----------|------|
| ① | 环境可用、连接正确 | Up 后，在 diting-core 使用 `prod.conn` 中的 DSN 配置 `.env`，执行 `make verify diting prod` | 退出码 0，能连通 L1/L2 |
| ② | Down 保留盘 | 执行 `make down diting prod` 后，在云控制台或约定方式确认数据盘未删除；或再次 Up 时可挂载同盘 | 数据盘仍存在 |
| ③ | 再次 Up 数据可访问一致 | 再次执行 `make deploy diting prod` 后，通过 psql 或约定查询确认 L1/L2 表数据与 Down 前一致（如行数、样例） | 数据继承成立 |

**前半部分·目标测试步骤**（功能/验收测试，与上表三项一一对应）：

| 测试项 | 目标 | 执行步骤（可复制执行） | 通过标准 |
|--------|------|------------------------|----------|
| **T1 环境与连接** | 验证 Up 后环境可用、DSN 正确 | 1）diting-infra：`make deploy diting prod`<br>2）diting-core：`cp <diting-infra>/prod.conn .env`<br>3）diting-core：`make verify diting prod` | 退出码 0；终端输出含 L1/L2 verify OK |
| **T2 Down 保留盘** | 验证回收资源且磁盘数据保留 | 1）diting-infra：`make down diting prod`<br>2）确认 `prod.disk_id` 文件仍存在且非空；或云控制台查看该盘未删除 | `prod.disk_id` 存在；或控制台可见数据盘 |
| **T3 再次 Up 数据继承** | 验证再次 Up 挂载同盘、数据可访问一致 | 1）diting-infra：`make deploy diting prod`<br>2）diting-core：再次 `cp prod.conn .env` 后 `make verify diting prod`<br>3）可选：psql 连接 L1/L2 查表行数或样例，与 Down 前对比 | `make verify diting prod` 退出码 0；可选：表行数/样例与 Down 前一致 |

## 生产数据环境：后半部分（采集生产级 + 落库 + 数据继承）

后半部分按以下顺序执行：**① 前置检查（五项）→ ② 采集逻辑与数量质量要求（先写）→ ③ 环境自检与按需部署 → ④ 运行与验证（数据继承 + 全量采集与质量）→ ⑤ 回收环境**。

**约定**：在**第一个依赖生产环境的操作**之前，须完成 [前置检查（五项）](#l4-stage2-06-phase2-precheck)；不满足项按表中「不满足时」修复后再继续。**任一步骤失败时**，须先执行 [回收环境](#l4-stage2-06-phase2-teardown)，再结束本流程并排查原因。**正常完成所有运行与验证后**，必须执行回收环境。

<a id="l4-stage2-06-phase2-precheck"></a>
### ① 前置检查（五项）

在运行 [环境自检](#l4-stage2-06-phase2-env-check) 或 [运行步骤](#l4-stage2-06-phase2-run) 之前，确认下列五项条件；任一项不满足时按「不满足时」处理后再继续。

| 条件 | 检查方法 | 不满足时 |
|------|----------|----------|
| **C1 生产环境已 Up** | **diting-infra** 存在 `prod.conn`，且含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 非空 | 执行 `make deploy diting prod`（或先配置 tfvars/yaml 再执行） |
| **C2 表已初始化（L1/L2）** | **charts/ingest** 含 pre-install/pre-upgrade **Hook**，`helm upgrade --install diting-ingest` 时会先执行 schema-init Job 再创建采集 Job，无需单独脚本 | 无需处理，由 Chart 生命周期保证 |
| **C3 采集镜像对 K3s 可用** | 在 **diting-core** 执行 `make build-images` 生成 `diting-ingest:test`（或 config 中 `data_ingestion.image` 指定 tag）；K3s 节点能拉取或已 load 该镜像 | 构建后推送到集群可访问的 registry，或在节点上 `docker load` 对应镜像；Chart 默认 `imagePullPolicy: IfNotPresent` 便于使用本地镜像 |
| **C4 集群出网** | K3s 内 Pod 能访问外网（AkShare 等行情源） | 检查节点网络/安全组/代理；无外网时无法拉取真实行情 |
| **C5 本机可连 prod** | 在 **diting-core** 将 `prod.conn` 复制为 `.env` 后执行 `make verify diting prod` 退出码 0 | 确保本机与 prod 公网 IP 或 VPC 互通、安全组放行；或在与 ECS 同 VPC 的机器上执行 |

**建议执行顺序**：先完成 C1（必要时 E3 部署）→ C2 由 Chart Hook 自动满足 → C3 在首次部署前完成 → C4/C5 按环境确认。

### ② 采集逻辑与数量质量要求（先写）

在部署生产环境或执行全量采集之前，须先在本节明确采集逻辑与数量、质量要求，并与 [02_采集逻辑与Dockerfile_实践](02_采集逻辑与Dockerfile_实践.md)、[11_ 数据采集与输入层规约](../../03_原子目标与规约/_共享规约/11_数据采集与输入层规约.md) 一致。

| 维度 | 要求 | 说明与引用 |
|------|------|------------|
| **采集逻辑** | F8 目标数据（单标日线≥5 年、universe、复权）、F10 universe 表+Job | 与 [02_ 实践#目标数据约定与真实结果](02_采集逻辑与Dockerfile_实践.md#l4-stage2-02-target-data)、[06_ 设计#OHLCV 生产级](../../03_原子目标与规约/Stage2_数据采集与存储/06_生产级数据要求_设计.md#design-stage2-06-points) 一致；写入 L1/L2 的表与 DSN 由 01_/02_ 约定 |
| **数量要求** | 单标日线 bar 数 ≥ 5 年对应日线数；标的集合与 universe 表一致 | 5 年约 ≥1200 根日线（按交易日）；标的 = `get_current_a_share_universe()` 与 universe 表一致，见 11_ 与 DNA `production.history_depth_years` |
| **质量要求** | 复权口径与回测/实盘一致；行业/板块与 11_ 一致；可执行验证 | 复权实现与 Stage4 回测/实盘配置对齐；验收通过 `make verify-data-production`（见下方可执行验证清单） |

**可执行验证（采集逻辑与目标数据）**：本流程**一律使用真实行情数据**，禁止 mock（生产数据无假数据）。执行前须在 diting-core 执行 **`make deps-ingest`**（安装 requirements-ingest.txt，含 akshare），且能访问行情源。**测试集**（步骤 3、7）：`make ingest-test-real` 拉取少量真实行情（约 15 标、约 30 日），再 `make verify-data-test`。**全量生产级**（步骤 8～9）：`make ingest-production`（全 A 股 + 单标≥5 年日线）后 `make verify-data-production`。

### ③ 环境自检与按需部署（首个依赖环境操作之前执行）

在首次依赖生产环境的操作（如复制 `prod.conn`、执行 `make verify diting prod`）之前，先完成 [前置检查（五项）](#l4-stage2-06-phase2-precheck)，再按下列顺序**自行判断**是否具备满足需要的基础环境；若不存在则实施部署。

| 步序 | 工作目录 | 判断/操作 | 说明 |
|------|----------|----------|------|
| **E1** | **diting-infra** | 检查 `prod.conn` 是否存在，且内容含 `TIMESCALE_DSN`、`PG_L2_DSN`、`REDIS_URL` 非空 | 若文件不存在或上述变量缺失/为空 → 基础环境未就绪，进入 E3 |
| **E2** | **diting-core** | 将 `prod.conn` 复制为 `.env`：`cp <diting-infra 路径>/prod.conn .env`；执行 `make verify diting prod` | 若命令**失败**（退出码非 0，如连接超时、连接拒绝）→ 基础环境不满足，进入 E3；若**成功** → 环境已就绪，进入 [运行与验证步骤](#l4-stage2-06-phase2-run) 步骤 1 |
| **E3** | **diting-infra** | 实施部署：`make deploy diting prod`。Up 后若 `data_ingestion.enabled=true` 且 `use_k3s_job=true`，会**先同步 Secret**（`make prod-sync-conn-secret`）再 **helm 安装 charts/ingest**；Chart 的 **pre-install Hook** 会自动执行 schema-init（建 L1/L2 表），再创建采集 Job 并等待完成；采集镜像须已构建且 K3s 可拉取（见 [#数据采集在 K3s 的部署方式与镜像](#l4-stage2-06-k3s-and-image)） | 若部署**失败** → 若有部分资源已创建，执行 [回收环境](#l4-stage2-06-phase2-teardown)，结束流程并排查；若部署**成功** → 回到 E2 再次执行 verify，通过后进入 [运行与验证步骤](#l4-stage2-06-phase2-run) 步骤 1 |

<a id="l4-stage2-06-phase2-run"></a>
### ④ 运行步骤与验证（数据继承 + 全量采集与质量验证）

**步骤 1 之前**：须已完成 [前置检查（五项）](#l4-stage2-06-phase2-precheck) 与 [环境自检](#l4-stage2-06-phase2-env-check)（必要时已执行 E3 部署；部署会自动执行 schema-init 再跑采集 Job）。

**前置条件**：本流程一律使用**真实行情数据**。**采集均在远程 K3s 以 Job 实现**：须先在 diting-core 构建采集镜像（`make build-images`），并确保 K3s 能拉取该镜像（推送到集群可访问的 registry，或在节点上 `docker load`）；config 中 `data_ingestion.use_k3s_job=true`、`data_ingestion.image` 与 target 正确。**禁止**设置 DITING_INGEST_MOCK。

以下步骤须**按序执行**。**任一步失败**：先执行 [回收环境](#l4-stage2-06-phase2-teardown)，再结束流程。

| 步序 | 工作目录 | 命令/操作 | 失败时 |
|------|----------|-----------|--------|
| **1** | **diting-core** | `cp <diting-infra 路径>/prod.conn .env`（若环境自检 E2 已执行且未重新部署可跳过） | 若复制失败 → [回收环境](#l4-stage2-06-phase2-teardown) 并退出 |
| **2** | **diting-core** | `make verify diting prod` | 退出码非 0 → 回收环境并退出 |
| **3** | **diting-infra** | **少量真实行情（远程 K3s Job）**：`make deploy-ingest-job WAIT=wait`（在 K3s 内跑 Job，默认 target=ingest-test-real，约 15 标/约 30 日真实数据；镜像见 config `data_ingestion.image`） | 退出码非 0 或 Job 未成功 → 回收环境并退出；可查 `kubectl get jobs -n default` 后 `kubectl logs job/<名称> -n default -f`（Job 名形如 diting-ingest-&lt;runId&gt;） |
| **4** | **diting-core** | `make verify-data-test`（仅验测试集约 15 标真实数据存在与口径，不要求 5 年深度） | 退出码非 0 → 回收环境并退出 |
| **5** | **diting-infra** | `make down diting prod`（回收环境但**保留数据盘**） | 失败则记录后尽量继续；若无法继续 → 回收环境并退出 |
| **6** | **diting-infra** | `make deploy diting prod`（再次 Up，挂载同盘；Up 后若启用 use_k3s_job 会再次跑采集 Job，可跳过或仅做 verify） | 退出码非 0 → 回收环境并退出 |
| **7** | **diting-core** | 再次 `cp …/prod.conn .env`；`make verify diting prod`；**③ 数据继承验证**：`make verify-data-test`（验测试集 15 标数据仍可访问）。若 Down 后数据未保留需补采：在 **diting-infra** 执行 `make deploy-ingest-job WAIT=wait` 后再在 diting-core 执行 `make verify-data-test` | 任一项退出码非 0 → 回收环境并退出 |
| **8** | **diting-infra** | **全量生产级数据采集（远程 K3s Job）**：`make deploy-ingest-job INGEST_TARGET=ingest-production WAIT=wait`（K3s 内 Job 执行全 A 股 + 单标≥5 年日线；**禁止**用 ingest-test 代替，**禁止** DITING_INGEST_MOCK） | 退出码非 0 或 Job 未成功 → 回收环境并退出 |
| **9** | **diting-core** | **全量数据的生产级数量质量检测**：`make verify-data-production`（L1 单标≥5 年、标的与 universe 一致等）；**必须在步骤 8 完成真实全量采集后再执行** | 退出码非 0 → 回收环境并退出 |
| **10** | **diting-infra** | **[回收环境]** `make down diting prod` | 正常结束时**必须**执行；见 [回收环境](#l4-stage2-06-phase2-teardown) |

<a id="l4-stage2-06-phase2-teardown"></a>
### 回收环境（正常结束或任一步失败时执行）

**以下两种情况均须执行回收**：  
- **正常结束**：完成步骤 1～9 后，执行步骤 10（`make down diting prod`）。  
- **中途失败**：步骤 1～9 中任一步失败时，先执行回收再结束流程。

| 操作 | 工作目录 | 命令 |
|------|----------|------|
| 回收生产数据环境 | **diting-infra** | `make down diting prod` |

说明：`make down diting prod` 仅回收 ECS/K3s/EIP，**数据盘保留**；`prod.disk_id` 保留供再次 Up 挂载同盘。完整销毁含 prod NAS 时使用 `FULL_DESTROY=1 make down diting prod`。

<a id="l4-stage2-06-phase2-checklist"></a>
### 后半部分可执行清单（复制执行）

在**能访问 prod 公网 IP** 的机器上（或与 ECS 同 VPC/已配置安全组放行），**先完成前置检查（五项）**，再按序执行下列命令。变量 `INFRA`、`CORE` 请替换为实际路径。**采集均在远程 K3s 以 Job 实现**，须先构建采集镜像并确保 K3s 可拉取（见 config `data_ingestion.image`）。

**建议执行顺序**：① 前置检查 → ② 若无 prod.conn 或环境未就绪则 `make deploy diting prod`（会先 schema-init 再跑采集 Job）→ ③ 步骤 1～10。

```bash
export INFRA=/path/to/diting-infra
export CORE=/path/to/diting-core

# ---------- ① 前置检查（五项）----------
# C1: 存在 prod.conn 且 DSN 非空 → 否则 make deploy diting prod
# C2: Chart 含 pre-install hook，deploy-ingest-job 会自动先建表再跑采集
# C3: 采集镜像可用 → cd "$CORE" && make build-images；镜像对 K3s 可拉取（registry 或 docker load）
# C4/C5: 集群出网、本机可连 prod（步骤 2 的 verify 通过即表示 C5 满足）
[ -f "$INFRA/prod.conn" ] && grep -q 'TIMESCALE_DSN=' "$INFRA/prod.conn" || echo "请先执行: cd $INFRA && make deploy diting prod"

# ---------- 步骤 1-2：连接与连通验证 ----------
cp -f "$INFRA/prod.conn" "$CORE/.env"
cd "$CORE" && make verify diting prod

# ---------- 步骤 3-4：少量真实行情（远程 K3s Job，默认 ingest-test-real）+ 测试集验证 ----------
cd "$INFRA" && make deploy-ingest-job WAIT=wait
cd "$CORE" && make verify-data-test

# ---------- 步骤 5-6：Down 保留盘 → 再次 Up ----------
cd "$INFRA" && make down diting prod
cd "$INFRA" && make deploy diting prod

# ---------- 步骤 7：数据继承验证；若 Down 后数据未保留需补采则先：cd "$INFRA" && make deploy-ingest-job WAIT=wait ----------
cp -f "$INFRA/prod.conn" "$CORE/.env"
cd "$CORE" && make verify diting prod && make verify-data-test

# ---------- 步骤 8-9：全量生产级数据采集（远程 K3s Job：ingest-production）→ 再验证 ----------
cd "$INFRA" && make deploy-ingest-job INGEST_TARGET=ingest-production WAIT=wait
cd "$CORE" && make verify-data-production

# ---------- 步骤 10：回收环境 ----------
cd "$INFRA" && make down diting prod
```

**说明**：本流程**一律使用真实行情数据**，生产数据无假数据；**采集一律在远程 K3s 以 Job 实现**。须先在 diting-core 构建采集镜像（`make build-images`）并确保 K3s 能拉取（推送 registry 或节点 `docker load`）；**禁止** DITING_INGEST_MOCK。部署流程（`make deploy diting prod`）会先同步 Secret 再 **helm 安装 charts/ingest**，Chart 的 **pre-install Hook** 会自动建表再跑采集 Job；无需单独 schema-init 脚本。步骤 3/7 为少量真实行情（Job 默认 `ingest-test-real`）；步骤 8 为全量（`make deploy-ingest-job INGEST_TARGET=ingest-production WAIT=wait`）。

**一键执行 10 步（任一步失败则先回收再退出）**：在**能访问 prod 的机器**上，可使用 diting-infra 提供的脚本按序执行全部 10 步；脚本会先检查本机是否可连通 prod，避免在错误环境执行导致误回收。采集镜像已推送到 registry 时，通过环境变量传入镜像地址即可。

```bash
export INFRA=/path/to/diting-infra
export CORE=/path/to/diting-core
# 若镜像已推送到 registry（如 ACR），设置 INGEST_IMAGE 使步骤 3、8 的 Job 可拉取
# export INGEST_IMAGE=registry.cn-hongkong.aliyuncs.com/<命名空间>/diting-ingest:test

bash "$INFRA/scripts/prod-run-phase2-steps.sh"
```

单独调用 `make deploy-ingest-job` 时也可覆盖镜像：`INGEST_IMAGE=<完整镜像地址> make deploy-ingest-job WAIT=wait`（见 diting-infra Makefile 与 `scripts/prod-apply-ingest-job.sh`）。

### ② 部署环境与依赖组件（参考）

生产数据环境与前半部分共用同一套 Up/Down 流程，依赖以下组件与配置。环境自检不通过时，通过 [环境自检与按需部署](#l4-stage2-06-phase2-env-check) 中的 E3 实施部署。

| 依赖项 | 说明 |
|--------|------|
| **部署流程** | **diting-infra**：`make deploy diting prod`（竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis）；`make down diting prod` 仅回收 ECS/K3s/EIP，**保留数据盘**。连接信息输出至 `prod.conn`。 |
| **配置** | **diting-infra** 的 `config/diting-prod.yaml`（部署哪些服务、存储与数据库参数）、`config/terraform-diting-prod.tfvars`（实例与数据盘）。数据继承由**静态 PV/PVC** 实现，配置与 Chart 见下方「静态存储与数据继承」。 |
| **落库触发（C3）** | Up 就绪后，**默认在远程 K3s 以 Job 实现**：Makefile 从 `prod.conn` 同步 Secret（`make prod-sync-conn-secret`），再 `helm upgrade --install diting-ingest charts/ingest`（Chart 含 pre-install Hook 建表 + 采集 Job）；集群内执行 `make <data_ingestion.target>`（默认 `ingest-test-real`）。镜像由 `data_ingestion.image` 指定。单独触发：`make deploy-ingest-job [WAIT=wait] [INGEST_TARGET=...]`。 |

#### 静态存储与数据继承（实施依据）

生产环境数据继承采用**混合存储策略**：需继承的库（TimescaleDB L1、PostgreSQL L2、Redis）使用**固定路径静态 PV/PVC**，Down 后数据保留在数据盘 hostPath，再次 Up 时重新创建同名 PV/PVC 并挂载同盘，实现继承。实现以 **diting-infra** 为准。

| 维度 | 约定与实现位置 |
|------|----------------|
| **架构** | 独立数据盘（如 `/mnt/titan-data`）下固定路径：`postgres/timescaledb`、`postgres/postgresql-l2`、`postgres/redis`；静态 PV `ReclaimPolicy: Retain`，PVC 名称固定以便数据库 Chart 通过 `existingClaim` 引用。 |
| **实现** | **diting-infra** 仓库内 Helm Chart **`charts/diting-stack`** 仅负责静态 PV/PVC 与存储目录初始化 Job；数据库由 Makefile 使用 Bitnami Chart 部署，并通过 `primary.persistence.existingClaim` / `master.persistence.existingClaim` 引用上述 PVC。配置由 `config/diting-prod.yaml` 的 `stack.storage`、`stack.databases` 驱动。 |
| **验证** | 见 [运行步骤与验证](#l4-stage2-06-phase2-run) 步骤 5～7：先写数据 → Down → 再 Up → 再 verify。 |
| **故障排查** | **PV 状态 Released**：`kubectl patch pv <pv-name> -p '{"spec":{"claimRef":null}}'` 后重新绑定。**权限问题**：Pod 无法写盘时，检查 hostPath 目录权限或重新执行 diting-stack 的 init-job（Helm Hook）。 |

更多背景与动态 PVC 无法继承的原因分析见 [StorageClass 配置与数据继承验证](StorageClass配置与数据继承验证.md)。diting-infra 侧说明见其 `charts/diting-stack/README.md` 与 `config/README.md`。
