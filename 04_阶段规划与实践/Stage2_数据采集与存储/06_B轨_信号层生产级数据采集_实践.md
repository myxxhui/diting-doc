## 步骤目标

- 在 diting-core（或采集侧约定仓库）实现 **信号层生产级数据采集**：接口 `refresh_segment_signals_for_symbols(symbols)`，按 12_ §3.2 与 [06_B轨_信号层生产级数据采集_设计](../../03_原子目标与规约/Stage2_数据采集与存储/06_B轨_信号层生产级数据采集_设计.md) 编排「解析细分 → 按细分拉取 → 信号理解打标 → 写 segment_signal_cache」。
- **接口返回缺数据信息**：接口**必须返回**（或等价日志/表可查）结构化结果，便于知道「哪些标的没有哪些数据」——见 [配置、触发与返回](#l4-stage2-b06-config-trigger)。
- **信号适配器**：至少一个按 segment_id 拉取原始摘要的适配器（可占位或 Mock）；适配器由人写代码或配置，不 AI 自动写抓取逻辑。
- **信号理解层**：对原始内容打标产出 JSON（type, direction, strength, summary_cn, risk_tags）；失败时写 direction=neutral、strength=0.5；可先规则/关键词，再可选 AI；**配置、触发、输出、写入与边界**见下节。
- **多层级（三层，可扩展）**：实现**三层** segment（第 1 层 domain、第 2 层 板块、第 3 层 具体业务）；代码与配置支持**层级抽象与扩展**（如 SegmentTier 枚举、按层级选适配器），便于后续增加第 4 层等仅扩展枚举与适配器；见下节「多层级实现要求」。
- **单测与集成测**：`refresh_segment_signals_for_symbols` 可被调用且能更新 segment_signal_cache；调用后对应细分行存在且 fetched_at 更新；**返回结果中** symbols_without_segments、segments_without_adapter 等字段可验证；**多层级**：同一标的含多层级 segment_id 时一次 refresh 拉取三层信号且可验证；与 12_ 验收一致。

## 配置、触发、输出与写入、边界（实现必读）

调度层与信号层均在**采集模块**（diting-core 或约定采集仓库）实现；信号理解层若使用 AI，也在此模块内完成「理解+打标」，以下为配置、触发、输出、写入与边界的实现约定。

### 多层级实现要求（含超出 B 轨生产要求，确保 AI 可完成扩展功能）

- **层级定义（三层）**：**第 1 层 domain 级**（如 tech、agri、geo）、**第 2 层 板块级**（如 tech_ai、tech_semiconductor）、**第 3 层 具体业务级**（如 tech_ai_factory、tech_ai_compute）。实现须支持此三层且可验证。
- **代码与配置**：实现时**必须**提供层级抽象，便于扩展：
  - 定义 **SegmentTier**（或等价）枚举：如 `DOMAIN=1`、`SECTOR=2`、`BUSINESS=3`（或 domain/sector/business）；可从 segment_id 解析或从 segment_registry 的 tier/segment_tier 字段读取。
  - **按层级选适配器**：根据 segment 所属层级（第 1/2/3 层）选择对应类型数据源（大盘/板块/垂直）；适配器注册表或配置中可带 tier 或 adapter_tier，实现时按 tier 分发。
  - **segment_registry.segment_tier**（1/2/3）已在 L2 DDL 与 `make init-l2-business-profile-tables` 落地；`diting.ingestion.segment_tier` 提供与 `model_override_by_tier` 的映射。披露哈希 `seg_bp_*` 缺省 tier=3；无库内 tier 时回退：`seg_bp_*`→business，命名型 segment_id 按段数粗分 domain/sector/business。
- **扩展**：上述抽象保证**未来增加第 4 层**（如子业务/产品线）时，仅需：在 SegmentTier 中新增枚举、在 segment_registry 中注册新 segment_id（及 tier=4）、为新层级挂接适配器，无需改 refresh 主流程。单测应覆盖「同一标的含第 1+2+3 层 segment_id 时，一次 refresh 拉取三层信号且 cache 中均有对应行」；可选单测「按 tier 过滤 segment 列表」以验证层级逻辑可扩展。

### 配置

| 配置来源 | 内容 | 说明 |
|----------|------|------|
| **segment_registry**（L2 表或 YAML） | segment_id、domain、sub_domain、name_cn、signal_adapter_id；**可选 tier/segment_tier**（1=domain,2=板块,3=具体业务） | 新细分在此注册；**三层**：第 1 层（tech/agri/geo）、第 2 层（tech_ai 等）、第 3 层（tech_ai_factory 等），见 [12_ §2.2](../../03_原子目标与规约/_共享规约/12_右脑数据支撑与Segment规约.md)；无 adapter 的 segment 会在返回的 `segments_without_adapter` 中体现 |
| **signal_adapter_registry**（L2 表或 YAML） | adapter_id、source_type、url/selector、解析函数或配置 | 新源在此挂接；人写适配器代码，不 AI 自动写抓取 |
| **信号理解层** | **mode**（rule_only/ai_only/rule_first_then_ai/ai_first_fallback_rule）、**prompt_path** 或 prompt_template_key、**model_id**/provider、**max_input_tokens**/max_output_tokens、**retry_times**/retry_backoff_sec、schema_version；规则关键词表 | 默认支持规则与 AI 双路径且可配置，见下「信号理解：规则与 AI 实现逻辑与边界」 |
| **边界与开关** | max_input_chars、strength_range、risk_tags_enum、fallback_on_failure、audit_enabled、ttl_sec；signal_understanding.* 见设计文档 §四「信号理解配置项」 | 建议从 YAML/ENV 读取 |

配置文件建议路径：`config/signal_layer.yaml`（或项目约定路径）；环境变量覆盖示例：`SIGNAL_LAYER_MAX_INPUT_CHARS=4096`。

### 触发

- **方式一（采集侧定时）**：cron 或调度 Job 读取 `b_track_candidate_snapshot`（B 轨）或 `quant_signal_snapshot`（A 轨）本批 symbol 列表，调用 `refresh_segment_signals_for_symbols(symbols)`；cron 表达式或 batch_id 由部署/配置约定。
- **方式二（编排/流水线内）**：B 模块写表后，同一流水线步骤中调用 `refresh_segment_signals_for_symbols(symbols)`，再执行 C；由 diting-core 编排或 diting-infra 流水线定义触发。
- **轨区分**：`DITING_TRACK=a` 时 `make refresh-segment-signals` 从 `quant_signal_snapshot` 取 symbols；`DITING_TRACK=b` 时从 `b_track_candidate_snapshot` 取（表需 `make init-l2-b-track-candidate` 建）；可显式传入 symbols 覆盖。
- **REFRESH_SEGMENT_SCOPE**：`snapshot`（默认）= 仅处理本批 B 档（quant_signal_snapshot 确认∪预警 或 b_track_candidate_snapshot）；`full` = 全量 a_share_universe。
- 实现时二选一或并存；需在配置或 README 中写明「当前触发方式」及调用入口（如 CLI 命令、API、cron 脚本路径）。

### 最佳工作流与 make 入口

| 项目 | 说明 |
|------|------|
| **推荐入口** | **make run-full-pipeline**：A→B→refresh→C 串联，一次命令完成；C 可拿到预期 segment_signal_cache。见 [06_B轨_信号层_1对1对1](../../03_原子目标与规约/Stage2_数据采集与存储/06_B轨_信号层_1对1对1.md)。 |
| **单步 refresh** | `make refresh-segment-signals`，调用 `scripts/run_refresh_segment_signals.py` |
| **所属模块** | 信号层（diting-core `diting/signal_layer/`） |
| **流程** | 读 symbols（snapshot 或 full）→ `refresh_segment_signals_for_symbols` → 解析主营细分 → 适配器从 L2 拉取新闻 → 规则/AI 打标 → 写入 `segment_signal_cache` |
| **输入** | 环境变量：`PG_L2_DSN`、`DITING_TRACK`（a/b）、`REFRESH_SEGMENT_SCOPE`（snapshot/full）；或命令行传入 symbols |
| **输出** | 终端：写入表 `segment_signal_cache`、标的中文名、通过/未通过标的列表；返回 `RefreshSegmentSignalsResult`（含 symbols_with_signal、symbols_without_segments、segments_written 等） |
| **依赖** | L2 可读写；symbol_business_profile、segment_registry、news_content（采集写入）就绪；可选 `make sync-prod-conn` |
| **被依赖** | Module C 右脑消费 segment_signal_cache；**流水线**：`make run-full-pipeline` 或 B 写表后 → `make refresh-segment-signals` → `make run-module-c` |

### 输出与返回（code 中必须体现缺数据）

接口**必须返回**结构化结果（如 `RefreshSegmentSignalsResult`），便于调用方与运维知道「哪些标的没有哪些数据」：

- **symbols_without_segments**：本次输入 symbols 中，在 symbol_business_profile 无主营构成的标的列表；
- **symbols_with_signal**：本次有细分信号的标的列表（含 segments_written 与 segments_skipped_ttl 对应的标的，即 TTL 跳过也算有信号）；
- **segments_without_adapter**：本次解析出的 segment_id 中，在 segment_registry 无 signal_adapter_id 或适配器未注册的列表；
- **segments_skipped_ttl**：因缓存未过期而未拉取的 segment_id 列表；
- **segments_written**：本次写入 segment_signal_cache 的 segment_id 列表；
- **segments_failed**（可选）：拉取或打标失败的 segment_id 及原因；
- **summary**：如 total_symbols、symbols_with_segments、segments_fetched、segments_cached 等计数。

返回结构需在代码中明确定义（如 dataclass/typed dict），单测与集成测应断言「无主营业的标的出现在 symbols_without_segments」「无适配器的细分出现在 segments_without_adapter」。

### 写入

- **多层级（三层）**：标的主营构成表可为同一标的挂**第 1/2/3 层** segment_id，一次 refresh 拉取「大方向+板块+具体业务」；适配器按层级：第 1 层→大盘/宏观源，第 2 层→板块指数/新闻，第 3 层→垂直政策/研报；实现时通过 SegmentTier 或 tier 字段按层级选适配器；见 12_ §2.2。
- **segment_signal_cache**：segment_id、signal_summary（JSON）、signal_at、fetched_at、ttl_sec；每次成功打标 upsert 一行。
- **segment_signal_audit**（可选，由 audit_enabled 控制）：segment_id、source_type、source_url、source_publish_time、raw_snippet、model_conclusion_json、created_at；每次信号理解写缓存时可选写一条。
- **失败降级**：打标失败时若 fallback_on_failure=true，仍写 cache：direction=neutral、strength=0.5、summary_cn 为原始或截断，保证 C 可跑。

### 信号理解：规则与 AI 实现逻辑与边界

- **分析边界**（规则与 AI 共用）：仅对「与该细分相关的一手信息」做利好/利空/中性 + 强度 + 风险标签；**禁止**预测价格、输出买卖建议、分析无关噪音；使用 AI 时 prompt 中须明确禁止上述输出。
- **输入边界**：单次调用信号理解，输入文本长度 ≤ max_input_chars（默认 4096）；AI 调用时同时遵守 max_input_tokens/max_output_tokens；多源时做摘要或取最近 N 条。
- **输出边界**：固定 JSON schema（type, direction, strength, summary_cn, risk_tags）；type 枚举 policy|price|order|rnd，direction 枚举 bullish|bearish|neutral，strength ∈ [0.0, 1.0]，summary_cn 非空，risk_tags 为字符串数组；实现**必须**做 schema 校验，不符合即视为失败并走 fallback。
- **模式与策略**：**mode** 可配置 `rule_only`、`ai_only`、`rule_first_then_ai`、`ai_first_fallback_rule`，**默认**为 `rule_first_then_ai`；实现须支持全部模式且可配置；可按 segment_tier 或 segment_id 配置「该 tier 用规则或 AI」（如 model_override_by_tier）。
- **Prompt 模板**：模板存放路径由 **prompt_path** 或 **prompt_template_key** 配置（如 `config/prompts/segment_signal_understanding_v1.md`）；模板中占位符建议含 `{{segment_id}}`、`{{raw_text}}`、`{{schema_instruction}}`；禁止在 prompt 中要求模型输出买卖建议或价格预测；模板外置便于迭代。
- **模型调用**：从配置读 **model_id**/provider；单次请求遵守 max_input_tokens、max_output_tokens；调用失败或超时：先按 **retry_times**、**retry_backoff_sec** 重试，仍失败则写 audit（raw_snippet + error 信息），再写 cache 为 direction=neutral、strength=0.5。
- **输出解析与 schema 校验**：解析模型返回 JSON 后**必须**做 schema 校验（type/direction 枚举、strength 范围、risk_tags 类型）；不通过即视为失败，同失败分支（写 audit → 写 cache neutral/0.5）。
- **失败与 fallback**：规则或 AI 任一路径失败（含校验不通过、超时、异常）时，若 **fallback_on_failure**=true：写 segment_signal_audit（若 audit_enabled）记录 raw_snippet 与错误，再写 segment_signal_cache 为 direction=neutral、strength=0.5、summary_cn 为原始或截断，保证 C 可跑。
- **审计与复用**：当 **audit_enabled**=true 时，每次信号理解调用（含失败）写 segment_signal_audit 一条；可选 **audit_reuse_same_day**：同一 segment_id、同一自然日若已有成功打标结论，可复用该结论写 cache、不再调模型，实现时在「按 segment 去重」后查 audit 或 cache 判断是否复用。

<a id="l4-stage2-b06-steps"></a>
## 实施步骤

**工作目录**：diting-core（或采集/信号层约定仓库）

1. **建表（若 01 未建）**：在 L2 创建 `segment_signal_cache`（及可选 `signal_adapter_registry`、`segment_signal_audit`）；DDL 见 [12_](../../03_原子目标与规约/_共享规约/12_右脑数据支撑与Segment规约.md) §2.3 与设计文档 §3.2。
2. **配置与边界**：增加 `config/signal_layer.yaml`（或等价）与 ENV 覆盖；实现从配置读取 segment_registry、signal_adapter_registry 来源及边界项（max_input_chars、audit_enabled 等）；见上节「配置」「边界」。
3. **层级抽象与多层级**：在代码中定义 SegmentTier 枚举（或等价）与「segment_id → tier」解析/查表；segment_registry 若支持 tier 字段则写入与读取，否则从 segment_id 命名约定解析；实现「按 tier 选适配器」逻辑，满足三层且可扩展（见上「多层级实现要求」）。
4. **适配器注册与实现**：在 signal_adapter_registry（或 YAML 配置）中注册至少一个 adapter_id 与 segment_id 绑定，适配器可按 tier 分类；实现至少一个**信号适配器**（按 segment_id 拉取原始摘要/文本），可先用 Mock 或占位（返回固定摘要），保证编排链路可跑通。
5. **信号理解层（规则 + AI）**：实现「原始摘要 → 打标」全流程，与设计 §3.3、§3.4 及上节「信号理解：规则与 AI 实现逻辑与边界」一致。
   - **5A 规则引擎**：实现规则/关键词 → 同一 JSON schema（type, direction, strength, summary_cn, risk_tags）；产出须通过 schema 校验。
   - **5B 配置加载**：从 config 读取 mode、prompt_path、model_id、max_input_tokens、max_output_tokens、retry_times、retry_backoff_sec、fallback_on_failure、audit_enabled；支持按 tier 选策略或 model（若配置了 model_override_by_tier）。
   - **5C Prompt 加载与渲染**：按 prompt_path 加载模板，占位符替换（如 segment_id、raw_text、schema_instruction）；禁止在模板中包含「输出买卖建议/价格预测」类指令。
   - **5D AI 调用**：按配置调用模型；遵守 token 上限；实现超时、重试与退避；返回原始 JSON 字符串。
   - **5E 输出解析与 schema 校验**：解析返回 JSON，校验 type/direction 枚举、strength 范围、summary_cn 非空、risk_tags 类型；不通过则视为失败，进入 5F。
   - **5F 失败与审计**：任一路径失败时写 segment_signal_audit（若 audit_enabled），再写 cache 为 direction=neutral、strength=0.5；可选实现同 segment 同日复用（audit_reuse_same_day）。
6. **编排接口与返回**：实现 `refresh_segment_signals_for_symbols(symbols) -> RefreshSegmentSignalsResult`：查 symbol_business_profile 得各标的细分列表（含多层级 segment_id）→ 合并去重得 S → 收集「无主营业标的」「无适配器细分」→ 对 S 中每个 segment_id 若缓存未过期则跳过并记录，否则**按层级选适配器**拉取 → 信号理解打标 → 写 segment_signal_cache（及可选 segment_signal_audit）→ 汇总 symbols_without_segments、segments_without_adapter、segments_skipped_ttl、segments_written、segments_failed、summary 并返回。
7. **触发入口**：实现触发方式之一（或两种）：cron/Job 调 refresh 或流水线步骤调 refresh；在配置或 README 中写明触发方式与入口。
8. **单测**：单测覆盖（1）解析标的→细分集合；（2）无主营业标的、无适配器细分在返回中正确；（3）**多层级**：同一标的含第 1+2+3 层 segment_id 时，解析得到三层 segment 且 refresh 后 cache 中均有对应行；（4）按 segment 去重与 TTL 跳过；（5）**信号理解**：规则路径打标产出符合 schema；（6）**AI 路径**（若实现）：Mock 模型返回合法 JSON 时写 cache、非法 JSON 或超时时 fallback 写 neutral/0.5 且可写 audit；（7）**schema 校验**：非法字段或越界时拒绝并 fallback；（8）写入 cache 后可按 segment_id 读出。可选：按 tier 过滤 segment 列表、按 mode 切换规则/AI 的单元测试。
9. **集成测 / 验收命令**：提供 `make refresh-segment-signals-test` 或等价命令：传入候选标的列表（含具多层级 segment_id 的标的），执行 refresh，验证 segment_signal_cache 对应行存在且 fetched_at 更新，**且返回结果中** symbols_without_segments、segments_without_adapter 等符合预期；**三层信号**均可被拉取并写入 cache；**信号理解**规则路径与 AI 路径（含失败 fallback）可验证；与 12_ 可执行验收一致。
