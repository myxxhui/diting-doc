## 一、L4 新定位（2026-05-16 重构）

L4 **不再**是"实施步骤设计"（已迁移到 L3 `stages/`），而是**实践运行的进展记录**：

| 维度 | L3 stages（设计） | L4（记录） |
|---|---|---|
| **承载** | "**应该怎么做**" | "**实际做了什么 + 结果**" |
| **频率** | 一次设计，跨阶段不变 | 每次实践运行一次记录 |
| **内容** | 目标、技术、数据、训练、验收标准 | 进展、代码 PR、测试输出、偏离决策 |
| **谁写** | 架构师 + AI（计划）| AI + 开发者（执行后记录）|

### 3.3 部署快照（推荐每份记录具备）

对齐目标运行时：**阿里云 ECS + K3s**、**Helm**、镜像 **阿里云 ACR**、`diting-infra` → **deploy-engine**（Chart 与子模块链路见 [_共享规约/16](../03_原子目标与规约/_共享规约/16_阿里云ECS_K3s_ACR_Helm部署与deploy-engine链路.md)）。

在每份实践中增加 **部署**（或并入「偏离与决策」）小节时，至少写清下列之一：**本周无上架**（并说明工作仅在 `diting-src`/`diting-infra` 何种程度），或已上架时的可追溯一行表：

| 项 | 填写 |
|---|---|
| **ACR 镜像 digest/tag**（各服务） | 无则 `—` |
| **helm release / namespace / values 覆盖文件名** | 无则 `—` |
| **验证命令与结果摘要**（如 curl / kubectl / helm status） | 必填一句 |

**约定**：不得在 **`diting-infra/deploy-engine/`** 子模块拷贝内做写操作或提交；须在 **deploy-engine 独立仓库** 修改、`push`，再在 **`diting-infra` 执行 `make update-deploy-engine`**（与仓库规则一致）。

## 六、写作流程

```mermaid
flowchart LR
  L3[L3 阶段设计] -->|实施前对照| PLAN[实施计划]
  PLAN -->|开发| CODE[代码 PR]
  CODE -->|测试| TEST[本地测试]
  TEST -->|完成后| L4[L4 实践记录]
  L4 -->|准出| L5[L5 验收]
  
  DNA[DNA YAML] -.->|引用键值| L4
```

**每份实践记录必须包含**：
- [ ] 计划部分：链接到 L3 stage 设计文档 + DNA 键
- [ ] 实际进展：✅/⚠️/❌ 状态
- [ ] 测试运行：实际命令 + 输出片段
- [ ] **部署**：按 §3.3 写明 **上架/无上架** 与快照字段（或无则占位句）
- [ ] 偏离与决策（如有）
- [ ] 引用：L3 设计、代码 PR、相关工单
