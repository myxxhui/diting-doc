## 本步依赖检查与数据来源

执行本步前，须确认以下依赖与数据来源可用。

### 1. 上游准出

| 检查项 | 说明 | 本步用途 |
|--------|------|----------|
| **Stage2-05 准出** | 采集模块已部署；`make ingest-test` 可写 L1/L2；行业/营收数据可被读取 | Module A 输入：申万行业、营收占比来自 L1/L2 或 MarketDataFeed；若 s0_data 未准出可先用 Mock 验证接口与结构，标注待补验 |

### 2. 数据输入来源（与 11_ 规约一致）

| 输入项 | 来源 | 说明 |
|--------|------|------|
| **标的池（universe）** | 执行前通过 [11_](../../03_原子目标与规约/_共享规约/11_数据采集与输入层规约.md) 约定的 **`get_current_a_share_universe()`** 获取当前全 A 股列表，或由调用方传入该列表 | 同批与 Module B 须使用**同一份**列表，见 [09_ Module A 标的池来源与同批一致](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)；对全部 N 只全量分类，不写死数量 |
| 标的代码 | 调用方传入（单标的）或来自上述 universe 列表 | 如 `"000998.SZ"`、`"688981.SH"` |
| 申万行业分类 | L2 或 Stage2 采集的 industry_revenue / 约定表 | 见 [11_ 数据采集与输入层规约](../../03_原子目标与规约/_共享规约/11_数据采集与输入层规约.md)、[Stage2-02 采集逻辑与 Dockerfile](../../03_原子目标与规约/Stage2_数据采集与存储/02_采集逻辑与Dockerfile_设计.md) |
| 营收占比数据 | L2 或 Stage2 采集的 industry_revenue | 同上；逻辑填充期可从 MarketDataFeed 或 L1/L2 读取 |

**生产逻辑对每只标的均处理**：执行时先通过 `get_current_a_share_universe()` 取得全量标的池（如 5000+ 只），再对 **universe 中每一只** 调用分类（`classify_batch` 遍历全量，不写死数量）。若大量结果为 UNKNOWN，通常是**数据源**问题：未配置 L2 或未接 L2 时使用默认 Mock，Mock 仅对 3 只示例（000998.SZ、688981.SH、601899.SH）返回行业/营收数据，其余标的均得到「未知」→ UNKNOWN。配置 `PG_L2_DSN` 后，run 会对**每只标的**从 L2 表 `industry_revenue_summary` 查询行业/营收；该表由采集任务 `ingest_industry_revenue` 写入（见 `diting-infra` 建表 `03_l2_industry_revenue_summary.sql` 及 diting-core 采集写入逻辑）。

**数据不可用时的策略**：先用 Mock 数据验证接口与结构、YAML 规则加载、UNKNOWN 路径；在文档中标注「待 s0_data / Stage2-05 准出后补验真实标的」。

<a id="l4-stage3-01-feature-practice"></a>
## 功能实践（功能实现要求详细描述）

本节给出 Module A 语义分类器的**功能实现要求**，作为开发与验收时的功能实践依据；实现须与 09_ 规约、DNA、设计文档一致。

### 1. 目录与包结构

| 要求 | 说明 | 与 DNA 一致性 |
|------|------|----------------|
| 包路径 | 在 diting-core 下建立 `diting/classifier` 包 | 与 09_ Module A、DNA delivery_scope 一致 |
| 配置路径 | 分类规则使用 YAML 配置，路径如 `config/classifier_rules.yaml`（或 DNA/设计文档约定的路径） | 规则不硬编码，可由配置驱动 |

### 2. 接口与契约

| 项目 | 要求 | 规约与设计引用 |
|------|------|----------------|
| 输入 | 标的代码、申万行业、营收占比（可从 L1/L2 或 MarketDataFeed 获取） | [09_ Module A 输入输出](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)、[dna_module_a.input](../../03_原子目标与规约/_System_DNA/core_modules/dna_module_a.yaml) |
| 输出 | `ClassifierOutput`：Domain Tag 列表 + 置信度 (0.0–1.0) | [ClassifierOutput.proto](../../03_原子目标与规约/_Design_Artifacts/protocols/classifier/classifier_output.proto)；可被 Module B/D 消费 |
| 实现类 | `SemanticClassifier`，提供 `classify(symbol)` 或等价接口，返回符合 Proto 的输出 | 09_ 规约、设计文档 [01_语义分类器_设计](../../03_原子目标与规约/Stage3_模块实践/01_语义分类器_设计.md#design-stage3-01-points) |

**标的池与全量分类**：执行入口先调用 **`get_current_a_share_universe()`** 获取当前全 A 股列表（或接收调度/调用方传入的同一列表），再对该列表**全部 N 只**全量分类（建议提供批量接口如 `classify_batch(universe)`）；与 Module B 使用同一接口或同一传入列表时，由调度保证同批一致。验证项可含「universe 来源与 11_/09_ 约定一致」或「同批与 B 使用同一列表」。

### 3. 分类规则（YAML 驱动）

规则须从 **YAML 配置** 加载，禁止在代码中硬编码规则阈值与行业关键词。与 `dna_module_a.classification_rules` 语义一致：

| Domain Tag | 规则描述（DNA） | 实现要点 |
|------------|-----------------|----------|
| **AGRI** | 申万行业=农林牧渔 或 营收占比>50% | 配置中可配置行业关键词与占比阈值 |
| **TECH** | 申万行业=电子/计算机/通信 或 研发投入>10% | 同上 |
| **GEO** | 申万行业=有色金属/石油石化 或 大宗商品营收>50% | 同上 |
| **UNKNOWN** | 无法匹配上述规则 | 默认出口；置信度可用 0.5 或配置值 |

配置变更后，分类结果应随之变化（**逻辑功能 100%** 验证项之一）。

### 4. 真实标的验证约定（可选，数据就绪时必做）

| 标的示例 | 预期 Tag | 说明 |
|----------|----------|------|
| 000998.SZ（隆平高科） | AGRI | 农林牧渔 |
| 中芯国际（688981.SH 等） | TECH | 电子/半导体 |
| 紫金矿业 | GEO | 有色金属/大宗 |
| 无法归类标的 | UNKNOWN | 走 Trash Bin 路径 |

单模块验证时至少用 1～2 个真实标的（或 Mock）跑通分类结果并核对 Tag 与置信度。

### 5. 单独运行、Dockerfile、镜像与版本管理、一键构建、一键部署

Module A 为**独立可部署单元**（见 [09_ 可部署单元清单](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)、[global_const.deployable_units.module_a](../../03_原子目标与规约/_System_DNA/global_const.yaml)），须支持单独运行、镜像构建与部署链，与 Stage2 采集模块实践方式对齐。

| 实践项 | 要求 | 与 DNA/规约一致 |
|--------|------|-----------------|
| **单独运行** | Module A 可作为独立服务运行：提供 CLI 或 HTTP/gRPC 入口（如 `python -m diting.classifier.run` 或等价）；本地或镜像内执行同一入口即可跑分类服务 | 09_ Module A 独立服务、镜像与验证链独立 |
| **Dockerfile** | 路径：`diting-core/docker/module_a`（与 `global_const.deployable_units.module_a.dockerfile_path` 一致）；多阶段构建、安装依赖、COPY 代码与 `config/classifier_rules.yaml`、设置 ENTRYPOINT/CMD；不硬编码业务密钥 | DNA dockerfile_path |
| **镜像与版本管理** | 镜像名：`diting-module-a`（可加 registry 前缀）；Tag 策略：dev 用 git-sha 或 latest、prod 用 SemVer；与 [dna_dev_workflow.release_bundle](../../03_原子目标与规约/_System_DNA/dna_dev_workflow.yaml)（tag_strategy: git_sha_or_semver）一致；同一 Release 内与其它可部署单元同 tag | global_const.image_name、release_bundle |
| **一键构建** | Makefile 提供 `make build-module-a`（或 `make build-images` 含 module_a）；在 **diting-core** 执行，产出可运行镜像；构建后可在**镜像内**执行 `make test` 或等价验证（V-IMAGE） | 与 Stage2-02 V-BUILD-ALL、02_量化扫描引擎 V-IMAGE 对齐 |
| **一键部署** | **diting-infra** 侧 diting-stack 含 module_a（Deployment + Secret）；`make deploy diting prod` 部署；本地验证用方式一（本地 build 运行），远程验证用方式二（K3s 部署） | DNA chart_path；见本步「生产数据实践测试」方式一/方式二 |

**说明**：镜像内须能执行分类逻辑（含加载 config/classifier_rules.yaml）；若需连 L1/L2，通过环境变量（如 `PG_L2_DSN`）或 Secret 注入，不写死镜像内。

**方式二 + 跑指定标的（远程跑 语义分类器-A 连远程库，仅对指定股票分类）**

指定标的**不通过 values**，而是通过 **ConfigMap 挂载**：Chart 会创建 ConfigMap `diting-semantic-classifier-a-symbols`，内容来自 chart 内 `config/diting_symbols.txt`，Pod 挂载到 `/config` 并设置 `DITING_SYMBOLS=/config/diting_symbols.txt`，分类器读该文件得到标的列表。

- **操作步骤**：保证 chart 内 `config/diting_symbols.txt` 已有标的列表后，执行 `make deploy diting prod` 即可；Helm 会将该文件打入 ConfigMap，Pod 挂载并设 `DITING_SYMBOLS=/config/diting_symbols.txt`，只对所列标的分类。若需与 diting-core 的名单一致，可先执行 `make sync-semantic-classifier-a-symbols` 再 deploy。
- **与 diting-core 同步（可选）**：在 diting-infra 根目录执行 `make sync-semantic-classifier-a-symbols`（默认从 `../diting-core` 读取）可将 diting-core 的 `config/diting_symbols.txt` 复制到 chart `config/`，再执行 `make deploy diting prod`。

```bash

make deploy diting prod
```

chart 内 `config/diting_symbols.txt` 仅含注释、无标的时，代码**不会**使用 Mock，会报错「DITING_SYMBOLS 已设置但解析结果为空」并得到 0 条结果。config 中 **timescaleHost / postgresL2Host** 须指向含生产数据的库（集群外库在 diting-prod.yaml 等中覆盖），部署后即连接数据库并使用指定标的的生产数据。

**执行结果输出到哪里、Module B 如何消费**：
- **当前**：Module A 的执行结果（每只标的的领域标签 + 置信度，即 `ClassifierOutput` 列表）**仅输出到 stdout**，通过 `kubectl logs deployment/diting-semantic-classifier-a -n default` 可见；日志中会输出「本批执行标的」列表（前 10 只 + 共 N 只）、「分类结果」摘要（前 10 条：标的 -> 领域标签中文名 + 置信度）。
- **同进程编排**：在 diting-core 内通过 `run_daily_scan.py` 一次性调用 Module A 与 Module B 时，Module A 的返回结果在内存中直接传给下游逻辑（当前脚本仅打日志，未再传给 Module B 做过滤；Module B 使用同一 universe 独立跑扫描）。
- **独立部署时 Module B 消费**：若语义分类器-A 与 Module B 分别部署（如 K8s 上各跑各的 Pod），当前**没有**落库或共享存储，Module B 无法直接读取 Module A 的本轮结果。后续若需「B 按 A 的分类结果过滤/加权」，需在 Module A 侧增加将 `ClassifierOutput` 写入 L2 表或共享存储（如 Redis/消息队列），Module B 再按约定读取；具体表结构或协议待 Stage3 后续规约。

**若日志出现「从 L1 读取全 A 股标的」而非「指定股票模式」**（即未按指定标的跑），按下列顺序逐项检查实际状态，不得臆断原因：

| 步骤 | 检查项 | 可执行命令 | 预期 |
|------|--------|------------|------|
| 1 | 本地 Chart 渲染出的 ConfigMap 是否含标的 | 在 diting-infra 根目录：`yq eval '{"storage": .stack.storage, "module_a": (.stack.module_a // {}), "schemaInit": (.stack.schemaInit // {"enabled": true})}' config/diting-prod.yaml > /tmp/v.yaml && helm template diting-stack charts/diting-stack -f /tmp/v.yaml 2>/dev/null \| sed -n '/name: diting-semantic-classifier-a-symbols/,/^---/p' \| head -40` | 输出中 `data.diting_symbols.txt` 下应有多行标的代码（如 002371.SZ） |
| 2 | 集群内 ConfigMap 实际内容 | `kubectl get cm diting-semantic-classifier-a-symbols -n default -o jsonpath='{.data.diting_symbols\.txt}' \| head -20` | 应含多行标的代码；若仅两行注释则说明某次 deploy 时 chart 内文件无标的或 Pod 需重启后挂载才更新 |
| 3 | Pod 内环境变量 | `kubectl exec deployment/diting-semantic-classifier-a -n default -- env \| grep -E 'DITING_SYMBOLS|MODULE_AB'` | 应有 `DITING_SYMBOLS=/config/diting_symbols.txt` |
| 4 | Pod 内挂载文件内容 | `kubectl exec deployment/diting-semantic-classifier-a -n default -- cat /config/diting_symbols.txt \| head -15` | 应含多行标的代码；若仅注释则与步骤 2 一致，需更新 ConfigMap 后重启 Deployment |
