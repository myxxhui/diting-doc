# 上部分 · 数据流与 L2 规约

## 1. make ingest-production 采集逻辑概览

`make ingest-production` 调用 `scripts/run_ingest_production.py`，按顺序执行：

| 步骤 | 开关 | 行为 |
|------|------|------|
| Step1 | `INGEST_PRODUCTION_UNIVERSE` | 刷新全 A 股标的池 → 写 L1 `a_share_universe` |
| Step2 | `INGEST_PRODUCTION_OHLCV` | 按批拉取 K 线 → 写 **L1**（TimescaleDB `ohlcv` 等） |
| Step3 | `INGEST_PRODUCTION_INDUSTRY_REVENUE` | 按标拉取行业/财务 → 写 **L2** `industry_revenue_summary` |
| Step3.3 | `INGEST_PRODUCTION_BUSINESS_PROFILE` | 按标拉取**主营构成（按产品）** → 写 **L2** `segment_registry` + `symbol_business_profile`（见 [12_右脑数据支撑与Segment规约](../../03_原子目标与规约/_共享规约/12_右脑数据支撑与Segment规约.md)） |
| Step4 | `INGEST_PRODUCTION_NEWS` | 按标拉取新闻 → 写 **L2** |

Step3 标的列表来自 `get_current_a_share_universe(force_refresh=False)`（建议带交易所后缀，与 Module A/B 一致）。

## 2. L2 数据落点与 Module A 输入

- **`industry_revenue_summary`**：每标的一行（symbol、industry_name、revenue_ratio、rnd_ratio、commodity_ratio、updated_at），供 Module A 按 symbol 查询分类所需输入。写入见 `diting/ingestion/industry_revenue.py` 的 `_upsert_industry_revenue_summary`。
- **`symbol_business_profile` + `segment_registry`**：每标的多行分部（主营构成披露）；采集见 `diting/ingestion/business_profile.py`。Module A 优先用该数据填充 **ClassifierOutput.segment_shares**（`business_segment_provider`）；无数据时回退单条占位（`seg_no_disclosure`），**不**用规则模拟多分部。建表：`make init-l2-business-profile-tables`。
- **`data_versions`**：仅存元数据（data_type、version_id、file_path 等），不存业务字段；Module A 不直接查此表得行业/营收。

**约定**：全链路统一使用**带交易所后缀**的 symbol（如 `000001.SZ`）。采集侧 Step3/Step4 传入带后缀的 `sym`，写入 `industry_revenue_summary` 时仍用该 symbol；Module A 的 universe 与 L2 查询键一致，即可命中。

## 3. ClassifierOutput 写入 L2 / 共享存储（Stage3 规约）

为支持**独立部署**下 Module B 按 A 的分类结果过滤/加权，需 **A 将 ClassifierOutput 写入 L2 或共享存储，B 按约定读取**。表结构/协议在 Stage3 规约如下。

### 3.1 表结构（L2 PostgreSQL）

| 表名 | 用途 |
|------|------|
| **classifier_output_snapshot** | 按批次存放 Module A 的分类结果，供 Module B 按 batch_id 或最新批次读取 |

**建议字段**（与 [ClassifierOutput.proto](../../03_原子目标与规约/_Design_Artifacts/protocols/classifier/classifier_output.proto) 对齐）：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigserial | 主键 |
| batch_id | varchar(64) | 本批执行唯一标识（如 correlation_id 或 run_id） |
| symbol | varchar(32) | 标的代码，如 000998.SZ |
| primary_tag | varchar(64) | 主领域标签：农业/科技/宏观/未知/自定义（与 YAML 自定义标签长度一致） |
| primary_confidence | float | 主 Tag 置信度 0.0–1.0 |
| tags_json | jsonb | 可选，完整 tags 列表（TagWithConfidence 序列化） |
| segment_shares_json | jsonb | 主营分部列表；每条含 `disclosure_present`（无 L2 主营行时为 false） |
| correlation_id | varchar(64) | 全链路请求 ID |
| created_at | timestamptz | 写入时间 |

**输出语义**：控制台与 [09_ Module A](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)「输出语义辨析」一致——**领域标签**（申万 + YAML）与 **主营披露**（财报分部）分列理解；`make run-module-a` 对每行打印「领域标签」与「主营披露 Top1 / 无」。

建表脚本归属 diting-infra L2 schema（如 `06_l2_classifier_output_snapshot.sql`，与 03/04/05 同级），与 `03_l2_industry_revenue_summary.sql` 同级。已有库需扩列/扩宽：`scripts/init_l2_classifier_table.py` 会执行 `ADD COLUMN segment_shares_json`、`ALTER COLUMN primary_tag TYPE VARCHAR(64)`。

### 3.2 写入责任（Module A）

- 分类完成后，在**同一批次**内将本批所有 `ClassifierOutput` 写入 `classifier_output_snapshot`（或等价共享存储）；batch_id 同一批一致，便于 B 按批消费。
- 若使用共享存储（如 Redis/消息队列）而非 L2 表，协议需约定 key 格式、TTL、序列化方式（如 JSON 与 Proto 兼容字段），并在 [03_量化扫描引擎_实践](03_量化扫描引擎_实践.md) 中与 B 的读取约定一致。

### 3.3 读取责任（Module B）

Module B 按约定从 L2 或共享存储读取 ClassifierOutput，用于按板块过滤、加权等。具体接口与实现见 [02_量化扫描引擎_实践](02_量化扫描引擎_实践.md#l4-stage3-03-classifier-input) 及对应设计文档。

<a id="run-module-a-and-query"></a>
## 一键本地运行 A 模块 / 一键查询 A 模块写入数据

### 一键本地运行 A 模块（make run-module-a）

在 **diting-core** 根目录执行：

```bash
make run-module-a
```

会加载 `.env`（含 `DITING_SYMBOLS`、`PG_L2_DSN`），按 **config/diting_symbols.txt**（或环境变量覆盖）全部标的执行分类，并在输出中直接看到：

- **执行标的**：本批共 N 只及完整列表（来源：DITING_SYMBOLS / config/diting_symbols.txt）
- **执行结果**：每条含 **领域标签**（申万 + `classifier_rules.yaml`）与 **主营披露**（L2 财报分部 Top1 或「无」）；二者数据源不同，见 [09_ 输出语义辨析](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)。  
  **置信度**：对领域标签首项的确信程度，0~1（与 ClassifierOutput.proto 中 TagWithConfidence.confidence 一致）。
- **写入**：若配置了 `PG_L2_DSN` 且写入成功，会打印「L2 表 classifier_output_snapshot（库: PG_L2_DSN），batch_id=xxx，行数=N」；否则打印未写入原因。

**连接的是 .env 里配置的数据库**：脚本和 `make run-module-a` / `make init-l2-classifier-table` 都会读取 `.env` 中的 `PG_L2_DSN` 并连接该地址。若需连**远程**库（例如已部署的生产 L2），请将 `.env` 中 `PG_L2_DSN` 的 host 改为远程地址（如从 diting-infra 的 **prod.conn** 复制 `PG_L2_DSN`，其为 `公网IP:NodePort`，例如 `postgresql://postgres:postgres@43.x.x.x:30002/diting_l2`）；不要用 `127.0.0.1`（除非 L2 就在本机）。

**若报「表可能未创建」或写入失败**：请先确保 L2 可达（上述 PG_L2_DSN 能连上），再执行 `make init-l2-classifier-table` 在 L2 库中创建表，然后重新 `make run-module-a`。

等价命令：`PYTHONPATH=. python3 scripts/run_module_a_local.py`（需先 `source .env` 或保证 .env 已被脚本加载）。

### 一键查询 A 模块写入的数据（make query-module-a-output）

在 **diting-core** 根目录执行：

```bash
make query-module-a-output
```

会加载 `.env` 中的 `PG_L2_DSN`，查询 L2 表 **classifier_output_snapshot**，输出：

- 最近若干批次的汇总（batch_id、行数、最新写入时间）
- 最新写入明细（最多 30 条：id、batch_id、symbol、primary_tag、primary_confidence、created_at）

需 L2 可达（PG_L2_DSN 可连）。若表为空或不存在，请先执行 `make init-l2-classifier-table` 建表，再执行 `make run-module-a` 写入数据后查询。

### 一键初始化 L2 表（make init-l2-classifier-table）

若 L2 已启动但报「表可能未创建」或写入失败，在 **diting-core** 根目录执行：

```bash

```

会在 PG_L2_DSN 指向的库中创建表 **classifier_output_snapshot**（若已存在则跳过），与 diting-infra 中 `06_l2_classifier_output_snapshot.sql` 一致。执行成功后即可正常 `make run-module-a` 并写入。

等价命令：`PYTHONPATH=. python3 scripts/init_l2_classifier_table.py`（需先加载 .env）。

**方式二 + 跑指定标的（远程跑 语义分类器-A 连远程库，仅对指定股票分类）**

指定标的**不通过 values**，而是通过 **ConfigMap 挂载**：Chart 会创建 ConfigMap `diting-semantic-classifier-a-symbols`，内容来自 chart 内 `config/diting_symbols.txt`，Pod 挂载到 `/config` 并设置 `DITING_SYMBOLS=/config/diting_symbols.txt`，分类器读该文件得到标的列表。

- **操作步骤**：保证 chart 内 `config/diting_symbols.txt` 已有标的列表后，执行 `make deploy diting prod` 即可；Helm 会将该文件打入 ConfigMap，Pod 挂载并设 `DITING_SYMBOLS=/config/diting_symbols.txt`，只对所列标的分类。若需与 diting-core 的名单一致，可先执行 `make sync-semantic-classifier-a-symbols` 再 deploy。

```bash
# 在 diting-infra 根目录：chart config/diting_symbols.txt 已有标的数据时，直接部署即可
make deploy diting prod
```

chart 内 `config/diting_symbols.txt` 仅含注释、无标的时，代码**不会**使用 Mock，会报错「DITING_SYMBOLS 已设置但解析结果为空」并得到 0 条结果。config 中 **timescaleHost / postgresL2Host** 须指向含生产数据的库（集群外库在 diting-prod.yaml 等中覆盖），部署后即连接数据库并使用指定标的的生产数据。

**执行结果输出与 Module B 消费**：见本文档 [上部分·数据流与 L2 规约](#l4-stage3-01-part1-dataflow)（ClassifierOutput 写入 L2/共享存储）与 [02_量化扫描引擎_实践](02_量化扫描引擎_实践.md#l4-stage3-03-classifier-input)（B 按约定读取）。实现 F9 后，A 每批分类结果写入 L2 表 `classifier_output_snapshot`，B 可按 batch_id 或最新批次读取；同进程编排时仍可在内存中传递，与写入 L2 并存。

**若日志出现「从 L1 读取全 A 股标的」而非「指定股票模式」**（即未按指定标的跑），按下列顺序逐项检查实际状态，不得臆断原因：

| 步骤 | 检查项 | 可执行命令 | 预期 |
|------|--------|------------|------|
| 1 | 本地 Chart 渲染出的 ConfigMap 是否含标的 | 在 diting-infra 根目录：`yq eval '{"storage": .stack.storage, "module_a": (.stack.module_a // {}), "schemaInit": (.stack.schemaInit // {"enabled": true})}' config/diting-prod.yaml > /tmp/v.yaml && helm template diting-stack charts/diting-stack -f /tmp/v.yaml 2>/dev/null \| sed -n '/name: diting-semantic-classifier-a-symbols/,/^---/p' \| head -40` | 输出中 `data.diting_symbols.txt` 下应有多行标的代码（如 002371.SZ） |
| 2 | 集群内 ConfigMap 实际内容 | `kubectl get cm diting-semantic-classifier-a-symbols -n default -o jsonpath='{.data.diting_symbols\.txt}' \| head -20` | 应含多行标的代码；若仅两行注释则说明某次 deploy 时 chart 内文件无标的或 Pod 需重启后挂载才更新 |
| 3 | Pod 内环境变量 | `kubectl exec deployment/diting-semantic-classifier-a -n default -- env \| grep -E 'DITING_SYMBOLS|MODULE_AB'` | 应有 `DITING_SYMBOLS=/config/diting_symbols.txt` |
