## Module B 设计目标（与 L1/L2 映射）

本步实现 **Module B 量化扫描引擎（左脑）**，设计目标来自 [01_顶层概念](../../01_顶层概念/01_一句话定义与核心价值.md) 与 [02_战略维度·产品设计](../../02_战略维度/产品设计/01_产品设计维度.md)，须在实践与验收中保持可追溯。

| 来源 | 设计目标 | 本步落实方式 |
|------|----------|--------------|
| **L1 核心公式** | $Alpha = (Quant_{Signal} \cap Router(Experts)) \times Kelly_{Position}$ 中的 **Quant_{Signal}** | Module B 产出量化信号（technical_score、strategy_source、sector_strength），供 Module C/D 消费；输出契约 [QuantSignal.proto](../../03_原子目标与规约/_Design_Artifacts/protocols/quant/quant_signal.proto) |
| **L1 双轨制** | 现金奶牛轨受「不可能三角」约束；量化信号需可解释、可验证 | 策略池与阈值由 YAML 驱动；技术指标统一基于 TA-Lib；Alphalens 质检作为策略池上线前防线（Phase2） |
| **L2 产品设计·功能范围** | 各 Phase 交付能力与 [09_ Module A–F](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md) 对应；Module B = 机会丰度、向量化扫描 | 三大策略池（Trend/Reversion/Breakout）、technical_score > 70、sector_strength > 1.1；全 A 股 universe 全量扫描，标的池与 A 同批一致 |
| **L2 Phase 目标与验收** | 阶段目标可验收、与 L1 价值映射 | 四项 100%（接口、结构、逻辑功能、代码测试）；L5 [l5-mod-B](../../05_成功标识与验证/02_验收标准.md#l5-mod-B)、[l5-stage-stage3_03](../../05_成功标识与验证/02_验收标准.md#l5-stage-stage3_03) 可更新 |

**09_ 规约对 Module B 的定位**：**机会丰度 (Opportunity Abundance)**；每日对当前全 A 股进行向量化扫描，输出技术面得分 > 阈值的候选池；独立可部署单元，见 [09_ Module B 小节](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)。

<a id="l4-stage3-03-classifier-input"></a>
### ClassifierOutput 输入约定（从 L2 / 共享存储读取）

Module B 在独立部署或与 A 分进程运行时，**按约定从 L2 或共享存储读取** Module A 产出的 ClassifierOutput，用于按板块过滤、加权等。表结构/协议由 Stage3 规约，见 [01_语义分类器_实践·上部分·数据流与 L2 规约](01_语义分类器_实践.md#l4-stage3-01-part1-dataflow)。

| 约定项 | 说明 |
|--------|------|
| **数据源** | L2 表 `classifier_output_snapshot`（或约定 Redis/消息队列 key 与序列化格式） |
| **批次** | 按 `batch_id` 或「最新一批」读取，与 Module A 当次写入的 batch_id 一致 |
| **字段** | symbol、primary_tag（AGRI/TECH/GEO/UNKNOWN）、primary_confidence、correlation_id 等，与 [ClassifierOutput.proto](../../03_原子目标与规约/_Design_Artifacts/protocols/classifier/classifier_output.proto) 对齐 |
| **实现** | B 的扫描入口或策略层：在执行前按 batch_id 查询 L2 表（或从共享存储拉取），得到 symbol → Domain Tag 映射；未命中时可按全量标的扫描或降级为无 Tag 过滤 |

本步功能清单中须包含「从 L2/共享存储读取 ClassifierOutput（按约定表或协议）」的实现与单测/集成验证（见 F2、F4 与 V-LOGIC）。

<a id="l4-stage3-03-ab-verify-and-l2"></a>
## 基于 A 模块数据做 B 验证、B 输出写 L2 与一键运行/查询

本步除实现 B 模块扫描逻辑外，须**基于 A 模块处理过的数据**（同源标的池、可选从 L2 读取 ClassifierOutput）执行 B 逻辑并做验证，且将 **B 的输出写入 L2 表**，供稍后 **Module C** 使用；并提供 **make 一键本地运行 B**、**一键验证输出是否符合预期**、**一键查询 B 模块输出数据**。

### 1. 基于 A 处理过的数据做 B 逻辑验证

| 项 | 说明 |
|----|------|
| **标的池同源** | B 与 A 使用**同一份**标的列表：`DITING_SYMBOLS` / `MODULE_AB_SYMBOLS` 或 `config/diting_symbols.txt`（与 [01_语义分类器_实践](01_语义分类器_实践.md) 一致）。建议先执行 `make run-module-a` 使 L2 有 `classifier_output_snapshot`，再执行 B。 |
| **可选读 A 输出** | B 运行时可从 L2 读取 A 的最新一批 `classifier_output_snapshot`（batch_id、symbol 集合），用于展示「基于 A 数据」或后续按板块过滤；单模块验证时可不强依赖 A 已写入。 |
| **验证方式** | `make run-module-b` 执行扫描并写入 L2，脚本内判断「执行标的数=扫描输出条数、L2 写入行数=输出条数」并输出**是否符合预期**；`make verify-module-b` 做功能验证并输出 PASS/FAIL。 |

**说明**：全量结果（含当前分数）始终写入 `quant_signal_scan_all`；通过条数可为 0（阈值 70 时仅得分≥70 写入 `quant_signal_snapshot`）。未配置 TIMESCALE_DSN 或未安装 TA-Lib 时 `make run-module-b` 会直接退出并提示。

**打分与通过**：① **为何 40 分没通过？** 配置里 `technical_score_threshold: 70`，只有得分≥70 的候选才会写入通过表。三池（趋势/反转/突破）打分为 **0 / 40 / 80**（满足两条件 80、满足一条件 40、否则 0），因此**只有 80 分会通过**，40 分不会通过。② **为何很多标的是 0 分？** 0 分表示该标在三个策略池中**均不满足条件**（既没有趋势/反转/突破的一条件 40，也没有两条件 80），属于「已打分、得分为 0」，并非未打分。可通过 L2 表 `quant_signal_scan_all` 查看全量分数。

### 2. B 输出写入 L2 表（通过/未通过分开存放，保存当前分数）

| 项 | 说明 |
|----|------|
| **通过表** | `quant_signal_snapshot`：仅写入**通过阈值**的候选，供 **Module C** 消费。 |
| **全量表** | `quant_signal_scan_all`：写入**全量**扫描结果（含 technical_score、passed）；通过/未通过按 `passed` 区分，可随时查询当前分数。 |
| **标的中文名** | 两表均含 `symbol_name` 列（VARCHAR(128)），运行与查询时一眼可见「标的 + 中文名 + 得分」。来源：`config/symbol_names.csv`（每行 `symbol,name_cn`）或 `diting_symbols.txt` 的「symbol,name」格式；缺失时由 akshare 自动补全。示例见 `config/symbol_names.csv.example`。 |
| **建表脚本** | diting-infra：`schemas/sql/07_l2_quant_signal_snapshot.sql`（含两表）；diting-core：`make init-l2-quant-signal-table`（含已有表补列 symbol_name）。 |
| **写入时机** | `make run-module-b` 在扫描完成后：先写全量到 `quant_signal_scan_all`，再写通过到 `quant_signal_snapshot`；同批共用同一 `batch_id`。 |
| **数据源与依赖** | **始终使用采集模块生产数据**：须配置 `TIMESCALE_DSN`（L1），禁止 Mock。**本地须安装 TA-Lib**：未检测到则脚本退出并提示执行 `make deps-scanner`。 |

### 3. make 一键本地运行 B 与验证输出是否符合预期

| 命令 | 说明 | 通过标准 |
|------|------|----------|
| **make run-module-b** | 加载 .env，**要求 TIMESCALE_DSN（生产数据）与 TA-Lib**；全量结果写 `quant_signal_scan_all`，通过写 `quant_signal_snapshot`；打印全量/通过条数及是否符合预期。 | 退出码 0：全量保存与通过表写入一致；未配置 TIMESCALE_DSN 或未安装 TA-Lib 时退出 1 并提示。 |
| **make verify-module-b** | B 模块功能验证：同源标的跑扫描，校验输出结构符合 QuantSignal 契约、L2 写入行数一致。 | 脚本输出「PASS: B 模块功能验证通过」且退出码 0。 |

**执行顺序建议**：先 `make init-l2-quant-signal-table`（若 L2 已由 infra 初始化可省略）→ 可选 `make run-module-a` → `make run-module-b` → 查看终端「是否符合预期」；再执行 `make verify-module-b` 做功能验证。

### 4. make 一键查询 B 模块输出数据

| 命令 | 说明 |
|------|------|
| **make query-module-b-output** | 查询 L2：**通过表** `quant_signal_snapshot`（供 Module C）与**全量表** `quant_signal_scan_all`（通过/未通过、当前分数）。需 `PG_L2_DSN` 可达。 |

**复现命令汇总**（在 diting-core 根目录）：先 `make deps-scanner`（安装 TA-Lib；本机需系统层 ta-lib C 库）→ 配置 **TIMESCALE_DSN**（L1 生产数据，必须）→ `make init-l2-quant-signal-table` → `make run-module-b` → `make query-module-b-output`；验证：`make verify-module-b`；单测：`make test-scanner`；镜像：`make build-module-b`。

## 实现要求与逻辑细节

### 1. 目录与包结构

### 2. 接口与契约

| 项目 | 要求 | 规约与设计引用 |
|------|------|----------------|
| 输入 | 标的池、OHLCV（L1 或 MarketDataFeed）；**Module A 的 Domain Tag**：从 L2 表 `classifier_output_snapshot` 或约定共享存储按 batch_id/最新批次读取，见 [ClassifierOutput 输入约定](#l4-stage3-03-classifier-input) | [09_ Module B](../../03_原子目标与规约/_共享规约/09_核心模块架构规约.md)、[dna_module_b](../../03_原子目标与规约/_System_DNA/core_modules/dna_module_b.yaml)、[01_语义分类器_实践·上部分](01_语义分类器_实践.md#l4-stage3-01-part1-dataflow) |
| 输出 | `QuantSignal` 列表：technical_score、strategy_source、sector_strength 等，符合 QuantSignal.proto | [QuantSignal.proto](../../03_原子目标与规约/_Design_Artifacts/protocols/quant/quant_signal.proto)；可被 Module C/D 消费 |
| 实现类 | `QuantScanner`，提供扫描接口，输入标的池、OHLCV 与可选 ClassifierOutput（从 L2/共享存储读取），输出 QuantSignal 列表 | 09_ 规约、设计文档 [设计要点](../../03_原子目标与规约/Stage3_模块实践/03_量化扫描引擎_设计.md#design-stage3-03-points) |

**标的池与全量扫描**：执行入口先调用 **`get_current_a_share_universe()`** 获取当前全 A 股列表（或接收调度/调用方传入的同一列表），再对该列表**全部 N 只**全量向量化扫描；与 Module A 使用同一接口或同一传入列表时，由调度保证同批一致。不写死标的数量（如不使用固定 5000）；验证与 03_ 设计一致，验证项可含「universe 来源与 11_/09_ 约定一致」或「同批与 A 使用同一列表」。

### 3. 策略池与 YAML 配置

三大策略池（Trend / Reversion / Breakout）及阈值须由 **YAML 配置** 驱动，与 `dna_module_b.strategy_pools`、`scanner` 语义一致：

| 策略池 | 条件描述（DNA） | 实现要点 |
|--------|-----------------|----------|
| **趋势** | 均线多头排列、MACD 水上金叉等 | 指标用 TA-Lib；条件与 score_range 写 YAML |
| **反转** | RSI&lt;20、价格触及 Bollinger 下轨等 | 同上 |
| **突破** | 价格突破 N 日高点、成交量&gt;2x 均量等 | 同上 |

- technical_score_threshold（如 70）、sector_strength_threshold（如 1.1）从配置或 DNA 引用，禁止硬编码。
- 配置变更后，扫描结果或过滤结果应随之变化（**逻辑功能 100%** 验证项之一）。

### 4. 依赖与构建（部署配套）

- 在 **Dockerfile / requirements** 中显式安装：**TA-Lib**（系统层 C 库 + Python 绑定）、**Qlib**、**VectorBT**、**Alphalens**（见 [设计-依赖与构建](../../03_原子目标与规约/Stage3_模块实践/03_量化扫描引擎_设计.md#design-stage3-03-deps) 与 dna_module_b.integration_packages）。
- 镜像构建后，**镜像内** `make test` 及本步单测须可运行。

### 5. 逻辑填充期开源接入点（必读设计文档小节）

| 接入点 | 实践重点 | 验收要点 |
|--------|----------|----------|
| **TA-Lib** | 全部技术指标（MACD、RSI、Bollinger 等）统一基于 TA-Lib；策略池与 TA-Lib API 映射明确 | 至少一策略池指标 100% 来自 TA-Lib；单测固定 OHLCV 与已知结果对比 |
| **VectorBT** | 全市场粗筛、向量化；technical_score_threshold、sector_strength_threshold 对应 | 给定规模跑通粗筛，输出候选池格式符合下游契约；性能可约定（如 &lt;30 分钟） |
