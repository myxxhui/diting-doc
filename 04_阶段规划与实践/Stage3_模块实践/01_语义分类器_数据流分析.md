# make ingest-production 采集逻辑与 L2 数据流分析（为何模块 A 拿不到）

## 1. make ingest-production 采集逻辑概览

`make ingest-production` 调用 `scripts/run_ingest_production.py`，按顺序执行：

| 步骤 | 开关 | 行为 |
|------|------|------|
| **Step1** | `INGEST_PRODUCTION_UNIVERSE` | 刷新全 A 股标的池 → 写 L1 `a_share_universe` |
| **Step2** | `INGEST_PRODUCTION_OHLCV` | 按批拉取 K 线 → 写 **L1**（TimescaleDB `ohlcv` 等） |
| **Step3** | `INGEST_PRODUCTION_INDUSTRY_REVENUE` | 按标拉取行业/财务 → 写 **L2** |
| **Step4** | `INGEST_PRODUCTION_NEWS` | 按标拉取新闻 → 写 **L2** |

Step3 对每个标的调用 `run_ingest_industry_revenue(sym)`，标的列表来自：

```python
symbols_full = [s.split(".")[0] for s in get_current_a_share_universe(force_refresh=False)]
# 即 ["000001", "000002", ...] 无交易所后缀
```

## 2. L2 数据写在哪里？

行业/营收（industry_revenue）在 L2 的落点有两处：

### 2.1 表 `data_versions`（元数据）

- **写入**：`diting/ingestion/industry_revenue.py` 中 `write_data_version(conn, data_type="industry_revenue", version_id=..., file_path=...)`
- **内容**：仅存**元数据**：`data_type`、`version_id`、`timestamp`、`file_path`、`file_size`、`checksum`
- **file_path** 形如：`l2/industry_revenue/000001.json`
- **重要**：表内**没有**行业名、营收占比等业务字段；实际 JSON 内容也未写入数据库（无 content 列），只是记录了“某版本在某路径”的元信息。**模块 A 无法通过查 data_versions 得到 (行业, 营收占比)**。

### 2.2 表 `industry_revenue_summary`（Module A 输入）

- **写入**：同一文件中的 `_upsert_industry_revenue_summary(conn, symbol, industry_name, revenue_ratio, rnd_ratio, commodity_ratio)`
- **内容**：每标的一行，字段为 `symbol, industry_name, revenue_ratio, rnd_ratio, commodity_ratio, updated_at`
- **用途**：供 Module A 的 L2 provider 按 `symbol` 查询，得到分类所需输入。

**结论**：对模块 A 有用的 L2 数据在 **`industry_revenue_summary`**；`data_versions` 只做版本/路径记录，不能直接用于分类。

### 2.3 JSON 内容「没存进库」——按哪里的设计？是否需要存？

- **规约依据**：[07_数据版本控制规约](../../03_原子目标与规约/_共享规约/07_数据版本控制规约.md)（L3）。其中约定：
  - **元数据**（版本 ID、时间戳、路径、大小、校验和）→ 存 **L2 PostgreSQL `data_versions`**；
  - **数据内容**（实际文件/快照）→ 存 **L3 Cold (OSS + DVC)**，通过 `file_path` 与元数据关联。
  - 即：规约**没有**要求把 JSON 内容存进**数据库**，而是要求把内容存到 **OSS**，数据库只记「在哪里」。
- **当前实现**：采集只写了 `data_versions` 元数据与 `industry_revenue_summary` 汇总表；**没有**把原始 JSON 上传到 OSS，也没有在 DB 中增加 content 列。因此「JSON 没存进库」有两层含义：
  1. **没进 DB**：符合 07_ 对 L2 的约定（L2 只存元数据）；
  2. **也没进 OSS**：与 07_ 的完整 DVC 流程不一致，即**原文/快照目前未落盘**，无法按版本回溯原始 payload。
- **是否需要存**：**需要**。规约的目标是「决策可回溯、数据可复现」，所以原始采集结果应当保留。可选做法：
  - **按 07_ 做全**：将 industry_revenue 的 JSON 写入 OSS（或通过 DVC 管理），`file_path` 指向 OSS 路径；回溯时通过元数据查路径再拉取文件。
  - **简化版**：若暂不接 OSS，可在 L2 增加 `data_versions` 的 `content` 列（或单独表）存 JSON 原文，用于审计与回溯，与 07_ 的「内容在 L3」有偏差但可接受。

**当前缺口小结**：采集阶段本应落盘的「数据内容」目前未存（既未上传 OSS，也未入库），导致无法按版本回溯原文。Module A 已通过 `industry_revenue_summary` 表正常分类，但审计/回放仍依赖补齐内容存储。

### 2.4 若内容存 OSS：保留时长与删除是否方便？

按 07_ 将数据内容放到 OSS 后，**更新保留策略、按时长清理、删除**都较方便：

| 操作 | 实现方式 | 是否方便 |
|------|----------|----------|
| **按保留时长自动过期** | OSS 生命周期（Lifecycle）规则：按前缀（如 `industry_revenue/`）配置「多少天后转为低频/归档」或「多少天后删除」 | ✅ 改 Bucket 生命周期配置即可，如 3 个月→10 个月、60 天删除等，无需改业务代码 |
| **按时间范围删除** | 用 OSS API/控制台按前缀 + 列出对象后按 last-modified 过滤，批量删除；或生命周期按「创建 N 天后删除」 | ✅ 前缀+时间即可批量处理 |
| **临时清空某类数据** | 按前缀删除（如删除 `industry_revenue/` 下全部对象） | ✅ 控制台或 API 一键/脚本即可 |
| **调整保留策略** | 修改同一 Bucket 的生命周期规则（如 90 天→300 天），仅影响新生效的对象；已有对象可配合规则更新或单独跑一次任务 | ✅ 配置驱动，无需迁库 |

**结论**：内容放 OSS 时，保留时长（3 个月、10 个月、60 天等）和删除、清理都通过**生命周期 + 前缀**即可管理，后续变更策略只需改配置或跑一次性任务，**操作方便**。若选「简化版」在 DB 存 content，则保留与删除需自行写 SQL/定时任务按 `timestamp` 清理，灵活性略逊但实现简单。

## 3. 为什么模块 A 没拿到？

### 3.1 标的代码格式不一致（根本原因）

| 环节 | 使用的 symbol 格式 | 来源 |
|------|-------------------|------|
| **ingest-production Step3** | `sym` = **无后缀**（如 `000001`） | `symbols_full = [s.split(".")[0] for s in get_current_a_share_universe()]` |
| **industry_revenue_summary 写入** | 存的是传入的 `sym` | 即 **`000001`** |
| **Module A universe** | **带后缀**（如 `000001.SZ`） | `get_current_a_share_universe()` 直接返回表内 symbol |
| **Module A L2 查询** | 用 `000001.SZ` 查表 | `l2_data.get("000001.SZ", missing)` → 表里只有 `000001` → **对不上** |

因此：**采集写的是 `000001`，模块 A 查的是 `000001.SZ`，键不一致，模块 A 拿不到任何一行。**

### 3.2 其他可能原因

- **未执行 Step3**：若 `.env` 中未开启 `INGEST_PRODUCTION_INDUSTRY_REVENUE=true`，则不会写 `industry_revenue_summary`。
- **表未建**：若未执行 L2 的 `03_l2_industry_revenue_summary.sql`（或等价本地 init），表不存在，upsert 会报错（当前实现里被 try/except 吞掉，只写 data_versions）。
- **历史采集**：在代码里增加 `industry_revenue_summary` 写入与 Module A 读取之前跑的采集，只写了 `data_versions`，没有写汇总表。

## 4. 修复与约定

- **约定**：全链路统一使用 **带交易所后缀** 的 symbol（如 `000001.SZ`），与 `get_current_a_share_universe()` 及 Module A 的 universe 一致。
- **采集侧**：Step3/Step4 使用「完整标的列表」`get_current_a_share_universe(force_refresh=False)`，向 `run_ingest_industry_revenue(sym)` 传入带后缀的 `sym`；`industry_revenue.py` 内调用 AkShare/JQ 时再转为无后缀（如 `sym.split(".")[0]`），**写入 `industry_revenue_summary` 时仍用传入的完整 symbol**。
- **模块 A**：已按带后缀的 universe 从 `industry_revenue_summary` 批量查询；修复采集侧后即可命中。

修复后，再跑一次 `make ingest-production`（或至少 Step3），模块 A 应能拿到 L2 行业/营收数据。
