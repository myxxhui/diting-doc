### 可追溯路径（缓解措施落地路径）

凡涉及**风险缓解、约束落实、执行保证**的表述（如阶段 2 风险评估的「缓解」、阶段 3 的补充方案），须同时给出**相关路径**，便于落地与审计。

| 路径类型 | 含义 | 示例 |
|----------|------|------|
| **规约路径** | 在何文档、何小节/锚点约定该约束或默认行为 | `11_数据采集与输入层规约.md` 的「全 A 股标的池」小节；`09_` Module A/B 的输入与编排描述 |
| **实践路径** | 在何 04_ 步骤、编排或执行流程中落实（谁在何时执行） | `04_/Stage3_模块实践/01_…` 的「依赖检查」；统一入口先获取 universe 再传入 A、B |
| **实现路径** | 在何代码/配置/接口中实现（若已确定） | `get_current_a_share_universe()` 实现；universe 表与更新 Job |

**强制要求**：阶段 2 的「风险评估」、阶段 3 的「补充优化方案」中，每条**缓解**或**建议动作**须至少包含下列之一且可定位：规约路径（文档+小节/锚点）、实践路径（04_ 或编排）、实现路径。不得仅写「在 11_ 约定…」而无具体小节或锚点；若涉及执行顺序或同批一致性，须写明实践路径（谁在何时调用/传入）。

## 使用方式

| 方式 | 用户 | AI |
|------|------|-----|
| **一键四阶段** | 描述需求+原始信息，说「按 @本规约 直接四阶段都执行」 | 先执行阶段 1→2→3，每阶段用 `--- 第N阶段输出 ---` 分隔；**阶段 3 结束后暂停**（见下「阶段 3 后暂停」），不自动进入阶段 4；收到用户回复「继续」或「执行阶段 4」后再执行阶段 4 产出待办清单 |
| **指定阶段** | 「只执行第 N 阶段」或「从第 N 到第 M 阶段」；无上文则粘贴 | 仅执行指定阶段；执行到阶段 4 则产出待办清单 |

**用户后续**：审待办清单 → 在 **diting-doc** 内按文档操作清单改文档 → 以 04_ 步骤为唯一入口在 **diting-core/diting-infra** 上按实践任务清单执行或 build，准出更 L5；问题依同一 04_ 步骤修正。按待办执行后须满足 [00_系统规则 §4.7](../00_系统规则_通用项目协议.md#47-新要求设计变更的-111-反写与可执行性验收) 的可执行性验收（仅凭 04_ 步骤与约定输入可完成全部验证、不偏离不臆造）；必要时做 1:1:1 核对。

**阶段 3 后暂停（避免切换太早/太晚）**：一键四阶段时，AI 在输出完阶段 3 后**必须暂停**，并输出以下提示（或等价表述）：*「阶段 1～3 已完成。请先切换到 Plan 模式，切换后回复「继续」或「执行阶段 4」，我将产出待办清单。」* 只有在用户**再次发送消息**（如「继续」）后，AI 才执行阶段 4。这样您有明确时间窗口切换，不会出现未切就跑完或切晚的情况。

**模式约定**：阶段 1～3 用 **Ask**（只读）；阶段 4 需 **Plan**（Agent）以便按清单执行。Cursor 无法自动切换模式，故用「阶段 3 后暂停」留出切换时间。若希望节奏完全自主，可**两轮提问**：第一轮 Ask 下「按本规约执行阶段 1～3，结束后暂停」；您切到 Plan 后第二轮「继续，执行阶段 4」。

## 阶段 1：需求精准表达

**目标**：原始需求 → 结构清晰、边界明确的需求文档。

| 项 | 内容 |
|----|------|
| **输入** | 用户需求描述与原始信息（可口语化）；可选：文档路径、上下文、优先级 |
| **处理** | 语义理解与结构化拆解；去冗余、消歧义、定边界与关联；涉 03_/04_ 时标出 L2/L3/L4 关联 |
| **输出** | 需求概述（一句+背景+预期成果）；需求详情清单（序号/需求项/描述/优先级/备注）；关联关系说明；待确认关键点（序号/待确认点/确认方式） |
| **自检** | 需求项无歧义可验证、边界清晰、待确认点可操作 |

**目标**：需求 → 完整技术方案 + 映射到 diting-doc 内具体文档与位置（03_ 设计、_System_DNA、04_、05_）。

| 项 | 内容 |
|----|------|
| **输入** | 本对话「第一阶段」输出；无则用户粘贴；可注明关联 Stage/步骤 |
| **处理** | 每需求项：前端方案、后端/接口/数据/配置与部署；每决策对应文档位置（03_/_System_DNA/04_/05_）、变更类型与要点；涉既有 Stage 时 1:1:1（设计 doc、DNA、04_、L5）；与 [03_ README 索引](../03_原子目标与规约/README.md) 一致 |
| **输出** | 技术方案总览；前端方案（变更点+依赖接口/数据）；后端与数据变更（API/Proto/数据模型/逻辑与 09_ 对应）；文档变更清单（路径相对 diting-doc/、类型、摘要、需求项；到文件或 #design-*/#l4-*/#l5-*）；前后端职责；**风险评估**（风险点/等级/缓解/**相关路径**）。每条缓解须带**相关路径**：规约路径（文档+小节或锚点）、实践路径（04_ 或编排）、实现路径（若适用）；至少填其一且可定位，不得仅写「在 XX 约定」而无具体位置。风险评估表列：风险点 / 等级 / 缓解 / 相关路径（规约路径、实践路径或实现路径至少其一） |
| **自检** | 每需求有方案与文档变更；路径可定位 03_/04_/05_；涉 Stage 则 1:1:1；风险有等级与缓解，**且每条缓解均带相关路径**（规约/实践/实现至少其一、可定位）；**文档变更清单中的 03_/04_/05_ 路径已与 [03_ README 设计–DNA–实践–L5 索引](../03_原子目标与规约/README.md) 及 dna_dev_workflow 核对后再输出** |

## 阶段 3：深度批判与补充优化

**目标**：审视前两阶段，发现遗漏/冲突/优化点 → 整合需求与执行建议。

| 项 | 内容 |
|----|------|
| **输入** | 本对话「第一+第二阶段」输出；无则粘贴；可补维度（合规/性能/可观测性等） |
| **处理** | 完整性/可行性/风险/可扩展性/与文档体系一致性 批判分析；发现不足与遗漏并补充；涉关键重构时联动 00_系统规则 §4.5 |
| **输出** | 问题分析报告（类别/问题/位置/严重程度）；**补充优化方案**（序号/内容/对应需求或文档/建议动作/**相关路径**：规约路径、实践路径或实现路径，至少其一可定位）；最终整合需求表达（在阶段 1 结构上修订，标新增/删除）；执行建议（顺序与依赖、§4.5 联动、单测/回滚/发布）。对阶段 2 风险评估中**未带相关路径或路径模糊**的缓解，在阶段 3 补充优化方案中**补全相关路径**。 |
| **自检** | 前两阶段假设与结论已审视；补充对应到需求或文档；**阶段 2 风险评估中每条缓解已带相关路径，阶段 3 补充方案中建议动作已带或已补全相关路径**；最终需求可作阶段 4 输入；执行建议与 workflow_stages 兼容 |
