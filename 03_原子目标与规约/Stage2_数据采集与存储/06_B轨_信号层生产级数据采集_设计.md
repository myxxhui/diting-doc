## 一、范围与边界

- **本步唯一实现归属**：采集模块的**信号层生产级数据采集**（按候选标的解析细分、按细分拉取原始摘要、信号理解打标、写入 segment_signal_cache）在 **Stage2-06_B轨** 实现与验收；不分散在 Stage3/Stage4 或 C 模块内。
- **上游依赖**：[01_B轨_数据与采集_实践](../../04_阶段规划与实践/Stage2_数据采集与存储/01_B轨_数据与采集_实践.md) 准出（标的主营构成表、细分注册表就绪；可选本步建 segment_signal_cache 或 01 步已建）。
- **下游消费**：Module C（右脑）从 L2 细分信号缓存表读取 segment_signals，调用 `unified_opinion` 时由编排或调用方传入；C 不实现拉取逻辑，见 [04_B轨_C模块右脑与信号设计](../Stage3_模块实践/04_B轨_C模块右脑与信号设计.md)。

<a id="design-stage2-b06-layers"></a>
## 二、信号层三层与职责

| 层级 | 职责 | 输出/存储 |
|------|------|-----------|
| **信号适配器** | 按 **segment_id** 从**L2 已采集数据**（news_content 等）拉取原始摘要/文本；**不调用 AkShare 等远程源**，只读采集写入的 L2 表；由人写代码或配置 | 原始摘要或文本，直接进信号理解 |
| **信号理解层** | 对原始内容做**理解+打标**：规则或 AI 产出 JSON（type, direction, strength, summary_cn, risk_tags）；失败时写 direction=neutral、strength=0.5 | 写入 **segment_signal_cache**（segment_id, signal_summary=JSON, signal_at, fetched_at, ttl_sec） |
| **信号层（编排）** | 候选标的列表确定后，解析涉及细分 S，对 S 中每个 segment_id 若缓存未过期则跳过，否则调「适配器拉取+信号理解打标」→ 写 segment_signal_cache | 缓存表就绪，供 C 读取 |

- **按细分去重**：同一 segment 在 TTL 内只拉取并打标一次，多标共享；见 12_ §3.2。
- **新细分/新源**：新细分在 segment_registry 注册，新源在 signal_adapter_registry 挂接适配器（人写代码），见 12_。
- **多层级 segment（三层，可扩展）**（见 [12_ §2.2](../_共享规约/12_右脑数据支撑与Segment规约.md)）：
  - **约定为三层**：**第 1 层 domain 级**（如 tech/agri/geo）、**第 2 层 板块级**（如 tech_ai/tech_semiconductor）、**第 3 层 具体业务级**（如 tech_ai_factory/tech_ai_compute）。标的主营构成表可为同一标的挂此三层的 segment_id，一次 refresh 拉取「大方向 + 板块 + 具体业务」。
  - **适配器按层级**：第 1 层 → 大盘/指数/宏观数据源；第 2 层 → 板块指数、板块新闻；第 3 层 → 垂直政策/研报/产品；仍为一个 segment_id 一个或一组数据源，人写适配器。
  - **扩展**：层级抽象由 `diting.ingestion.segment_tier`（数值 1/2/3）与 `segment_registry.segment_tier` 列实现；`refresh` 将 tier 映射为 `model_override_by_tier` 的 `domain`/`sector`/`business` 键。无库内 tier 时对 `seg_bp_*` 默认按业务层处理。后续可在不改核心流程下支持第 4 层（如子业务/产品线）等。

<a id="design-stage2-b06-interface"></a>
## 三、接口与表

### 3.1 接口约定（与 12_ §3.2 一致，扩展返回）

- **方法名**：`refresh_segment_signals_for_symbols(symbols: List[str], ...) -> RefreshSegmentSignalsResult`（或等价结构；为便于调用方排查，**建议有返回值**）。
- **输入**：候选标的列表（及可选：batch_id、options）。
- **输出**（建议）：结构化结果，便于知道「哪些标的没有哪些数据」：
  - **symbols_without_segments**：在 symbol_business_profile 中无行的标的列表（无主营构成/无细分映射）；
  - **segments_without_adapter**：本次涉及的 segment_id 中，在 segment_registry 无 signal_adapter_id 或适配器未注册的列表；
  - **segments_skipped_ttl**：因缓存未过期而跳过的 segment_id 列表；
  - **segments_written**：本次写入 segment_signal_cache 的 segment_id 列表；
  - **segments_failed**：拉取或打标失败的 segment_id 及原因（可选列表/字典）；
  - **summary**：如 total_symbols、symbols_with_segments、segments_fetched、segments_cached 等计数。
- **副作用**：更新 L2 表 `segment_signal_cache`（及可选 segment_signal_audit）。
- **内部步骤**：解析标的→细分集合 S（查 symbol_business_profile）→ 去重 → 对 S 中每个 segment_id 若未过期则跳过，否则适配器拉取 → 信号理解打标 → 写 segment_signal_cache；并汇总上述缺数据/跳过/写入/失败信息填入返回值。

### 3.2 存储表

- **segment_signal_cache**：见 [12_ §2.3](../_共享规约/12_右脑数据支撑与Segment规约.md) DDL；本步若 01 未建则在本步建表。
- **signal_adapter_registry**（可选 L2 表或 YAML 配置）：见 12_ §2.3；适配器与 segment 的绑定。
- **segment_signal_audit**（可选）：segment_id, source_type, source_url, source_publish_time, raw_snippet, model_conclusion_json, created_at；每次打标写一条，便于复查与复用，见 [04_B轨_C模块右脑与信号设计 §2.3](../Stage3_模块实践/04_B轨_C模块右脑与信号设计.md)。

### 3.3 信号摘要 schema 与校验（与 C 消费一致）

- **signal_summary** 固定 JSON 结构（实现必须校验，不符合即视为打标失败，走 fallback）：
  - **type**：string，枚举 `policy` | `price` | `order` | `rnd`；
  - **direction**：string，枚举 `bullish` | `bearish` | `neutral`；
  - **strength**：number，闭区间 [0.0, 1.0]；
  - **summary_cn**：string，中文摘要，非空；
  - **risk_tags**：string[]，可选风险标签，建议取自配置 risk_tags_enum。
- **校验规则**：缺字段、类型错误、枚举/范围越界一律视为失败，触发 fallback（写 audit 后写 cache 为 direction=neutral、strength=0.5）；见 04_A轨_MoE议会_设计「细分信号解析约定」。

<a id="design-stage2-b06-signal-understanding"></a>
### 3.4 信号理解层：规则与 AI 实现逻辑要求

- **实现要求**：支持**规则/关键词**打标与 **AI 打标**双路径，产出上述固定 JSON；**默认**采用「规则优先、AI 补充」或可配置模式（见下「信号理解配置项」）。输出须经 schema 校验；失败必 fallback（写 cache 为 neutral/0.5），保证 C 可跑。
- **AI 实现逻辑要求**：AI 路径可配置；支持多种模式（见下「信号理解配置项」）；可配置 prompt 模板、model、token 上限、重试与退避；可按 tier 或 segment 选策略或 model；输出严格 schema 校验；审计支持，且支持「同 segment 同日复用」以减少重复调用。
- **分析边界**（规则与 AI 共用）：仅对「与该细分相关的一手信息」做利好/利空/中性 + 强度 + 风险标签；**禁止**预测价格、输出买卖建议、分析无关噪音；prompt 中须明确禁止模型输出上述内容。

<a id="design-stage2-b06-config-trigger"></a>
## 四、配置、触发、输出与写入、边界（实现必读）

- **配置**：segment_registry、signal_adapter_registry 来源为 L2 表或 YAML（路径/表名可配置）；信号理解层 **mode、prompt、schema、model** 等由配置指定；**通用边界**与**信号理解（规则与 AI）**配置项见下表，实现时从 YAML/ENV 读取。
- **触发**：两种方式二选一或并存——(1) **采集侧 cron/Job**：定时或事件触发，从 L2 读取 `b_track_candidate_snapshot` 本批 symbol 列表，调用 `refresh_segment_signals_for_symbols(symbols)`；(2) **编排/流水线**：**推荐** `make run-full-pipeline`（A→B→refresh→C 串联）；或 diting-core 日批流水线中 B 写表后同一流程内调用 refresh，再调 C。触发方式与参数（如 cron 表达式、batch_id）由配置或部署约定。
- **输出与写入**：写入 `segment_signal_cache`（及可选 `segment_signal_audit`）；接口**返回** RefreshSegmentSignalsResult（见 §3.1）；打标失败时写 direction=neutral、strength=0.5 仍写 cache，保证 C 可跑。
- **通用边界与配置项**（AI 边界 6.4 落地）：

| 配置项 | 说明 | 建议默认 | 用途 |
|--------|------|----------|------|
| **signal_layer.max_input_chars** | 单次信号理解输入最大字符数 | 4096 | 输入边界，控 token 成本 |
| **signal_layer.strength_range** | strength 合法范围 | [0.0, 1.0] | 输出边界校验 |
| **signal_layer.risk_tags_enum** | risk_tags 建议枚举（可选） | 如 ["高风险","中风险","低风险"] | 输出边界、C 降权一致 |
| **signal_layer.fallback_on_failure** | 打标失败时是否写 neutral | true | 保证 C 可跑 |
| **signal_layer.audit_enabled** | 是否写 segment_signal_audit | false/true | 审计与复查 |

- **信号理解（规则与 AI）配置项**（实现时须可配置，默认见下表）：

| 配置项 | 说明 | 建议默认 |
|--------|------|----------|
| **signal_understanding.mode** | 模式：rule_only / ai_only / rule_first_then_ai / ai_first_fallback_rule | rule_first_then_ai |
| **signal_understanding.prompt_path** 或 **prompt_template_key** | Prompt 模板文件路径或配置键 | config/prompts/segment_signal_v1.md |
| **signal_understanding.schema_version** | 输出 schema 版本，用于校验 | v1 |
| **signal_understanding.model_id** 或 **provider** | AI 模型标识或 provider | 使用 AI 时必配 |
| **signal_understanding.max_input_tokens** | 单次请求最大输入 token | 1024 |
| **signal_understanding.max_output_tokens** | 单次请求最大输出 token | 256 |
| **signal_understanding.retry_times** | 调用失败重试次数 | 1 |
| **signal_understanding.retry_backoff_sec** | 重试退避秒数 | 2 |
| **signal_understanding.model_override_by_tier** | 按 tier 选用不同 model（可选） | {} |

- **审计与复用**：segment_signal_audit 写入时机为每次信号理解调用（含 AI 失败）；字段至少含 segment_id、source_type、source_url、source_publish_time、raw_snippet、model_conclusion_json、created_at。**同 segment 同日复用**：同一 segment_id、同一自然日（或同一 batch_id）若已有成功打标结论，可优先复用该结论写 cache、不再调模型，具体策略由实现与配置约定（如 audit_reuse_same_day=true）。

- **按轨区分**：`DITING_TRACK=a`（A 轨短线）时从 `quant_signal_snapshot` 取 symbols，信号层用 `tracks.a_track` 配置（行业+标的新闻）；`DITING_TRACK=b`（B 轨中线）时从 `b_track_candidate_snapshot` 取 symbols，用 `tracks.b_track` 配置（可扩展周全垂直信号）。配置 `config/signal_layer.yaml` 的 `tracks.a_track` / `tracks.b_track` 可按轨覆盖 understanding、adapter_prefixes 等。
