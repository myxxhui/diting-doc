<a id="design-stage2-06-goal"></a>
## 本步目标

定义数据采集**生产级**约束与验收项，使 Module A、Module B 生产运行可验收；历史深度、复权、标的覆盖、行业/板块口径与 11_、06_dna 一致，并与 L1 保留策略衔接。

<a id="design-stage2-06-points"></a>
## 设计要点

### OHLCV（生产级）

- **历史深度**：单标的日线 **≥5 年**（明确为「**单标的日线可追溯长度**」）；与 MA60、20 日高点/均量等指标所需最少 60 根日线兼容。5 年与 **L1 数据保留/归档策略一致**（保留周期见基础设施或 DNA）。
- **主粒度**：日线；可选粒度：分钟（按需求）。
- **复权**：与**回测、实盘一致**；实现与 Stage4 回测/实盘文档或配置对齐。

### 标的覆盖与运维一致

- 生产扩展时标的范围与 `get_current_a_share_universe()` 一致。
- 覆盖率、数据新鲜度、采集失败告警与**运维/告警**约定一致（见运维手册或 DNA）。

### 行业/营收与板块映射

- **Module A**：申万层级、营收占比维度等**最小字段集**在规约中列出或引用（见 11_ 与 Stage2-02）。
- **Module B Sector Strength**：板块划分与 Module A 行业数据**同源**（如申万），同表或同接口；由 L2 或约定表供给，避免 A/B 口径不一致。

### Module C（右脑）与 L2 快照一致性（衔接 Stage3-04）

- **生产右脑输入（DITING_TRACK=a）**：**L2 `a_track_signal_cache`** 提供「标的新闻/公告 + 申万行业新闻/公告」双路打标（`sym:*` / `ind:*`）；Module C 与 `segment_signal_cache` 合并，主营 `segment_id` 无 B 轨缓存时用 A 轨标的级信号填充。详见 [07_行业新闻与标的新闻分离存储_设计](07_行业新闻与标的新闻分离存储_设计.md)、`diting.signal_layer.a_track_refresh`、`diting.moe.a_track_signal_reader`。
- **生产右脑输入（DITING_TRACK=b）**：细分垂直信号以 **L2 `segment_signal_cache`** 为准，与 [12_右脑数据支撑与Segment规约](../_共享规约/12_右脑数据支撑与Segment规约.md) 一致；**禁止**在默认配置下以占位 stub 冒充生产意见。
- **批次对齐**：`classifier_output_snapshot` 与 `quant_signal_scan_all` 应对同一跑批/业务日一致；Module C 写入 **`moe_expert_opinion_snapshot.moe_run_metadata`** 记录批次 id 与对齐告警，便于与 OHLCV/采集新鲜度审计并列查验。详见 [04_研究议会与Agent编排_实践 · 生产级运行与 L2 元数据](../../../04_阶段规划与实践/Stage3_模块实践/04_研究议会与Agent编排_实践.md#l4-stage3-04-prod-moe-metadata)。

<a id="design-stage2-06-prod-data-env"></a>
## 生产数据环境与两段实现

本步通过「生产数据环境」在云端（竞价实例 + 独立数据盘 + K3s + 数据库）实现生产级数据的承载与验收，并拆为**前半**（环境与数据继承）与**后半**（采集逻辑生产级 + 落库 + 数据继承）两段，每段各有核心目标实现验证。实现路径：**diting-infra** 的 Make 与脚本（`make deploy diting prod`、`make down diting prod`）；配置为 `config/terraform-diting-prod.tfvars`、`config/diting-prod.yaml`（部署 PG/Redis、K3s 存储等由 YAML 控制）；Up 后由 `scripts/prod-write-conn.sh` 将连接信息写入约定文件 `prod.conn`，供本步与 01_/02_ 使用。

### 关键约定（C1 / C2）

- **C1 数据盘 ID 持久化**：数据盘 ID 的持久化方式由 **diting-infra 约定**（如 Terraform state 隔离或约定文件 `prod.disk_id`），Down 后保留供再次 Up 挂载同一块盘。
- **C1a prod 单独 NAS**：prod 使用独立 NAS，tfvars 中不设置或设置 `nas_use_existing_access_group = false`，Terraform 会创建 prod 专用 File System、Access Group（`diting_nas_group_prod`）及 Mount Target。`FULL_DESTROY=1 make down diting prod` 会按依赖顺序销毁，无需 state rm。
- **C1b 后续环境复用 prod NAS**：后续若有其他 diting 环境（如 prod2、staging）需复用 prod 的 NAS，在其 tfvars 中设置 `nas_use_existing_access_group = true`、`nas_existing_access_group_name = "diting_nas_group_prod"`。该类环境执行 `FULL_DESTROY=1 make down` 前，须从 state 移除 `module.nas.alicloud_nas_access_group.main[0]`（或由 diting-infra Make 按 tfvars 自动处理），避免误删 prod 共享资源导致 `InvalidAccessGroup.AlreadyAttached`。
- **C2 云与规格**：云厂商、竞价实例与数据盘规格 **与 Stage1-03 一致或使用单独 tfvars**（如 `config/terraform-diting-prod.tfvars`），见 [02_基础设施与部署规约](../../节奏与交付/02_基础设施与部署规约.md)。

### 前半部分：环境与数据继承

- **目标**：Up = 竞价实例 + 独立数据盘 + K3s + TimescaleDB/L2/Redis（数据落该盘）→ NodePort + 公网 IP → 输出连接信息；Down = 仅回收 ECS、K3s、EIP，**保留数据盘不删**；再次 Up = 新起竞价实例挂载同一块数据盘，K3s 与数据库继续使用该盘上的数据，实现数据继承。
- **前半核心验证（三项）**：① Up 成功后环境可用、连接信息正确，可从 diting-core 使用 `prod.conn` 连通 DB（如 `make verify-db-connection`）；② Down 后数据盘未删除，云上可查或再次 Up 可挂载；③ 再次 Up 后挂载同一块盘，K3s+DB 启动后能访问到与 Down 前一致的数据（如 psql 查表行数或样例一致）。

<a id="design-stage2-06-ingest-image"></a>
### 采集镜像与 ACR/Chart 配置（C3 配套）

- **镜像仓库（ACR）**：本地构建的采集镜像须推送到项目指定的 **ACR**（阿里云容器镜像服务个人版）；地址与凭证见 [06_ 实践#镜像仓库（ACR）与本地构建推送](../../../04_阶段规划与实践/Stage2_数据采集与存储/06_生产级数据要求_实践.md#l4-stage2-06-acr-and-push)。**diting-core** 提供 **`make push-images`**（构建 + 打 ACR tag + 推送），一键完成；密码通过 `DITING_ACR_PASSWORD` 或 docker login 配置，勿写入文档或版本库。
- **Chart 使用最新镜像**：**diting-infra** 的 **config/diting-prod.yaml** 中 **`stack.ingest.image`** 配置为 ACR 全路径且 **tag 为 `latest`**、**pullPolicy 为 `Always`**，即每次部署/重建 Pod 时拉取仓库最新镜像；不在 values 中写死版本号，本地 `make push-images` 后重新部署即可生效。
- **拉取凭证（imagePullSecrets）**：ACR 为私有仓库时，K3s 需拉取凭证。**部署类 manifest 仅放在 charts/ 下**（见 diting-infra 目录边界）：**charts/diting-stack/manifests/acr-pull-secret.yaml**（含凭证，已 .gitignore）、**acr-pull-secret.yaml.example** 示例；部署前 apply 到集群或执行 **`make apply-acr-pull-secret`**。**config/** 仅放配置（如 diting-prod.yaml），**禁止**在 config/ 下放置 K8s 部署用 manifest。

### 后半部分：采集生产级 + 落库 + 数据继承

- **目标**：先在 **diting-core** 实现采集逻辑达到生产级设计目标（单标日线≥5 年、标的=universe、复权、universe 表+Job 等）；再在同一套 Up 流程中，**每次部署时**由 **diting-infra** 的 **diting-stack** Chart 通过 post-install/post-upgrade Hook 在 K3s 内创建采集 **Job**，Job 执行 **`make ingest-deploy`**（diting-core 的 `scripts/run_ingest_deploy.py`）：检查 L1 是否为空或数据过期（可配天数阈值），**自动选择全量或增量**；连接信息由 Chart 生成 Secret、集群内 Service DNS。采集镜像由 **`stack.ingest.image`** 指定（见上「采集镜像与 ACR/Chart 配置」），须已通过 **`make push-images`** 推送到 ACR 且集群已配置 imagePullSecrets。NodePort+公网 IP+`prod.conn` 仍供本机/CI 验证；Down 仍只回收 ECS/K3s/EIP、保留数据盘；再次 Up 挂载同盘后，库内已有此前落库的生产级数据（**C3**：落库触发方式为**每次部署时 K3s Job 执行 make ingest-deploy（检查后全量/增量）**；可选保留本机 REPO_I_ROOT + make ingest-test 用于本地/CI 验证。ingest 失败时是否仍输出连接信息、是否视为 Up 失败由 06_ 实践约定）。
- **推荐执行顺序**（06_ 实践细化）：① 先写采集逻辑与数量质量要求 → ② 部署环境与依赖组件 → ③ 验证依赖组件是否具备数据继承能力（先写若干标的真实数据→回收→再部署→确认继承）→ ④ 若继承成立则按采集逻辑全量采集，并对采集数据做数据质量验证。
- **后半核心验证（三项）**：① 采集逻辑生产级设计目标在 code 层验证通过（如 `make verify-data-production` 或 02_ 目标数据约定）；② 首次 Up+执行采集后，库内具备生产级数据（L1 单标≥5 年、universe 等）；③ Down 后再次 Up 挂载同盘，库内数据仍在，且生产级验证仍通过（数据继承+生产级数据同时成立）。

- L5 02_验收标准 中 stage2_06 行已更新。
