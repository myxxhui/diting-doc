## 1. 内容理解服务（content_comprehension_service）

### 主责
- 从 L2 文本（公告 / 新闻 / 研报）抽取实体、事件、主营关联、产业链关系
- 对齐到统一的 `subject` ID（标的 / 行业 / 主营 segment）
- 构建时间线索引

### 输入
- L2 文档（PDF / HTML / 结构化 JSON）
- 旧版抽取结果（增量更新）

### 内部组件
```
┌─────────────────────────────────────────────┐
│ DocPreprocessor (清洗 / 分段 / OCR / 编码)  │
│         ↓                                    │
│ EntityExtractor (NER；公司/人/产品/事件)    │
│         ↓                                    │
│ EventClassifier (事件分类 + 时间锚点)        │
│         ↓                                    │
│ SubjectAligner (实体 → 标准 subject_id)     │
│         ↓                                    │
│ SegmentAligner (主营 → segment_id)          │
│         ↓                                    │
│ TimelineBuilder (事件 → 标的时间线)          │
│         ↓                                    │
│ Persister (写 L2 派生表 + 嵌入索引)         │
└─────────────────────────────────────────────┘
```

### 输出
- L2 派生表：`comprehension_entities` / `comprehension_events` / `subject_segment_map`
- 向量索引（OpenSearch / Milvus）

### 失败模式与保护
| 失败 | 保护 |
|------|------|
| 推理网关超时 | 退化到规则抽取 + 标注 `degraded=true` |
| 实体对齐失败 | 写入 `unresolved_entities` 表，议会可见但权重降级 |
| 编码异常 | 直接调极寒防御 `input_sanitizer` |

## 2. 信号特征工程（signal_feature_engine）

### 主责
- 把"理解后的事件 + 行情 + 知识"转为可被议会推理使用的特征
- 维护**特征仓库（Feature Store）**：批特征（每日 / 每周）+ 流特征（实时）

### 内部组件
```
┌──────────────────────────────────────────────┐
│ BatchPipeline (Spark / Ray)                   │
│   ├─ 行情衍生（动量 / 波动 / 流动性）         │
│   ├─ 事件聚合（窗口计数 / 主营加权）          │
│   ├─ 行业 / 产业链聚合                        │
│   └─ 嵌入向量（Embedding）                    │
│         ↓                                     │
│ FeatureStore (写入 Feast 或自建)             │
│                                                │
│ StreamPipeline (Flink / Faust)                │
│   ├─ 实时事件触发                              │
│   ├─ 跨源关联                                  │
│   └─ 写入 hot 区                               │
└──────────────────────────────────────────────┘
```

### 输出
- Feature Store 中可被议会按 `subject_id + ts` 查询的特征
- 触发式特征事件（流式）→ 议程编排器

## 3. 研究议会服务（research_council_service）

### 主责
- 编排多个 Agent / Expert 完成议题；含工具调用、检索、人审钩子
- 输出 `ResearchCard / Candidate / CouncilSession`

### 内部组件
```
┌────────────────────────────────────────────────┐
│ SessionInit (从 Agenda 拿议题 + 上下文)        │
│         ↓                                       │
│ ContextRetriever (RAG: 知识库 + 特征仓库)      │
│         ↓                                       │
│ AgentOrchestrator (LangGraph / 自建)           │
│   ├─ MoE Router (按议题路由专家)               │
│   ├─ Tool Calling Runtime (browser/computer)   │
│   ├─ 多 Agent 协作 + 投票                      │
│   └─ 失败兜底（规则路径）                       │
│         ↓                                       │
│ ConsensusBuilder (聚合多 Agent 输出 + 解释)    │
│         ↓                                       │
│ ConfidenceScorer                               │
│         ↓                                       │
│ EvidenceLinker (拉证据 → evidence_chain)       │
│         ↓                                       │
│ Output (ResearchCard / Candidate / Session)    │
└────────────────────────────────────────────────┘
```

### 关键设计
- **每个 Agent 是无状态的；状态在 Session 里**
- **工具调用全程录音**：调用、参数、返回、耗时全记录到 `tool_call_log`
- **失败兜底**：任何 Agent 失败 → 转规则路径并标注 `degraded=true`
- **人审钩子**：低置信度议题自动挂起，等待人审

### 输出
- `ResearchCard` / `Candidate` → 候选注册表
- `CouncilSession` → 超级个体进化的评测 / 知识沉淀

### 失败模式与保护
| 失败 | 保护 |
|------|------|
| 推理网关熔断 | 转规则路径 + degraded |
| 工具调用失败 | 记录失败 + Agent 重新规划 |
| Session 超时 | 终止并输出"未达共识"卡片 + 触发议程重排 |

## 4. 议程编排器（agenda_orchestrator）

### 主责
- 维护周期性议程（每日 / 每周 / 月度）
- 接收触发式议题（重大事件、用户实时议题）
- 排程到议会服务

### 内部组件
```
┌───────────────────────────────────────────┐
│ Scheduler (cron / 事件订阅)                │
│         ↓                                  │
│ PriorityQueue (用户 > 事件 > 周期)         │
│         ↓                                  │
│ Dispatcher (调 research_council_service)   │
│         ↓                                  │
│ Tracker (Session 状态追踪 / 重排 / 取消)   │
└───────────────────────────────────────────┘
```

### 输出
- 议会任务调度
- 议程执行报告

---

## 5. 预期差量化器（expectation_gap_quantifier）

### 主责
- 度量"我们的研究结论 vs 市场共识"的差距
- 按时间衰减折现（动量折现）

### 输入
- `ResearchCard.conclusion`
- 市场共识来源：行情衍生 + 卖方研报抓取（数据层）

### 输出
- `ExpectationGap { subject, our_view, market_consensus, gap_score, momentum_decay }`

### 部署形态
- 既可作为 `research_council_service` 的子组件（同进程），也可独立服务

---

## 6. 研究候选注册表（candidate_registry）

### 主责
- 候选生命周期管理：创建 / 更新 / 去重 / 失效
- 提供按 `subject / kind / status / priority` 的查询 API

### 数据模型
- `candidates` 表（详见 [04_数据契约_设计.md](./04_数据契约_设计.md)）
- 状态：`draft` → `pending_gate`（送极寒防御）→ `active`（门禁通过）→ `archived`

### 与状态机监控的衔接
- 候选附带 `suggested_state_machine_template`（建议绑定的状态机模板 ID）
- 候选 `active` 后，状态机监控可拉起对应状态机实例
