## 一、章节定位

本章是 L3 超级个体进化的"**后端服务子模块工程实现**"权威规约，回答四个问题：

1. **L2 维度五的 13 MLOps 组件**（Teacher LLM/DVC/Label Studio/LLaMA-Factory/vLLM/MLflow/...）**如何下沉为 7 个 service 的工程实现**？
2. **L2 维度五的 8 象限路由、Kappa 标定、DPO 偏好对**等核心规则如何在哪个 service 内实现？
3. **来自维度零反馈闭环的 verified 数据 / DPO 偏好对**如何被 feedback_collector 接收？
4. **阶段 3 的"自动驾驶限额"**在 external_action_boundary 如何实现？

## 二、总览

| # | 子模块 | 英文键 | 部署形态 | 承接的维度五组件 |
|---|--------|--------|---------|---|
| 1 | 评测中心 | `eval_center` | 独立服务 / 多实例 + 离线 worker | 组件 5 (K8s GPU Job 触发评测) / 组件 8 (评测回归集与回放器) / 组件 13 (A/B 测试与灰度) |
| 2 | 反馈采集 | `feedback_collector` | 独立服务 / 多实例 | 组件 1 (Teacher LLM 蒸馏) / 组件 3 (Label Studio) / 组件 9 (DPO 偏好对齐) |
| 3 | 知识库 + 长期记忆 | `knowledge_base` | 独立服务 / 多实例 + 向量索引 | 组件 1 (Teacher 蒸馏知识) / 组件 11 (数字分身 RAG) |
| 4 | 复盘服务 | `retrospective_service` | 独立服务 / 单实例（HA）+ worker | （承接维度四引擎 7·退出回放归因） |
| 5 | 版本管理 | `version_manager` | 独立服务 / 多实例 | 组件 2 (DVC) / 组件 4 (LLaMA-Factory) / 组件 5 (K8s GPU Job) / 组件 6 (vLLM) / 组件 7 (MLflow) / 组件 9 (DPO) / 组件 10 (vLLM 多 LoRA) / 组件 13 (A/B 灰度) |
| 6 | 外部动作边界 | `external_action_boundary` | 独立服务 / 多实例（关键路径） | 组件 12 (gVisor / Firecracker 沙箱) |
| 7 | 用户成长档案 | `user_profile_service` | 独立服务 / 多实例 | 组件 11 (数字分身 RAG 个人风格) ; 维度零·SCS/EV 计算 |

> **关键观察**：维度五 13 组件大部分由 `version_manager` 承担（8 个），因为 version_manager 是"模型生命周期总管"。

## 1. 评测中心（eval_center）

### 1.1 主责
- 管理评测集（离线 dataset 与在线 traffic 分流）
- 自动评测任务调度与执行
- 评测报告生成 + 趋势监控
- **承接维度五组件 8**：评测回归集与回放器（Holdout 任意指标退化 > 5% 自动 Block）
- **承接维度五组件 5**：K8s GPU Job 自动训练触发后的评测启动
- **承接维度五组件 13**：A/B 灰度发布的对照组评测

### 1.2 内部组件

```
┌────────────────────────────────────────────────────────────┐
│ DatasetManager (CRUD + 版本)                                │
│ TaskScheduler (cron / 触发 / K8s Job 完成事件订阅)          │
│ Runner                                                      │
│   ├─ Offline (跑评测集 → 议会 / 状态机 / 推理)             │
│   ├─ Online (A/B 流量分流；可逆)                            │
│   └─ Holdout (回放守门，>5% 退化自动 Block)【组件 8 实现】  │
│ MetricCollector (按指标汇总)                                │
│   ├─ 6 守门指标（与 L2 §LoRA 守门人协议一致）：             │
│   │  ① accuracy ② recall@k ③ precision@k                   │
│   │  ④ false_positive_rate ⑤ p99_latency ⑥ cost_per_call  │
│   └─ 自定义指标插件                                          │
│ ReportBuilder (HTML / PDF / Markdown)                       │
│ TrendMonitor (与历史对比；告警)                              │
│ CanaryReleaseController【组件 13 实现】                     │
│   ├─ 5% → 25% → 50% → 100% 渐进式                          │
│   └─ 任一守门指标退化触发自动回滚                            │
└────────────────────────────────────────────────────────────┘
```

### 1.3 关键设计
- **评测集是数据资产**：版本化（[07_数据版本控制](../_共享规约/07_数据版本控制规约.md)），变更走审计；**底层用维度五组件 2 DVC 实现**
- **离线 + 在线 + Holdout 三层**：离线先验证 → Holdout 守门 → 在线 A/B
- **6 守门指标**与 L2 演进策略规划 §LoRA 守门人协议严格一致
- **指标可扩展**：核心指标内置；自定义指标走插件

### 1.4 失败模式
| 失败 | 保护 |
|------|------|
| 评测集加载失败 | 锁定版本，禁用任务，触发风险事件 |
| 在线 A/B 流量异常 | 自动收敛流量到对照组 |
| Holdout >5% 退化 | 自动 Block 候选版本 + 通知架构师 |

## 2. 反馈采集（feedback_collector）

### 2.1 主责
- 接收前端 / 用户的反馈：显式（点赞 / 踩 / 评分 / 备注）+ 隐式（点击 / 停留 / 忽略）
- 接收外部世界回馈（如新闻证伪、状态机长期表现）
- **承接维度零反馈闭环**：接收 verified 标注 + DPO 偏好对（来自 events:co_pilot:verified_dpo_pair_ready）
- **承接维度五组件 1**：调用 Teacher LLM 蒸馏服务批量生成训练候选
- **承接维度五组件 3**：把样本推送到 Label Studio 等待人工 verified
- **承接维度五组件 9**：DPO 偏好对齐流水线的数据采集端
- **8 象限路由器（核心新增）**：按 [L2 演进策略规划 §3.1 八象限路由](../../02_战略维度/05_维度五_演进飞轮/04_演进实践策略规划.md) 实现

### 2.2 内部组件

```
┌────────────────────────────────────────────────────────────┐
│ Receiver (前端 SDK 埋点 + 显式反馈 + 维度零 stream 推送)    │
│ TeacherDistillClient【组件 1 实现】                         │
│   └─ 调 Claude 3.5 Sonnet / GPT-4o 批量生成；速率限制 +    │
│      成本上限 + 错误重试                                     │
│ LabelStudioConnector【组件 3 实现】                         │
│   ├─ Push: 样本推送到 Label Studio                          │
│   └─ Pull: 拉取 verified 标注                                │
│ VerifiedDataReceiver【新增·承接维度零】                     │
│   └─ 订阅 events:co_pilot:verified_dpo_pair_ready          │
│ EightQuadrantRouter【核心新增】                             │
│   ├─ 输入：verified 决策记录                                 │
│   ├─ 计算 quadrant ∈ {A, B, C, D, E, F, G, H}              │
│   │  （逻辑链状态 × 价格状态 × 时间窗口 三轴矩阵）           │
│   └─ 路由：                                                  │
│     - A/F 象限 → gold_library (高质量样本，训练正向)        │
│     - H 象限 → failure_library (高 loss 样本，DPO 训练)     │
│     - 其他象限 → general_library                             │
│ DPOPreferencePairBuilder【组件 9 实现】                     │
│   ├─ 从 gold_library + failure_library 配对                 │
│   └─ 输出 (chosen, rejected) 对                              │
│ KappaCalibrator【新增·承接维度零】                          │
│   ├─ 双盲标注采集                                            │
│   ├─ 计算架构师 vs 系统的 Cohen's Kappa                     │
│   └─ Kappa < 0.4 触发标定告警                                │
│ Persister (写库 + 派发到 eval_center / knowledge_base)      │
└────────────────────────────────────────────────────────────┘
```

### 2.3 8 象限路由表（与 L2 严格一致）

| 象限 | 逻辑链 | 价格 | 时间窗口 | 路由目标 | 训练用途 |
|---|---|---|---|---|---|
| A | 验证 | 上涨 | 在窗内 | gold_library | 正向强化（SFT 主样本）|
| B | 验证 | 上涨 | 出窗 | general_library | 持有耐心的负样本（DPO chosen）|
| C | 验证 | 下跌 | 在窗内 | general_library | 持仓策略边界 |
| D | 验证 | 下跌 | 出窗 | failure_library | DPO rejected（验证但失利）|
| E | 失效 | 上涨 | 在窗内 | general_library | 卖飞豁免（不训练）|
| F | 失效 | 下跌 | 在窗内 | gold_library | 卖出决策正向（防御正样本）|
| G | 失效 | 上涨 | 出窗 | general_library | 长期市场扰动 |
| H | 失效 | 下跌 | 出窗 | failure_library | 高 loss 样本（DPO 重点）|

### 2.4 接入点
- 前端 SDK：埋点 + 显式反馈按钮
- 状态机长期数据：定期跑批从 `state_watch` 拉迁移日志 / Advisory 执行结果
- 数据层 webhook：新事件验证旧结论
- **新增**：订阅 `events:co_pilot:verified_dpo_pair_ready`（维度零反馈闭环推送）

### 2.5 失败模式
| 失败 | 保护 |
|------|------|
| 入口高并发 | 队列 + 限流 + 异步持久化 |
| 反馈池数据偏倚 | 评测中心做加权 / 去偏 |
| Teacher LLM 调用失败 | 本地降级到 rule-based 标签 + 标记 degraded |
| Kappa < 0.4 | 暂停训练触发；通知架构师重新校准 |

## 3. 知识库 + 长期记忆（knowledge_base）

### 3.1 主责
- 维护结构化知识（实体 / 关系 / 事件 / 规则）+ 嵌入索引
- 提供 RAG 检索接口
- 知识衰减（时效性 / 置信度）
- **承接维度五组件 1**：吸收 Teacher LLM 蒸馏的知识条目
- **承接维度五组件 11**：数字分身 RAG 系统（基于架构师历史决策日志构建个人风格 RAG）

### 3.2 数据结构

```yaml
KnowledgeEntry:
  id: string
  type: fact | rule | playbook | glossary | personal_style  # 新增 personal_style 用于数字分身
  subject_ids: [string]
  content: string
  embedding_ref: string
  source_refs: [string]
  confidence: float
  valid_from: timestamp
  valid_to: timestamp
  decay_policy: fast | medium | slow | none
  version: string
  owner: string  # 新增：personal_style 类型必填，仅本人可见
```

### 3.3 内部组件

```
┌──────────────────────────────────────────────────────────┐
│ Ingester (来自评测 / 复盘 / 反馈 / 议会 / Teacher 蒸馏)  │
│ Deduplicator (相似度去重)                                │
│ Embedder (嵌入向量；接 Inference Gateway)                │
│ DecayWorker (定期降权 / 归档)                            │
│ Retriever (RAG: hybrid 检索 + rerank)                    │
│ FeedbackHandler (用户对知识条目的反馈)                   │
│ DigitalTwinRAGEngine【组件 11 实现】                     │
│   ├─ 索引架构师历史决策（决策日志 + 月报）                │
│   ├─ 隔离：personal_style 仅 owner 可见                  │
│   └─ 风格辅助检索（不替代决策）                          │
└──────────────────────────────────────────────────────────┘
```

### 3.4 关键设计
- **去重在写入时**：相似度 > 阈值 → 合并而非新增
- **衰减可配置**：政策类高衰减；术语类低衰减
- **检索默认带可解释性**：返回每条结果的 source_refs + score breakdown
- **数字分身严格隔离**：personal_style 类型不参与跨用户聚合

### 3.5 失败模式
| 失败 | 保护 |
|------|------|
| 嵌入失败 | 写规则索引兜底；议会接受降级 |
| 检索超时 | 议会拿到部分结果 + 标记 partial |
| 数字分身 RAG 失败 | 退化到通用知识库 |

## 4. 复盘服务（retrospective_service）

### 4.1 主责
- 周期复盘（每日 / 每周 / 每月）+ 触发式复盘（重大事件 / 用户请求）
- 自动归因：哪些事件 / 工具 / 模型版本影响结果
- **承接维度四引擎 7·退出回放归因**：每次卖出决策的事后归因
- **承接维度零·价值账本月报**：8 象限归因数据 + LoRA 训练贡献页生成
- 输出复盘报告并归档

### 4.2 内部组件

```
┌──────────────────────────────────────────────────────────┐
│ Scheduler (cron + 事件订阅 + 用户请求)                   │
│ DataLoader (拉评测 / 反馈 / 状态机 / 议会 / 决策日志)    │
│ AttributionAnalyzer (归因模型)                           │
│   ├─ 8 象限分析【新增】                                  │
│   ├─ counterfactual 反事实分析                           │
│   └─ LoRA 版本贡献分析（哪个版本影响哪些决策）           │
│ EightQuadrantReportBuilder【新增·维度零月报】            │
│   ├─ 输入：本月所有决策                                   │
│   ├─ 按 8 象限归因                                        │
│   └─ 输出：HTML/PDF 月报                                  │
│ Storyteller (LLM 写报告)                                 │
│ Archiver (s3 + 知识库回写)                               │
└──────────────────────────────────────────────────────────┘
```

### 4.3 关键设计
- **归因不只看相关，要看反事实**：尽可能跑 counterfactual（如"若使用旧 prompt 会怎样"）
- **写报告必带证据 + 数据出处**
- **8 象限是月报的核心呈现方式**：每个象限的决策数 + EV 贡献 + 关键 case

### 4.4 失败模式
| 失败 | 保护 |
|------|------|
| 数据源缺失 | 跳过该维度并标注 |
| LLM 写报告失败 | 退化到模板报告 + 标注 degraded |
| counterfactual 失败 | 仅给相关分析 + 标注 |

## 5. 版本管理（version_manager）

### 5.1 主责
- 注册 / 灰度 / 上线 / 回滚 模型 / prompt / 规则 / 评测集
- 提供版本元数据查询（哪个版本来自哪条评测、谁批准的）
- **承接维度五 8 个组件**（核心模块，承担最重）：
  - 组件 2 (DVC)：数据集版本化
  - 组件 4 (LLaMA-Factory)：本地手动微调（P0 阶段）
  - 组件 5 (K8s GPU Job)：自动训练触发（P1 起）
  - 组件 6 (vLLM 推理网关)：模型服务化
  - 组件 7 (MLflow Model Registry)：核心实现底座
  - 组件 9 (DPO)：DPO 训练流水线
  - 组件 10 (vLLM 多 LoRA 多路复用)：P2 高阶
  - 组件 13 (A/B 测试与灰度)：灰度策略

### 5.2 数据结构

```yaml
VersionEntry:
  artifact_type: model | prompt | rule | dataset
  artifact_id: string
  version: string              # SemVer
  status: draft | staging | canary | prod | rolled_back | archived
  approver: string
  approved_at: timestamp
  eval_report_ref: string      # 必须有评测才能从 staging → canary
  rollout_strategy:
    canary_pct: float          # 5% → 25% → 50% → 100%
    ramp_steps: [int]
    auto_rollback_on:
      - any_holdout_metric_degrade > 5%   # 与 eval_center 守门一致
  signature: string            # 上线版本带签名
  base_artifact: string        # 基模型（如 Qwen2.5-7B）
  lora_id: string              # 若是 LoRA，关联到 base_artifact
  vllm_endpoint: string        # 推理网关端点（组件 6/10）
  mlflow_run_id: string        # MLflow 运行 ID（组件 7）
  dvc_data_snapshot: string    # 训练数据 DVC 快照（组件 2）
  training_method: sft | dpo | lora_sft  # 训练方法（组件 4/9）
```

### 5.3 内部组件

```
┌──────────────────────────────────────────────────────────┐
│ Registry (核心 CRUD；底层 MLflow Model Registry【组件 7】) │
│ DataVersionController【组件 2 实现】                       │
│   └─ DVC CLI 嵌入；每次训练前锁定数据快照                 │
│ TrainingTriggerService【组件 5 实现】                     │
│   ├─ K8s Job 模板（GPU 资源配额）                         │
│   ├─ Argo Workflows 编排                                  │
│   └─ 自动触发条件：新增 verified 样本 ≥ 100 条             │
│ LocalTrainingService【组件 4 实现，P0 阶段用】             │
│   └─ LLaMA-Factory CLI 调用；本地 GPU 微调                │
│ DPOTrainingService【组件 9 实现】                         │
│   ├─ 输入 (chosen, rejected) 偏好对                       │
│   └─ DPO 训练脚本                                          │
│ InferenceGatewayController【组件 6/10 实现】              │
│   ├─ vLLM 单 LoRA 部署（P0/P1）                           │
│   └─ vLLM 多 LoRA 多路复用（P2）                          │
│ RolloutController【组件 13 实现】                         │
│   ├─ canary 渐进式发布                                     │
│   ├─ 与 eval_center.CanaryReleaseController 协同           │
│   └─ 任意守门指标退化自动回滚                              │
│ SignatureService (上线版本签名校验)                        │
└──────────────────────────────────────────────────────────┘
```

### 5.4 关键设计
- **强制评测前置**：从 staging → canary 必须有评测报告（守门通过）
- **回滚一键**：任意版本可回到上一个 prod
- **签名机制**：上线版本带签名，防篡改
- **数据 + 模型双锁定**：DVC 数据快照 + MLflow 模型 = 完全可复现
- **MLflow 是事实底座**：version_manager 是 L3 抽象层，底层用 MLflow 实现

### 5.5 训练触发频率（与 L2 一致）

| 训练类型 | 触发频率 | 触发条件 |
|---|---|---|
| Teacher 蒸馏 → 候选生成 | 每日 | 新增 verified 样本 ≥ 100 条 |
| LoRA SFT（增量微调）| 每周 | gold_library 新增样本 ≥ 500 条 |
| LoRA DPO（偏好对齐）| 每月 | failure_library 新增配对 ≥ 200 对 |
| 全量基模型重训 | 不主动触发 | 仅当多个 LoRA 合并失败时 |

### 5.6 成本约束（与 L2 一致）

| 资源 | 月度上限 | 超限处理 |
|---|---|---|
| Teacher LLM 调用 | $300 | 暂停蒸馏；改人工 verified |
| GPU 训练时长 | 100 小时 | 暂停训练；通知架构师 |
| MLflow 存储 | 50 GB | 触发归档；老版本压缩 |
| DVC 数据快照 | 200 GB | 旧快照转冷存储 |

### 5.7 失败模式
| 失败 | 保护 |
|------|------|
| 灰度异常（守门指标退化）| 自动收敛流量；触发风险事件；通知架构师 |
| 回滚失败 | 多副本快照；强制人审 |
| K8s Job 训练失败 | 重试 3 次；超过则告警 |
| MLflow Server 不可用 | 临时锁定发版；本地 cache 维持推理服务 |

## 6. 外部动作边界（external_action_boundary）

### 6.1 主责
- 全系统对外动作的**唯一出口**
- 接收任意模块的 `ExternalActionRequest` → 调极寒防御门禁 → 通过后调 [External API Port](../_共享规约/05_接口抽象层规约.md)
- 强制 idempotency_key + 签名 + 审计
- **承接维度五组件 12**：gVisor / Firecracker 沙箱执行（Agent 生成的代码必须在沙箱）
- **承接维度零阶段 3·自动驾驶限额**（核心新增）

### 6.2 内部组件

```
┌──────────────────────────────────────────────────────────┐
│ Receiver (接 ExternalActionRequest)                      │
│ IdempotencyGuard (去重 + 缓存首次结果)                   │
│ AutopilotLimiter【新增·承接维度零阶段 3】                │
│   ├─ 单笔限额（按用户 profile）                           │
│   │  - 阶段 3 默认：单笔 ≤ 5% 持仓；日累计 ≤ 15%         │
│   │  - 用户可在 user_profile 自定义                      │
│   ├─ 异常熔断：连续 N 次失败自动降级到全人审            │
│   └─ 推送 PENDING_GATE 状态给前端 autopilot-bff          │
│ GateClient → 极寒防御 decision_gate                      │
│   (失败 / reject → 拒绝业务 + 通知)                      │
│ SandboxExecutor【组件 12 实现】                          │
│   ├─ gVisor / Firecracker Runtime                        │
│   ├─ 网络隔离 / 文件系统隔离 / 资源限制                  │
│   └─ 仅 Agent 生成的代码必经                             │
│ ApiPortClient (调 External API Port)                     │
│ AuditEmitter (写极寒防御 audit)                          │
│ ResultDispatcher (返调用方)                              │
└──────────────────────────────────────────────────────────┘
```

### 6.3 自动驾驶限额规则（新增）

```yaml
autopilot_limits:
  phase_3_default:
    per_order_max_pct: 0.05    # 单笔 ≤ 5% 持仓
    daily_max_pct: 0.15        # 日累计 ≤ 15% 持仓
    auto_circuit_break_after_consecutive_failures: 3
    require_user_pre_approval: true   # 默认仍需用户预批
  user_customizable:
    - per_order_max_pct
    - daily_max_pct
    - require_user_pre_approval  # 极信任用户可关闭
```

### 6.4 失败模式
| 失败 | 保护 |
|------|------|
| 门禁失败 | 业务拒绝 + 通知调用方 |
| 第三方 API 失败 | 重试（带 idempotency）；超阈值熔断 |
| idempotency 冲突 | 返回首次结果，不重复调用第三方 |
| 沙箱执行超时 | 强制 kill + 标记失败 + 写审计 |
| 自动驾驶限额触发 | 拒绝订单 + 推送 PENDING_GATE 状态 |
| 自动驾驶熔断 | 切换到全人审模式 + 紧急告警 |

## 7. 用户成长档案（user_profile_service）

### 7.1 主责
- 维护单用户的偏好 / 历史 / 习得能力
- **承接维度零·SCS（System Capability Score）+ EV（Economic Value）计算**
- **承接维度零·Kappa 标定记录**
- **承接维度五组件 11·数字分身的个人风格档案**
- 接受用户对档案的查看 / 编辑 / 导出（GDPR 友好）

### 7.2 数据

```yaml
UserProfile:
  user_id: string
  preferences:
    notification:
      channels: [wechat, telegram, email, voice]
      red_alert_channels: [wechat, telegram, voice]
      digest_schedule: weekly | monthly
    autopilot:
      enabled: bool
      per_order_max_pct: float
      daily_max_pct: float
      require_user_pre_approval: bool
    risk_tolerance: conservative | balanced | aggressive
  habits:           # 系统观察到的习惯，用户可见可改
    typical_review_time: string    # "Saturday 09:00"
    avg_decision_time_seconds: int
    most_active_battlefield: string
  growth_metrics:
    scs:                            # System Capability Score
      current: float                # 0-100
      history: [{date, value}]
      trend: improving | stable | declining
    ev:                             # Economic Value
      current: float                # 累计真实经济价值（元）
      monthly: [{month, value}]
    kappa:                          # 双盲标定 Kappa
      current: float                # Cohen's Kappa
      history: [{date, value}]
      last_calibration_at: timestamp
    decisions_count: int
    accuracy_rate: float
  digital_twin:                     # 组件 11 数字分身
    enabled: bool
    rag_index_ref: string           # 关联 knowledge_base 中的 personal_style 条目
    style_consistency_score: float  # 个人风格一致性
  updated_at: timestamp
```

### 7.3 关键设计
- **用户拥有数据**：用户可导出 / 删除（GDPR 友好）
- **不跨用户共享**（除非用户显式同意）
- **SCS 与 EV 解耦**：SCS 反映系统能力，EV 反映财务收益；二者独立追踪
- **Kappa 是双盲标定唯一指标**：低于 0.4 触发标定告警
