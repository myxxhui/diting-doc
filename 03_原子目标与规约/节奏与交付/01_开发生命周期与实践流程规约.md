> - **本文档**: L3 层级，定义双目标系统的环境阶段、准入准出与执行节奏

## 一、原则

- 生命周期服务双目标：每一步既要推进产品价值，也要沉淀岗位训练证据。
- 机器真相源以 `_System_DNA/dna_dev_workflow.yaml` 为准；本文档给出执行原则与解释口径。
- 不再使用 `s0/s1/s1b` 一类旧环境叙事作为主入口；统一按 `Stage1` 到 `Stage5` 表达。

## 二、五段式生命周期

| 阶段 | 核心目标 | 主要环境 | 准出标准 |
|------|----------|----------|----------|
| **Stage1 仓库与骨架** | 仓库、接口、协议、文档、基础设施入口统一 | 本地 | 仓库结构可用，基础接口存在，DNA 与 README 对齐 |
| **Stage2 数据底座与采集** | 数据采集、存储、容器化、基础观测打底 | 本地 + Docker + K3s 验证 | 数据链路可跑通，采集任务可复现，基础组件可部署 |
| **Stage3 研究与智能体主链** | 内容理解、信号引擎、研究议会、决策主链与 runtime 落地 | Docker + K3s | 研究主链可运行，关键模块有测试和运行证据 |
| **Stage4 评测、外部边界与发布** | 评测/仿真、外部系统接入、发布门禁、回滚机制 | K3s + 流水线 | 评测与发布门禁可执行，回滚与灰度条件明确 |
| **Stage5 稳定性与效率优化** | 成本、性能、可观测性、复盘、漂移治理 | K3s + 生产预备 | 关键指标可持续观测，优化与治理方案落地 |

## 三、每阶段都必须输出两类结果

### 1. 产品结果

- 本阶段新增了什么用户可感知能力。
- 该能力的质量指标和最低门槛是什么。
- 验证截图、日志、快照、报告或演示链路在哪里。

### 2. 成长结果

- 对应岗位能力是什么。
- 这项能力是否有代码、配置、压测、评测、演练或安全验证证据。
- 是否沉淀成可复用的架构模板、脚本、runbook 或 ADR。

任何只交付其中一类结果的阶段，都视为未准出。

## 四、开发与测试循环

| 环节 | 强制要求 |
|------|----------|
| **设计前** | 明确主责文档、指标、风险、依赖与退出条件 |
| **实现中** | 先补接口与配置，再补逻辑，再补验证；避免直接跳到“最终部署” |
| **验证时** | 至少覆盖单元、集成、阶段验收三层；必要时补压测、安全演练、恢复演练 |
| **准出前** | 更新 L5 验收项、同步 L6 映射、记录风险与未决项 |

## 五、岗位能力如何嵌入生命周期

### Stage2 到 Stage5 的强制训练重点

- **岗位 1**：Stage2 开始接触资源治理和容器化；Stage3/5 必须覆盖推理接入、GPU 调度、性能优化、checkpoint 与观测。
- **岗位 2**：从 Stage1 就开始建立指标、优先级与上线门禁；Stage4/5 必须参与灰度、回滚、ROI 判断与复盘。
- **岗位 3**：Stage3 开始引入 `MLflow`；Stage4/5 必须补齐流水线编排、模型注册、服务化与 drift detection。
- **岗位 4**：Stage3 开始落地 Agent runtime；Stage4/5 必须覆盖 WebSocket、会话恢复、browser runtime 与安全隔离。

## 六、失败与回退

- 单步失败优先回退到上一步最近稳定点，而不是继续累积脏状态。
- 阶段失败必须同时说明：失败指标、影响面、当前阻塞、回退动作、是否影响下一阶段。
- 进入 Stage4 以后，任何面向外部系统的变更都必须有灰度条件与回滚条件。

## 七、交接规则

阶段准出时必须同步完成以下动作：

1. 更新对应的 L4 阶段实践文档。
2. 更新 L5 验收标准中的相关锚点。
3. 更新 L6 追溯映射或影响表。
4. 把新增风险、债务和待补实验写入阶段记录。
