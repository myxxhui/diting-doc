<a id="b-track-data-requirement"></a>
## 一、B 轨数据需求清单

**基本数据（行情/技术、标的池、行业板块）**：由 [11_数据采集与输入层规约](../_共享规约/11_数据采集与输入层规约.md) 约定并写入 L1；B 轨**复用**该部分，不在此重复列出。下表仅列 **B 轨专用** 数据需求。

| 类别 | 内容 | 必选/可选 | 说明 |
|------|------|-----------|------|
| **标的主营构成** | 标的→细分列表（segment_id、revenue_share、is_primary） | 必选 | 来自 A 的 segment_shares 或 L2 标的主营构成表；供 C 右脑对齐与 B 轨候选解析 |
| **细分一手信号** | 按 segment_id 的垂直一手信号（政策/价格/情绪等） | 必选 | 经适配器+信号理解层产出 direction/strength/risk_tags，写入 segment_signal_cache；供 C 右脑消费，见 [12_右脑数据支撑与Segment规约](../_共享规约/12_右脑数据支撑与Segment规约.md)、C 模块 B 轨设计 |
| **阶段/趋势信号** | B 模块产出的阶段得分、趋势确认、板块强度等 | 必选 | 供 B 轨通过判断与 b_track_candidate_snapshot 写入 |

<a id="b-track-storage"></a>
## 二、存储表与契约

- **标的主营构成表**：L2 `symbol_business_profile`（或等价）；与 12_ 一致；消费方为 Module A 扩展、信号层、Module C。
- **细分信号缓存表**：L2 `segment_signal_cache`；与 12_ 一致；写入方为信号层（适配器+信号理解）；消费方为 Module C。
- **B 轨候选表**：`b_track_candidate_snapshot`（或等价）
  - **用途**：仅存放 **B 轨阶段/趋势策略通过** 的标的。
  - **建议字段**：symbol、batch_id、phase_score、trend_confirm、sector_strength、created_at；与 [01_B轨系统设计](01_B轨系统设计.md)#2.3 一致。
  - **写入方**：B 模块；**消费方**：Module C（右脑）仅或优先从此表读取 B 轨候选。

<a id="b-track-consumption"></a>
## 三、消费方与读接口

| 消费方 | 读取内容 | 约定 |
|--------|----------|------|
| B 模块 | 标的主营构成（或 A 的 segment_shares）、行情/技术数据 | 用于阶段/趋势打分与 B 轨通过判断 |
| 信号层 | 标的主营构成、细分注册表；拉取后写 segment_signal_cache | 按 12_ 按需拉取、按细分去重 |
| Module C（右脑） | b_track_candidate_snapshot、segment_list、segment_signals | 统一分析入口 unified_opinion；输出 MEDIUM_TERM ExpertOpinion |

<a id="b-track-layers"></a>
## 四、调度层 / 信号层 / 采集层 / 存储层（设计规范与实践归属）

| 层级 | 职责 | 设计规范 | 实践与验收 |
|------|------|----------|------------|
| **采集层** | 标的主营构成写入（ingest_business_profile）、细分注册表初始化；可选补写 domain/板块级 segment_id | [12_ §2.1、§3.1](../_共享规约/12_右脑数据支撑与Segment规约.md)、[01_B轨_数据与采集_设计](../Stage2_数据采集与存储/01_B轨_数据与采集_设计.md)、本规约 §一§二 | [Stage2 01_B轨_数据与采集_实践](../../04_阶段规划与实践/Stage2_数据采集与存储/01_B轨_数据与采集_实践.md) |
| **存储层** | L2 表定义与契约：symbol_business_profile、segment_registry、segment_signal_cache、b_track_candidate_snapshot | 12_ §2（DDL）、本规约 §二、[01_B轨系统设计](01_B轨系统设计.md)#2.1 | 表创建与 schema 在 01_B轨、06_B轨 实践中落地；无单独「存储层」步骤，合在规约与各步 |
| **信号层** | 按候选标的解析细分 → 按 segment_id 去重拉取 → 信号理解打标 → 写 segment_signal_cache；支持多层级 segment（domain/板块/具体业务） | [12_ §2.2 分层约定、§3.2](../_共享规约/12_右脑数据支撑与Segment规约.md)、[06_B轨_信号层生产级数据采集_设计](../Stage2_数据采集与存储/06_B轨_信号层生产级数据采集_设计.md) | [Stage2 06_B轨_信号层生产级数据采集_实践](../../04_阶段规划与实践/Stage2_数据采集与存储/06_B轨_信号层生产级数据采集_实践.md) |
| **调度层** | 在 B 产出候选后触发信号层 refresh；或定时/事件触发读 b_track_candidate_snapshot 后调 refresh，再调 C | 06_B轨_信号层设计 §四「触发」、[01_B轨系统设计](01_B轨系统设计.md)#2.1 采集模块 | 无单独「调度层」文档；触发方式与入口在 06_B轨 实践「配置、触发」中约定，由 diting-core 编排或 diting-infra 流水线实现 |

- **标的主营多层级**：segment_id 分层约定（domain 级 / 板块级 / 具体业务级）见 12_ §2.2；标的主营构成表可为同一标的挂多层级 segment_id，一次 refresh 拉取「大方向+板块+具体业务」；实现难度低、可逐步加 segment 与适配器。
