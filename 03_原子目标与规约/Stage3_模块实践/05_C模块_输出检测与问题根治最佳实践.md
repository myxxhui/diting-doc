## 一、输出检测暴露的问题清单

`make run-module-c` 执行后，可通过终端输出与 L2 写入结果**系统性地检测**以下问题。每一类对应根因与根治方案见下文。

| 检测项 | 输出表现 | 根因归类 |
|--------|----------|----------|
| **segment_signal_cache 表不存在** | `relation "segment_signal_cache" does not exist`（已解决：见 `make init-l2-segment-signal-cache`） |  infra/表未建 |
| **segment_signal_cache 为空** | 所有标的摘要「主营细分无信号（认知边界）；缺失依赖：segment_signal_cache 需先写入该标的 segment_id 的条目」 | **信号层未就绪** |
| **短轨 OK=0** | `MoE 产出 短轨 OK=0`；明细列「短轨」全部为 `主营无信号，不…` | 同上，主营一票否决 |
| **不予支持=全量** | `不予支持 43`（或等于本批只数） | 同上 |
| **A/B 行数不一致** | `[警告] L2 classifier 行数(154) ≠ quant_scan_all(152)` | **批次漂移** |
| **A/B symbol 漂移** | `[警告] universe 内 A 快照与 B 量化 symbol 不一致：仅 A 有 2 只、仅 B 有 0 只` | **批次漂移** |
| **stub=关 仍大量不支持** | 与「信号层未就绪」重叠；`stub=0` 且无真实信号时属**预期**，需补信号层 | 设计预期 |

## 二、根因与根治方案

### 2.1 信号层未就绪（segment_signal_cache 为空）

**根因**：`refresh_segment_signals_for_symbols` 尚未实现或未被调用；`segment_signal_cache` 表存在但无写入。

**依赖链**：
```
symbol_business_profile（主营构成）→ segment_registry（细分注册）→
refresh_segment_signals_for_symbols（信号层）→ segment_signal_cache（细分信号）→
Module C 读取 segment_signals
```

**根治方案**：

| 阶段 | 动作 | 说明 |
|------|------|------|
| **短期（联调）** | `MOE_STUB_SEGMENT_SIGNALS=1 make run-module-c-dev` | 占位细分信号，仅用于管道验证，**不得作为生产依据** |
| **生产** | `make refresh-segment-signals`（[06_B轨_信号层生产级数据采集_实践](../../04_阶段规划与实践/Stage2_数据采集与存储/06_B轨_信号层生产级数据采集_实践.md) 已实现） | 信号层已实现：L2 news_content + AkShare 个股新闻 → 规则/AI 打标 → segment_signal_cache；**流水线编排**：B 写表后 → `make refresh-segment-signals` → `make run-module-c` |

**验证**：`segment_signal_cache` 中有本批标的涉及 segment_id 的行，且 `signal_summary` 含合法 JSON（direction/strength 等）。

---

### 2.2 A/B 批次与行数不一致

**根因**：未设 `MOE_CLASSIFIER_BATCH_ID` / `MOE_QUANT_BATCH_ID` 时，脚本从 L2 取「最近一批」A 和 B；若 A、B 来自不同业务跑批，则行数与 symbol 集合不一致。

**根治方案**：

| 动作 | 说明 |
|------|------|
| **同一编排** | 同一业务日、同一流水线内：`make run-module-a` → `make run-module-b` → `make run-module-c` |
| **显式锁批** | 传入 `MOE_CLASSIFIER_BATCH_ID=<A_batch_id>`、`MOE_QUANT_BATCH_ID=<B_batch_id>` 锁定同批 |
| **告警写入** | 脚本已将 `alignment_warnings` 写入 `moe_run_metadata`，下游可按需过滤或审计 |

**验证**：`classifier 行数` 与 `quant_scan_all 行数` 相等或可解释；`仅 A 有/仅 B 有` 为空或符合业务预期。

---

### 2.3 短轨 OK=0 与不予支持=全量

**根因**：主营无信号 → 主营一票否决 → 全部 `is_supported=False` → 短轨 OK=0、不予支持=全量。

**说明**：在 `stub=0` 且 `segment_signal_cache` 无数据时，此结果**符合预期**；不属 bug，属数据依赖未满足。根治即补信号层（见 §2.1）。

---

### 2.4 stub 联调 vs 生产

| 模式 | 命令 | 用途 |
|------|------|------|
| **联调** | `MOE_STUB_SEGMENT_SIGNALS=1 make run-module-c-dev` | 管道验证、CI；细分信号为占位 |
| **生产** | `make run-module-c`（默认 stub=0） | 使用真实 segment_signal_cache；无数据时输出「上游无数据」 |

**禁止**：用 stub 冒充生产右脑依据。

## 三、前置检查与提示（可选实现）

在 `run_module_c_local.py` 中，当 `stub=0` 且从 L2 读 segment_signals 时，可增加**非阻断性**检查：

1. **segment_signal_cache 空表检测**：对本批涉及 segment_id 做一次 `SELECT COUNT(*)`；若为 0，打印：
   ```
   [提示] segment_signal_cache 无本批 segment 数据；建议先 make refresh-segment-signals-test（若已实现）或执行信号层 refresh。无数据时 C 将输出「主营细分无信号」。
   ```
2. **建表检测**：若表不存在，现有 `segment_signal_reader` 已静默返回 `{}` 并打 warning；可补充：
   ```
   [提示] 若报 relation "segment_signal_cache" does not exist，请先 make init-l2-segment-signal-cache。
   ```

## 四、流水线编排推荐顺序

**A 轨（默认）**：
```
同一业务日 / 同一批次：
  1. make run-module-a          # 写 classifier_output_snapshot
  2. make run-module-b          # 写 quant_signal_*
  3. make refresh-segment-signals # DITING_TRACK=a，从 quant_signal_snapshot 取 symbols
  4. make run-module-c          # 消费 A+B+segment_signal_cache → horizon=SHORT_TERM
```

**B 轨**：
```
  export DITING_TRACK=b
  1. make run-module-a
  2. make run-module-b          # 需写入 b_track_candidate_snapshot（make init-l2-b-track-candidate 建表）
  3. make refresh-segment-signals # 从 b_track_candidate_snapshot 取 symbols
  4. make run-module-c          # 从 b_track 表取 universe，输出 horizon=MEDIUM_TERM
```

可选：显式传 `MOE_CLASSIFIER_BATCH_ID`、`MOE_QUANT_BATCH_ID` 锁定 A、B 同一批。
