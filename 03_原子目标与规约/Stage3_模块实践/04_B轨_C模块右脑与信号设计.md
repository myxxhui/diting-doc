# L3 · B 轨 C 模块右脑与信号设计

> [!NOTE] **[TRACEBACK]**
> - **顶层概念**: [04_B轨系统概念与目标](../../01_顶层概念/04_B轨系统概念与目标.md)、[00_B轨整体设计](../B轨/00_B轨整体设计.md)
> - **原子规约**: [12_右脑数据支撑与Segment规约](../_共享规约/12_右脑数据支撑与Segment规约.md)、[09_核心模块架构规约](../_共享规约/09_核心模块架构规约.md) Module C
> - **本文档**: B 轨（中线）下 **Module C 右脑** 与 **信号层（适配器+信号理解）** 的推荐方案：实现难度低、功能足、性能好、成本低；与 [04_A轨_MoE议会_设计](04_A轨_MoE议会_设计.md) 共用同一套右脑管道，B 轨仅输出 **horizon=MEDIUM_TERM** 与消费 **B 轨候选表**。

<a id="b-track-c-goal"></a>
## 一、目标与定位

- **B 轨** 目标：捕获 1～2 月内 **30%～100%+** 涨幅标的，年化复利目标 200%；识别依赖 **阶段信号（B 模块）+ 政策/情绪/景气（C 右脑）**。
- **C 模块在 B 轨的职责**：对 **B 轨候选**（来自 `b_track_candidate_snapshot`）做**利好与主营对齐、景气强度、风险分级**，输出 **ExpertOpinion（horizon=MEDIUM_TERM）**；与 A 轨共用「按股配置+统一分析」管道，仅 **horizon** 与 **候选来源**（B 轨表 vs A 轨全量通过）不同。
- **信号层职责**：为右脑提供 **segment_signals**（按细分、已打标 direction/strength/risk_tags）；采用「适配器按细分拉取 + 信号理解层打标 → segment_signal_cache」，C 只消费已结构化结果，**不**在 C 内做自然语言理解。

<a id="b-track-c-signal"></a>
## 二、信号层方案（适配器 + 信号理解，适合 B 轨）

### 2.1 分层

| 层级 | 职责 | 输出/存储 |
|------|------|-----------|
| **信号适配器** | 按 **segment_id** 从多源（东方财富、新闻、政策、国际组织、研报等）拉取原始摘要/文本；由人写代码或配置，不 AI 自动写抓取逻辑 | 原始摘要或文本，可写中间表或直接进信号理解 |
| **信号理解层** | 对原始内容做**理解+打标**：规则或 AI 产出 JSON（type, direction, strength, summary_cn, risk_tags）；失败时写 direction=neutral、strength=0.5 | 写入 **segment_signal_cache**（segment_id, signal_summary=JSON, signal_at, fetched_at, ttl_sec） |
| **信号层（编排）** | B 轨候选确定后，解析涉及细分 S，对 S 中每个 segment_id 若缓存未过期则跳过，否则调「适配器拉取+信号理解打标」→ 写 segment_signal_cache | 缓存表就绪，供 C 读取 |

### 2.2 按细分去重与成本

- **按细分** 拉取与打标：同一 segment 一天/一周期只打标一次，多标共享；B 轨 50 标可能只对应 10～20 个细分，成本可控。
- **新细分/新源**：新细分在 segment_registry 注册，新源由人在 signal_adapter_registry 挂接适配器（人写代码），**不**依赖 AI 自动生成抓取代码，实现难度低、可控。

### 2.3 审计与复用

- 可选表 `segment_signal_audit`（segment_id, source_type, source_url, source_publish_time, raw_snippet, model_conclusion_json, created_at）；每次打标写一条，便于复查与复用。

<a id="b-track-c-right-brain"></a>
## 三、C 模块右脑在 B 轨的用法

- **入口**：与 A 轨相同，**unified_opinion(symbol, quant_signal, segment_list, segment_signals, config, domain_tag)**；B 轨调用时 **quant_signal** 来自 B 模块阶段/趋势通过结果，**segment_list/segment_signals** 由上游按 12_ 与信号层提供。
- **输出**：ExpertOpinion 字段与 A 轨一致（is_supported, direction, confidence, reasoning_summary, risk_factors 等）；**horizon 固定为 MEDIUM_TERM**（B 轨）。
- **逻辑**：与 [04_A轨_MoE议会_设计](04_A轨_MoE议会_设计.md) 策略详规完全一致（解析信号→对齐与主营否决→认知边界→多细分加权与四维度→风险降权→摘要与 risk_factors→direction→输出）；**不**做共识落库、不调 VC-Agent。
- **Router**：若请求来自 B 轨候选（symbol 在 b_track_candidate_snapshot），则走统一分析并设置 **horizon=MEDIUM_TERM**；否则按 A 轨（SHORT_TERM）或 C 轨（LONG_TERM，走 VC-Agent）分支。

<a id="b-track-c-dna"></a>
## 四、与 DNA / 配置的对应

- **dna_module_c.yaml**：mode、entry、router.supported_tags、strategy、pipeline_steps、risk_factor_templates 与 A 轨共用；B 轨仅通过 **horizon=MEDIUM_TERM** 与 **候选来源（b_track_candidate_snapshot）** 区分。
- **moe_router.yaml**：supported_tags、alignment、veto_threshold、risk_factor_templates 与 A 轨共用；Router 内按「是否 B 轨候选」选择输出 horizon。

<a id="b-track-c-exit"></a>
## 五、准出

- 信号层（适配器+信号理解+按细分缓存）与 12_、00_B轨整体设计、01_B轨系统设计 一致；**信号层实现与测试**见 [Stage2 06_B轨_信号层生产级数据采集_实践](../../04_阶段规划与实践/Stage2_数据采集与存储/06_B轨_信号层生产级数据采集_实践.md)，不在此处或 Stage3/Stage4 重复实现。C 右脑 B 轨输出与 09_ Module C、04_A轨_MoE议会_设计 一致，仅 horizon=MEDIUM_TERM；L4 实践见 B轨 相关步骤与 04_A轨_MoE议会_实践（复用同一实现，B 轨为参数/分支）。
