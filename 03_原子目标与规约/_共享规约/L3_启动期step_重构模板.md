本文件是**所有维度启动期 step 的统一权威模板**。新建或重写 L3 启动期 step 时**必须**满足本模板的全部 13 个小节；旧 step 升级时按本模板对齐。

## 一、为什么需要本模板

在过往实践中出现过：
- 用 Mock 数据"跑通"步骤，实际无法用于真实分析或训练；
- 步骤未指明"在本机 / Docker / K3s 哪一层跑"，导致开发与部署节奏割裂；
- 没有"用户需要提供哪些凭证"的清单，下一步行动靠 AI 反复追问；
- 没有"启动期数据量"约束，导致首跑就要全 A 数据，跑不通也无人发觉；
- 把"采到 N 行就算完成"作为准出，但**字段粒度根本不够引擎做实质分析**——例如只采三表 raw 就声称"采到了财报"，但应收账款账龄、关联方网络、定价方法、承诺履约等"真正决定引擎判断力"的维度全缺。

新模板**强制**每个 step 显式回答 **6 件事**：
1. 采集什么真实数据（§3）；
2. **数据质量对得起目标引擎吗**（§3.5 验收矩阵 · 按引擎分析需求反推 · 巴菲特原则 · **仅启动期负责**）；
3. 要什么凭证（§4.2）；
4. **如何执行 + 一键复现**（§7 实施规划 · L3 给设计与合约 · L4 / 后续模型落地）；
5. 在哪里运行（§8 部署节奏）；
6. 下一期触发条件（§6 · **一行**，不展开扩展期细节）。

> **L3 的根本定位**：以**设计规划推演**为主 — 靠谱地描述"做什么 / 如何执行 / 验证标准"，但**不**嵌入完整 Makefile / 脚本代码、**不**试图本步把所有事 100% 完成。具体落地由 **L4 实践记录 / 后续执行模型**按 L3 规划完成；L3 是"给后面模型的工作指引 + 验证标准"，不是"自己把活全干了"。

### §3.5 数据质量验收矩阵（按目标引擎反推 · 强制小节 · 仅启动期负责）

> **核心原则**：数据采集是为给**模型引擎分析**的。引擎的分析需求决定字段粒度，**不是反过来**。**禁止**先采"行数 / 条数能凑齐"的表面数据，再问"够不够分析"。
> **巴菲特原则**："不看财报的投资就是自杀"，但只看三表 raw json **不算看了财报**——必须扩展到附注、披露、事件链、质性披露、关联方网络、定价方法、承诺履约、监管处罚等具体分析维度，让引擎能像专业分析师那样判断。
> **本步范围**：**只记录启动期负责的数据类型与质量检测**。每行要么 ✅ 启动期已覆盖，要么 ⚠️ 启动期内有降级路径。**不展开**扩展期 / 完善期清单——那是 `stage_2_扩展期 / stage_3_完善期` 文档的事，与本 step 无关；启动期不做的维度**根本不列**在矩阵里。
> **质量门槛 > 数量门槛**：§5 行数门槛是必要不充分；本节矩阵全绿（含 ⚠️ 有降级）才能 §9 准出。

**写作要求**（每个采集 step 强制 4 列结构）：

| # | 分析维度（一句话由功能反推） | 本步必产字段 / 衍生 | 启动期覆盖状态 | 启动期降级路径 |
|---|---|---|---|---|
| `X#` | 例：现金流-账面利润背离（财务测谎信号） | `operating_cash_flow / net_profit` 多年比值 | ✅ 已有 / ⚠️ 启动期有降级 | ✅ → 抽样验证非 null 率；⚠️ → 给出启动期可用的最低能力线 + 下游 step 中具体抽取动作 |

**分子节组织建议**（按下游消费 step 分块）：
- §3.5.1 引擎 A 所需的分析维度（列出每个引擎所需的所有字段，逐行说明启动期覆盖状态）
- §3.5.2 引擎 B 所需的分析维度
- §3.5.3 引擎 C 所需的分析维度
- §3.5.4 共用 / 跨维度补充
- §3.5.5 质量门槛（合并到 §9 准出）

**典型分析维度示例**（数据采集 step 应至少覆盖以下类别，**不止于**三表 raw json）：

| 数据大类 | 表面层（不够） | 引擎需要的深度（够） |
|---|---|---|
| **财报** | 三表 raw + 几个归一化数字 | 附注关联交易、应收账款账龄、研发资本化口径、商誉构成、坏账政策、政府补助归类、季度连续趋势、行业基线对比、现金流分布异常 |
| **公告** | 标题 + 日期 + 5 类粗分类 | PDF 全文、承诺履约进度、累计变动序列、董监高变动密度、解禁时间表、问询函与监管处罚、对赌履约、诉讼记录 |
| **关联交易** | 关联方文本行列表 | 关联方网络图（穿透实控人）、金额占比时间序列、定价方法异常（市价/协议/成本+）、循环交易识别、明股实债条款、关联担保资金占用 |
| **舆情新闻** | 标题与时间 | 情绪打分、来源权重、跨源同事件去重、与披露的一致性核对 |
| **行情** | 收盘价 | 复权、换手率、北向资金、龙虎榜、大单流向、行业动量 |
| **持仓** | 数量 + 成本 | 时间序列变动、加减仓事件、对账与历史对照 |

> 上表给"启动期 step 应该考虑哪些深度维度"的全景；实际矩阵只列**启动期负责**的子集。某项启动期不做，则不出现在本矩阵；不需要"❌ 扩展期承接"行。

**质量门槛准出条款（§9 复用）**：矩阵每行须满足之一：
- **✅ 覆盖**：本步已落库，§9 抽样验证字段非 null 率 ≥ 90%；
- **⚠️ 启动期降级**：本步未完美但启动期可接受，已给出 §7 实施规划中具体抽取动作 + 引擎在该降级下仍能达到 baseline 的能力线。
- **禁止**：①把字段假装填 null 视为达标；②在本矩阵列扩展期 / 完善期内容；③仅以"行数 / 条数"作为本步达标判定。

### §7 实施规划（L3 设计 · 给后续执行模型的工作指引）

> **L3 定位**：本节是**设计规划 + 实现要点 + 验证标准**——靠谱地描述"做什么 / 如何执行"，但**不嵌入完整 Makefile / 脚本代码**。具体命令、调试日志、参数填值由 L4 实践记录 / 后续执行模型按本节规划自行落地。

#### §7.1 实现要点（按交付物拆分 · 4 列表格）

| 实现要点 | 涉及代码位置 | 关键设计决策 | 验证标准 |
|---|---|---|---|
| **A 配置 / SoT 驱动** | 配置文件路径 + 加载器 | 唯一真相源；变更（增减标的 / 改阈值 / 调参数）只改本文件 | 配置加载器自检命令的预期输出 |
| **B ORM 或 schema 扩展**（若适用） | `apps/<module>/db/models.py` | 列名 / 表名 / 约束设计；alembic autogenerate | `.tables` / `.schema` 反映 |
| **C 业务逻辑改动**（若适用） | 具体脚本 / 模块 | 关键策略：分类 / 抽取 / 计算公式的设计意图 | 单测或抽样验证标准 |
| **D 新建采集脚本 / 模块**（若适用） | 路径 | 设计要点：调用什么 API / 怎么 throttle / 落库语义 | 验证 SQL 或脚本期望输出 |
| **E 验证脚本** | 新建 `training/scripts/validate_xxx.py` 之类 | 自动按 §3.5 18 项扫 DB 覆盖率，输出 ✅/⚠️ + 证据 | 退出码 0；输出每行状态 |
| **F 单测覆盖** | `tests/<module>/test_xxx.py` | 关键路径覆盖：分类 / 抽取 / 边界 / 异常 | `pytest -q` ≥ N passed |

> 写作要点：**只写"做什么 / 设计决策 / 验证标准"，不写完整代码、不写每个命令的命令行参数**。后者由 L4 实践记录 / 后续执行模型自行落地。

#### §7.2 Makefile 一键复现合约（若本步涉及多步流水线 · 强制）

**设计目的**：架构师**只改配置文件**（如 `data/config/my_holdings.yaml`）就能扩 / 缩 / 调整范围，跑**一条命令** `make <module>-step<N>-all` 完成端到端"采集 → 质量检测 → 锁库 → DVC → 单测"全套。**禁止**在 Makefile 或脚本中 hardcode 标的代码、年份、阈值等。

**target 合约表**（L3 此处只定义合约 · 实现交 L4 / `diting-src/Makefile`）：

| target | 用途（一句话） | 入参（环境变量 · 均有默认） | 验证标准 |
|---|---|---|---|
| `make <module>-step<N>-prep` | 前置（migration / 自检） | 关键环境变量 | 退出码 0 |
| `make <module>-step<N>-collect` | **一键采集** | 关键环境变量 | 数据落库；行数 ≥ 期望 |
| `make <module>-step<N>-quality-check` | §3.5 矩阵复核 | — | 18 行 ✅/⚠️；退出码 0 |
| `make <module>-step<N>-test` | 单测 | — | `pytest -q` ≥ N passed |
| `make <module>-step<N>-all` | **端到端一键**（含上述 + 其他必须步骤） | 同上合并 | 全部退出码 0；耗时 ≤ X min |
| `make <module>-step<N>-status` | 只读 · 进度快照 | — | 打印 COUNT + §3.5 覆盖率 |
| `make <module>-step<N>-clean` | 清产出（保留不可删的，如 Holdout） | — | 待删的产出已删 |

**合约要求**（L4 实现时必须遵守）：
1. **入参全部环境变量化**：命令行 / `.env` / Makefile 顶部三层叠加优先级；
2. **target 是薄包装**：核心动作在 Python 脚本里，Makefile 只串环境变量与调用顺序，**不**写业务逻辑；
3. **可重入幂等**：重跑 `*-all` 不破坏已有数据（按业务主键 upsert 而非 insert）；
4. **配置驱动**：变更范围只改配置文件，**禁止**改 Makefile 或脚本内 hardcode；
5. **失败可观察**：每个 target 输出"做了什么 / 期望什么 / 实际什么"3 行摘要（中文优先）。

> 若本 step 仅一两条命令、不涉及多步流水线（如 step_01 服务骨架），可省略本小节；其余采集 / 训练 / 评估类 step **强制**列 Makefile 合约。

#### §7.3 给后续执行模型的指引（步骤与边界）

L4 / 执行模型按以下顺序，**不偏离 §7.1 实现要点 + §7.2 合约**：

1. **核对前置**：配置文件 / 上游 step 产出 / 凭证齐备 → 否则不开工；
2. **逐项落地 §7.1 表**：每项产出独立可跑，落地后跑对应"验证标准"；
3. **集成 Makefile**（若适用）：按 §7.2 合约表实现 target，过 `*-all` 端到端验证；
4. **§9 准出清单逐项打勾** + 同会话给证据；
5. **回写 L4 实践记录**：⚠️ 项实际状况、§3.5 矩阵填表、commit hash、耗时；
6. **遇问题**：按 `00_系统规则` §7.2 第 6 条（Verify First）先验证再改；同问题修复重试 ≥ 2 次仍失败再回收（§8.4f）。

> **L3 责任边界**：本节给规划 + 实现要点 + 验证标准；**不**给完整 Makefile / 脚本代码、不写每个命令。具体落地由 L4 实践记录 / 后续执行模型按本节规划自行完成。
> **禁止 Mock**：所有实现要点与 target 均不允许 `*_MOCK=1` / `--use-mock` / `mock_*.json`；上游真流未就绪前**等待**而非伪造。占位 fixture **仅**用于 `tests/` 单元测试，**禁止**入业务库 / 禁止上 staging。

### §8 部署节奏（本步在哪里跑）

## 三、AI 写作守则

- **禁止臆造数据源**：用 Akshare / 巨潮 / Eastmoney 等已在仓内验证过的源；新源须先在 §4.1 注明并通过单测。

- **凭证清单要"用户可勾选"**：让用户读完 §4.2 后可一次性把 key 填进 `.env`，然后让 AI 接管执行。
- **部署节奏要"按本步层级"**：不要写"启动期所有步骤都需要上 K3s"，那是 §六里的环境节奏闸；本步只写"本步**必须**做到哪一层"。
- **准出要"同会话可执行"**：每条命令 AI 必须能直接运行；不能用"建议执行" / "请你确认"等收尾。
