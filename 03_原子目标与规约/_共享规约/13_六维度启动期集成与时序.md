## 一、为什么需要本文档

启动期 6 维度并行开发，**单独看每个维度都能跑通，但合起来未必跑得通**。本文档解决：

1. **端到端流程**：用户打开 App 到看到一个推荐，背后 6 维度如何协作？
2. **依赖时序**：哪个维度必须先就绪？哪个可以并行？
3. **事件契约**：6 维度之间通过什么事件流通信？
4. **集成测试边界**：每个维度的输入/输出契约是什么？

## 三、典型端到端流程（4 个核心场景）

### 场景 A：新机会进入推荐池（用户主动浏览）

```mermaid
sequenceDiagram
  participant U as 👤 用户
  participant D2 as 02·纵深进攻
  participant D1 as 01·极寒防御
  participant D0 as 00·副驾驶
  
  Note over D2: 剧本扫描，发现潜在标的
  D2->>D1: 调用 decision_gate.check(symbol)
  D1->>D1: 3 引擎并行评分 + 一票否决
  alt 通过防御
    D1-->>D2: PassEvent(symbol)
    D2->>D2: 生成 thesis 卡片（5 必填）
    D2->>D0: events:thrust:thesis_proposed
    D0->>D0: 写入 thesis_cards 表，进入推荐池
    U->>D0: 浏览推荐池
    D0-->>U: 显示推荐 + thesis 详情
  else 防御拒绝
    D1-->>D2: RejectEvent(symbol)
    D2->>D2: 写入"已研究未持仓"日志（负样本）
  end
```

**端到端 SLO**：从 D2 剧本命中到 D0 推荐池可见 ≤ 5 分钟（含 D1 检查）

---

### 场景 B：持仓健康度突变（系统主动告警）

```mermaid
sequenceDiagram
  participant D3 as 03·持仓监控
  participant D4 as 04·卖出决策
  participant D0 as 00·副驾驶
  participant U as 👤 用户
  
  Note over D3: SLI 探针采集 → 健康度计算
  D3->>D3: 节点状态从 stable → warning
  D3->>D0: events:monitor:health_change(push_level=2)
  D0->>U: 微信/Telegram/邮件橙色告警
  
  alt 健康度继续恶化 < 30
    D3->>D3: 节点状态 warning → exit
    D3->>D4: events:monitor:health_change(thesis_status='invalid')
    D4->>D4: 协议 SP3 触发：Thesis 失效
    D4->>D0: events:exit:sell_signal
    D0->>U: 红色告警（5 分钟内）
  end
```

**端到端 SLO**：健康度突变到用户收到红色告警 ≤ 5 分钟（5 分钟到达率 ≥ 99.5%）

---

### 场景 C：止损线触发（系统主动告警）

```mermaid
sequenceDiagram
  participant D4 as 04·卖出决策
  participant D0 as 00·副驾驶
  participant U as 👤 用户
  
  Note over D4: 行情数据每 30 分钟刷新
  D4->>D4: 检查所有持仓 (current_price / cost_price - 1)
  D4->>D4: SP1·止损协议触发（涨跌 ≤ -15%）
  D4->>D0: events:exit:sell_signal(signal_type='stop_loss')
  D0->>U: 红色告警 + 详情页
  U->>D0: 用户决策（卖/不卖）
  D0->>D0: 记录用户决策（用于 8 象限归因）
```

---

### 场景 D：模型自我进化（基础设施支撑）

```mermaid
sequenceDiagram
  participant D0 as 00·副驾驶
  participant D5 as 05·演进飞轮
  participant D1 as 01·极寒防御
  
  Note over D0: 用户标注暴雷案例（启动期手动）
  D0->>D5: 写入 Label Studio
  D5->>D5: 双盲 Kappa 校准 ≥ 0.80
  D5->>D5: Teacher 蒸馏新案例
  D5->>D5: LLaMA-Factory 训练新 LoRA
  D5->>D5: Holdout 评测 ≥ 0.90
  
  alt 评测通过
    D5->>D5: 灰度发布（手动）
    D5->>D1: 加载新 LoRA（vLLM 热加载）
    D5->>D0: events:flywheel:lora_updated
  else 退化 > 5%
    D5->>D5: CI Block，回退
  end
```

### 关键依赖链（自上而下）

```
05·演进飞轮（Teacher + LLaMA-Factory）
  ↓ 提供 LoRA 能力
01·极寒防御（3 引擎需要 LoRA）
02·纵深进攻（thesis LoRA）
03·持仓监控（叙事 NLI LoRA）
  ↓ 提供事件流
04·卖出决策（依赖 D3 的 health_change）
  ↓ 提供事件流
00·AI 投资副驾驶（消费所有事件）
```

**严格依赖**：
- 没有 05 就没有任何 LoRA（卡 D1/D2/D3 训练）
- 没有 D3 的 health_change，D4 的 SP3·Thesis 失效无法触发
- 没有 D1/D2/D3/D4 的事件流，D0 的 4 子模块没东西显示

**并行机会**：
- D5 蒸馏 + D1/D2/D3 数据采集 可以并行
- D4 规则引擎可以先用 mock 数据开发，等 D3 事件流就绪后接入

## 六、集成测试边界

每个维度对外暴露的契约（必须稳定）：

### 维度零（消费者）
- **输入**：6 个 Redis Stream
- **输出**：用户界面 + 推送
- **测试边界**：用 mock 事件流驱动 UI 验证

### 维度一
- **输入**：HTTP POST `/api/decision-gate/check {symbol, name, context}`
- **输出**：`{decision, audit_id, engine_results}` + RejectEvent/PassEvent
- **测试边界**：50 案例 Holdout + 100 正常公司白名单

### 维度二
- **输入**：HTTP POST `/api/playbooks/{id}/scan {symbol}` + PassEvent
- **输出**：ThesisProposedEvent
- **测试边界**：5 必填完整性 + 与人工一致率

### 维度三
- **输入**：用户持仓列表 + thesis 卡片
- **输出**：HealthChangeEvent
- **测试边界**：10 持仓模拟测试（含状态切换）

### 维度四
- **输入**：持仓数据 + HealthChangeEvent
- **输出**：SellSignalEvent
- **测试边界**：100 笔历史回测，触发准确率

### 维度五
- **输入**：标注数据 + Holdout 案例
- **输出**：LoRA 权重 + LoraUpdatedEvent
- **测试边界**：双盲 Kappa ≥ 0.80 + Holdout 不退化

## 七、首日用户体验场景（产品价值锚定）

为确保启动期"用户第一天打开就有用"，按以下顺序确认：

| 用户场景 | 最小可用版本 | 涉及维度 | 必须就绪日 |
|---|---|---|---|
| **第一眼**：看到持仓状态 | 4 色卡片 + 持仓列表（**允许 Mock 或未接真流**：与 [14§4.3](./14_六维度启动期统一节奏表.md) Mock 一致；**接入 D3 `health_change` 真流的体验以 M3/W7 为准**）| D0 + D3 | **W5 体感完整**；**W4 可为骨架/占位/mock** |
| **第二眼**：看到推荐 | thesis 卡片（即使只有 1 条）| D0 + D2 + D1 | W5 末 |
| **响应突变**：红色告警 | 至少微信通道可达 | D0 + D4 | W6 末 |
| **月底回顾**：月报 | SCS + EV + 月报 PDF | D0 + 全维度 | W10 末 |

> **说明**：本节「首日」强调的是**可被用户解释的页面与数据**，不强制当日已是生产级真流事件；与 [§二 里程碑表](./14_六维度启动期统一节奏表.md) 列「维度零」的周序对齐时——**时间表以 §14 矩阵与本表「必须就绪日」列为准**。上架路径同步见 **§八**。
