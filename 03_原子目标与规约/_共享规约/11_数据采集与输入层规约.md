## 问题：数据源缺失

**现状**：09_ 核心模块架构将市场数据、新闻数据作为输入层，但未定义**如何采集、写入、供 Module A/B/C 消费**。Module A-F 验证依赖 Mock，无法用真实市场数据验证系统。

**目标**：将数据采集与输入层作为**第一优先**，在 Module A-F 实现之前落地，为全链路真实数据验证提供前提。

### 与 07_、09_ 的衔接

- **07_ 数据版本控制**：采集写入时遵循 DVC 版本化策略；版本标识规则见 07_。
- **09_ 输入层**：本规约定义的采集输出，对应 09_ 架构图中「市场数据 OHLCV」「新闻数据 公告/研报」及 Module A 所需的申万行业、营收占比。
- **05_ MarketDataFeed**：消费层通过 MarketDataFeed 从 L1 读取 OHLCV；采集层负责**写入** L1。

### 全 A 股标的池（universe）

Module A、Module B 的标的池 = **当前全 A 股**，数量随上市/退市变化，不写死。以下约定标的池的存储、写入、更新策略与读取方式。

| 维度 | 约定 |
|------|------|
| **存储** | **L1/L2 表**：表名与字段由项目约定（建议至少含 `symbol`、`market`、`updated_at`；可选 `count`、`source`）。与 L1 OHLCV、行业数据同库，便于同批口径一致。 |
| **写入方** | **Stage2 某采集 Job** 或 **专用 universe 更新 Job**；Job 内可调 AkShare/OpenBB 等获取当前全 A 股列表并写入上表。实现与触发方式见 [Stage2_数据采集与存储](../Stage2_数据采集与存储/) 对应设计/实践。 |
| **更新策略** | **按需刷新（check-before-use）**：每次需要获取标的池时，先根据**有效条件**判断表内数据是否仍有效；若有效则直接读表返回，若无效则先执行更新（调用写入 Job 或等价逻辑）再读表返回。 |
| **有效条件** | 默认建议 **「当日已更新」**（`updated_at` 为当日）或 **「当前交易日已更新」**；具体判断规则在实现中读取（可配置或按环境扩展，扩展说明可置于本小节或 DNA）。 |
| **读取方式** | diting-core 通过统一接口 **`get_current_a_share_universe() -> List[str]`** 读取；该接口内部实现「检查有效条件→若无效则触发更新→读表返回」。接口语义与 09_ Module A/B 的标的池来源一致。 |

同一批次/同一日任务中，Module A 与 Module B 须使用**同一份**标的池；由调度/编排层一次调用 `get_current_a_share_universe()` 并将结果传入 A、B，或在同一执行流程入口处一次获取后分别传入，见 [09_核心模块架构规约](./09_核心模块架构规约.md) Module A、Module B 的「标的池来源与同批一致」。

### 数据采集与存储可部署单元（模块级边界）

本规约对应的**可部署单元**为「数据采集与存储」，与 [09_ 可部署单元清单](./09_核心模块架构规约.md#可部署单元清单) 一致。

| 维度 | 约定 |
|------|------|
| **边界** | 本单元**仅负责**：从外部数据源采集、DVC 版本化、写入 L1/L2/L3；不包含消费逻辑。 |
| **消费方与契约** | **Module A**：通过约定表或 Redis 缓存读取申万行业、营收占比；L2 标的主营构成表供 A 扩展输出主营占比列表（见 [12_右脑数据支撑与细分规约](./12_右脑数据支撑与Segment规约.md)）。**Module B**：通过 MarketDataFeed 从 L1 读取 OHLCV。**Module C**：从 L2 知识库与 L3 冷归档读取新闻/研报；并消费标的细分列表与**细分信号缓存表**（垂直一手信号由信号层按候选按需拉取，见 12_）。消费方不部署在本单元内。 |
| **镜像与部署** | 独立 Dockerfile、独立镜像（如 `diting-ingestion`）；部署为 K3s Job/CronJob 或常驻 Worker，由 DNA 与 [02_基础设施与部署规约](../开发与交付/02_基础设施与部署规约.md) 约定。 |
| **依赖** | 依赖 TimescaleDB、Redis、L2 知识库、OSS（L3）；不依赖 Module A/B/C 进程。 |
| **Scale-to-Zero** | 本单元可 Scale-to-Zero（任务结束后缩容）；见成本治理与 DNA。 |

L4 实践与 1:1:1 设计-DNA-步骤见 [03_/Stage2_数据采集与存储](../Stage2_数据采集与存储/)、[04_/Stage2_数据采集与存储](../../04_阶段规划与实践/Stage2_数据采集与存储/)。

### 生产级数据要求

以下约定为 **Module A、Module B 生产级运行**所需的数据深度、口径与验收；详细要求与可执行验收见 [Stage2 步骤 06_生产级数据要求_设计](../Stage2_数据采集与存储/06_生产级数据要求_设计.md) 与 [06_生产级数据要求_实践](../../04_阶段规划与实践/Stage2_数据采集与存储/06_生产级数据要求_实践.md)。

| 维度 | 约定 |
|------|------|
| **OHLCV 历史深度** | 单标的日线 **≥5 年**（单标的日线可追溯长度）；与 L1 数据保留/归档策略一致（保留周期见基础设施或 DNA）。主粒度日线，可选分钟。 |
| **复权** | 与**回测、实盘一致**；实现与 Stage4 回测/实盘文档或配置对齐。 |
| **标的覆盖** | 生产扩展时与 `get_current_a_share_universe()` 一致；覆盖率、数据新鲜度、采集失败告警与**运维/告警**约定一致。 |
| **行业/营收与板块** | Module A：申万层级、营收占比等最小字段集见 Stage2-02 与约定表。Module B Sector Strength 所用板块划分与 Module A 行业数据**同源**（如申万），同表或同接口。 |

## A. 数据源定义

| 数据类型 | 用途 | 目标存储 | 采集方式 | 备注 |
|---------|------|---------|---------|------|
| **市场行情（OHLCV）** | Module B 量化扫描 | L1 TimescaleDB | API 定时拉取（Tushare/Wind/自建） | 见 data_architecture.l1_hot |
| **新闻/公告/研报** | Module C 专家推理、知识库 | L2 知识库、L3 冷归档 | API 或爬虫；写入 Agri-KG/Tech-KG/Macro-KG；**行业新闻与标的新闻分离存储**见 [Stage2-07_行业新闻与标的新闻分离存储_设计](../Stage2_数据采集与存储/07_行业新闻与标的新闻分离存储_设计.md) | 见 data_architecture.l2_knowledge |
| **申万行业、营收占比** | Module A 语义分类 | L1 或 L2；可缓存在 Redis | API（Wind/Tushare 等） | Module A 输入 |
| **主营业务构成（细分+占比）** | Module A 扩展输出、Module C 右脑 | L2 标的主营构成表（symbol_business_profile） | 采集任务（见 [12_右脑数据支撑与细分规约](./12_右脑数据支撑与Segment规约.md)） | Module A、信号层、Module C |
| **配置/策略参数** | 全局 | L2 PostgreSQL | 见 06_ 动态配置 | 非采集主路径 |

### 逻辑填充期数据源选型（国内 / 国际）

逻辑填充期采集层**必须**接入以下开源数据接口，与 [01_顶层概念/05_谛听优先借鉴的十大开源选型](../../01_顶层概念/05_谛听优先借鉴的十大开源选型.md) 一致；具体接口边界、错误与限流、写入契约及验收要点见 [Stage2_数据采集与存储/02_采集逻辑与Dockerfile设计](../Stage2_数据采集与存储/02_A轨_采集逻辑与Dockerfile_设计.md) 中「逻辑填充期开源接入点」小节。

| 选型 | 用途 | 供给关系 |
|------|------|----------|
| **国内：AkShare** | 国内行情、行业、宏观、公告等数据的**统一 Python 接口** | **Module A**：申万行业、营收占比（约定表或 Redis）；**L2 知识库**：Agri-KG/Tech-KG/Macro-KG 国内侧数据；**Module B**：A 股 OHLCV 写入 L1。无国内数据则 Module A 与全链路真实数据验证无法跑通。 |
| **国际/宏观/基本面：OpenBB** | 宏观、大宗、经济、基本面（财报、营收等）作为 Module A 与 L2 的**国际侧数据源**，与 AkShare 形成国内外互补 | **Module A**、**L2 Macro-KG**、**B 轨基本面**：财报、营收增速、研发占比、大宗、汇率、宏观数据。采集层采用「轻量 Core + Provider 可插拔」设计，与 05_ 选型一致。 |

## B. 采集任务与调度

### 任务类型

| 任务 ID | 数据源 | 写入目标 | 调度频率 | 超时 |
|---------|--------|---------|---------|------|
| `ingest_ohlcv` | 行情 API | TimescaleDB | 日级/分钟级（按需求） | 见 DNA |
| `ingest_industry_revenue` | 行业/财报 API | 约定表或 Redis 缓存 | 日级 | 见 DNA |
| `ingest_business_profile`（主营构成） | AkShare `stock_zygc_em`（东方财富主营构成） | L2 `segment_registry` + `symbol_business_profile` | 日级/季报后（`INGEST_BUSINESS_REFRESH_DAYS`） | 见 12_、Stage2-02 |
| `ingest_news` | 新闻 API | 知识库 + L3 冷归档 | 小时级 | 见 DNA |

上述任务中，国内数据（A 股行情、申万行业、营收占比、国内新闻/公告）由 **AkShare** 供给；国际/宏观/基本面数据由 **OpenBB** 供给。实现细节与验收见 [Stage2-02 采集逻辑与Dockerfile设计](../Stage2_数据采集与存储/02_A轨_采集逻辑与Dockerfile_设计.md#design-stage2-02-exit) 中「逻辑填充期开源接入点：AkShare / OpenBB」小节。

### 调度与执行

- 采集任务由 **cron / Airflow / 自研调度** 触发；具体选型由项目在 DNA 中约定。
- 采集失败时：重试（指数退避）、告警；不阻塞下游模块读取缓存数据（见 09_ fallback_strategies）。

### 降级策略（数据源不可用）

当 AkShare、OpenBB 等主数据源**长期不可用**（如 IP 封禁、接口下线）时：

| 策略 | 说明 |
|------|------|
| **缓存/历史** | 在约定范围内可使用最近一次成功写入的缓存或历史数据，保证下游 Module A/B 可读；须在可观测性中标注数据新鲜度。 |
| **备用数据源** | 若项目约定备用数据源（如另一 Provider），可在 L3 或 DNA 中配置切换条件与优先级；未约定则仅告警与人工介入。 |
| **告警与人工介入** | 须触发告警并记录；运维须有 Runbook 或人工决策流程（切换源、暂停采集、降级展示等）。 |

## C. 写入契约

### L1/L2 与 PostgreSQL、TimescaleDB 关系

默认部署中 **L1 和 L2 均为 PostgreSQL 系**：

| 存储 | 实现 | 与 PG 关系 |
|------|------|-------------|
| **L1** | TimescaleDB | **TimescaleDB = PostgreSQL + 时序扩展**（同一进程、同一协议，可视为 PG 的扩展）；本地 Compose 可用 `postgres` 镜像模拟，生产/K3s 用 `timescale/timescaledb-ha`。 |
| **L2** | PostgreSQL | **独立 PostgreSQL 实例**（Bitnami 或官方镜像），存知识库、`data_versions` 等；与 L1 协议兼容、均为 PG。 |

因此「默认部署 L1/L2 是否有 PG」：**有**，L1 是 PG+TimescaleDB，L2 是纯 PG。

### Redis：默认部署且必验

**Redis** 为默认部署组件（**非可选**），用于缓存/队列（如行业、营收缓存；后续 Module 与执行网关 Redis Streams）。默认部署须包含 Redis；`make verify diting prod`（及 `verify-db-connection`）须校验 Redis 连通性并执行**缓存/队列效果与简单性能测试**（如 PING、SET/GET 延迟），未通过不得准出。

### L1 TimescaleDB（OHLCV）

- Schema 由 `diting-core/design/schemas/sql/` 或项目约定路径定义。
- 写入延迟须满足 `data_architecture.l1_hot.timescaledb.write_latency_ms_max`。
- 采集脚本/任务须通过 Great Expectations 或等价校验，再写入。

### L2 知识库（新闻/事件）

- 写入 Agri-KG、Tech-KG、Macro-KG；向量维度与 DeepSeek-R1 嵌入一致。
- 新闻/公告按 07_ 版本化粒度写入 L3。

### 与 MarketDataFeed 的契约

- MarketDataFeed（TimescaleDBFeed 等）**读取** L1；采集层**写入** L1。
- 采集完成并提交 DVC 后，消费层可通过 `get_history`、`get_snapshot` 获取数据。

## D. 可执行验证

采集与输入层准出时，须满足：

1. **写入可验证**：至少一种数据源（如 OHLCV）可成功写入 L1，并可通过 MarketDataFeed 读取。
2. **Module A 输入可获取**：申万行业、营收占比可通过约定接口或缓存获取。
3. **可执行命令**：`make ingest-test` 或 DNA 约定的验收命令退出码 0。
