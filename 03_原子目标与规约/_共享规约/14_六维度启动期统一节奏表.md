## §1 三句话上手

1. **持仓 SoT**：所有维度从 `diting-src/data/config/my_holdings.yaml` 读取持仓清单；启动期默认**只**对 SoT 中 `active=true` 的标的拉真实数据（典型 1～10 只），扩展期再扩到候选池。模板：[`my_holdings.example.yaml`](../../../../diting-src/data/config/my_holdings.example.yaml)。
2. **L3 step 模板**：每个 step 必须遵守 [`L3_启动期step_重构模板`](./L3_启动期step_重构模板.md)，含 13 个强制小节（数据对象 / 真实数据源 / 凭证清单 / 启动期目标 / 下一步一行 / 部署节奏 / 准出 / [Deploy] / 依赖 / 风险 / 修订记录）。
3. **执行序与日历行**：开发**不以周次执行**，以各维 `steps/README.md` 顺序为准（§3）；§4 的"日历行 Wn"仅是产品视角的并行节拍对齐，便于跨维契约冻结与里程碑（Mn）排期。
4. **用户可用路线（MVP）**：若目标是「先邮件/告警能用上」，以 L4 **[实践记录_MVP_W1-W4_用户可用路线](../../04_阶段规划与实践/00_维度零_AI投资副驾驶/stage_1_启动期/实践记录_MVP_W1-W4_用户可用路线.md)** 为纵向切片执行序；**本表仍管**六维并行、凭证与 Mn。

### §8.1 数据质量优于数据量（启动期通则 · 强制）

> **巴菲特原则**：不接受"看了三表就算看了财报"——数据采集是**为给模型引擎做实质分析**，**引擎的分析需求决定字段粒度**，不是反过来用"行数 / 条数能凑齐"作为准出。

**总则**：启动期标的可以少（典型 1～10 只持仓 SoT），但**对这些标的的数据采集深度必须达到引擎能做实质分析的程度**。任何采集 step（D1 step_02 / D2 step_02 / D3 step_02~03 等）须在 L3 §3.5 写**数据质量验收矩阵**，按下游消费引擎反推所需的分析维度（参见 [`L3_启动期step_重构模板`](./L3_启动期step_重构模板.md) §3.5）。

**典型分析维度（不止于"表面层"）**：

| 数据大类 | 表面层（启动期**不达标**） | 引擎要的深度（启动期**达标**） |
|---|---|---|
| **财报** | 三表 raw + 几个数字 | 附注关联交易、应收账款账龄、研发资本化口径、商誉构成与减值、坏账政策、政府补助归类、季度连续趋势、行业基线对比、现金流分布异常 |
| **公告** | 标题 + 日期 + 粗分类 | PDF 全文、承诺履约进度、累计变动序列、董监高变动密度、解禁时间表、问询函与监管处罚、对赌履约、诉讼记录 |
| **关联交易** | 关联方文本行 | 关联方网络图（穿透实控人）、金额占比时间序列、定价方法异常、循环交易识别、明股实债条款、关联担保与资金占用 |
| **舆情新闻** | 标题与时间 | 情绪打分、来源权重、跨源同事件去重、与披露的一致性核对 |
| **行情** | 收盘价 | 复权、换手率、北向资金、龙虎榜、大单流向、行业动量 |
| **持仓** | 数量 + 成本 | 时间序列变动、加减仓事件、对账与历史对照 |

**违规**：
- 仅以"行数 / 条数"作为采集 step 的准出（如 `financial_reports ≥ 16` 而不核验字段非 null 率与粒度）；
- 把 ❌ 字段假装填 null 然后视为达标；
- 下游引擎需要 X 字段而本步只产出 Y 字段，且未在 §3.5 给出降级或扩展期承接路径。

**修复**：按 L3 §3.5 模板补质量矩阵，并对照本节"引擎要的深度"列扩展采集脚本。
