## 一、Schema First

- 所有跨模块消息都必须先定义 Schema，再进入实现。
- Schema 可以是 `proto`、JSON Schema、Pydantic 模型或等价契约，但必须有版本号和兼容策略。
- 代码里临时拼装、无法追踪版本的数据结构，不允许越过模块边界。

## 二、协议类型

| 协议类型 | 作用 | 必带字段 |
|----------|------|----------|
| **Research Protocol** | 研究输入、检索结果、摘要、候选、评测对象 | `request_id`、`session_id`、`source_ref`、`schema_version` |
| **Decision Protocol** | 决策建议、优先级、门禁结果、拒绝原因 | `decision_id`、`correlation_id`、`policy_result` |
| **Runtime Protocol** | WebSocket / SSE / task 状态同步 / resume | `session_id`、`resume_token`、`heartbeat_ts` |
| **Ops Protocol** | 训练、评测、注册、部署、回滚事件 | `run_id`、`model_version`、`environment` |
| **External Action Protocol** | 通知、工单、外部 API、执行请求 | `action_id`、`idempotency_key`、`audit_ref` |

## 三、强制字段

所有跨服务协议都必须支持以下最小字段集合：

- `correlation_id`：串起一次请求的全链路日志和追踪。
- `schema_version`：识别协议版本与兼容范围。
- `created_at`：用于排查时序与重放。
- `producer` / `consumer`：用于审计依赖关系。

## 四、长连接和运行时协议要求

- WebSocket/SSE 协议必须显式定义心跳、断线重连、恢复令牌与错误码。
- browser/computer-use runtime 的任务协议必须定义权限、资源上限、产物位置与失败原因。
- 任何“需要人工确认”的协议都必须可审计，可复放，不允许只有前端状态没有后端证据。

## 五、与岗位能力的关系

- **岗位 1**：关注推理与跨服务协议的吞吐、延迟、批量化与可观测字段。
- **岗位 2**：关注决策协议里的指标、门禁、灰度与回滚语义。
- **岗位 3**：关注 Ops Protocol 的实验、注册、部署与漂移字段。
- **岗位 4**：关注 Runtime Protocol 的长连接、恢复、安全与执行证据。
