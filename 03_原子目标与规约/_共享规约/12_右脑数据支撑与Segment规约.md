## 一、目标

`Segment` 不再只是旧 B 轨叙事的补充数据，而是研究议会理解“一个标的到底靠什么业务、受什么事件影响”的关键中间层。

## 二、主责

- 为 Module A / Module C 提供更细粒度的主营构成与细分语义。
- 让研究议会能判断“事件是否真正作用到主营业务”。
- 为后续 feature store、检索增强和标签扩展提供稳定键空间。

## 三、最小落地要求

| 能力 | 说明 |
|------|------|
| **主营构成** | 标的与主营业务的结构化映射，至少可用于画像和研究解释 |
| **Segment 注册** | 细分领域必须有统一 `segment_id`，不能在不同模块里随意命名 |
| **补充上下文** | 可为研究议会按需提供更细分的新闻、事件、行业补充信息 |
| **按需拉取** | 避免无上限全量预抓，优先服务当前候选和活跃研究对象 |

## 四、与岗位能力的关系

- **岗位 1**：按需拉取、缓存与性能成本权衡。
- **岗位 2**：判断哪些细分值得建、哪些只需保留接口。
- **岗位 3**：把 Segment 变成可复用特征与评测资产。
- **岗位 4**：让 Agent 能安全地消费这些细分上下文，而不破坏 runtime 边界。

## 五、约束

- Segment 体系必须有注册表或等价真相源。
- Segment 相关数据必须能回到原始来源或采集任务。
- 不能再把它写成旧 `B轨` 的专属概念；它是共享研究能力的一部分。
