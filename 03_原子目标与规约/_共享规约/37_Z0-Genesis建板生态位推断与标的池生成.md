## 一、设计目标

### 1.1 问题

| 问题 | 描述 |
|------|------|
| **概念标签粗粒度** | A 股概念标签（同花顺/东方财富）是第三方人工标注分类，不存在层级关系，不具备「AI 产业链」的语义。中际旭创属于「光模块」而非「AI」，但其 800G 光模块需求 100% 由 AI 驱动 |
| **选股无工程标准** | 旧版本 Prompt 只要求「选最有成长潜力的 5-10 只」，没有多因子筛选、没有证据链、没有排除规则，选出结果无法审计 |
| **打分数值不可验证** | 旧版 confidence 仅凭 LLM 感觉，无因子拆解、无判断理由、无法交叉验证 |
| **人类无介入点** | LLM 决定了一切产业链范围和标的名单，人类只能在结果出来后被动接受或拒绝，没有过程纠偏能力 |
| **权重无来源** | 打分权重若存在则纯凭经验设定，无回测/校准路径 |

### 1.2 目标

```
政策确认方向 (Z0+) → BOM 定义范围 (L3 白名单) → LLM 映射标的 → 多因子打分 + 证据链 → 人审确认 → Z1 验证
```

**核心原则**：
1. 标的范围由 **BOM 拆解 (产业链结构)** 定义，而非 A 股概念标签
2. 每只标的附带 **5 因子打分 + 可验证证据链**，所有判断可审计
3. **人类架构师**定义 BOM 白名单和边界规则，LLM 在白名单内执行
4. A 股概念标签 **仅用于 policy_bond 加分**，不限制候选范围
5. 排除规则（ST/*ST、立案调查、商誉暴雷、审计非标）一票否决

### 2.3 5 因子打分体系

| 因子 | 权重 | 含义 | 评分范围 | 证据要求 |
|------|------|------|---------|---------|
| **moat** | 0.30 | 市占率 / 技术壁垒 / 客户锁定 | 0-1 | 至少2条证据：市占率/排名、专利/技术认证、客户绑定 |
| **growth** | 0.25 | 赛道增速 / 产能扩张 / 政策加速度 | 0-1 | 下游CAGR、公司扩产计划、政策直接扶持 |
| **profit** | 0.20 | 毛利率趋势 / ROE / 经营现金流 | 0-1 | 毛利率及趋势、ROE、经营现金流/净利润 |
| **localize** | 0.15 | 进口依赖 / 出口管制 / 国产化率 | 0-1 | 进口依赖程度、管制状态、国产化现状 |
| **policy_bond** | 0/0.2 | A股概念标签是否在Z0+信号中 | 0或0.2 | 匹配到的概念标签名称 |

**总分公式**：`composite = moat × 0.30 + growth × 0.25 + profit × 0.20 + localize × 0.15 + policy_bond`

**权重声明**：当前 v1.0.0 为初始设定。Z1 阶段累积 ≥30 条 ground_truth 后，跑 grid_search 校准，校准结果人工审批后替换。

### 2.4 排除规则（一票否决）

```
以下任意一条触发 → 该标的从候选池中排除，记录在 excluded_stocks：

1. ST / *ST 标注
2. 近 12 个月被证监会立案调查
3. 商誉 / 净资产 > 50%（谨慎限制，conf ≤ 0.6）
4. 近 2 年审计意见为「非标准无保留」
```

### 2.5 概念标签的使用范围

| 场景 | 使用方式 |
|------|---------|
| 限定候选范围 | ❌ **不得使用**。BOM 白名单才是范围定义来源 |
| policy_bond 加分 | ✅ 标的所属概念标签命中 Z0+ 政策信号 → +0.2 |
| 前端展示 | ✅ 展示在标的详情中作为参考信息 |

### 2.6 LLM 行为约束

```
Prompt 硬约束（不可覆盖）：

1. 你必须在白名单所列节点范围内进行标的映射。
2. 你不得擅自新增/删除节点。
3. 如果你认为白名单外有值得考虑的新节点，写入 suggested_additions 字段。
4. 每个标的必须输出 scoring_detail（含5因子打分 + 证据链）。
5. 每个因子必须附带至少2条可验证的 evidence。
6. 概念标签信息仅用于 policy_bond 加分判断，不可用于限制候选范围。
7. 排除规则检查必须逐项执行，结果记录在 exclusion_check。
```

### 2.7 BOM 白名单生命周期

| 事件 | 触发 | 动作 |
|------|------|------|
| 首次创建 | 架构师编写 | 基于产业研究报告定义初始节点 |
| LLM 建议新增 | 每次标的映射输出 suggested_additions | 架构师审批后入库 |
| 定期复审 | 每季度 CI 提醒 | 检查新技术方向、调整 tier |
| 版本升级 | 节点增删 | MAJOR: 核心节点变动 · MINOR: tier 调整 · PATCH: 修正描述 |
| 边界争议 | 架构师仲裁 | 记录在 edge_case_log |

### 2.8 可验证性的三步闭环

```
Step A (LLM 生成):
  输出 judgment + evidence + evidence_source

Step B (架构师即时检查):
  对每个 evidence 做合理性判断
  标注: verified / suspicious / needs_update

Step C (Z1 自动验证):
  拉取真实财务数据，对比 profit.evidence
  不一致 → 标记为 stale，降级或重新评估
```

## 六、前端 UX 交互设计（v2.1 · 用户操作闭环）

### 6.1 设计目标

标的池结果不只是「只读展示」，用户需要能**审核、筛选、操作和向下游推送标的**。覆盖三级操作：

```
浏览（Read）   → 审核（Review）   → 推送（Promote）
├ 节点分组     ├ 代表/增补区分    ├ 加入猎物池
├ 因子打分表   ├ 接受/排除标的    ├ 推送到 CVM
└ 证据链       └ 调整评分         └ Z1 Gate-A
```

### 6.2 标的来源视觉区分

每只标的在 `<summary>` 行右侧增加来源徽章：

| 徽章 | 来源 | 条件 |
|------|------|------|
| 🏷 **代表标的** | 紫色 badges · `bg-purple-100 text-purple-700` | 标的 `stock_source == "curated"`，来自 YAML 白名单 |
| 🤖 **LLM 增补** | 灰色 badges · `bg-gray-100 text-gray-600` | 标的 `stock_source == "llm"`，LLM 自行生成 |
| ⚠ **待复核** | 琥珀色 badges · `bg-amber-100 text-amber-700` | 缺部分证据或分数低于阈值 |

**存储字段**：`stock_pool_json.bom_nodes[].stocks[].stock_source: "curated" | "llm"`

### 6.3 标的操作按钮

每只标的摘要行右侧增加三个按钮：

| 按钮 | 动作 | 后端端点 | 视觉反馈 |
|------|------|---------|---------|
| ✓ **加入猎物池** | 将标的加入 `barbell_config_json.hunt_pool[]` | `POST /api/strategic/boards/{id}/stock/accept` | 变为 ✅ 已加入（emerald），不可再点 |
| ✗ **排除** | 标记为 `rejected` 并隐藏 | `POST /api/strategic/boards/{id}/stock/reject` | 标的上方弹入灰色 + 🚫 已排除 |
| 📝 **调整评分** | 切换因子评分为可编辑 input | 前端本地，提交后 `POST .../stock/override-score` | 编辑图标变为 💾 保存 |

### 6.4 批量操作工具栏

在结果区域顶部添加工具条：

```
┌─ 工具栏 ──────────────────────────────────────────────────────────┐
│ [📋 全部展开] [📁 全部折叠] [筛选: 综合≥80 ▼]   [🎯 一键晋级 (47只)]  │
└───────────────────────────────────────────────────────────────────┘
```

- **全部展开/折叠**：JS `document.querySelectorAll('details').forEach(d => d.open = !d.open)`
- **筛选**：按 `composite` 值域过滤（全部 / ≥80 / ≥60 / <60）
- **一键晋级**：将当前展示的所有「未操作」标的批量加入猎物池

### 6.5 向下游衔接

结果区域底部增加衔接操作区：

```
┌─ 下级衔接 ──────────────────────────────────────────────────────────┐
│ 🎯 已选标的将自动进入 CVM 矩阵进行 Gate-A 财务验证                   │
│ [👁 预览已选标的 (47只)]  [🚀 推送到 CVM 矩阵]                        │
└─────────────────────────────────────────────────────────────────────┘
```

- **预览**：弹窗展示已选标的列表
- **推送 CVM**：将 `hunt_pool` 中的标的写入 `CvmScorecard` 表，触发 Z1 Gate-A

### 6.6 编辑模态框增强

在编辑模态框的 BOM 节点列表中，每个节点右侧显示代表标的预览（代码 + 名称缩写），让用户在编辑时就能看到该节点下的核心标的：

```html
☑ AI训练/推理芯片 (核心 · L2)  🏷 688256寒武纪, 688041海光信息, 300474景嘉微
```

同时增加二次确认对话框：修改赛道或概念时弹出「将清除已有生态位分析结果，是否确认？」。

### 6.7 用户完整操作旅程

```
新建板块 → 确认创建
     ↓
命令中心 → [触发生态位分析]
     ↓ （异步 3-6 分钟）
生态位分析结果
     ├─ 浏览 BOM 节点 × 标的池
     ├─ 对每个标的操作：
     │   ├ ✓ 加入猎物池 (代表标的 · 快速采纳)
     │   ├ ✗ 排除        (触发排除规则的 · 确认排除)
     │   └ 📝 调整评分    (LLM 评分有争议的 · 人工覆盖)
     ├─ 批量操作：筛选 + 一键晋级
     ↓
```
