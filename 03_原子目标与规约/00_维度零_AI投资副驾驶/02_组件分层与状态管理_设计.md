## 一、章节定位

本章是 L3 前端工程的"**架构分层 + 状态管理 + product_module 工程下沉**"权威规约，回答三个问题：

1. **维度零 5 个 product_modules（持仓体检/推荐池/告警/价值账本/反馈闭环）在前端工程里如何分层落地**？
2. **每个 product_module 的页面 / Feature 组件 / domain hooks / API 客户端 / 实时通道**如何组织？
3. **如何按 3 阶段路线（启动期/扩展期/完善期）渐进式上线 product_module**？

> 历史上"7 大场景 S1-S7"是面向**功能模块**的高阶视图（适合管理员驾驶舱与高阶用户）；本规约新增的"5 product_modules"是面向**普通投资副驾驶用户**的产品形态（默认入口）。两者并存，对应不同用户角色。

## 二、整体分层（保持不变 + 新增 product_module 编排层）

```
┌────────────────────────────────────────────────────────┐
│ 0. Product (维度零·5 product_modules 编排层) 【新增】    │
│    - 持仓体检 / 推荐池 / 紧急告警 / 价值账本 / 反馈闭环  │
│    - 每个 product_module = 1 路由前缀 + 1 主页面 + N    │
│      Feature 组件 + 1 套 domain hooks + 1 套事件流订阅  │
├────────────────────────────────────────────────────────┤
│ 1. View (路由 + 页面)                                    │
│    - 按 product_module 划分一级路由（/dashboard /research│
│      /alerts /ledger /verified）+ 高阶路由（/admin）    │
├────────────────────────────────────────────────────────┤
│ 2. Feature (业务组件)                                    │
│    - 每个 Feature 组件标注归属的 product_module         │
├────────────────────────────────────────────────────────┤
│ 3. Domain Hooks (业务 hooks)                             │
│    - useResearchCards / useStateMachineInstance / ...   │
├────────────────────────────────────────────────────────┤
│ 4. UI Kit (设计系统组件)                                 │
│    - Button / Card / Table / EvidenceTrail / ...        │
├────────────────────────────────────────────────────────┤
│ 5. Service Layer (BFF 客户端 + SSE / WebSocket / Stream) │

│    - 鉴权 / i18n / 日志 / 错误处理 / 埋点 / Feature Flag│
└────────────────────────────────────────────────────────┘
```

## 五、状态管理策略

### 5.1 状态分类

| 类别 | 工具 | 例子 |
|------|------|------|
| 服务端状态（远程数据 + 缓存） | **TanStack Query v5** | useResearchCards / useAdvisories |
| 实时流（Redis Stream / SSE） | **自建 Stream 消费者 + Query 集成** | useStreamSubscribe（**新增**）、useCouncilSessionStream |
| 表单本地状态 | **React Hook Form** + Zod | 提问表单 / 偏好编辑 / 双盲标注 |
| 全局 UI 状态 | **Zustand** | useUIStore |
| 鉴权 / 用户信息 | Zustand + 持久化 | useAuthStore |
| URL 状态 | URL Search Params | useFilters |
| **本地决策日志（价值账本核心）** | **本地 SQLite + Dexie**（**新增**） | useLocalLedger |

### 5.2 数据获取约定

```ts
useQuery({
  queryKey: ['research-cards', 'list', { subjectId, status }],
  queryFn: () => api.deepStrike.cards.list({ subjectId, status }),
  staleTime: 30_000,
});

useStreamSubscribe('events:cryo_guard:reject', (event) => {
  queryClient.setQueryData(['risk-feed'], (old) => prepend(old, event));
  if (event.severity === 'critical') {
    notificationService.popup(event);
  }
});
```

### 5.3 错误处理与降级

- 全局 `<ErrorBoundary>` 捕获 + 上报到超级个体进化反馈采集
- 局部错误：组件内显示"重试"按钮 + 友好降级文案
- **极寒防御 reject 类错误**：以"系统拒绝原因"卡片展示，给用户明确"为什么"
- **stream 断流**：3s 内自动重连；超过 30s 显示"实时数据暂时不可用，已切换为轮询"
- **后端服务降级**（events:cryo_guard:degrade）：在受影响的 Feature 组件上叠加 `<DegradedBadge>` 标记

## 九、约定与禁止

### 必须
- **Feature 层不直接 fetch；统一走 domain hooks**
- **每个 Feature 组件必须标注归属的 product_module**（注释或 Storybook story）
- **每个 product_module 的 BFF 必须独立路径前缀**
- **类型强制**：Stream 事件 schema 必须与 [04_后端契约](../../02_战略维度/00_维度零_AI投资副驾驶/04_与5维度后端的契约.md) 一致；用 zod 校验
- **可访问性优先**：UI Kit 组件须通过 axe 检查
- **埋点统一**：通过 `useTrack` hook，写入超级个体进化的 `feedback_implicit`
- **本地决策日志不可丢**：使用 IndexedDB + 自动同步到云（用户授权后）

### 禁止
- 在 Feature 层直接组合 Tailwind 原子类
- 在组件内直接调浏览器 fetch
- 在前端实现业务规则（应在后端；前端仅展示与汇总）
- 直接读后端数据库或绕过 BFF
- **跨 product_module 直接调用其他 product_module 的内部组件**（必须通过 domain hooks 暴露的公共接口）

## 十、与维度零 stages/ 阶段路线的强约束

| 阶段切换信号 | 前端动作 |
|---|---|
| 阶段 1 → 阶段 2 | feature flag `verified_loop_enabled=true` 启用 product_module 5 + Kappa 校准 UI 上线 |
| 阶段 2 → 阶段 3 | feature flag `autopilot_limited=true` 启用自动驾驶限额预览 + 议会模式 UI |

阶段切换由架构师在管理员驾驶舱手动触发，前端通过 feature flag 渐进式启用。
