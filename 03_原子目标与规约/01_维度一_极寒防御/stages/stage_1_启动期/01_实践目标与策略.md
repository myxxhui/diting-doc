## 一、本阶段目标

### 1.1 一句话目标

> **用 3 个 P0 引擎 + decision_gate 实现"任何对外输出必须先过防御门禁"的最小闭环。**

### 1.2 量化目标

| 目标项 | 指标 | 阈值 |
|---|---|---|
| 引擎数量 | 3 P0 引擎上线 | 财务测谎 + 大股东诚信 + 关联交易 |
| 召回率（Recall）| 50 案例 Holdout 评测 | ≥ 0.90（综合） |
| 精确率（Precision）| 50 案例 Holdout 评测 | ≥ 0.70（综合） |
| 门禁漏判 | decision_gate 漏放高风险 | 0（零容忍）|
| 系统可用性 | 单日可用性 | ≥ 99.5% |

### 1.3 本阶段交付物

| 交付物 | 描述 | 验收方式 |
|---|---|---|
| 3 P0 引擎微服务 | 财务测谎/大股东诚信/关联交易，各自独立部署 | K8s Pod Running + 健康检查通过 |
| decision_gate 服务 | 一票否决聚合 + 认知边界检查 | API 可调用 + 聚合逻辑正确 |
| 3 LoRA 模型（v1）| 各引擎专属 LoRA（Qwen2.5-7B + rank=16）| 50 案例 Holdout 通过 |
| 审计日志表 | cryo_guard.audit_log | 每次判决可查 + 不可篡改 |
| 50 案例 Holdout | 永久锁库，不可用于训练 | 按引擎分布 + 标注完成 |
| 实践文档 | 本阶段 5 份实践设计文档 | 完整可执行 |

## 二、总体策略

### 2.1 核心策略：防御优先 + 宁严勿松

```
┌─────────────────────────────────────────────────────────────┐
│                    策略一：一票否决                          │
│  任意引擎判 reject → 整体 reject                            │
│  任意 2 个引擎判 degrade → 升级为 reject                    │
│  全部 pass → 放行                                           │
└─────────────────────────────────────────────────────────────┘
```

**为什么宁严勿松？**
- L1 基石⑤：防御优先于进攻
- 误报（false positive）的代价远低于漏报（false negative）
- 启动期容忍更高的 reject 率，后续阶段通过 DPO 降低过敏

### 2.2 引擎分工策略

| 引擎 | 检测目标 | 数据主依赖 | 模型特点 |
|---|---|---|---|
| **财务测谎** | 财务造假/粉饰 | 财报 + 附注 | 数值特征 + LLM 审讯 |
| **大股东诚信** | 言行不一 | 公告 + 实际行为 | RAG 长上下文 + NLI |
| **关联交易** | 资金腾挪/明股实债 | 股权穿透 + 附注 | 图算法 + LLM |

### 2.3 技术选型策略

| 层面 | 选型 | 理由 |
|---|---|---|
| 基座模型 | Qwen2.5-7B（32K 上下文）| 中文能力强 + 开源 + 私有部署 |
| 微调方式 | LoRA（rank=16）| 显存友好 + 快速迭代 |
| 推理框架 | vLLM | 高吞吐 + 多 LoRA 复用 |
| 服务框架 | FastAPI + Uvicorn | 轻量 + 异步 |
| 编排层 | LangGraph | Agent 工作流编排 |
| 向量存储 | Milvus | 生产级向量检索 |
| 图数据库 | Neo4j | 股权穿透 + 关联方图谱 |

### 2.4 数据策略：质量 > 数量

| 策略 | 说明 |
|---|---|
| **Holdout 永久锁库** | 50 案例（3 引擎各 ~17 案例）用于评测，永不训练 |
| **Teacher 蒸馏** | 用 Claude-3.5-Sonnet 蒸馏，生成 JSONL |
| **架构师 Verified** | 每条训练数据必须架构师人工 verified |
| **不做 DPO** | 启动期不做偏好对齐，留待扩展期 |

### 2.5 部署策略：单节点起步

| 策略 | 说明 |
|---|---|
| **单节点部署** | 启动期在单台 4090 服务器部署 |
| **K3s 而非 K8s** | 轻量级 Kubernetes，降低运维成本 |
| **SQLite → PostgreSQL** | 启动期用 SQLite，扩展期迁移 |
| **本地 Milvus** | Standalone 模式，不上集群 |

## 四、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|---|---|---|---|
| 50 案例 Holdout Recall < 0.90 | 中 | 高 | 增加训练数据 + 调整 LoRA rank |
| 关联交易图谱构建困难 | 中 | 中 | 降级为规则匹配 + LLM 补充 |
| vLLM 显存不足 | 低 | 中 | 降低 batch size / 使用量化模型 |
| Neo4j 性能瓶颈 | 低 | 低 | 索引优化 + 查询缓存 |

## 五、本阶段不做什么（明确边界）

| 不做的事 | 留待阶段 | 原因 |
|---|---|---|
| ❌ DPO 偏好对齐 | 扩展期 | 先跑通 SFT，再做偏好 |
| ❌ 多 LoRA 行业细分 | 扩展期 | 先验证单 LoRA 效果 |
| ❌ 议会模式 | 完善期 | 先验证单引擎，再做多引擎投票 |
| ❌ 4-10 号引擎 | 扩展期/完善期 | 聚焦 3 P0 引擎 |
| ❌ reject 配额管理 | 扩展期 | 启动期容忍高 reject 率 |
| ❌ 5 维认知边界完整检查 | 扩展期 | 启动期只做行业白名单 |
