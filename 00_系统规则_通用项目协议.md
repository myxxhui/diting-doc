# 通用项目文档系统规则 (Universal Doc Protocol)

**版本**：v1.1  
**适用范围**：任意项目的文档仓（可配合代码仓、部署仓使用）  
**设计目标**：建立「顶层概念 → 战略维度 → 原子规约 → 阶段实践 → 成功验证 → 追溯与审计」的可追溯文档体系，适配 AI 协作与人类架构师协作。

**文档定位**：本文档为文档仓内**完整协议正文**，与工作区根目录 `.cursorrules` 中的同名规则保持同步；AI 与人类在文档仓内操作时以本文件为权威引用。

## 规则生效要点（AI 必读）

以下要点确保本协议在每次协作中生效，AI 须在相关操作时自动遵守。

**AI 默认遵循（1:1:1 同步）**：凡用户提出**设计需求、实践修改需求、设计修改需求、或由疑问引起的修改需求**时，AI 在给出解决方案的**同时**须默认遵循 §4.6/§4.7，不仅解决问题且**同步补充或更新** 1:1:1 文档（03_ 设计、步骤级 DNA、dna_dev_workflow、04_ 实践、L5 对应行）；无需用户额外提醒。适用 §4.6 时修复后执行 1:1:1 核对并输出结果；适用 §4.7 时当次完成 1:1:1 反写。

| 场景 | 强制动作 |
|------|----------|
| **生成/重构文档** | 先读 L3 规约或 `_System_DNA/`；输出含 `[TRACEBACK]` 的追溯锚点；结束时输出「一致性检查表」；涉及结构或全局约定变更时含对 00_系统规则 的同步（见 §4.0/§4.5） |
| **修改/新增 L1 哲学文档** | **仅承载哲学性陈述**（核心立场 + 哲学边界 + 哲学性判据 + 不做什么 + 承接位置）；**禁止**写具体阈值数字、Python/伪代码、详细规则矩阵动作列、完整 DNA YAML；具体规则均移交对应 L2 `04_XX实践策略规划.md`。每个哲学基石内统一为「①核心立场 / ②哲学边界 / ③哲学性判据 / ④实践规划承接位置」四节。见 §3.5.1/§3.5.2。 |
| **修改/新增 L2 实践策略规划** | 须含「①层级定位 / ②承接 L1 待办清单（含状态）/ ③具体规则/阈值/矩阵 / ④DNA 键落地建议 / ⑤一致性检查」；`[TRACEBACK]` 须上溯到具体 L1 哲学基石 + 下沉到 L3 规约 + DNA 键；**禁止重新定义哲学边界**（哲学引用 L1）。见 §3.5.1。 |
| **L4 实践/代码** | 步骤中明确 **工作目录**（如 `{project}-src`）；逻辑密集走 5D；代码注释含 `[Ref: 03_XX]` 或 `[Ref: 便利贴_XXX]` |
| **架构图** | 仅用 Mermaid 嵌入 Markdown；与 L3 规约一致；同一文档内选用 §7.4 一种可视化风格并保持一致 |
| **术语与能力** | 不臆造规约外技术栈；L3 中文业务语→代码用项目约定英文（如 积分→Credits） |
| **英文简写与技术名词** | 见 §十三：先检测对照表是否存在；使用简写或技术名词时查表、缺则补表；同一文档内首次出现附中文或释义，后续不重复 |
| **关键重构（任意层）** | 将更新后的关键逻辑（如目录约定、轴线划分、链接约定等）同步到本协议相应小节（如 §3.2、§4、§7.1）；并输出一致性检查表 |
| **产出涉及关键重构的变更计划/待办** | **生成计划时须先自检**：本次变更是否涉及关键重构（含 L4 阶段/步骤模板、Phase 结构等全局约定）？**若是，待办须显式包含**「按 §4.5 同步 00_系统规则（及 .cursorrules）」；执行顺序见 §4.5。未包含则违规。 |
| **L4 实践文档与核心指令** | 实践文档与核心指令以 **DNA 为核心实践目标**；核心指令内对文档/DNA 的**逻辑引用须以 AI 可读方式**呈现（完整路径、DNA 键路径或结构化引用块），见 §8.4d、§12。 |
| **L4 实践文档新增/修改** | 凡在 **04_阶段规划与实践** 下**新增或修改实践文档**（各实践阶段内 01_～0N_ 步骤文档）时，AI 须**判读是否为实践文档**；若是，产出须满足「**按文档完成所有验证且不偏离、不臆造**」之目标，见 §8.4e；**不把任务留给人**（所有动作作为步骤或「不满足时」命令写入文档），**尽量不专门写脚本**（优先文档内命令/Make），见 §8.4e；**步骤失败时**须约定「先分析、修复、重试；同一问题修复重试超过 2 次仍失败则回收/退出」，见 §8.4f。 |
| **L3 启动期 step 文档新增/修改（六维度共享模板）** | 凡在 **03_原子目标与规约/0X_维度*/stages/stage_1_启动期/steps/** 下新增或修改 `step_*.md` 时，**须**遵守权威模板 [`_共享规约/L3_启动期step_重构模板`](03_原子目标与规约/_共享规约/L3_启动期step_重构模板.md) 的 **13 个强制小节**（数据对象 / 真实数据源 / 凭证清单 / 启动期目标 / 下一步一行 / 实施步骤 / **部署节奏** / 准出 / [Deploy] / 依赖 / 风险 / 修订记录）；**禁止**省略 §4.2「用户须提供的凭证」、§5「启动期数据量预期」、§8「本步在哪里跑」三节。全局开发地图见 [`_共享规约/14_六维度启动期统一节奏表`](03_原子目标与规约/_共享规约/14_六维度启动期统一节奏表.md)。 |
| **持仓 SoT（六维度共享单一真相源）** | 启动期所有维度的数据采集（D1 财报/公告 / D2 thesis 候选池 / D3 持仓监控 / D4 卖出协议 / D0 副驾驶 CRUD）须以 **`diting-src/data/config/my_holdings.yaml`** 为唯一来源（环境变量 `MY_HOLDINGS_YAML`）；加载器在 `apps/common/holdings_sot.py`；模板见 `data/config/my_holdings.example.yaml`。**禁止**在 step 文档或代码中硬编码 symbol 列表或另起 SoT；持仓 yaml 私有副本已在 `.gitignore`，禁止提交。启动期默认仅对 `active=true` 标的拉真实数据；扩展期触发条件写入对应 step §6「下一步」。 |
| **no-mock-policy（启动期六维度统一）** | 启动期 L3 step 与 L4 实践记录中**禁止**出现 `CRYO_MOCK=1` / `--use-mock` / `mock_*.json` 作为业务事件流 / `THESIS_GENERATOR_MODE=stub` 进入生产路径等表达。占位 fixture **仅**在 `tests/` 单元测试中合法，且**不得**入业务库、**不得**入训练集、**不得**上 staging。上游真流未就绪时下游 step **应等待**或仅以 fixture 单测推进；缺凭证时须写「待用户提供 `XXX_API_KEY` 后由 Cursor 直接执行」，**禁止**伪造 LLM 输出或行情/财务数字。详见 [`_共享规约/14_六维度启动期统一节奏表`](03_原子目标与规约/_共享规约/14_六维度启动期统一节奏表.md) §8 占位 fixture 合法边界。 |
| **数据采集质量优先于数据量（启动期六维度通则）** | **巴菲特原则**：不看财报的投资是自杀，**但看了三表 ≠ 看了财报**。启动期 L3 数据采集 step 须按**下游模型引擎的分析需求**反推所需字段粒度，**禁止**仅以"行数 / 条数"作为准出。须在 step §3.5 写**数据质量验收矩阵**（按目标引擎分块 · 4 列：分析维度 → 必产字段 / 衍生 → 启动期覆盖状态 ✅/⚠️ → 启动期降级路径），下游每个消费 step 都至少列 1 行。**禁止**：①只采"表面层"（如仅三表 raw / 仅公告标题 / 仅关联方文本行），而不扩展到附注 / 披露 / 事件链 / 网络穿透 / 定价方法 / 承诺履约等分析必需的细分；②把字段假装填 null 即视为达标；③数据量达 §5 行数门槛但 §3.5 矩阵不绿就准出；④**在 §3.5 矩阵列扩展期 / 完善期内容**——本 step **只对启动期负责**，扩展期数据类型与质量要求在 `stages/stage_2_扩展期/` 文档独立维护。修复：按 [`_共享规约/L3_启动期step_重构模板`](03_原子目标与规约/_共享规约/L3_启动期step_重构模板.md) §3.5 补质量矩阵。**启动期标的可以少，但数据质量必须达到引擎能做实质分析的程度。** |
| **L3 以设计规划推演为主（不嵌入完整代码 · 给后续模型工作指引）** | **L3 根本定位**：以**设计规划推演**为主——靠谱地描述"做什么 / 如何执行 / 验证标准"，**不**嵌入完整 Makefile / 脚本代码、**不**试图本 step 把所有事 100% 完成。具体命令、调试日志、参数填值由 **L4 实践记录 / 后续执行模型**按 L3 规划自行落地。L3 是"给后面模型的工作指引 + 验证标准"，不是"自己把活全干了"。L3 启动期 step 文档 §7 须为"实施规划"风格：①§7.1 实现要点（4 列表：实现要点 → 涉及代码位置 → 关键设计决策 → 验证标准）；②§7.2 Makefile 一键复现合约（若涉及多步流水线 · 强制 · 见下条）；③§7.3 给后续执行模型的指引。**禁止**：①在 L3 step 嵌入完整 Makefile 实现代码或完整脚本；②把每一条命令、每一行参数都写在 L3（那是 L4 实践记录的事）；③试图在 L3 step 完成所有具体落地（L3 给规划，L4/执行模型按规划落地）。 |
| **采集 / 训练 / 评估类 step 须 Makefile 一键复现合约（配置驱动 · 可重入幂等）** | 涉及多步流水线的 L3 step（数据采集 / 模型训练 / 质量评估等）须在 §7.2 写 **Makefile target 合约表**：一组 `<module>-step<N>-prep / -collect / -quality-check / -test / -all / -status / -clean` target，**让架构师只改配置文件**（如 `data/config/my_holdings.yaml` 增减 active 标的、改年份、调阈值）就能跑 `make *-all` 一键端到端完成全套，无需改 Makefile / 脚本 / 命令行。**合约要求**：①入参全部环境变量化（命令行 / `.env` / Makefile 顶部三层叠加优先级）；②target 是"脚本 + 环境变量"的薄包装，**不**在 Makefile 写业务逻辑；③可重入幂等（重跑 `*-all` 不破坏已有数据，须按业务主键 upsert 而非 insert）；④配置驱动（变更范围只改配置文件，**禁止** Makefile / 脚本内 hardcode 标的代码、年份、阈值等）；⑤失败可观察（每个 target 输出"做了什么 / 期望什么 / 实际什么"3 行摘要，中文优先）。**L3 只定义合约**（target 名 / 用途 / 入参 / 验证标准）；**实现交 L4 / `diting-src/Makefile`**。准出须含"配置驱动验证"项：临时增 1 个 active 标的，跑 `make *-all` 端到端通过；移除并跑 `make *-status` 验证可重入。 |
| **`实践记录_*.md` 与 L3 步骤索引** | **实际开发与追溯**以 L3 **`steps/step_*.md`（或项目约定的可执行步骤清单）** 与 L4 **`实践记录_*.md`** 的 **映射表** 为权威；**节奏表、日历周次（如 W1、W3）仅为计划/沟通别名**，**不得**据此推断「一周 = 一份独立实践记录」或「实践记录文件数 = 自然周数」。详见 **§8.4h**。 |
| **按实践文档执行** | 按 **04_阶段规划与实践** 下所有实践阶段的所有实践文档执行步骤时：**遇问题则分析问题、解决问题并重试**；仅当**同一问题**在**修复重试两次以上**仍无法解决时，才执行文档规定的回收/退出。不得在首次失败时直接回收退出。见 §8.4f。 |
| **Debug/修复/疑问并给出解决方案** | 修复**后**须对该步骤执行 1:1:1 核对（03_、DNA、04_、L5）；发现引用不清、要求不具体或验证缺失（问题穿透）时**须一并修复**并说明已联动更新；回复须含「1:1:1 核对结果」与已修改项。见 §4.6。 |
| **新要求/新想法/设计变更** | 当次反写到 03_、DNA、dna_dev_workflow、04_、L5（§4.7）；若某验证项待实现须在文档中写明代用动作；变更准出/必做项/验证命令或设计要点时须完整 1:1:1。见 §4.7。 |
| **按四阶段规约提问** | 用户按 04_ 下 `00_四阶段提问体系_完整规约.md`（新需求/设计变更入口）执行时，须**先**完成阶段 1～4 产出待办清单；**执行**待办中的文档操作与实践任务时须满足 §4.7 的 1:1:1 反写与可执行性验收。 |
| **部署内容调整** | **部署内容**（Service 类型、NodePort、副本数等）由 **配置文件与 Chart** 控制，**不在 Makefile 中写死**；Makefile 仅从配置读取并传递给 helm。见 §十一「部署内容由配置与 Chart 控制」。 |
| **修改部署仓 Makefile/脚本** | 涉及**某组件的部署方式、暴露、端口、类型**时：**先**改 config 或 values 覆盖文件，Makefile/脚本仅 `-f` 或从配置传参；**禁止**在 Makefile 中为单组件写专属分支、临时文件或写死值；**禁止**为每次功能/波次/step 新增 `copilot-*-deploy` 类 Make 目标（日常发版统一 `make copilot-deploy`，见 §十一「Copilot 统一部署」）。修改前自检：该决策是否已在配置中？见 §十一「部署内容由配置与 Chart 控制」。 |

| **代码/配置变更后的同会话验证（禁止可选收尾）** | 当次任务修改了**代码仓或部署仓**内可执行制品（Python/Shell/Makefile/Helm values 等），且用户目标含**实现、修复、联调或验收**时：在本环境具备 shell/Python 等前提下，须**同一轮会话内**执行与变更**直接相关**的最小可重复验证（如受影响路径的 `pytest`、文档或 Makefile 约定的 `make <target>`、导入/语法检查）。**禁止**用「若你希望」「可下一步再联调」「建议你本地跑」等**代替**上述执行；**禁止**将本环境可完成的验证列为用户待办。因无 DSN/凭证/外网等**确实无法执行**时，须在回复中区分**已执行验证**与**未执行项及阻塞原因**；不得暗示已完成需库/集群才能做的验收。见 §7.2 第 10 条。 |

| **依赖链与数据路径验收（禁止只改代码不验就绪）** | 当能力依赖 **L2/L1 表、采集任务、外部 API、密钥或上游数据** 时，须**同一会话**内：**(1)** 列出**最小依赖清单**（如 `PG_L2_DSN`、相关表名、须先执行的 `ingest`/init）；**(2)** 对可自动化项做**就绪检查**（连接/表存在/环境变量/相关 `pytest`）；**(3)** 缺表或缺数据时在目标路径上**主动** init/采集重试后再跑**最小成功路径**（与 §7.2 第 9 条一致）；**(4)** 仅当环境不可达时，在回复中写明**未就位项**、**已执行命令与输出摘要**，**禁止**未核验依赖即宣称「已联调/已验证/生产级完成」。见 §7.2 第 11 条。 |
| **数据/模块验证结论（须有证据）** | 断言「数据符合预期」「模块验证通过」或「符合 AB/C 模块要求」时，**必须同时提供**：① **数据库查询证据**（库 DSN 说明、表名、查询条件、结果行数或样本）；② **模块执行证据**（实际运行命令与输出摘要）。**禁止**在仅看到降级/回退结果（如 Module A 全部输出「未知」、L2 行业名为空未补全）时断言「符合预期」，除非规约/文档已明确该行为为预期。见 §7.2 第 8 条。 |
| **答复事实、全量宣称与输出语义（禁止打太极、禁止误导占位）** | 用户问「是否完成/是否符合/是否全量」时须给出**可核验事实**（命令、范围、输出/行数）；**禁止**用「原则上」「抽样代替全量」**除非用户明确同意**；未跑全量/未查库时**不得**作肯定结论。缺数据时输出须**显式**（如「无披露」、约定 `segment_id`），**禁止**用行业标签伪造分部或冒充已验收。见 §7.2 第 12 条。 |
| **`.env` 与 `*.env.template` 同步** | 任务依赖的配置说明已写在仓库 `*.env.template`（或项目约定模板）时，须**同一代码仓**将**注释与非密钥占位**合并进**已存在**的 `.env`；**不得**覆盖用户已填 DSN/密钥/密码；**禁止**以「请自行复制 template」为唯一收尾。见 §7.2 第 13 条。 |
| **面向人的输出以中文为主** | 终端/CLI、文档解释、Cursor 对话、**代码输出（print/log/脚本终端输出）**等**优先中文**；环境变量名、配置键、代码标识符等技术必要处可英文。见 §7.2 第 14 条。 |
| **目标路径上的前置项与顺手优化（主动完成）** | 在完成用户**已声明目标**的前提下，凡属**达成该目标所必需**的前置动作（如缺表、缺依赖、配置未就绪、**从 template 合并 `.env` 中与任务相关的非密钥块**、验证命令失败）、**明显阻塞的小问题**，以及**低成本、低歧义、与当前改动同一上下文**可完成的优化，AI 须**主动分析、修复或实现，并执行可重复的验证**，**不得**默认停顿、仅列「请你执行」清单或等用户逐条追加指令。典型示例包括：`relation … does not exist`、`ImportError`、`make`/`pytest` 失败、文档与实现不一致等。**边界**：仍须遵守 §7.2 第 6 条（先验证再改）、禁止臆造能力或擅自扩大与目标无关的大规模重构；生产/集群以 Chart/schema-init 为权威，本地可与仓库脚本对齐。 |

完整条文见下方各节。

**关于「判断/修复时禁止臆想」的触发与防穿透**：① **触发**：问题涉及「为什么不对」「修 bug」「排查原因」时，上表「判断根因/修复问题/Bug 分析」行生效；或显式写「先按 §7.2 Verify First 排查」「不要臆想原因，先按排查清单逐项检查」。② **防穿透**：若仅说「修一下」「改好就行」，AI 可能直接改而跳过验证；建议涉及原因判断或修 bug 时带上「先排查/先验证/按规则不要臆想」或引用规则生效要点「判断根因」行。③ **直接执行与验收穿透**：排查、修复、部署相关验证**由 AI 直接执行**；**实现或修复完成后**，可执行的验收步骤（make、脚本、查库、看结果等）也**由 AI 直接执行**，任务穿透至可执行验收完成，**禁止默认以「你需要做的」「建议您」「请执行」收尾**；仅在本环境无法执行时（如用户独占集群/凭证）才给出用户侧操作清单。④ **同会话最小验证**：凡修改代码/配置且目标含交付或验收，须**同会话**跑与变更相关的**最小** pytest/make 等（见上表「代码/配置变更后的同会话验证」）；**禁止**用「可选下一步」「若你希望」替代可本地完成的验证，避免把本应由 AI 执行的检查留给用户。⑤ **依赖链与数据路径**：凡涉及**库表、采集、外部数据**的能力，须**先核验依赖就绪**再宣称完成（见上表「依赖链与数据路径验收」与 §7.2 第 11 条）；**禁止**只合并代码或只改 YAML 而不执行表存在性/单标的采集/最小端到端命令，亦**禁止**在未说明缺数据、缺表或未跑通路径的情况下断言「已验证」。⑥ **全量与占位**：用户声明「全部标的」时**不得**擅自改为抽样；缺披露/缺表时**不得**用占位冒充真实分部或全量验收（见上表「答复事实、全量宣称与输出语义」与 §7.2 第 12 条）。⑦ **代码修改闭环**：写/修/加代码须满足「有理解→有预期目标→有验证→验证失败则修复再验证」；**禁止**验证未通过时收尾，**禁止**以「请本地跑一下」替代（见上表「写/修/加代码（代码修改闭环）」与 §7.2 第 10a 条）。

## 一、核心理念 — 双核共生

解决「人类思维局限」与「系统复杂度」之间的矛盾：**人类定义战略，AI 补全实施**。

| 角色 | 层级 | 职责 |
|------|------|------|
| **人类架构师** | L1～L2 | 定义愿景、战略维度、关键约束；审查推演结果；裁决冲突 |
| **AI** | L3～L6 | 推演架构细节、生成代码/文档；维护追溯矩阵与审计；执行便利贴 GC |

**四大原则**：
1. **定义者 vs 补全者**：人类负责 L1～L2；AI 负责 L3～L6。
2. **配置即文档**：L3 优先用结构化表达（表格、列表、YAML），便于 AI 解析。
3. **惰性一致性**：不要求实时全量同步；用「增量便利贴」+ 定期合并。
4. **垂直切片**：L4 按特性打通数据→API→UI，避免横向断层。

## 二、共识前提

- **多仓库策略**（可选）：常见为三仓库 — **文档仓**、**代码仓**、**部署仓**。本协议约束文档仓。
- **文档仓定位**：仅存放项目文档，不含代码或运行时配置。
- **记录范围**：从顶层概念到战略维度、原子规约、阶段实践、成功验证与追溯审计。
- **层级原则**：上层定「为什么、做什么」，下层定「怎么做、做到哪」；下层须能溯源到上层。
- **本协议作为受约束文档**：`00_系统规则_通用项目协议.md` 是文档仓内受本协议约束的文档之一；当文档仓结构或全局约定发生变更时，本协议须与现状一致（见 §4.0、§4.5）。

## 三、六级架构层次

```
L1 (顶层概念) ↔ L2 (战略维度) ↔ L3 (原子规约) ↔ L4 (阶段实践) ↔ L5 (成功验证) → L6 (追溯与审计)
```

| 层级 | 目录 | 职责 |
|------|------|------|
| **L1** | `01_顶层概念/` | 定义项目根本定位、价值主张、**哲学边界**（含哲学性判据：什么算正确/什么算失败/不做什么）；**仅承载哲学性陈述**，不写具体参数、矩阵、代码、DNA YAML |
| **L2** | `02_战略维度/` | 拆解战略维度 + **承接 L1 哲学边界的实践策略规划**（具体阈值、规则矩阵、健康范围、告警规则、协作 schema、伪代码思路）；不写代码实现细节与字段类型 |
| **L3** | `03_原子目标与规约/` | 原子能力规约、技术规约、架构共识；**侧重实现设计与规约**（接口定义、协议、配置、可执行规约、最终 DNA YAML）；不写单次迭代步骤 |
| **L4** | `04_阶段规划与实践/` | 阶段规划、实施步骤、**实践情况记录**；须引用 L3 规约 + DNA 键；不重新设计规约 |
| **L5** | `05_成功标识与验证/` | 完成情况、验收标准、验证方法 |
| **L6** | `06_追溯与审计/` | 追溯矩阵、审计报告、增量便利贴 |

### 3.5 层级职责边界

> **核心分层原则（哲学 → 规划 → 规约 → 实施）**：
>
> | 层级 | 回答的问题 | 不回答的问题 |
> |---|---|---|
> | **L1**（哲学）| 是什么 / 为什么 / 什么算正确 / 不做什么 | 用什么数字 / 按什么步骤 / 写什么代码 |
> | **L2**（实践策略规划）| 具体阈值 / 规则矩阵 / 健康范围 / 协作 schema / 伪代码思路 | 字段类型 / 代码实现 |
> | **L3**（规约）| 接口 / 协议 / Schema / 最终 DNA YAML | 单次迭代步骤 |
> | **L4**（实施记录）| 实施情况 / 验证结果 | 重新设计规约 |
>
> **判据**：若一条内容回答"为什么/是什么/算不算正确"→ 属 L1；若回答"用什么数字/按什么矩阵/几个交易日"→ 属 L2；若回答"字段叫什么/接口怎么调"→ 属 L3；若回答"按规约做了什么、跑出什么结果"→ 属 L4。

- **L1**：只写「是什么、为什么做、核心价值、**哲学边界与判据**（什么算正确/什么算失败/不做什么）」；**不写**功能清单、具体阈值数字、规则矩阵、Python 代码、DNA YAML 配置。
- **L2**：写「从哪几个维度落地、每个维度覆盖什么、**承接 L1 哲学边界的具体实践策略规划**（具体阈值、规则矩阵、健康范围、告警规则、审计逻辑思路、协作 schema、DNA 键落地建议）」；**侧重技术/逻辑架构 + 实践策略规划**；不写接口字段类型或单次迭代排期；维度文档须有 `[TRACEBACK]` 上溯到 L1 哲学基石。
- **L3**：写「原子能力规约、技术规约、架构共识」；**侧重更细致化的实现设计与规约**（接口定义、协议矩阵、配置项、可执行规约、**最终 DNA YAML**）；不写单次迭代步骤；规约须有 `[TRACEBACK]` 上溯到 L2 实践规划。
- **L4**：写阶段目标、实施步骤、**实践情况记录**；须引用 L3 规约 + DNA 键 + 工作目录 + 验证命令；AI 须遵守第八节 L4 实践文档创建与 AI 协作规范；不重新设计规约。
- **L5**：验证战略维度对应功能的完成情况、验收标准、验证方法。
- **L6**：维护追溯矩阵、审计报告、便利贴；不承载业务内容。

#### 3.5.1 L1 哲学 → L2 实践策略的承接模式（强制）

> **触发**：任何 L1 哲学基石的具体落地（含阈值、矩阵、规则、流程、参数）须通过 L2 实践策略规划文档承接，**不得在 L1 内堆砌**。

| 项 | 强制要求 |
|---|---|
| **L1 内部结构** | 每个哲学基石内统一为：①核心立场、②哲学边界、③哲学性判据、④实践规划承接位置（→ L2 文档路径 + L3 规约路径 + DNA 键提示）|
| **L2 文档命名** | 维度目录下增设 `04_XX实践策略规划.md`（或与维度命名对齐），文件首部明确"本文档承接 L1 基石 X 的全部实践层规则" |
| **L2 文档结构** | 须含：①本文档的层级定位（L1→L2→L3→L4 简表）、②承接 L1 基石的待办清单（含状态）、③具体规则/阈值/矩阵、④DNA 键落地建议、⑤一致性检查 |
| **L1 严禁内容** | 具体阈值数字（如 0.95、5 个交易日、30%）、Python 代码、详细规则矩阵的动作列、完整 DNA YAML、字段 Schema |
| **L2 严禁内容** | 重新定义哲学边界（哲学引用 L1）、单次迭代步骤、代码实现细节 |
| **TRACEBACK 链** | L2 文档须含上溯 L1 哲学锚点（具体到基石 X）+ 下沉 L3 规约 + DNA 键 |

#### 3.5.2 违规与修复（哲学/实践层混淆类）

凡发现 L1 哲学文档内出现以下内容均视为违规：
- 具体阈值表（如健康度 [0.3, 0.5)、≥ 0.95、5 个交易日 …）
- Python/伪代码实现
- 详细规则矩阵的具体动作列（如"减仓 50%+"、"加仓 30%"）
- 完整 DNA YAML 配置块

修复方式：
1. 将上述内容迁移到对应维度的 `04_XX实践策略规划.md`；
2. 在 L1 内仅保留"哲学边界 + 承接位置指向 L2"；
3. 在 L1 修订记录中标注本次重构；
4. 同步更新 [TRACEBACK] 链（上溯/同层/下沉）。

### 4.0 系统规则自身的同步（元规则）

- 本协议（`00_系统规则_通用项目协议.md`）是文档仓内的一份文档，且为文档仓的**单一事实来源**。

- 本条款不替代 §4.5，而是与 §4.5 共同构成「关键重构 → 必须回写」与「任意相关变更 → 须自检并视情况回写」的双重保障。

### 4.1 向下传播

- **L2 变更** → 查 `06_追溯与审计/00_L2_L3_DNA_映射.md`（或项目约定的 L2↔L3↔DNA 映射）得到受影响 L3 与 DNA 列表 → 按 **变更传播顺序** 执行（见下）；新增 L2 维度时须同步维护该映射表。
- **L2 维度变更传播顺序**：当 L2 某维度内容变更时，（1）先更新该维度的**主责 L3 文档**；（2）再同步对应 **DNA 子树**（`_System_DNA/global_const.yaml` 或约定 dna_*.yaml）；（3）最后检查 L4 阶段/步骤是否仍引用正确 DNA 键。若某维度由多份 L3 联合支撑，以映射表中的「牵头规约」为变更第一落点。
- L1 变更 → 检查 L2/L3/L4/L5 是否有对应表述。
- L3 变更 → 同步 L4 阶段、L5 验收项；涉及 DNA 键时须同步 global_const 或对应 dna_* 文件。**DNA 的 workflow_stages 或 product_scope.phases[].steps 变更**时，须同步 L4 阶段/步骤文档中的引用（如 work_dir、verification_commands、本步骤落实的 DNA 键）及 L5 表体（或重新从 DNA 生成/校验一致）。
- **DNA 的 workflow_stages 新增或删除**时，须在 [05_成功标识与验证/02_验收标准](05_成功标识与验证/02_验收标准.md) 中同步新增或删除对应行及显式锚点（如 `<a id="l5-stage-stageX_0Y">`）；可选步骤仍保留行与锚点，状态可为「跳过/不适用」。
- **某 stage 的 dna_file 或 dna_dev_workflow 对应条目发生变更时**（含 delivery_scope、exit_criteria、work_dir、verification_commands、l5_stage_anchor），须在**当次或紧随的 MR** 中复核并更新对应 04_ 步骤文档（「本步骤落实的 DNA 键」、核心指令、可执行验证清单、工作目录等）；可选：使用 06_/03_ 或 scripts/ 下提供的「按 stage_id 列出对应 04_ 文件」脚本在 MR 前自检或 CI 提示。

### 4.6 Debug/修复/疑问时的 1:1:1 检查与问题穿透防护

- **适用场景**：用户提出 **debug 请求、修复要求、或对某阶段/步骤的疑问**，AI 或人类给出解决方案（含对文档或代码的修改）时。与「用户提出新功能、新步骤或明确要求改设计/准出/必做项」区分：后者适用 §4.7；若同时涉及，两条均适用。
- **问题穿透定义**：**「问题穿透」**指因 03_ 设计、DNA、04_ 实践或 L5 中**引用不明确、要求不具体或验证覆盖缺失**，导致执行者仅凭文档操作仍失败或不得不臆造路径、命令或是否必做。
- **强制动作**：在**修复后必须执行一次**该步骤的 1:1:1 核对；**范围**至少为该 step 的 03_ 设计、对应步骤级 DNA、该 04_ 步骤、L5 对应行；若变更可能影响上下游（如准出或准入），须同时检查**相邻 step** 的 1:1:1。
  - **03_ 设计**：本步目标、设计要点、准出是否与 04_ 及 L5 一致；对 DNA、L4、L5 的引用是否带锚点、路径是否完整可解析。
  - **步骤级 DNA**：exit_criteria、delivery_scope、verification_commands、work_dir 是否与 04_ 和 L5 一致；是否覆盖所有必做验证项。
  - **04_ 实践**：本步输入是否明确（缺则报错、禁止臆造）；可执行验证命令是否无未替换占位符、与代码仓 Makefile/CI 一致；准出清单是否包含设计/DNA 中所有必做项；是否存在「文档写了但无对应命令或实现」的缺口；AI 执行约束是否显式写出。
  - **L5**：对应 stage 行是否与 03_ 准出、DNA exit_criteria、04_ 准出一致。
- **发现问题时**：若存在引用不明确、要求不具体、验证覆盖缺失（或因此导致问题穿透），须在**本次解决方案中一并修复** 03_、DNA、04_、L5 中受影响处，并在回复中说明「已按 1:1:1 联动更新」。
- **与代码仓不一致时**：若可执行验证命令与代码仓 Makefile/CI 不一致，须在本次解决方案或待办中**列出代码仓需同步的项**（如新增/重命名 make target、CI 步骤），或**注明不一致原因与后续处理**（与 §十一 可执行验证与代码仓一致 呼应）。
- **输出要求**：完成修复后，回复中须包含**「1:1:1 核对结果」**：已核对 03_/DNA/04_/L5 四项（可简表）；**已修改项**（若有）：列出文件名与修改性质（如「03_ 设计：准出补充步骤 8」）。
- **与 §8.8 的衔接**：若本次修改涉及 04_ 实践文档，1:1:1 核对后须同时满足 §8.8 L4 实践文档一致性检查表中与本次修改相关的勾选项。
- **责任主体**：人类在执行修复或采纳新需求/设计变更时，应遵循同一 1:1:1 检查要求。
- **L6 矩阵（可选）**：当因 Debug 或修复而修改了 03_/DNA/04_/L5 且影响阶段产出或准出时，须在当次或紧随其后的 MR 中更新 `06_追溯与审计/02_战略追溯矩阵.md` 受影响行（状态或备注）。

<a id="47-新要求设计变更的-111-反写与可执行性验收"></a>
### 4.7 新要求/设计变更的 1:1:1 反写与可执行性验收

- **适用场景**：用户提出**新要求、新想法**，或对某阶段/步骤的**设计或准出做变更**（例如将某步骤从可选改为必做、新增必做验证、调整准出条件）。与 debug/修复/疑问区分：后者适用 §4.6；若同时涉及，两条均适用。
- **与四阶段提问体系的关系**：当用户**按四阶段提问体系规约**（如「按 @00_四阶段提问体系_完整规约.md 执行」）提出新需求/设计变更时，须**先**按该规约完成阶段 1～4 并产出待办清单；**执行**待办清单中的文档操作与实践任务时，须满足本条（§4.7）的 1:1:1 反写与可执行性验收（仅凭实践文档与约定输入可完成全部验证、不偏离不臆造）。
- **强制动作**：新要求或设计变更必须在**当次**同步到该步骤的 1:1:1 文档：**03_ 设计**（设计要点、准出）；**步骤级 DNA**（exit_criteria 等）；**dna_dev_workflow** 对应 stage 条目；**04_ 实践**（步骤表、验证与准出表、准出后自检）；**L5 02_验收标准**对应行。**范围**：至少该 step；若准出/必做项变更影响上一 stage 产出或下一 stage 准入，待办须**显式列出受影响的其他 stage** 并复核其 03_/DNA/04_/L5 是否需同步。
- **变更量级例外**：若变更**仅限文案、错别字或格式**且**不改变准出条件、必做项或验证命令**，可仅更新直接涉及的文档；若变更**准出、必做项、验证命令或设计要点**，须完整执行 1:1:1 反写。
- **验收标准**：变更后须满足：**任何 AI 或人类执行者仅凭实践文档与约定输入，即可完成文档规定的全部验证步骤，且不偏离、不臆造**。**「完美完成」**指完成文档中规定的全部**可执行验证**与**已写明的替代动作**；不要求完成文档中已标明「**待实现**」且已提供替代动作（如单测+记录待补验）的项。若某验证项依赖尚未实现的实现（如某 Feed、某 make target），文档中须**显式写出**「待实现」及**当前可执行的替代动作**，不得留下未定义动作。
- **与 §8.8 的衔接**：若本次修改涉及 04_ 实践文档，1:1:1 反写后须同时满足 §8.8 L4 实践文档一致性检查表中与本次修改相关的勾选项。
- **与代码仓不一致时**：若 1:1:1 反写中发现可执行验证命令与代码仓 Makefile/CI 不一致，须在待办中列出代码仓需同步的项或注明原因/后续处理。
- **计划/待办**：若变更影响多个 stage 或多份 04_ 文档，待办中须列出「同步 03_、DNA、dna_dev_workflow、04_、L5」的逐项检查或修改项；跨 stage 影响时须列出受影响 stage 并复核。
- **L6 矩阵（可选）**：修改 03_/DNA/04_/L5 且影响阶段产出或准出时，须在当次或紧随 MR 更新 02_战略追溯矩阵 受影响行。

## 五、追溯锚点格式

### 5.1 推荐格式

```markdown
> [!NOTE] **[TRACEBACK] 战略追溯锚点**
> - **顶层概念**: [一句话定义](01_顶层概念/01_xxx.md)
> - **战略维度**: [维度A](02_战略维度/07_平台与产品/02_xxx.md) 或 [开发与交付](02_战略维度/10_节奏与交付/01_xxx.md)
> - **原子能力**: [规约A](03_原子目标与规约/01_xxx.md)
> - **阶段**: [Phase1](04_阶段规划与实践/Phase1_xxx/)
```

### 5.2 L3 规约文档须包含战略维度链接

- **要求**：每份 L3 规约的 TRACEBACK 中必须包含「**战略维度**：链接到具体 L2 维度文档」；若一份 L3 支撑多个 L2 维度，至少写主责维度（主责见 [06_追溯与审计/00_L2_L3_DNA_映射](06_追溯与审计/00_L2_L3_DNA_映射.md)）。
- **目的**：便于 L2 变更时定位主责 L3，且可从 L3 反查所属 L2 维度。

### 5.3 禁止

- 追溯锚点放在代码块或 HTML 注释内；
- 无 `[TRACEBACK]` 标识的零散链接。

### 5.4 锚点命名约定

**目的**：全仓引用与指向均有可解析的索引锚点；命名单点约定，便于 AI、人工与脚本一致使用。

- **L3 阶段设计文档**：`design-{stage}-{step}-goal`、`design-{stage}-{step}-exit`、`design-{stage}-{step}-dna-keys`（stage/step 如 stage1、02，与 DNA stage_id 对应）。
- **L4 步骤文档**：`l4-{stage}-{step}-goal`（本步目标）、`l4-{stage}-{step}-exit`（验证与准出/本步骤准出）；可选：`l4-{stage}-{step}-dna-keys`、`l4-{stage}-{step}-verify`、`l4-{stage}-{step}-fail`；步骤导航区块 id 建议 `l4-step-nav`。
- **L5**：`l5-stage-{stage_id}`（如 `l5-stage-stage1_02`）；功能验收表保持 `l5-mod-A`、`l5-func-02` 等。
- **格式**：仅小写字母、数字、连字符；stage 与 step 在锚点中用连字符，与 DNA 的 stage_id 下划线区分（如 stage_id 为 stage1_02，锚点为 l4-stage1-02-exit）。**复制或拆分步骤文档时须替换文档内全部锚点 id 为当前步骤对应 id，禁止重复 id。**

### 5.5 设计文档索引锚点（L3 阶段设计文档）

**目的**：使设计文档对 DNA、L4、L5 的引用可深链到具体段落，便于读者与脚本跳转；设计文档自身关键节可被 04_、06_、中央索引表反链。

- **TRACEBACK 中的引用**：
  - **DNA**：使用可点击的 Markdown 链接指向步骤级 DNA 文件（文件级，YAML 无段落锚点），禁止仅写行内代码路径。
  - **L4 实践**：链接须带目标文档的显式锚点，如 `#l4-{stage}-{step}-exit` 或 `#l4-{stage}-{step}-goal`（与 04_ 步骤文档内显式锚点一致，见 §5.4 锚点命名约定）。
  - **L5**：若提及「准出时更新 L5」或「L5 对应行」，须链到 [05_成功标识与验证/02_验收标准](05_成功标识与验证/02_验收标准.md) 对应行锚点 `#l5-stage-stageX_0Y`。
- **设计文档内关键节**：小节「本步目标」「设计要点」「准出」「本步落实的 _System_DNA 键」（若有）须可作为**索引锚点**被外链。推荐在标题前使用显式 HTML 锚点，如 `<a id="design-stage2-01-goal"></a>`；或采用统一命名约定：`design-{stage}-{step}-goal`、`design-{stage}-{step}-exit`、`design-{stage}-{step}-dna-keys`（其中 stage/step 如 stage2、01）。04_ 步骤文档引用「本步设计文档」时须使用上述锚点（如 `#design-stage2-01-exit` 或 `#design-stage2-01-goal`）。
- **04_ 步骤文档**：须具备可外链的「本步目标」「准出」显式锚点（`l4-{stage}-{step}-goal`、`l4-{stage}-{step}-exit`）及步骤导航（上一步/下一步），且步骤导航中的链接须带 #。
- **中央索引表**：03_ README 或 06_ 应维护「设计–DNA–实践–L5」索引表，每行 stage_id 对应设计文档（带 #）、DNA 文件、L4 步骤（带 #l4-*-exit 或 #l4-*-goal）、L5 锚点，链接均可点击；表旁注明 L4 列为准出锚点（#l4-*-exit）。

### 7.2 AI 守则（通用 + L4 强化）

1. **先读规约（Read First）**：生成前先读 L3 相关规约（若有 `_System_DNA` 则优先喂给该目录，省 Token、防歧义）；
2. **禁止臆造（No Hallucination）**：不臆造规约中未定义的技术栈或能力；
3. **可追溯（Traceability）**：代码/文档注释含 `[Ref: 03_XX]` 或 `[Ref: 便利贴_XXX]`；
4. **语言约定（Language）**：L3 可用中文业务术语；生成代码时须译为项目约定的英文（如规约中的 `积分` → 代码中的 `Credits`）；
5. **deploy-engine 路径（Path Check）**：凡编辑或对 deploy-engine 执行写操作前，**必须先确认目标路径**：若为 `diting-infra/deploy-engine` 或其子路径则**禁止**操作，仅允许在**与 diting-infra 平级的 deploy-engine** 目录内修改。见 §十一。
6. **判断/修复时禁止臆想（Verify First）**：判断逻辑或修复问题时，**禁止凭假设断言原因**；须逐步检查、根据逻辑核对每一份实际文档/配置/集群状态（如读文件、执行可验证命令、对比渲染结果），先验证再下结论。
7. **排查/修复/部署由 AI 直接执行，验收穿透（No “需要你做的”）**：① 涉及排查、修复、实现与部署时，**由 AI 直接执行**可执行的验证与修复验证（读文件、make/helm/kubectl、集群状态与日志等）。② **实现或修复完成后**，须由 AI **直接执行**可执行的验收或验证步骤（如运行 make、脚本、查库、查看分类/采集结果等），**任务穿透至可执行验收完成**；**禁止默认以「你需要做的」「建议您」「请执行」作为收尾**。仅当本环境无法执行时（如用户独占集群、凭证或交互）才给出用户侧操作清单。
8. **数据/模块验证须有证据**：断言「数据符合预期」「模块验证通过」时，须同时提供**数据库查询结果**（表、条件、行数或样本）与**模块执行输出**（命令与日志摘要）；不得在模块明显为降级结果（如全「未知」）时断言符合预期，除非规约已明确该行为。
9. **目标路径主动完成（Proactive completion）**：在完成用户已声明目标的路径上，对**必需的依赖项**（环境、表、包、配置）、**小阻塞**与**可顺手完成的优化**（同文件/同模块、低风险、与目标一致），须**主动完成并验证**，不得默认等待用户专门下「优化一下」类指令。仍须遵守第 6 条先验证再改；**禁止**无依据扩大范围或大规模无关重构。
10. **同会话本地验证闭环（禁止用「可选下一步」代替执行）**：当次任务涉及**代码或配置**变更（含实现、修复、文档中约定的可执行命令所涉脚本/Makefile），且用户目标包含落地或验收时，在本环境可运行 shell/Python 的前提下，须**同一轮会话内**执行与变更**直接相关**的最小可重复验证（例如：受影响模块的 `pytest`、仓库已约定的 `make <target>`、`python -m compileall` 或导入自检）。**禁止**以「若你希望」「可以下一步再跑」「建议你本地验证」等**代替**上述执行而将验证列为用户任务；**禁止**在能跑单测/编译检查时不跑而仅写「建议联调」。**确实无法执行**时（如无 `PG_L2_DSN`、无外网、无集群凭证）：须在回复中写明**已执行的本地验证**、**依赖外部环境且未执行的项**及**阻塞原因**；若断言端到端数据验收仍须满足第 8 条证据要求。本条与第 7 条配合：第 7 条约束收尾话术，本条约束**改代码须带最小可自动化验证**。

10a. **代码修改闭环（有理解 → 有预期目标 → 有验证 → 验证失败 → 修复 → 再验证）**：凡**写代码、修复代码、修改或添加**实现时，须满足闭环：**(1) 有理解** — 明确当前改动所解决的问题、依赖与边界；**(2) 有预期目标** — 明确期望输出或验收标准（如单测通过、make 退出码 0、某断言成立）；**(3) 有验证** — 同一会话内执行最小可重复验证（pytest、make、导入检查等）；**(4) 验证失败 → 修复 → 再验证** — 若验证失败，须分析原因、修复、重新执行验证直至通过；**禁止**在验证未通过时收尾，亦**禁止**将「请本地跑一下」作为替代。本条与第 7、10 条叠加，防止「只改代码不跑验证」的问题穿透。
11. **依赖链与数据路径验收（禁止只改实现不验依赖）**：当本次交付的能力**依赖数据库表、采集流水线、外部 API、密钥或上游业务数据**时，AI 须在**同一会话**内完成：**(1) 依赖清单**——在回复或实现说明中列出**最少**依赖（例如：`PG_L2_DSN`、`segment_registry` 已建、`run_ingest_business_profile` 已对该标的成功写入等）；**(2) 就绪核验**——对环境中可检查项执行实际检查（如测试连接、`information_schema` 或约定脚本/`make init-*` 是否已执行）；**(3) 最小成功路径**——在依赖满足或已补齐后，执行**至少一条**与本次改动相关的端到端或准端到端验证（如相关 `pytest` + 单标的采集或 `run_module_a_local` 对单标）；**(4) 禁止**在未执行上述检查、未补齐可本地补齐的缺表/缺依赖时，以「已修改代码」代替「已验证可用」；若因凭证/独占集群无法执行，须**逐项**列出**未验证项**与**已执行的命令及输出**，不得笼统收尾。本条与第 8、9、10 条叠加：**无证据不断言数据正确；能补齐的依赖不得留给用户；话术上不穿透为「已完成」而实际未验路径。**
12. **答复事实、全量宣称与输出语义（禁止打太极、禁止误导性占位）**：**(1) 用户问「是否完成 / 是否符合 / 是否覆盖全量」时**，答复须**可核验**：写明**已执行**的**命令/脚本**、**范围**（如标的个数、表名、时间条件）、**原始输出**或**查询行数**；**禁止**用「原则上」「基本可以」「从结果看」等**替代事实**；**禁止**用「抽样」代替用户声明的「**全部标的**」**除非用户明确同意抽样**。**(2) 未跑全量、未查库、未在同一环境复现时**，**不得**作肯定结论；须写**未执行**项与**阻塞原因**（如未跑批采、未连 DSN）。**(3) 代码与对外输出**中，缺数据须**显式**（如 `无披露`、`segment_id=seg_no_disclosure`），**禁止**用与事实不符的占位（如用行业主 Tag 伪造 segment、用未执行的「全量验收」冒充已验证）。**Module A 等**：无 `symbol_business_profile` 行时不得假装有分部占比。本条与第 8 条证据要求、第 11 条依赖验收叠加。
13. **代码仓 `.env` 与 `*.env.template` 合并（禁止「请复制 template」收尾）**：当任务依赖的配置说明已写入仓库内 `*.env.template`（或项目约定的环境模板），且目标为**可本地运行/联调**时，AI 须在**同一代码仓**内将对应**注释块与非密钥占位**合并进**已存在的** `.env`（或项目约定的本地环境文件）；**不得**覆盖用户已填写的 DSN、密钥、账号、密码；**禁止**以「请自行从 template 复制」作为**唯一**收尾。与第 9、10 条及 §十一「文档仓与代码仓契约」中本款配合。
14. **面向人的输出以中文为主（非必要不使用英文）**：凡**终端与 CLI、文档说明、Cursor 对话中的逻辑解释与结论、代码审查说明**等面向人类的内容，**优先使用中文**（简称、术语与项目约定一致）；**非必要不堆英文**。**代码输出**（`print`、`logging`、脚本/命令的终端输出、`summary` 等面向用户的消息）**须以中文表达为主**；新增或修改 CLI、脚本输出时，字段标签、状态描述、汇总信息等应使用中文（如「无主营业标的人数」「已写入细分数」「失败细分数」），避免用户看到纯英文字段名。下列情形可保留英文：**环境变量名、配置文件键名、代码标识符与类型名、协议字段名、须与日志或上游系统一致的枚举**；**首次出现的英文简写**仍按 §十三 附中文释义。新增或修改 CLI 时须在输出中体现关键策略是否生效（如占位开关、门控状态），避免用户误以为未执行策略逻辑。
15. **同会话共享资源批处理（禁止中间过早释放）**：当**同一会话或同一待办包**内有多项任务复用**同一运行时资源**（典型：P 轨 Spot ECS + K3s + `prod.conn` 所指集群、同一 GPU 节点、三轮 persist 测试环境）时，**禁止**在**单项**任务（如 Makefile 复验、L4 回填、单轮 smoke）完成后立即执行 tier-1 `down-stack`、Terraform 销毁 ECS 或等价回收，**除非**用户**明确要求**释放，或该项已是该资源的**最后一项**消费者。**须**在会话内**全部**依赖该资源的任务完成后再统一回收；中间可并行推进不依赖 down/up 的工作（L4/L5/14 表、本地 pytest、代码修改）。**例外**：用户单条指令明确要求「测完立刻 down」；tier-3 `FULL_DESTROY=1` 等已二次确认操作；某项任务本身的设计就是「down→up 验证」（如 persist 脚本内嵌的 down/up 轮次）——该 down **仅**服务于该测试合约，**不**代表会话结束即释放。与 `.cursorrules` §15.4、`14_六维度启动期统一节奏表` §6.4「AI 行为约定」一致。

## 八、L4 实践文档创建与 AI 协作规范

本节适用于 **AI 参与创建或补全 L4 阶段实践文档、实施步骤、代码实现** 的场景，确保高效、准确、可追溯。

### 8.1 核心心法

**你是架构师，AI 是初级程序员。**

- **永远不要让 AI 替你做架构决策**：输入/输出结构、计算规则、边界条件、数据模型由人定义；AI 在约束内补全实现与测试。
- **设计左移 + 测试锚定**：先锁死数据结构与用例，再让 AI 写代码；验收标准是「测试从红变绿」，而非「跑一遍感觉对」。

### 8.2 5D 闭环模型（AI 须遵守）

| 步骤 | 英文 | 人的动作 | AI 的动作 |
|------|------|----------|-----------|
| **Design** | 设计左移 | 写伪代码、定义接口与结构体；用自然语言 + 代码片段锁死逻辑路径 | 理解意图，补全类型定义与流程，确认逻辑无歧义 |
| **Drive** | 测试驱动 | 定义测试用例：输入、期望输出、边界；不写实现 | 生成 Table-Driven 测试（先红灯） |
| **Decompose** | 原子拆解 | 将大功能拆成单一职责函数（建议 <50 行/函数） | 实现具体原子函数 |
| **Dispatch** | 模型路由 | 难逻辑给聪明模型，脏活给便宜模型 | 按分配执行 |
| **Defense** | 人工防御 | 运行测试、Code Review、确认业务规则覆盖后 Commit | 根据失败日志与 Review 意见修复 |

**口诀**：Design 锁逻辑 → Drive 锚测试 → Decompose 拆原子 → Dispatch 选模型 → Defense 人把关。

### 8.3 何时走全 5D vs 简化

| 场景 | 建议 | 说明 |
|------|------|------|
| **逻辑密集**（公式、算法、聚合、判定、证据链） | **必须走齐 5D** | 公式、边界、零值处理需测试锚定 |
| **CRUD、HTTP 绑定、DTO 映射、静态页** | Design + Decompose 后可直交 | 体力活可简化 |
| **NFR 敏感**（精度 <1%、性能、安全） | 全 5D + 专门验证 | 如成本核算、财务口径 |

### 8.4 工作目录强制

L4 实践文档中的实施步骤须**明确标注工作目录**，避免在错误仓库操作。**工作目录与多仓路径的权威定义**见 [04_阶段规划与实践/README](04_阶段规划与实践/README.md) 的「工作目录与多仓路径约定」；步骤内仍须明确标注工作目录（如 `diting-src`）。

| 类型 | 工作目录示例 | 示例说明 |
|------|--------------|----------|
| 源码 | `{project}-src` | 后端、前端、共享库 |
| 部署 | `{project}-deploy` | Helm、Values、K8s、CI 配置 |

每个实践步骤的 Prompt 或验收项须包含 `工作目录: xxx`。

**书写规范**：工作目录须统一为**仓库名**（如 `diting-src`、`diting-infra`）；若执行依赖特定分支，须注明「branch: xxx」；多仓库场景下避免歧义（可约定「相对工作区根或文档仓的说明」）。格式建议符合上述约定。

**工作目录与 DNA 对齐**：步骤内工作目录须与 DNA `workflow_stages[].work_dir` 的 description 一致，或在「本步骤落实的 _System_DNA 键」表中列出 `work_dir`；核心指令首句或「工作目录」小节须出现与 DNA 一致的工作目录表述（见 §8.4a、§12）。

### 8.4a 步骤文档模板（04_ 阶段目录内）

阶段目录内**步骤文档统一命名为** `01_本阶段实践与验证.md`，须包含：

- **本步骤落实的 _System_DNA 键**：须含小节「本步骤落实的 _System_DNA 键」，列出本阶段**消耗/产出的 DNA 根节点或键路径**（如 `core_formula`、`trinity_repos`、`dna_dev_workflow.workflow_stages[s0]`）；可用表格（DNA 键 | 用途）或「DNA 根节点：xxx、yyy」形式，便于与 [06_追溯与审计/00_L2_L3_DNA_映射](06_追溯与审计/00_L2_L3_DNA_映射.md) 联合做覆盖度检查。实施内容与验证步骤**按这些键展开**（如目录列表从 DNA 来、验收命令从 make_targets 来）。**工作目录**、**可执行验证命令**须从 DNA 的 `workflow_stages[].work_dir`、`workflow_stages[].verification_commands` 引用或与之一致（见 §12）；准出时更新的 L5 行锚与 DNA 的 `l5_stage_anchor` 一致。

### 8.4b 阶段目录内结构化块约定（两文档结构）

阶段目录采用 **2 个核心文档**：`README.md`（阶段总览、关键最佳实践目标、AI 推荐模型、行动入口、本阶段关联 Phase）、`01_本阶段实践与验证.md`（实践指令、目标验证步骤、验证结果说明与 L5 同步指向）。

README 或 01_ 须包含**可解析结构**（如 YAML front matter 或固定标题），便于脚本/AI 检查「本阶段是否具备最小闭环验证」：

- **best_practice_goals**：可为同目录文件名或**锚点**（格式：`README.md#关键最佳实践目标`）；解析约定：按 `#` 拆分为「文件」与「锚点」，再定位到该文件内对应标题。
- **minimal_closure_verification**：可为同目录文件名或**锚点**（格式：`01_本阶段实践与验证.md#目标验证步骤`）；解析同上。
- **verification_result_ref**：可为「L5 02_验收标准 workflow_stages 对应行」或阶段内「验证结果记录.md」（仅当保留最小独立验证结果文件时）；**不得**指向 01_ 内嵌可填表。

各阶段 README 中须包含「**本阶段关联的 Phase 步骤**」列表（链接到 Phase1_xxx/ 等），与双轨执行顺序一致；Phase 未创建时可写占位。README 须在显眼处提供**行动入口**：本阶段执行与验证见 `01_本阶段实践与验证.md`。

**按角色阅读与使用路径**：人类执行者先读阶段 README 再按 01_ 执行；AI 执行者优先喂 01_ + 本阶段 DNA 键 + 可执行验证清单，README 作策略与选型补充；脚本从 01_ 固定标题（如「可执行验证清单」）解析验证命令与工作目录。详见 [04_阶段规划与实践/README](04_阶段规划与实践/README.md)「按角色阅读与使用路径」。

**Phase–Stage 双向约定**：Phase 文档须在开头或元数据中声明「依赖的 Stage」（stage_id + 链接）；Stage 的「本阶段关联的 Phase」在 Phase 创建后由负责人或 AI **回填链接**，回填时机为 Phase 合并或定稿时。详见 [04_阶段规划与实践/README](04_阶段规划与实践/README.md)「Phase–Stage 双向约定」。

### 8.4c AI 实践最佳（性价比）推荐模型

L4 阶段 README 须含「**AI 实践最佳（性价比）推荐模型**」小节。推荐格式为表格或列表：**用途**（如设计/架构、代码生成、评审/Defense）、**推荐模型或档次**、**一句理由**（成本/质量/性价比）；具体推荐由各阶段填写，格式由本协议统一，便于横向对比与自动化。

- **「推荐模型或档次」须填写具体模型名或可识别产品**（如 Claude 3.5、GPT-4o、DeepSeek-R1、豆包、Cursor 默认模型等）；**禁止**仅填写泛化能力描述（如「具备长上下文的模型」「性价比高的代码模型」）而不给出具体名称；若项目暂未选定，可填「待项目选定」并注明**决策责任人或决策时机**（如「待架构师在 Phase0 前选定」）。
- **维护约定**：AI 推荐模型表须在**固定周期或触发条件**下复审（如每季度/每 Release/模型重大变更时），**责任人**为架构师或项目负责人；若某模型停产或不再推荐，须更新为当前推荐型号或「待项目选定」。
- **自检**：生成或修订 L4 阶段 README 的「AI 实践最佳（性价比）推荐模型」后，须自检「推荐模型或档次」列是否每行均含具体模型名/产品名或明确的「待项目选定」说明；若仅含能力描述且无具体名称，须补全或改为占位。
- **Phase 步骤级可选**：各 Phase 的步骤文档（01_、02_…）可含「本步骤推荐模型」或引用阶段 README 的推荐模型表。

### 8.4d DNA 为核心实践目标与核心指令引用 AI 可读

- **DNA 为核心实践目标**：04_ 下所有实践文档（含 Stage 01_、Phase 步骤 01_～0N_）及其中「核心指令（The Prompt）」块所描述的目标、交付范围、验收与准出，**必须以 _System_DNA 为唯一权威依据**；不得与 DNA 冲突，冲突时以 DNA 为准。核心指令应依据本步 DNA 的 `delivery_scope`、`exit_criteria`、`artifacts`、`verification_commands` 生成任务与验收项（与 §8.4a、§12 一致）。
- **核心指令块内禁止项**：核心指令块内**禁止**出现以下不可读形式：仅「0X_」开头的文档名（如 `04_全链路…`、`02_三位一体`）；仅「XX_规约」「XX_协议」等无路径指代；仅 DNA 文件名（如 `dna_module_a.yaml`）而无键路径或完整 YAML 路径。
- **核心指令块内必须项**：核心指令块内**必须**满足：**文档引用** = 从文档仓根起的完整相对路径（可带 `#锚点`），如 `03_原子目标与规约/_共享规约/04_全链路通信协议矩阵.md`；**DNA 引用** = 与 YAML 结构一致的键路径（如 `dna_dev_workflow.workflow_stages[stage1_02]`、`global_const.data_ingestion`）或完整 YAML 文件路径（如 `03_原子目标与规约/_System_DNA/Stage1_仓库与骨架/dna_stage1_02.yaml`）。
- **核心指令内引用须 AI 可读**：核心指令块内对任意文档、DNA 节点或 L5 锚点的**逻辑引用**，须以 **AI 可读方式** 呈现，便于脚本与 AI 无歧义解析，禁止仅用口语化简称或「见上文」类指代。满足以下任一即视为 AI 可读：
  - **文档**：使用从文档仓根起的**完整相对路径**（如 `03_原子目标与规约/_共享规约/02_三位一体仓库规约.md`），或带锚点的路径（如 `05_成功标识与验证/02_验收标准.md#l5-stage-stage1_01`）；
  - **DNA**：使用与 YAML 结构一致的**键路径**（如 `dna_dev_workflow.workflow_stages[stage1_01]`、`global_const.trinity_repos`）；
  - **结构化引用块**：在步骤文档中提供固定标题（如「本步逻辑引用（AI 可读）」）下的表格或 YAML 块，列出本步 `stage_id`、`dna_file`、`design_doc`、`l3_docs`、`l5_anchor` 等，且核心指令内引用与该表中路径/键一致。
- **自然语言指示时的自动约束**：凡以**自然语言命令**修改或新增实践文档、核心指令及其中引用时，**系统规则自动要求按 AI 完全可读方式实践**（文档引用=从文档仓根起的完整相对路径或路径#锚点，DNA 引用=键路径或完整 YAML 路径；禁止 0X_ 简称、仅文件名等）。AI 与人在执行此类指示时须默认遵守，无需另行声明。

### 8.4e 实践文档可执行性与「按文档完成所有验证且不偏离、不臆造」

凡在 **04_阶段规划与实践** 下**新增或修改**的文档，若为**实践文档**（阶段步骤文档：各 Stage 下 01_ 本阶段实践与验证、01_～0N_ 步骤文档，含「核心指令」「验证与准出」的文档），AI 须确保该文档满足以下目标：**任何 AI 或人类执行者仅凭该文档与约定输入，即可完成文档中规定的全部验证步骤，且不偏离、不臆造**。

**判读**：当 AI 被要求在 L4（04_）添加或修改文档时，须先判读是否为实践文档（是否含步骤目标、核心指令、验证与准出、工作目录等）；若是，则适用本条。

**实践文档须包含（满足可执行、不臆造）**：

| 要素 | 要求 |
|------|------|
| **本步输入** | 执行前必须提供的变量/路径（如 REPO_A_ROOT、KUBECONFIG）；**缺则报错中止，不得臆造**；以表格或列表明确列出，并注明「禁止臆造」。 |
| **可执行验证命令** | 验证表中的命令须**可复制执行**；路径、release 名、Secret 名、namespace 等仅使用本文档、DNA 或对应 L3 设计文档中已写明的值；**禁止占位符未替换即交给执行者**（如 `/path/to/xxx` 须改为「由本步输入 REPO_A_ROOT 提供」或 DNA 中的约定路径）。 |
| **准出检查清单** | 文档内须有**准出检查清单**（可勾选项），便于执行者与 AI 自检；末条须为「已更新 L5/06_ 对应行」（与 §8.4a 一致）。 |
| **验证结果去向** | 明确写出验证结果同步到的**具体位置**（如 L5 02_验收标准 #l5-stage-xxx、06_追溯与审计/02_战略追溯矩阵.md 某行）；不得仅写「更新 L5」而无锚点或文件名。 |
| **AI 执行约束** | 文档中须显式写出：所有路径、名称、命令仅使用本文档、DNA 或 L3 设计已出现的值；**不得臆造**；本步输入缺失时**中止并报错**。 |

**不把任务留给人、尽量不专门写脚本**：
- **所有可执行动作须写入文档**：实践文档不得将任务「留给人」或「需另行完成」；前置条件、依赖准备（如镜像推送、凭证配置）须作为**流程内步骤**或**「不满足时」的明确命令**写入文档，执行者按文档顺序执行即可完成全部验证，无需记忆或额外操作。
- **优先文档内命令/Make**：流程宜用文档内的**逐条命令**或** Make 目标**表达；**尽量避免单独编写专用脚本**——若流程可由文档中的可复制命令块（含 `export`、`cd`、`make` 等）完整表达，则不新增独立脚本；仅当多仓/多环境重复序列过长且需参数化时，可保留简短脚本并确保文档中仍提供等效的逐条命令说明。

**步骤失败时的处理**：见 [§8.4f 步骤失败时的处理原则](#84f-步骤失败时的处理原则)。

**环境前提**：若步骤涉及多仓（如 diting-doc、diting-infra、diting-src）或集群（KUBECONFIG），须在「本步输入」或「工作目录」中写明；执行者无对应环境时仅完成可完成部分并报错，不臆造环境。
- **与 TRACEBACK 的关系**：步骤文档顶部的 [TRACEBACK] 已包含设计文档、DNA 文件等可点击链接时，核心指令内可引用「本步设计文档」「本步 DNA 文件」等，但须在 TRACEBACK 或上述结构化引用块中提供**完整路径或键**，确保 AI/脚本可解析。
- **「本步逻辑引用（AI 可读）」统一格式**（若采用该块）：步骤文档可含固定标题「**本步逻辑引用（AI 可读）**」；若含，则至少包含字段：`stage_id`、`dna_file`（完整相对路径）、`design_doc`（路径#锚点）、`l3_docs`（分号分隔的完整路径）、`l5_anchor`（路径#锚点）；形式为表格或 YAML 块二选一。示例（表格）：

| stage_id | stage1_01 |
| dna_file | 03_原子目标与规约/_System_DNA/Stage1_仓库与骨架/dna_stage1_01.yaml |
| design_doc | 03_原子目标与规约/Stage1_仓库与骨架/01_三位一体仓库设计.md#design-stage1-01-exit |

| l5_anchor | 05_成功标识与验证/02_验收标准.md#l5-stage-stage1_01 |

### 8.4f 步骤失败时的处理原则

**[04_阶段规划与实践](04_阶段规划与实践/) 下所有实践阶段的所有实践文档**须遵循本条约定；**按上述实践文档执行**时遇步骤失败须按此处理。

- **遇问题则分析、修复、重试**：任一步骤执行失败时，**先分析问题、解决问题，然后重新执行**（或从该步重试）；**不直接回收环境退出**（或文档中不得将「失败则立即回收并退出」作为唯一动作）。
- **仅当同一问题两次循环以上仍无法解决时回收/退出**：若**同一问题**在**修复并重试两次以上**仍无法解决，则执行文档规定的回收/退出动作（如回收环境、结束流程）。上述实践文档中步骤的「失败时」描述须体现：先修复重试；同一问题修复重试超过 2 次仍失败则回收/退出。
- **适用**：04_阶段规划与实践 下各 Stage（及 Phase）中，含运行步骤表、可执行验证清单或涉及环境/部署的实践文档；执行者（人或 AI）按 04_ 下任一实践文档执行时须遵守本条。

### 8.4g 实施记录正文维护：单一最佳结果

**适用于**：`04_阶段规划与实践/` 下名为 **`实践记录_*.md`（及项目约定的同角色实施记录）** 的正文，以及 AI 回填验证输出时的纪律。

- **单一事实来源**：每份实施记录应维护**一份**与当前代码仓、当前规约一致的**最佳可复现结论**（命令、期望输出、环境前提）。新一轮验证若**推翻或细化**旧结论，须在正文内 **编辑、替换** 旧段落与终端摘录，使读者打开文档即见**当前**结论。
- **禁止堆叠**：**禁止**在文末无限制追加「第 N 轮」「首轮/二轮/…」全文、重复粘贴过期命令输出，或以「会话纪要不删」为名保留多条互斥准出。确需保留审计痕迹时，写入 **`06_追溯与审计/`**（报告、ADR、矩阵变更说明），**不**将多轮全文堆在同一 `实践记录_*.md` 内。
- **跨周引用**：多份实施记录共用同一环境证据时（如全局 Redis/`/health`），**一处写深、他处链接**，避免每份文档各抄一套数字。
- **「实际进展」不是「未验证清单」**：`实践记录_*.md` 的 **「二、实际进展」**（或项目约定的等效小节）默认承载 **与本步 L3 子步骤对齐、且已按「说明」列所述方式核验过的当前准出**。✅ 表示该格对应验证链 **已跑通**（pytest、TestClient、真机 `curl`、Redis 探活等，以该格文字为准）。若主证据在别份记录，须在说明中给出 **可点击路径与锚点**，仍视为闭环，**不是**「省略验证」。
- **L3 变更或真实 Redis 复验仍须写入 L4**：用户更新 L3、改用真实 Redis、或提高重验标准后，须 **编辑并覆盖** 本文件「三、测试运行」「当前最佳验证结果」等小节中的 **日期、命令、期望与摘录**，并同步刷新「二、实际进展」相关行的表述。§8.4g **禁止**的是文末 **append** 多轮互斥全文；**不禁止**、且 **要求** 用覆盖式更新把**唯一当前有效**结论写清楚。
- **AI 执行**：回填 L4 实施记录时，默认动作是 **search_replace / 重写相关小节**，而非仅在文件末尾 `append`；除非用户或规约明确要求追加 changelog 表，且表内仍须以「当前有效」行为准。

### 8.4h L3 步骤为权威索引：周次仅为别名

**适用于**：采用 **`steps/step_*.md` + `实践记录_*.md`** 成对维护的 Stage（如维度零 `stage_1_启动期`）；其他项目可在 §十一声明等价结构。

- **权威索引**：统计「做到第几步」「缺哪份实施记录」时，以 **L3 步骤序号 / `step_NN_*.md` 文件** 与 **`steps/README.md`（或项目唯一映射表）** 为准，并以 **L4 是否已按该 step 回写** 为核验依据。**实际开发顺序**可与最初「第几周」计划不一致；**文档与看板不得假装仍按日历周真实发生**。
- **周次（W1、W3、W7-W8…）的定位**：出现在文件名或表格中时，多为 **与共享节奏表 / 产品沟通对齐的别名或标签**，**不是**「每个日历周必须产出一篇独立 `实践记录`」的定义。**多个节奏周合并进一个 L3 step**（如 W1+W2 → `step_01`）时，**仍只对应一份**约定名称下的 L4 `实践记录_*.md`；**禁止**因周次数字相加而要求文件数与周数相等。
- **TRACEBACK / 索引表写法**：`实践记录_*.md` 的 `[TRACEBACK]`（或等效块）须 **显式含 L3 `step_*.md` 可点击路径**；看板表格宜分列 **「L3 step（权威）」** 与 **「节奏周次（别名）」**，避免单用「周次」列计数步骤。
- **AI / 人类**：解读「已完成 7 步为何只有 5 个 md」类问题时，**先**核对 **step↔实践记录** 映射与 **是否合并命名**，**禁止**仅按「周」数或文件个数臆断缺写。

### 8.5 Context 优化（防幻觉、省 Token）

- **优先喂给**：`03_原子目标与规约/_System_DNA/` 下的 YAML（若有），或与该步骤相关的 L3 文档；
- **禁止**：一次性塞入全仓文档；无依据的臆造技术栈。
- **本步骤推荐 Context 清单**：各阶段 01_ 宜包含「本步骤推荐 Context 清单」小节，列出：**必读 YAML 路径**（如本 stage 对应的 dna_dev_workflow 子树）、**必读 L3 文档路径**（1～2 个）、**可选文档**；便于脚本/AI 组装 Prompt 时一致注入、避免遗漏关键约束。
- **量级建议**：单步建议 Context 总字符量不超过约 50k 字符（或按项目约定 Token 量级），超出时优先保留必读 YAML + 主责 L3 摘录，按需引用可选文档；可在 01_ 或 DNA 中标注「单步最大建议 Token」供调用方裁剪。

## 九、违规与修复

**视为违规**：错层堆放、单点更新未联动、断链、追溯格式错误；关键重构后未将目录/轴线/链接等约定同步至本协议（§4.5）；**在 diting-infra 内对 `deploy-engine/` 做任何会改变其 Git 状态或工作树的操作**；**在代码仓（diting-src）内放置用于部署/编排或「连接远程库验证」的 docker-compose 等文件**（此类文件归属 diting-infra，见 §十一「部署与编排文件归属」）。（含编辑文件、git commit/push、**git stash**；须在 deploy-engine 独立仓库操作，diting-infra 仅通过子模块更新，见 §十一）；**在部署仓 Makefile 或脚本中为单一组件写死部署/暴露逻辑**（如为某组件写专属 if、临时 values 文件、或写死端口/类型），而未将决策放在 config 或 Chart values 覆盖文件中（见 §十一「部署内容由配置与 Chart 控制」）；**为每次 Copilot 功能/波次/step 新增 `copilot-*-deploy` 类 Make 目标**（日常发版须统一 `make copilot-deploy`，见 §十一「Copilot 统一部署」）；**在本环境可执行最小验证（如 pytest、约定 make）的情况下，修改代码或配置后以「若你希望」「建议下一步」等将验证留给用户，而未在同一会话执行或未说明阻塞原因**（见 §7.2 第 7、10 条）；**写/修/加代码后验证失败时未分析修复再验证即收尾，或以「请本地跑一下」替代实际执行**（见 §7.2 第 10a 条）；**实现依赖库表/采集/外部数据的能力时，未做依赖清单与就绪核验、未执行可本地完成的最小成功路径即宣称已验证或生产级完成，亦未说明未就位项与已执行命令**（见 §7.2 第 11 条）。  
**修复**：先归类到正确层级，再补全上下游引用与追溯矩阵；若为关键重构遗漏，须按 §4.5 补写本协议相应小节；若已在 diting-infra 内误改或误操作 deploy-engine（含 stash），须到 **deploy-engine 独立仓库** 恢复或迁移改动并提交、再在 diting-infra 执行 `make update-deploy-engine` 同步子模块；若在 Makefile/脚本中为单组件写了专属逻辑，须将相关决策**迁移到 config 下的 values 覆盖文件或主配置**，Makefile/脚本改为仅 `-f` 或从配置读值传递；若新增了 per-feature deploy target，须**删除或标注弃用**，文档改为 **`make copilot-deploy` + 镜像 tag**；若在 diting-src 内放置了部署/编排用 compose 文件，须将该文件**迁移到 diting-infra/compose/**，并更新 04_ 实践文档中的路径与工作目录（见 §十一「部署与编排文件归属」）；若写/修/加代码后验证失败即收尾，须**分析失败原因、修复实现、重新执行验证直至通过**（见 §7.2 第 10a 条）。

## 十一、项目定制化

采用本协议时，可在项目内补充：

- **仓库命名**：仓库名统一**小写**，如 `diting-doc`、`diting-src`、`diting-infra`；文档中首次出现可写「diting-doc（文档仓）」等说明，避免 Diting-doc 与 diting-doc 混用导致链接或理解错误。通用模板为 `{project}-doc`、`{project}-src`、`{project}-deploy`；
- **L2 维度**：根据业务定义（如 成本/质量/体验/安全 等）；
- **L4 阶段命名**：如 Phase1_骨架、Phase2_核心逻辑、Phase3_集成 等；
- **工作目录**：L4 步骤中标注代码在哪个仓、部署配置在哪个仓。

### 云原生优先（部署仓）

- **默认实现方式**：凡涉及部署仓内「建表、初始化、一次性/定时 Job、ConfigMap/Secret 创建」等能力时，**默认采用云原生方式**：在 Chart 内用模板定义（ConfigMap、Secret、Job/CronJob），由 `helm install/upgrade` 统一渲染并应用；配置与开关通过 values/config 注入。
- **禁止**：不新增独立 shell 脚本（如 `scripts/xxx-init.sh`）去执行 `kubectl create configmap`、`kubectl create secret`、`kubectl apply -f job.yaml` 等；不在 Makefile 中为上述能力写专属步骤调用脚本。
- **Makefile 职责**：仅负责调用 helm 并传入 config/values（如 `helm upgrade -f $(TMP)`）；不承载资源定义逻辑。
- **例外**：若某能力必须由集群外触发、或与 helm 生命周期解耦（如 Sealed Secrets、外部 CI 注入），可在文档中说明并保留最小脚本；新能力优先评估是否可纳入 Chart。与「部署操作的云原生实践要求」一致，本条作为其**默认约定**的显式强化。

### 部署与编排文件归属（部署仓）

- **原则**：与**部署、编排、本地验证用 Compose** 相关的文件归属 **部署仓（diting-infra）**；**代码仓（diting-src）** 仅保留业务代码、**构建镜像用 Dockerfile**（如 `docker/module_a/Dockerfile`）、单测与 Make 目标（如 `make build-module-a`），**不放置**用于「连接远程库验证」「本地编排运行某可部署单元」的 docker-compose 等部署编排文件。
- **归属 diting-infra**：本地 Docker Compose 文件（如 `compose/docker-compose.ingest.yaml`、`compose/docker-compose.module-a.yaml`）、本地建表/初始化脚本、`make local-deps-up` / `local-deps-down` / `local-deps-init` 等编排与部署验证用制品；连接信息（如 `prod.conn`）由部署仓产出或存放，供 Compose 或 Chart 使用。
- **归属 diting-src**：业务代码、Dockerfile（仅用于构建镜像）、单元测试、`make test` / `make build-module-a` 等构建与测试目标；不包含「以连接远程库或启动某服务栈」为目的的 compose 文件。
- **实践流程**：先在一仓构建镜像（如 diting-src 内 `make build-module-a`），再在**部署仓**执行 Compose 或 Helm 进行本地验证/K3s 部署；L4 实践文档中须明确**工作目录**（编排与部署步骤的工作目录为 diting-infra）。与 [02_三位一体仓库规约](03_原子目标与规约/_共享规约/02_三位一体仓库规约.md)#本地开发与部署文件 一致。

### 实践测试方式约定（L4 全局）

### .cursorrules 与仓库提交

- **允许提交到仓库**：各仓库（**deploy-engine**、**diting-src**、**diting-doc**、**diting-infra**）**允许并将 `.cursorrules` 提交到仓库**；根目录 `.cursorrules` 用于存放本仓的 Cursor/AI 操作规则，供团队与 AI 共享。
- **禁止忽略**：不得将 `.cursorrules` 加入各仓的 `.gitignore`，以便规则随仓版本化与协作一致。

### 文档与配置同步（部署/基础设施）

- **配置真相源**：部署仓（如 `diting-infra`）中的配置文件为配置的**单一真相源**；文档仓不存放可被运行时直接消费的部署 config。
- **文档仓职责**：仅描述配置的**语义、可选范围与约束**（如 L3 基础设施与部署规约、部署配置说明章节）。
- **同步约定**：部署仓中 config 结构或关键字段发生变更时，需同步更新文档仓的对应说明（如「部署配置说明」或 L3 基础设施与部署规约），避免文档与配置脱节。

### diting-infra 目录边界（部署仓）

- **config/**：**仅配置**。仅存放环境与部署**控制用**文件（如 `*.tfvars`、`diting-prod.yaml`、`deploy.yaml`）。**禁止**在 config/ 下放置 K8s 部署用 manifest（如 Secret、Job、Deployment 的 YAML）或与某 Chart 绑定的附属资源；此类文件归属 **charts/** 下对应 Chart。
- **charts/**：Chart 及附属 manifest。与某 Chart 配套、需单独 apply 的 K8s YAML（如 ACR 拉取凭证 Secret）放在该 Chart 的 **manifests/** 子目录（如 `charts/diting-stack/manifests/`）。**禁止**将部署用 manifest 放在 config/ 或仓根。
- **scripts/**：脚本；不存放 K8s 资源定义。**docs/**：文档；不存放可执行部署或配置真相源。
- 违规示例：在 config/ 下创建 `config/manifests/acr-pull-secret.yaml` 属**违规**；正确位置为 `charts/diting-stack/manifests/acr-pull-secret.yaml`。部署仓（diting-infra）的 `.cursorrules` 须与本约定一致。

### 部署操作的云原生实践要求（部署仓与 L4）

- **原则**：凡涉及部署、发布、初始化或运维类操作，**须优先采用符合云原生要求的方式实践**；文档与实现一致，避免以脚本为中心、 imperative 式流程。
- **优先方式**：**声明式资源**（Helm Chart、Kustomize、Kubernetes 原生资源）优于手写 YAML 或脚本生成；**Helm Hook**（pre-install、pre-upgrade 等）优于单独脚本触发初始化 Job；**kubectl/Helm 原生能力**（如 `kubectl create secret --from-env-file`、`helm upgrade --install`）优于自定义脚本拼装，仅在过滤或复杂逻辑时辅以最小脚本或 Makefile。
- **避免**：新增「仅做部署/初始化」的脚本时，须先评估是否可用 Chart/Hook、Makefile + kubectl 替代；禁止以脚本堆叠实现本可由 Chart 生命周期或 K8s 资源表达的能力。
- **L4 实践文档**：04_ 中涉及部署、建表、采集 Job、Secret 同步等步骤时，描述与可执行清单须体现上述优先方式（如「Secret 由 make prod-sync-conn-secret（kubectl --from-env-file）提供」「schema-init 由 charts/ingest 的 pre-install Hook 执行」），并与部署仓实际实现一致。

### 部署内容由配置与 Chart 控制（部署仓）

- **单一事实来源**：**部署内容**（如 Service 类型 ClusterIP/NodePort、NodePort 端口号、副本数、资源限制、暴露方式等）由 **Helm Chart 与配置文件**（如部署仓 `config/*.yaml`、Chart 的 `values.yaml` 或 **values 覆盖文件**）**控制**，**禁止在 Makefile 或通用脚本中写死**。Makefile/脚本仅从配置**读取**并**传递**给 helm/terraform（如 `-f config/xxx.yaml` 或从配置读值后 `--set`），不承载业务端口、类型等决策。
- **禁止单组件专属逻辑**：**禁止**在 Makefile 或脚本中为**单一组件**（如 Redis、某数据库）写专属分支逻辑，包括：根据「是否 NodePort」等条件生成临时 values 文件、对某组件单独写死 `--set ...=端口/类型`、或为满足某 Chart 的 schema（如某键须为 string）而在 Makefile 里拼装临时 YAML。若某 Chart 的 values 需特殊格式或类型，应**在 config 下新增或维护该组件的 values 覆盖文件**（如 `config/redis-values-prod.yaml`），Makefile 仅执行 `helm ... -f config/该文件`，不包含任何该组件专属的 if/临时文件/字面量。
- **正确做法**：新增或调整某组件的暴露方式、端口、类型时，**先**在 config 下新增或修改 **values 覆盖文件**或主配置中对应段落；Makefile 仅通过 `-f config/xxx.yaml` 或从配置读取后 `--set`（且 `--set` 的值必须来自配置文件，不得使用字面量）。同一组件的所有部署相关决策（含 schema 要求的类型）集中在**同一份配置/覆盖文件**中。
- **违规示例**：在 Makefile 中为 Redis 写「若 NodePort 则生成临时 YAML 并 -f」、或对某组件写 `--set master.service.nodePorts.redis=30379` 等，均视为**违规**；应改为在 config 下独立 values 文件（如 `config/redis-values-prod.yaml`）中写 `master.service.nodePorts.redis: "30379"`，Makefile 仅 `-f config/redis-values-prod.yaml`。
- **修改前自检（部署仓）**：凡修改部署仓的 **Makefile、部署脚本**或任何「如何部署/暴露某组件」的逻辑时，须**先自检**：该决策（端口、Service 类型、副本数等）是否已落在**配置文件或 Chart values 覆盖文件**中？若否，**先**在 config 或 values 中新增/修改，再改 Makefile/脚本为「仅 -f 或从配置读值传递」；不得在 Makefile 中为单组件补写分支、临时文件或写死值。
- **与云原生实践一致**：本约定与「部署操作的云原生实践要求」一致；配置即文档、配置即单一事实来源，便于审计与多环境复用。

### Copilot 统一部署（diting-infra · AI/日常标准）

<a id="copilot-统一部署diting-infra--ai日常标准"></a>

**原则**：Copilot（及同 Chart 内可部署服务模块）的**日常发版只有一条命令**；版本差异由 **Helm values 中的镜像 tag** 表达，**禁止**为每次功能、波次或 L3 step 在 Makefile 新增 `copilot-stepNN-deploy` / `copilot-waveN-deploy` / `copilot-xxx-deploy` 类目标。

**工作目录**：`diting-infra` · `KUBECONFIG=~/.kube/config-diting-prod`（或 `prod.conn` 约定路径）。

#### 日常发版（默认路径）

| 你改了什么 | 做什么 | 镜像 tag |
|------------|--------|----------|
| **`diting-src` 代码**（UI/后端/依赖） | `make copilot-deploy` | 自动：`copilot-resolve-image-tag.sh` → git short sha；脏工作区 → `sha-d<hash7>` |
| **仅 Helm/env**（副本、env、Cron、非镜像 values） | `make copilot-deploy-rollout` | 不变（或 `COPILOT_IMAGE_TAG=已有tag` 显式 rollout） |
| **指定已在 ACR 的镜像** | `COPILOT_IMAGE_TAG=<tag> make copilot-deploy-rollout` | 手设 tag，不 build |

**配置真相源**：`config/diting-prod.yaml` → `stack.copilot.image.repository` / **`stack.copilot.image.tag`**（Helm 覆盖 Chart `diting-stack`）。`make copilot-deploy` 成功后会由 `copilot-sync-image-tag-to-config.sh` **自动回写 tag**；正常发版**不必手改** yaml。

**smart 决策链**（`make copilot-deploy` = `scripts/copilot-deploy.sh smart`）：① ACR 已有同 content tag → 仅 Helm + rollout（~30s）；② 本地 `diting-copilot:<tag>` 存在 → push + Helm；③ 否则 full build + push + Helm（~5–10min）。

**进阶档位**（同一入口的不同模式，**非**新功能专用 target）：`copilot-deploy-rollout` / `copilot-deploy-push` / `copilot-deploy-full`；或 `COPILOT_FORCE_BUILD=1 make copilot-deploy`。`copilot-deploy-fast` 为 **`copilot-deploy` 别名**，禁止再增同类别名。

#### 禁止与例外

| 类别 | 规则 |
|------|------|
| **禁止** | 每次 UI/后端/Helm 小改就新增 `make copilot-<feature>-deploy`；在 L3/L4 文档把「生产部署」写成必须跑某 step 专用 deploy target |
| **允许（历史遗留）** | 带**一次性** bootstrap / 数据迁移 / 专项验收的 target（如 `executing-bars250-deploy`、`copilot-executing-workspace-deploy`）— **仅首次环境初始化**，文档须标注「非日常发版」 |
| **验收** | 生产 HTTP/单测验收用 **`diting-src` 的 `make copilot-stepNN-tier2-verify`** 等；**不**与 deploy 绑成新 Make 目标 |
| **executing 补采** | 与 deploy **解耦**：`make executing-t0-bootstrap-sync`（默认异步）；**禁止**默认绑进 deploy（避免 90min 级阻塞） |

**环境变量**（节选）

| 变量 | 默认 | 含义 |
|------|------|------|
| `COPILOT_IMAGE_TAG` | 自动解析 | 显式指定 tag |
| `COPILOT_FORCE_BUILD=1` | 0 | 强制 full build |
| `COPILOT_SKIP_BUILD=1` | 0 | 跳过 build/push |
| `COPILOT_WAIT_ROLLOUT` | 1 | 0=helm 后不等待 Pod Ready |
| `EXECUTING_T0_BOOTSTRAP` | 0 | 1=部署末尾跑 executing bootstrap Job |

**凭证合并**：`copilot-sync-ai-from-src-env.sh` 先读 `diting-infra/.env` 再读 `diting-src/.env`（src 覆盖 infra）。

**AI 部署守则**：① 改代码后先跑相关 pytest；② **仅** `make copilot-deploy`（或 rollout 档）；③ 禁止用「请本地 deploy」收尾；④ **禁止**为本次功能新增 Makefile deploy 目标；⑤ Pod 标签 `app=diting,component=copilot`。

**与 L3 对齐**：[16_阿里云ECS_K3s_ACR_Helm部署与deploy-engine链路](03_原子目标与规约/_共享规约/16_阿里云ECS_K3s_ACR_Helm部署与deploy-engine链路.md) · Chart `diting-stack.copilot.image.tag`。

### deploy-engine 子模块约定（部署仓）

- **禁止在子模块拷贝内做任何“写”操作**：在部署仓（如 `diting-infra`）内，**禁止在 `deploy-engine/` 目录下进行任何会改变其 Git 状态或工作树的操作**，包括但不限于：编辑/添加/修改文件，`git add/commit/push`，**`git stash`**。deploy-engine 在部署仓中为 **Git 子模块**，真相源为 **deploy-engine 独立仓库**（与 diting-infra 平级）；在子模块拷贝内 stash/commit 会污染该拷贝、影响后续 `make update-deploy-engine` 或 pull main，视为**违规**（见 §九）。
- **路径判定**：工作区若同时存在 `diting-infra/deploy-engine`（子模块）与**与 diting-infra 平级的** `deploy-engine`（独立仓库），则**仅平级 deploy-engine 为可编辑源**，`diting-infra/deploy-engine` 为只读引用；编辑或写操作**必须**使用平级路径，禁止使用子模块路径。
- **AI 编辑前路径强制自检**（防忽略）：凡对 deploy-engine 内文件执行**编辑或任何写操作**前，必须依次确认：① 目标完整路径是什么？② 该路径是否位于 `diting-infra/deploy-engine` 下（或是否以 `diting-infra/` 开头且含 `deploy-engine`）？③ 若是 → **禁止操作**，改为在**与 diting-infra 平级的 deploy-engine** 目录下编辑。未完成自检不得执行写操作。
- **唯一允许的更新方式**：① 在 **deploy-engine 独立仓库** 中修改、**commit 并 push**；② 在部署仓（diting-infra）中**仅**通过子模块同步拉取，例如 `git submodule update --init --remote deploy-engine` 或该仓 Makefile 约定的 `make update-deploy-engine`。**仅修改不提交推送则子模块无法拉取到变更**。
- **正确做法（完整流程）**：① 在 **deploy-engine 独立仓库** 内编辑；② 在独立仓库根目录执行 `git add`、`git commit`、`git push`；③ 再在 diting-infra 执行 `make update-deploy-engine`。**不得**在 `diting-infra/deploy-engine/` 内做任何写操作（含编辑、commit、stash、push）。
- **AI 执行前自检**：凡需改动 deploy-engine 或在其内执行 git 写命令（含 stash）时，**先判断操作是否发生在 `diting-infra/deploy-engine/`**：若是，则**不得在该路径下执行**，须改为在 **deploy-engine 独立仓库**内操作，再在 diting-infra 中执行子模块更新。
- **AI 行为**：凡需改动 deploy-engine 逻辑或配置时，须在 deploy-engine 独立仓库中修改、**commit 并 push**，再在 diting-infra 中执行子模块更新；不得在 diting-infra 内对 `deploy-engine/` 做任何写操作（含编辑、stash、commit、push）。

### charts/dependencies 目录约定（部署仓）

- **禁止直接修改 Chart**：在部署仓（如 `diting-infra`）的 **`charts/dependencies/`** 目录下，**禁止直接更改、添加或修改** Chart 内文件（如 `templates/`、`Chart.yaml`、`values.yaml` 等）。Chart 须从官方/上游（如 Helm 仓库）**下载到本地**后原样存放使用。
- **允许的定制方式**：仅允许通过 **values 覆盖** 定制行为（如在本仓新建或维护 `values-*.yaml`，或在 `helm install/upgrade` 时通过 `-f`、`--set` 传入），不得修改 `charts/dependencies/<chart-name>/` 下的 Chart 源码。
- **版本与更新**：Chart 版本在 `Chart.yaml` 或 `Chart.lock` 中显式固定；升级时通过重新下载同路径覆盖或按项目约定的更新流程执行，仍不得在 dependencies 下手改 Chart 内容。

- **Ref 纪律与检查点**：代码注释须含 `[Ref: 03_XX]` 或 `[Ref: 便利贴_XXX]`（见 §7.2）；Code Review 时至少检查新增/修改逻辑是否引用对应 L3 规约或便利贴。
- **可选**：代码仓 README 或 ADR 中链回文档仓的 L3/L4 文档；文档仓 L4 步骤中链到对应 PR/branch（若可访问），以利双向追溯。
- **代码仓本地执行与前置项（diting-src 等）**：凡用户目标依赖本地可执行验证（make、脚本、单测），AI 应按 §7.2 第 7、9、**10**、**11**、**13** 条**主动**完成路径上**必需**的步骤（含但不限于：**`make init-l2-b-module-tables`**、**`make deps-scanner`**、**`pip install -r requirements-*.txt`**、修复缺表/缺包导致的失败；**代码变更后同会话内**跑相关 **`pytest` / 约定 `make`**；**涉及 L2/采集/外部数据时**须做**依赖清单 + 就绪检查 + 最小成功路径**，见第 11 条；**template 中已有说明块时**须合并进 `.env` 见第 13 条），并**重跑**直至通过；**不得**将可本地完成的修复留成「请你执行」清单。生产环境以 **diting-infra** Chart / schema-init 为权威。
- **`.env` 与 template 同步**：当任务要求运行依赖某配置节（如 Module C、采集开关）且仓库已提供 `*.env.template` 时，AI 须在**同一代码仓**将对应**注释与非密钥占位**合并进**已存在的** `.env`（或项目约定文件名）；**不得**覆盖用户已填 DSN/密钥；**禁止**以「请自行复制 template」作为唯一收尾。见 §7.2 第 13 条。

### 可部署单元与运行形态（多服务部署）

- **可部署单元**：实盘多服务时，可部署单元清单（如 数据采集与存储、Module A/B/C、热路径 D+E+F）在 L3（如 09_）与 DNA（`global_const.deployable_units`、`dna_dev_workflow.deployable_units`）中一致定义；热路径 D+E+F 为同一进程、同一镜像。
- **三种运行形态**：回测 = 单进程 + 历史数据；实盘 = 多服务；开发期连调 = 本地运行部分服务 + 远程 K3s 基础组件与相关服务 + 线上/类生产数据。见 L2 评测一致性与发布闭环与 L4 各 Stage README「开发期连调」。
- **Release bundle**：一次发布使用同一 tag 的全体可部署单元镜像；部署时禁止混用不同 build 的镜像。见 L3 基础设施与部署规约与 DNA `release_bundle`。

## 十二、DNA 机制 — `_System_DNA`

本项目在 `03_原子目标与规约/_System_DNA/` 目录下维护**机器可读的 YAML「骨架」**，用于承载从 L2/L3 推导出的所有关键参数与约束，为 L4 阶段实践与 L5 验收提供统一「单一真相源」。**步骤级 DNA**（`StageX_*/dna_stageX_0Y.yaml`）与本步 03_ 设计文档、04_ 步骤文档形成 **1:1:1**；规约级配置收敛至 `global_const.yaml` 各子树，_共享规约 01_～11_ 对应 `global_const#xxx` 键，不设独立 dna_0X 文件。

- **L2（战略维度）职责**：以自然语言说明「从哪些维度落地」，对应到 `_System_DNA/global_const.yaml` 中的**主题节点**（如：`tech_stack`、`constraints`、`production_requirements`、`governance_and_dr` 等），并在 YAML 中通过 `strategic_dimensions.*` 显式声明「维度 ↔ DNA 节点」映射。
- **L3（原子规约）职责**：在各规约文档中解释「每个原子能力/规则的含义与设计」，同时将会影响实现与验收的**阈值、开关、路径、比例、频率等变量**收敛到 `_System_DNA/*.yaml` 的对应子树（如：`core_modules.*`、`dynamic_config.*`、`data_version_control.*`、`heartbeat_protocol.*` 等），并在 `03_原子目标与规约/README.md` 顶部维护一张「L3 文档 ↔ DNA 子树」总对照表。
- **L4（阶段规划与实践）职责**：所有阶段与步骤文档只引用 YAML 中的键值，而不再硬编码具体数值；步骤目标、实施内容与验收标准应显式标注「本步骤落实了哪些 `_System_DNA` 键」（例如：`core_modules.module_b_quant_engine.scanner.technical_score_threshold`），确保实践与配置一一对应、可追溯。**工作目录**、**可执行验证命令**、**验证结果预期**、**准出时更新的 L5 行**须从 DNA 的 `workflow_stages[].work_dir`、`workflow_stages[].verification_commands`、`workflow_stages[].l5_stage_anchor` 引用或与之一致；核心指令宜由 DNA 的 delivery_scope、exit_criteria、verification_commands 生成任务与验收项（见 §8.4）。
- **L5（成功标识与验证）职责**：L5 的 workflow_stages 表行集与 DNA 的 `workflow_stages` 的 `stage_id` 一一对应；每行锚点（如 `l5-stage-stage1_01`）与 DNA 的 `l5_stage_anchor` 一致。功能验收表与 `product_scope.phases[].steps`（含 `l5_anchor`、验收标准、验证方法）一致；表体可由 DNA 生成或严格校验，仅「状态」列在 L5 文档维护。**DNA workflow_stages 新增或删除**时，须在 02_验收标准 中同步新增或删除对应行及显式锚点；可选步骤仍保留行与锚点，状态可为「跳过/不适用」。

通过上述分工，形成「L2/L3 写规则 → `_System_DNA` 收敛为键值对 → L4 按键执行与验收、L5 以 DNA 为源或强一致」的闭环，方便 AI 与脚本自动消费这些配置。**实践文档与核心指令**：所有 04_ 实践文档及其中核心指令块，**以 DNA 为核心实践目标**；核心指令内对文档与 DNA 的引用须符合 **§8.4d（AI 可读）** 约定。

## 十三、英文简写与技术名词对照表规则（AI 自动执行）

**目的**：统一**英文简写**与**技术名词**的释义与文档内引用方式，避免同一简写或技术名词在不同文档中释义不一致或重复堆砌；技术名词与 L3 规约、代码命名保持一致。

**对照表约定位置**：文档仓根目录 `00_英文简写对照表.md`（若项目另有约定，以项目约定为准）。该对照表可同时包含两类条目：
- **英文简写**：简写、英文全称、中文释义；
- **技术名词**：中文术语、英文对应、释义/说明（可选）、代码/标识用名（可选，与 L3、代码一致）。

### 13.1 执行流程（AI 在操作文档仓时自动执行）

1. **检测对照表是否存在**
   - 在生成、修改或引用文档仓内 Markdown 内容之前，先检查是否存在 `00_英文简写对照表.md`。
   - **若不存在**：提示用户「未检测到英文简写对照表（00_英文简写对照表.md），是否创建？」，待用户确认后可按本协议附录模板创建并写入默认简写条目（可同时预留技术名词章节）。
   - **若存在**：继续后续步骤。

2. **使用英文简写或技术名词时查表并补表**
   - 当在文档中**将要写入**某一英文简写（如 ROI、MoE、RTO、DVC 等）或技术名词（如 混合专家、路由权重、证据链、数据版本控制 等）时，先读取 `00_英文简写对照表.md`，检查是否已在表中。
   - **英文简写**：若表中不存在该简写，将「简写、英文全称、中文释义」按现有表格格式追加到对应分类下；当前文档内**首次出现处**书写为「简写（中文释义）」；若已存在则按表中释义书写。
   - **技术名词**：若表中不存在该名词，将「中文术语、英文对应、释义/说明」按表格格式追加到对应分类下（分类可与简写对齐）；当前文档内**首次出现处**书写为「中文术语（英文对应）」或「中文术语（释义）」；若已存在则按表中定义使用，并与 L3/代码命名一致。

3. **同一文档内引用规则**
   - **英文简写**：首次出现须为 `简写（中文释义）`，例如 `RTO（恢复时间目标）`、`MoE（混合专家）`；后续重复出现仅写简写。
   - **技术名词**：首次出现须附英文对应或释义，格式为「中文术语（英文对应）」或「中文术语（释义）」；后续重复出现仅写已定义的术语，同一概念不混用多种说法。

### 13.2 对照表维护约定

- 对照表可同时包含「英文简写」与「技术名词」两类；分类建议一致（业务与指标、架构与模型、运维与灾备、安全与存储、成本与资源、产品与组件等），必要时可新增「其他」。
- 简写：若同时属于多类，归入最常被引用的那一类；产品/品牌名可只列「说明」列，不强制英文全称。
- 技术名词：英文对应/代码标识用名应与 `_System_DNA`、L3 规约及代码仓一致（如 积分 → Credits）；若某技术名词有简写（如 MoE），可在表中注明「简称：MoE」并与简写表一致。

### 13.3 与 §7.2 第 14 条的关系（文档仓与代码仓）

- **文档仓 Markdown** 仍按 §十三 1～2 节处理英文简写与技术名词（首次附释义等）。
- **凡面向人的表述**（含终端、协作对话中的策略与逻辑解释）与 §7.2 第 14 条一致：以中文为主；环境变量名、配置键、代码标识符等技术必要处可保留英文。

## 十四、同会话共享资源批处理（§十四）

与 **§7.2 第 15 条**、`.cursorrules` §15.4、`14_六维度启动期统一节奏表` §6.4「AI 行为约定」一致。

**适用**：同会话多步任务复用 P 轨 ECS+K3s、同一 GPU 节点、三轮 persist 环境等。

| 场景 | 做法 |
|------|------|
| 多项任务共用 base ECS + K3s | 全部消费者完成后再 tier-1 `down-stack`（或用户明确要求提前释放）|
| 单项 Makefile 绿、后续还有 persist | **不** down；继续下一项 |
| persist 脚本内嵌 down/up | 测试合约内轮次，**不**代表会话结束即释放 |
| 用户说「测完立刻 down」 | 按用户指令 |
